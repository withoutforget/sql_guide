# lab010 — Оконные функции: основы (`OVER`, `PARTITION BY`, ранги)

Десятая лаба открывает тему **оконных функций** внутри **модуля C — «Агрегация и
аналитика»**. Всё, что мы считали до сих пор — `GROUP BY` из [lab001](../lab001/),
`GROUPING SETS`/`ROLLUP`/`CUBE` из [lab008](../lab008/), статистические и
сортирующие агрегаты из [lab009](../lab009/), — **схлопывало** строки в итог.
Оконные функции меняют правило игры:

> оконная функция считает значение по **набору связанных строк** («окну»), но, в
> отличие от `GROUP BY`, **не схлопывает** строки — каждая исходная строка остаётся,
> а рядом появляется вычисленный столбец (ранг, доля, итог по группе).

Это **первая из двух** лаб про окна. Здесь — **фундамент**: идея окна, синтаксис
`OVER (PARTITION BY ... ORDER BY ...)`, ранжирующие функции (`row_number`/`rank`/
`dense_rank`/`ntile`/`percent_rank`/`cume_dist`), агрегаты как оконные (`sum`/`avg`/
`count`/… `OVER`), правило «где можно писать окна» и базовые паттерны (top-N на
группу, дедупликация, доли, ранги). Рамки, `LAG`/`LEAD`, скользящие средние — в
[lab011](../lab011/).

Теория разбита на короткие документы в папке [`theory/`](theory/) — читайте по
порядку. Общая подборка курсов и тренажёров — в [`../RESOURCES.md`](../RESOURCES.md).

## Доменная область

Тот же интернет-магазин. Данные **специально** подобраны под тему (все числа
проверены прогоном — см. подробную шапку [`init.sql`](init.sql)). Четыре таблицы:

### `products` — товары (главный источник числовых мер)
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор |
| `name` | TEXT | название |
| `category_id` | INT → `categories` | категория (для `PARTITION BY`) |
| `price` | NUMERIC | цена за штуку, ₽ |
| `units_sold` | INT | продано штук |
| `revenue` | NUMERIC (**GENERATED**) | выручка = `price * units_sold` — метрика ранжирования |

### `categories` / `customers` / `orders`
Категории; клиенты (с городом — для `PARTITION BY city`); заказы (каждая строка —
один товар в заказе, сумма строки = `price * qty` — для рангов по клиентам).

> **Как подобраны данные (главное):**
> - **Ничьи по выручке товара** (ядро лабы — на них различаются `row_number`/`rank`/
>   `dense_rank`): в *Электронике* Планшет и Наушники по **30000** (делят 2-е место);
>   в *Доме* Робот-пылесос и Кофеварка по **24000** (делят **1-е** место).
> - **Итоги для долей:** категории 132000 / 57000 / 20500 / 13500, магазин **223000**.
> - **Траты клиентов различны** (для `NTILE`-квартилей): 55000, 48000, 42000, 38000,
>   22000, 10500, 7500.
> - **Ничьи между заказами одного клиента** (для «второго по величине»): у Глеба два
>   заказа по 3000 (ничья-максимум → `rank` даёт 1,1,3, а честный «второй» находит
>   только `dense_rank`); у Дарьи два по 12000 (ничья на втором месте → «второй» это
>   оба).
> - **Города:** в Москве три клиента, в Казани два — осмысленный `PARTITION BY city`.

| 13 | 🔥 `13_second_largest_order_per_customer.sql` | [03](theory/03-ranking-functions.md) · [06](theory/06-filtering-patterns-boundaries.md) | **Второй по величине заказ каждого клиента** с честной обработкой ничьих — почему нужен именно `dense_rank`, а не `row_number`/`rank`. |

> **Про хардкор.** Задача 12 собирает два приёма в один отчёт: top-N на группу
> (`row_number` + `WHERE rn <= 2`) **и** долю в разделе (`sum OVER PARTITION`). Изюм
> — увидеть, что оба окна надо посчитать **до** фильтрации по `rn`, иначе доля
> считалась бы от суммы двух товаров, а не всей категории (окна вычисляются раньше
> внешнего `WHERE`). Задача 13 — ловушка «второй по величине»: `row_number` назвал бы
> «вторым» один из заказов-**максимумов** при ничьей, `rank` вовсе пропустил бы ранг
> 2 (у Глеба стало бы 1,1,3), и лишь `dense_rank` честно находит **второе различное**
> значение — и корректно показывает **обе** строки, когда его делят (два заказа Дарьи
> по 12000). Обе решаются ровно материалом lab001–lab010, без `LAG`/`LEAD` и рамок.

## Что дальше

Мы освоили **основы** окон: разбиение (`PARTITION BY`), порядок (`ORDER BY`), ранги
и агрегаты рядом с каждой строкой. В [lab011](../lab011/) — **вторая половина** темы:
оконные **рамки** (`ROWS`/`RANGE`/`GROUPS`) и их поведение по умолчанию, нарастающие
и **скользящие** агрегаты (running/moving total, скользящее среднее), функции
**смещения** `LAG`/`LEAD`, `FIRST_VALUE`/`LAST_VALUE`/`NTH_VALUE` и именованные окна
(`WINDOW`). Всё это надстраивается ровно над тем, что здесь усвоено.
