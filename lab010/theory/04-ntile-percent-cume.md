# 04 — `NTILE`, `PERCENT_RANK`, `CUME_DIST`: корзины и относительное положение

[← ранжирующие функции](03-ranking-functions.md) · [оглавление](../README.md) · [дальше: агрегаты как окна →](05-aggregate-windows.md)

---

Ещё три оконные функции того же семейства — они отвечают не «какой номер по
порядку», а «в какую **корзину** попала строка» и «насколько она **впереди/позади**
относительно остальных». Всем нужен `ORDER BY` внутри `OVER`.

## `NTILE(n)` — разбить упорядоченный раздел на `n` корзин

`NTILE(n)` делит строки раздела (в порядке `ORDER BY` окна) на `n` **примерно
равных** групп и возвращает номер корзины `1..n`. Классическое применение —
**квартили** (`n = 4`), **децили** (`n = 10`), **процентили** (`n = 100`): разбить
клиентов/товары на равные доли по какой-то метрике.

```sql
WITH spend AS (
    SELECT cu.name, cu.city, sum(p.price * o.qty) AS total_spent
    FROM orders o
    JOIN products p  ON p.id  = o.product_id
    JOIN customers cu ON cu.id = o.customer_id
    GROUP BY cu.name, cu.city
)
SELECT name, total_spent,
       ntile(4) OVER (ORDER BY total_spent DESC) AS quartile
FROM spend
ORDER BY total_spent DESC;
```
```
 name  | total_spent | quartile
-------+-------------+----------
 Егор  |    55000.00 |        1   ┐ Q1 — самые щедрые
 Дарья |    48000.00 |        1   ┘
 Анна  |    42000.00 |        2   ┐ Q2
 Борис |    38000.00 |        2   ┘
 Вера  |    22000.00 |        3   ┐ Q3
 Жанна |    10500.00 |        3   ┘
 Глеб  |     7500.00 |        4     Q4 — самый скромный
```

**Как делится, если не поровну.** Клиентов **7**, корзин **4**: `7 = 4·1 + 3`.
Нацело по одному в каждую, а остаток `3` **раздаётся первым корзинам** — по одному.
Значит размеры корзин: `2, 2, 2, 1` (первые три получили «лишнюю» строку, последняя
осталась с одной). Правило: **первые `остаток` корзин на одну строку больше
остальных**; ранние корзины никогда не меньше поздних.

Полезно помнить:

- `NTILE` смотрит на **позицию** строки, а не на её значение. Две строки с
  **одинаковым** значением метрики могут попасть в **разные** корзины (если они на
  границе) — `NTILE` про равные доли по количеству строк, а не по значению.
- Если строк **меньше**, чем `n`, лишние корзины просто пустуют: первые строки
  получат номера `1, 2, …`, а корзин с бо́льшими номерами не будет.

## `PERCENT_RANK()` — относительный ранг `0..1`

`PERCENT_RANK()` показывает, **какую долю строк раздела данная строка обошла**.
Формула:

```
percent_rank = (rank - 1) / (n - 1)
```

где `rank` — обычный оконный `rank()` этой строки, `n` — число строк в разделе. Даёт
число от `0` (самая первая в порядке) до `1` (самая последняя). Первая строка всегда
`0`, последняя всегда `1`.

## `CUME_DIST()` — кумулятивная доля `(0..1]`

`CUME_DIST()` (cumulative distribution) — **какая доля строк идёт не позже данной**,
включая её саму:

```
cume_dist = (сколько строк с значением ≤ текущего) / n
```

Даёт число от «чуть больше 0» до `1`. У самой последней строки — ровно `1`
(все строки «не позже неё»). Отличие от `percent_rank`: `cume_dist` **включает**
текущую строку в числитель, а `percent_rank` считает только «строго обошедших».

## Оба на одних данных: товары по цене

```sql
SELECT name, price,
       round(percent_rank() OVER (ORDER BY price)::numeric, 4) AS percent_rank,
       round(cume_dist()    OVER (ORDER BY price)::numeric, 4) AS cume_dist
FROM products
ORDER BY price;
```
```
         name         |  price   | percent_rank | cume_dist
----------------------+----------+--------------+-----------
 Пазл 1000 деталей    |   500.00 |       0.0000 |    0.0667   ← самый дешёвый
 Книга «SQL за месяц» |  1000.00 |       0.0714 |    0.2000   ┐ ничья: две цены по 1000
 Мяч футбольный       |  1000.00 |       0.0714 |    0.2000   ┘
 Плед тёплый          |  1500.00 |       0.2143 |    0.3333
 ...                  |      ... |          ... |       ...
 Планшет Tab 10       | 15000.00 |       0.9286 |    0.9333
 Смартфон Nova        | 30000.00 |       1.0000 |    1.0000   ← самый дорогой
```

Разберём числа (`n = 15` товаров):

- **Пазл** (дешевле всех): `percent_rank = (1−1)/(15−1) = 0` — никого не обошёл.
  `cume_dist = 1/15 = 0.0667` — одна строка (он сам) «не дороже него».
- **Две цены по 1000** (ничья): у обеих `rank = 2`, значит
  `percent_rank = (2−1)/14 = 0.0714` — одинаково. `cume_dist = 3/15 = 0.2` (три
  товара стоя́т ≤ 1000: сам Пазл за 500 и оба по 1000) — тоже одинаково у обеих.
  Как и в [файле 03](03-ranking-functions.md), равные значения получают равный
  относительный ранг.
- **Смартфон** (дороже всех): `percent_rank = 14/14 = 1`, `cume_dist = 15/15 = 1`.

Смысл словами: `percent_rank ≈ 0.93` у Планшета — он дороже примерно 93%
«предшественников»; `cume_dist ≈ 0.93` — доля товаров «не дороже него» ≈ 93%.

## Связь с гипотетическими `percent_rank`/`cume_dist` из lab009

Это ровно те же меры, что в [lab009](../../lab009/theory/06-hypothetical-set-rank.md),
но с другой стороны. Там был **гипотетический агрегат**
`percent_rank(10000) WITHIN GROUP (ORDER BY price)` — «в какой перцентиль попал **бы
один** товар за 10 000 ₽», **одно** число на группу. Здесь `percent_rank() OVER
(ORDER BY price)` даёт перцентиль **каждому реальному** товару — **строка на каждую
запись**. Тот же приём «сравнить значение с распределением», но:

| | `percent_rank(10000) WITHIN GROUP` (lab009) | `percent_rank() OVER` (эта лаба) |
|--|---------------------------------------------|----------------------------------|
| Про кого | про **один** примеряемый аргумент | про **каждую** строку |
| Результат | одно число на группу | столбец рядом с каждой строкой |

## Крайние случаи

- **Раздел из одной строки:** `percent_rank = 0` (формула `(1−1)/(1−1)` доопределена
  как 0), `cume_dist = 1`, `ntile(любое n) = 1`.
- **`NULL` в ключе сортировки:** строка участвует, попадая в конец/начало порядка
  (управляется `NULLS FIRST/LAST`), как и у ранжирующих функций.
- `percent_rank` и `cume_dist` возвращают тип `double precision` — при
  необходимости округляйте (`round(... ::numeric, 4)`), как в примерах.
