# 02 — Анатомия `OVER`: `PARTITION BY` и `ORDER BY`

[← что такое окно](01-what-is-window.md) · [оглавление](../README.md) · [дальше: ранжирующие функции →](03-ranking-functions.md)

---

Всё поведение оконной функции задаётся тем, что стои́т **в скобках `OVER (...)`** —
это и есть «определение окна». Разберём его по частям.

```sql
функция(...) OVER ( PARTITION BY  выражения      -- на какие разделы бить
                    ORDER BY      выражения )     -- как упорядочить внутри раздела
```

Обе части **необязательны**. Отсюда три базовые формы, от самой простой к полной.

## Форма 1: `OVER ()` — пустое окно = все строки как одно окно

Пустые скобки означают: **окно — это весь результат запроса целиком**, без деления
и без внутреннего порядка. Функция считается один раз по всем строкам и
приписывается каждой.

```sql
SELECT name, revenue,
       sum(revenue)  OVER () AS store_total,   -- сумма по ВСЕМ товарам
       count(*)      OVER () AS n_products,     -- сколько всего товаров
       round(avg(revenue) OVER (), 2) AS avg_revenue
FROM products
ORDER BY revenue DESC, id
LIMIT 4;
```
```
       name        | revenue  | store_total | n_products | avg_revenue
-------------------+----------+-------------+------------+-------------
 Смартфон Nova     | 60000.00 |   223000.00 |         15 |    14866.67
 Наушники TWS      | 30000.00 |   223000.00 |         15 |    14866.67
 Планшет Tab 10    | 30000.00 |   223000.00 |         15 |    14866.67
 Кофеварка Barista | 24000.00 |   223000.00 |         15 |    14866.67
```

`store_total`, `n_products`, `avg_revenue` одинаковы во **всех** строках: окно у
всех одно и то же — вся таблица. Это самая частая форма для «доли от общего итога»
(см. [файл 05](05-aggregate-windows.md)).

## Форма 2: `OVER (PARTITION BY ...)` — разбить на независимые разделы

`PARTITION BY` делит строки на **разделы** (partitions) — группы с одинаковыми
значениями указанных выражений. Функция считается **отдельно внутри каждого
раздела**, но строки при этом **не схлопываются** (в отличие от `GROUP BY`).

```sql
SELECT c.name AS category, p.name AS product, p.revenue,
       sum(p.revenue)   OVER (PARTITION BY c.name) AS cat_total,
       count(*)         OVER (PARTITION BY c.name) AS n_in_cat
FROM products p JOIN categories c ON c.id = p.category_id
WHERE c.name IN ('Дом', 'Игрушки')
ORDER BY c.name, p.revenue DESC, p.id;
```
```
 category |      product        | revenue  | cat_total | n_in_cat
----------+---------------------+----------+-----------+---------
 Дом      | Кофеварка Barista   | 24000.00 |  57000.00 |       4
 Дом      | Робот-пылесос       | 24000.00 |  57000.00 |       4
 Дом      | Настольная лампа    |  6000.00 |  57000.00 |       4
 Дом      | Плед тёплый         |  3000.00 |  57000.00 |       4
 Игрушки  | Конструктор «Город» |  9000.00 |  13500.00 |       3
 Игрушки  | Мяч футбольный      |  3000.00 |  13500.00 |       3
 Игрушки  | Пазл 1000 деталей   |  1500.00 |  13500.00 |       3
```

Внутри «Дома» окно — четыре товара «Дома» (`cat_total = 57000`, `n_in_cat = 4`);
внутри «Игрушек» — свои три (`13500`, `3`). Разделы **независимы**: функция никогда
не заглядывает из одного раздела в другой.

> **Аналогия с `GROUP BY` — и ключевое различие.** `PARTITION BY` разбивает строки
> ровно так же, как `GROUP BY` (те же «корзины»). Но `GROUP BY` затем **сворачивает**
> каждую корзину в одну строку, а `PARTITION BY` — **оставляет** все строки, лишь
> приписывая каждой значение по её корзине. `PARTITION BY` можно опустить —
> тогда весь результат это один раздел (форма 1).

Разбивать можно **по нескольким** выражениям — раздел определяется их комбинацией
(например `PARTITION BY city, product_id`), как и в `GROUP BY`.

## Форма 3: `OVER (PARTITION BY ... ORDER BY ...)` — упорядочить внутри раздела

`ORDER BY` **внутри `OVER`** задаёт порядок строк **внутри раздела**. Он нужен там,
где важна последовательность:

- для **ранжирующих** функций (`row_number`/`rank`/`dense_rank` и др.) — они по
  определению нумеруют строки «по порядку», значит порядок обязателен;
- для «нарастающих» вычислений у агрегатов (running total) — но это уже тема рамок,
  см. ниже и [lab011](../../lab011/).

```sql
SELECT c.name AS category, p.name AS product, p.revenue,
       row_number() OVER (PARTITION BY c.name ORDER BY p.revenue DESC) AS place
FROM products p JOIN categories c ON c.id = p.category_id
WHERE c.name = 'Книги'
ORDER BY place;
```
```
 category |      product        | revenue | place
----------+---------------------+---------+------
 Книги    | Книга «Алгоритмы»   | 8000.00 |     1
 Книги    | Книга «Рефакторинг» | 5000.00 |     2
 Книги    | Книга «Чистый код»  | 4500.00 |     3
 Книги    | Книга «SQL за месяц»| 3000.00 |     4
```

`ORDER BY p.revenue DESC` внутри окна выстроил товары «Книг» от дорогих к дешёвым, и
`row_number` пронумеровал их 1..4.

### `ORDER BY` внутри `OVER` ≠ `ORDER BY` всего запроса

Их **два разных**, и они независимы:

| | `ORDER BY` внутри `OVER (...)` | `ORDER BY` в конце запроса |
|--|-------------------------------|----------------------------|
| Что делает | упорядочивает строки **внутри окна** — чтобы функция знала «кто раньше» | упорядочивает **итоговую выдачу** на экране |
| На что влияет | на **значение** оконной функции (ранг, нарастающий итог) | только на **порядок строк** результата |
| Сколько раз | своё у каждого окна | одно на весь запрос |

Их можно комбинировать по-разному — например, посчитать ранг по убыванию выручки,
а вывести строки по алфавиту:

```sql
SELECT p.name,
       rank() OVER (ORDER BY p.revenue DESC) AS rank_by_revenue
FROM products p
ORDER BY p.name;         -- вывод по алфавиту, а ранг — по выручке
```

Ранг посчитается по `revenue DESC` (внутри окна), а строки на экране лягут по имени.
Если убрать финальный `ORDER BY`, порядок строк на выходе **не гарантирован** —
даже если внутри окна есть свой `ORDER BY` (он влияет только на вычисление функции,
не на раскладку результата). Хотите предсказуемый вывод — пишите финальный
`ORDER BY` явно.

## Мысленная модель «строка → что видит её окно»

Для запроса `row_number() OVER (PARTITION BY category ORDER BY revenue DESC)`
разложим по шагам, что происходит с каждой строкой:

```
Шаг 1. PARTITION BY category  — разложить строки по разделам:
        Электроника: [Смартфон 60000, Планшет 30000, Наушники 30000, Колонка 12000]
        Книги:       [Алгоритмы 8000, Рефакторинг 5000, Чистый код 4500, SQL 3000]
        ...

Шаг 2. ORDER BY revenue DESC  — упорядочить ВНУТРИ каждого раздела (уже так и стои́т)

Шаг 3. для КАЖДОЙ строки функция смотрит на своё окно и берёт номер по порядку:
        строка «Смартфон»  → окно = [Электроника], её позиция = 1  → row_number = 1
        строка «Планшет»   → окно = [Электроника], её позиция = 2  → row_number = 2
        строка «Алгоритмы» → окно = [Книги],       её позиция = 1  → row_number = 1
        ...
```

Каждая строка «видит» только свой раздел, выстроенный своим `ORDER BY`, и получает
значение по этому окну. Ни одна строка не исчезает.

## Про рамки — забегая вперёд

Когда у **агрегатной** оконной функции (`sum`, `avg`, …) появляется `ORDER BY`
внутри `OVER`, включается понятие **рамки** (frame): по умолчанию функция считается
не по всему разделу, а «от начала раздела до текущей строки» — получается
**нарастающий итог**. Управление рамками (`ROWS`/`RANGE`/`GROUPS`), их поведение по
умолчанию и тонкости с `ORDER BY` — это [lab011](../../lab011/). В **этой** лабе у
агрегатных окон мы `ORDER BY` внутри `OVER` **не используем** — берём **весь
раздел** целиком (см. [файл 05](05-aggregate-windows.md)). А `ORDER BY` внутри
`OVER` применяем только с **ранжирующими** функциями, где он задаёт нумерацию.
