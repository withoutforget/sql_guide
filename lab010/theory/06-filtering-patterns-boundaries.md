# 06 — Где можно писать окна, как фильтровать по ним, и базовые паттерны

Осталось собрать всё вместе: **где** в запросе оконным функциям место, почему по их
результату нельзя фильтровать напрямую, и какие **паттерны** из этого вырастают
(top-N на группу, дедупликация, доли, ранги). А в конце — что осознанно оставляем на
[lab011](../../lab011/).

## Где разрешены оконные функции: только `SELECT` и `ORDER BY`

Оконную функцию можно писать **лишь в двух местах** запроса: в списке `SELECT` и в
`ORDER BY`. В `WHERE`, `GROUP BY`, `HAVING` — **нельзя**:

```sql
-- ❌ ОШИБКА
SELECT name, revenue
FROM products
WHERE row_number() OVER (ORDER BY revenue DESC) = 1;
```
```
ERROR:  window functions are not allowed in WHERE
```

Это не каприз — это следствие **порядка исполнения**.

## Полный конвейер: куда встают окна

Мы уже собирали конвейер в [lab001](../../lab001/theory/06-group-having.md) и
[lab006](../../lab006/). Впишем в него оконные функции — они считаются **поздно**:

```
FROM / JOIN     — взять и соединить таблицы
      ↓
WHERE           — отфильтровать строки        ← окон ещё НЕТ
      ↓
GROUP BY        — свернуть в группы           ← окон ещё НЕТ
      ↓
HAVING          — отфильтровать группы        ← окон ещё НЕТ
      ↓
ОКОННЫЕ ФУНКЦИИ — посчитать OVER(...)         ← вот здесь!
      ↓
SELECT (список) — вычислить/выбрать столбцы
      ↓
DISTINCT        — убрать дубликаты
      ↓
ORDER BY        — отсортировать выдачу         ← тут окна уже посчитаны, можно
      ↓
LIMIT / OFFSET  — обрезать
```

Отсюда всё встаёт на места:

- **Почему нельзя в `WHERE`/`HAVING`:** они отрабатывают **раньше**, чем окна
  посчитаны. На момент `WHERE` значения `row_number()` ещё не существует.
- **Почему можно в `ORDER BY`:** он отрабатывает **позже** окон.
- **Важное следствие:** окно **видит** только те строки, что дожили до этого шага
  (после `WHERE`/`HAVING`). Если отфильтровать строки в `WHERE`, они **не попадут**
  в окно и не повлияют на суммы/ранги. Это удобно (окно считается по «нужному»
  подмножеству), но об этом надо помнить: `WHERE revenue > 0` перед
  `sum(...) OVER ()` изменит и итог.

## Как отфильтровать по результату окна: обёртка (подзапрос/CTE)

Раз в `WHERE` окно писать нельзя, а отобрать «строки, где `row_number = 1`» надо
постоянно — применяют **универсальный приём**: посчитать окно во **внутреннем**
запросе (подзапрос или CTE из [lab005](../../lab005/)), а фильтровать по нему —
уже **снаружи**, где это обычный столбец:

```sql
WITH ranked AS (
    SELECT ...,
           row_number() OVER (PARTITION BY ... ORDER BY ...) AS rn   -- считаем окно ВНУТРИ
    FROM ...
)
SELECT ...
FROM ranked
WHERE rn = 1;                 -- фильтруем СНАРУЖИ: тут rn — обычный столбец
```

Внутри `ranked` оконная функция стои́т в `SELECT` (разрешено); снаружи `rn` —
готовое число, по нему можно фильтровать как угодно. На этом приёме держатся почти
все практические паттерны ниже.

## Паттерн 1: top-N строк на группу

Задача-архетип: «топ-1 товар в каждой категории», «топ-3 заказа каждого клиента».
Нумеруем строки внутри группы по нужному порядку и снаружи оставляем первые N.

```sql
-- топ-1 товар по выручке в каждой категории
WITH ranked AS (
    SELECT c.name AS category, p.name AS product, p.revenue,
           row_number() OVER (PARTITION BY c.name ORDER BY p.revenue DESC, p.id) AS rn
    FROM products p JOIN categories c ON c.id = p.category_id
)
SELECT category, product, revenue
FROM ranked
WHERE rn = 1                  -- для top-N: WHERE rn <= N
ORDER BY revenue DESC;
```
```
  category   |      product      | revenue
-------------+-------------------+---------
 Электроника | Смартфон Nova     | 60000.00
 Дом         | Кофеварка Barista | 24000.00
 Игрушки     | Конструктор «Город»| 9000.00
 Книги       | Книга «Алгоритмы» |  8000.00
```

Для top-2/top-3 — меняем условие на `WHERE rn <= 2` (`<= 3`). В «Доме» ничья за
1-е место (Кофеварка и Робот-пылесос по 24 000): `row_number` с доводчиком `, p.id`
выбрал **ровно одного** (меньший `id` — Кофеварку). Это и есть чистое разрешение
ничьих.

### Сравнение с приёмами из прошлых лаб

Ту же задачу «top-N на группу» мы уже решали двумя способами — сравним:

| Приём | Как | Плюсы | Минусы |
|-------|-----|-------|--------|
| `= MAX(...)` через CTE ([lab005](../../lab005/queries/12_top_customer_per_category.sql)) | посчитать максимум группы, оставить строки, равные ему | просто для «топ-1 по одному полю» | **ловушка ничьих**: при равном максимуме вернёт **несколько** строк; только для top-**1** |
| `LATERAL ... LIMIT N` ([lab006](../../lab006/queries/10_top_n_products_per_category.sql)) | для каждой группы — подзапрос с `ORDER BY ... LIMIT N` | гибко, любой N, любые колонки | по подзапросу на группу; синтаксис тяжелее |
| **`row_number() ... WHERE rn <= N`** (эта лаба) | пронумеровать в группе, отсечь снаружи | один проход, ничьи разруливаются доводчиком, легко менять N | нужна обёртка (CTE/подзапрос) |

Главный выигрыш оконного варианта над `= MAX`: **чистое разрешение ничьих**. `= MAX`
при двух одинаковых максимумах вернёт **обоих** (иногда это и нужно — тогда берут
`rank() = 1`); `row_number() = 1` вернёт **ровно одного**, и доводчик в `ORDER BY`
делает выбор предсказуемым.

> Хотите «топ-1, но при ничьей показать **всех** лидеров» — берите `rank() = 1`
> вместо `row_number() = 1`. Хотите «ровно один лидер» — `row_number() = 1`. Выбор
> ранжирующей функции = выбор политики ничьих (см. [файл 03](03-ranking-functions.md)).

## Паттерн 2: дедупликация (одна строка на ключ)

«Оставить по одному заказу на клиента — самый последний». Нумеруем строки внутри
ключа в нужном порядке и оставляем `rn = 1`:

```sql
WITH ranked AS (
    SELECT o.id, cu.name AS customer, o.ordered_at,
           row_number() OVER (PARTITION BY o.customer_id
                              ORDER BY o.ordered_at DESC, o.id DESC) AS rn
    FROM orders o JOIN customers cu ON cu.id = o.customer_id
)
SELECT customer, ordered_at
FROM ranked
WHERE rn = 1
ORDER BY customer;
```

`PARTITION BY` — по **ключу**, который дедуплицируем (клиент); `ORDER BY` внутри
окна — правило «какую из дублей оставить» (последнюю по дате; `id DESC` — доводчик
на случай двух заказов в один день). Снаружи `rn = 1` оставляет ровно одну строку на
ключ. Тот же приём убирает дубли по любому ключу (последняя цена товара, последняя
версия записи и т. п.).

## Паттерн 3: ранг/доля рядом с каждой строкой

Здесь обёртка **не нужна** — если фильтровать не надо, а просто показать ранг или
долю рядом со строкой (отчёт «каждый товар + его место в категории + его доля»),
оконная функция живёт прямо в `SELECT`:

```sql
SELECT c.name AS category, p.name AS product, p.revenue,
       rank() OVER (PARTITION BY c.name ORDER BY p.revenue DESC) AS rank_in_cat,
       round(100.0 * p.revenue / sum(p.revenue) OVER (PARTITION BY c.name), 2) AS pct_in_cat
FROM products p JOIN categories c ON c.id = p.category_id
ORDER BY c.name, p.revenue DESC;
```

Ранг и доля появляются рядом с каждым товаром, строки не теряются. Долевые окна
подробно — в [файле 05](05-aggregate-windows.md).

## Несколько окон в одном запросе — это нормально

В одном `SELECT` можно смешивать сколько угодно оконных вызовов с **разными**
`OVER (...)` — каждый считается по своему окну (мы так делали в
[файле 03](03-ranking-functions.md): `row_number` по одному порядку, `rank` по
другому). Разные окна независимы. А чтобы не повторять длинное `OVER (...)` много
раз, в SQL есть **именованные окна** (клауза `WINDOW`) — но это уже
[lab011](../../lab011/).

## Границы темы: что осознанно оставили на lab011

Мы прошли **основы** окон. Намеренно **не** трогали (это следующая лаба):

- **рамки** (frames) — `ROWS`/`RANGE`/`GROUPS`, поведение по умолчанию и его ловушки
  при `ORDER BY` у агрегатов;
- **нарастающие и скользящие** агрегаты (running/moving total, скользящее среднее);
- функции **смещения** — `LAG`/`LEAD` («значение из предыдущей/следующей строки»);
- `FIRST_VALUE`/`LAST_VALUE`/`NTH_VALUE` — «первое/последнее/N-е значение окна»;
- именованные окна — клауза **`WINDOW`**.

В этой лабе — только `PARTITION BY`/`ORDER BY`, ранжирующие функции и агрегаты как
окна **без явных рамок**. Этого достаточно для top-N, дедупликации, долей и рангов —
самых ходовых аналитических задач.
