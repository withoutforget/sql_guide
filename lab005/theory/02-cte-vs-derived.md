# 02 — CTE против подзапроса в `FROM` (derived table)

[← что такое CTE](01-what-is-cte.md) · [оглавление](../README.md) · [дальше: несколько CTE и цепочки →](03-multiple-and-chains.md)

---

CTE — это **прямая эволюция** подзапроса в `FROM` (производной таблицы,
derived table), который мы разбирали в lab004 ([файл 04 той лабы](../../lab004/theory/04-from-derived-row.md)).
Оба дают «временную таблицу внутри запроса». Разница — в **читаемости** и в
одной важной возможности, которой у derived table нет. Разберём подробно.

## Та же задача двумя способами

Вспомним задачу из lab004: «какова **средняя** сумма покупок в расчёте на
клиента» (а заодно максимум и минимум). Это **агрегат от агрегата** — среднее
**от** клиентских сумм; напрямую `AVG(SUM(...))` писать нельзя, нужен
промежуточный шаг «сумма на клиента».

### Способ 1 — подзапрос в `FROM` (как в lab004)

```sql
SELECT
    ROUND(AVG(total), 2) AS avg_total,
    MAX(total)           AS max_total,
    MIN(total)           AS min_total
FROM (
    SELECT o.customer_id, SUM(p.price * o.quantity) AS total
    FROM orders   AS o
    JOIN products AS p ON p.id = o.product_id
    GROUP BY o.customer_id
) AS per_customer;               -- ← производная таблица прямо внутри FROM
```

### Способ 2 — то же самое через CTE

```sql
WITH per_customer AS (
    SELECT o.customer_id, SUM(p.price * o.quantity) AS total
    FROM orders   AS o
    JOIN products AS p ON p.id = o.product_id
    GROUP BY o.customer_id
)
SELECT
    ROUND(AVG(total), 2) AS avg_total,
    MAX(total)           AS max_total,
    MIN(total)           AS min_total
FROM per_customer;               -- ← тот же результат, но он ВЫНЕСЕН и НАЗВАН выше
```

Оба возвращают **абсолютно одинаковый** результат:

```
 avg_total | max_total | min_total
-----------+-----------+-----------
  17165.00 |  34640.00 |   2070.00
(1 row)
```

Разница — чисто в форме записи. В первом случае «внутренний шаг» вложен прямо в
`FROM` главного запроса; во втором — вынесен наверх, назван `per_customer`, и
главный запрос стал коротким и очевидным: «посчитать avg/max/min по таблице
`per_customer`». Что такое `per_customer`, читатель видит **отдельно и заранее**,
а не выкапывает из середины `FROM`.

### Почему CTE-версия читается лучше

- Главный запрос (`SELECT AVG ... FROM per_customer`) свёлся к сути — не отвлекает
  на то, как именно считается `per_customer`.
- Промежуточный шаг **назван** осмысленно. `per_customer` объясняет замысел; у
  анонимного подзапроса имени нет.
- Читается **сверху вниз**: сначала «вот шаг per_customer», потом «а вот что мы с
  ним делаем». Для одного уровня вложенности разница невелика, но с ростом
  сложности (2–3 шага, [файл 03](03-multiple-and-chains.md)) CTE выигрывает
  разительно: вложенные подзапросы превращаются в нечитаемую матрёшку.

## Главное отличие: переиспользование

Это не только про красоту. У CTE есть возможность, которой у derived table
**нет**: один и тот же промежуточный результат можно использовать в запросе
**несколько раз**, объявив его **один раз**.

С производной таблицей так нельзя — она существует ровно в том месте `FROM`, где
написана. Если тот же набор нужен ещё раз (в другом `JOIN`, в подзапросе,
для сравнения самого с собой), пришлось бы **скопировать весь подзапрос
повторно**. Это дублирование: длиннее, и при правке легко забыть поправить одну
из копий.

Пример: «клиенты, потратившие **больше среднего** по клиентам, и на сколько». Тут
`per_customer` нужен **дважды**: один раз — чтобы взять строки клиентов, второй —
чтобы посчитать по нему же среднее (`AVG(total)`).

```sql
WITH per_customer AS (
    SELECT o.customer_id, SUM(p.price * o.quantity) AS total
    FROM orders   AS o
    JOIN products AS p ON p.id = o.product_id
    GROUP BY o.customer_id
)
SELECT c.name,
       pc.total,
       ROUND(pc.total - (SELECT AVG(total) FROM per_customer), 2) AS diff_from_avg
FROM per_customer AS pc                                  -- ← использование №1
JOIN customers AS c ON c.id = pc.customer_id
WHERE pc.total > (SELECT AVG(total) FROM per_customer)   -- ← использование №2
ORDER BY pc.total DESC;
```
```
 name  |  total   | diff_from_avg
-------+----------+---------------
 Егор  | 34640.00 |      17475.00
 Анна  | 30550.00 |      13385.00
 Борис | 21980.00 |       4815.00
(3 rows)
```

`per_customer` объявлен **один раз**, а сослались на него **три раза** (в `FROM`
и в двух скалярных подзапросах `(SELECT AVG(total) FROM per_customer)`). С derived
table пришлось бы трижды выписать один и тот же агрегирующий подзапрос. Именно
эту задачу мы решаем в [задаче 3](../queries/03_above_avg_customers.sql).

> Как PostgreSQL исполняет CTE, использованный несколько раз (посчитает ли он его
> один раз и переиспользует, или встроит в каждое место) — это про
> `MATERIALIZED` / `NOT MATERIALIZED`, разбираем в [файле 04](04-materialized.md).
> Здесь важно, что **записать** это одним объявлением можно только с CTE.

## Сравнительная таблица

| Свойство | Подзапрос в `FROM` (derived table) | CTE (`WITH`) |
|----------|-----------------------------------|--------------|
| Что даёт | временную таблицу внутри `FROM` | именованную временную таблицу |
| Имя | нет (только алиас в месте использования) | да, осмысленное, объявлено заранее |
| Порядок чтения | «изнутри наружу» (матрёшка) | сверху вниз, слева направо |
| Переиспользование в запросе | ❌ надо копировать целиком | ✅ объявил раз — используй сколько нужно |
| Ссылка на предыдущий шаг | ❌ только вложенностью | ✅ следующий CTE видит предыдущий ([файл 03](03-multiple-and-chains.md)) |
| Живёт | в пределах запроса | в пределах запроса |
| Создаёт объект в схеме | нет | нет |

Обратите внимание: обе конструкции **эквивалентны по смыслу** и по результату —
CTE не «мощнее» математически, он **удобнее и выразительнее** для человека.
Любой derived table переписывается в CTE и обычно наоборот.

## Когда что выбирать

- **CTE** — почти всегда, когда шаг хоть немного нетривиален: если он **назван**,
  становится понятнее; если шагов **несколько** — читаемость растёт кратно; если
  один результат нужен **дважды** — без CTE вообще не обойтись без дублирования.
- **Подзапрос в `FROM`** — уместен для совсем короткого одноразового шага, где
  выносить его наверх и придумывать имя — избыточно (например, крошечная
  производная таблица внутри большого `JOIN`). Дело вкуса и локальной читаемости.

Практическое правило: **как только запрос перестаёт читаться с первого взгляда —
раскладывайте его на CTE**. Это самый дешёвый способ сделать сложный SQL
понятным (в том числе будущему себе).

## Итог

- CTE и подзапрос в `FROM` дают одно и то же — «временную таблицу внутри запроса»,
  и взаимно переписываются; результат идентичен.
- CTE **читается лучше** (именованный шаг, чтение сверху вниз) и, в отличие от
  derived table, позволяет **переиспользовать** один результат в нескольких
  местах, объявив его один раз.
- Задачу lab004 «агрегат от агрегата» (avg от sum) на CTE записать нагляднее;
  для многошаговой аналитики CTE становится практически обязательным.
