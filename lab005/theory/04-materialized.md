# 04 — `MATERIALIZED` и `NOT MATERIALIZED`

[← несколько CTE и цепочки](03-multiple-and-chains.md) · [оглавление](../README.md) · [дальше: область видимости и подводные камни →](05-scope-pitfalls.md)

CTE — про читаемость. Но у него есть и второй смысл: **как СУБД физически
исполняет именованный шаг** — вычисляет его отдельно «один раз и складывает в
буфер» или **встраивает** его определение прямо в главный запрос. Этим управляют
ключевые слова `MATERIALIZED` и `NOT MATERIALIZED`. Здесь — только про **поведение
и семантику**; как читать план запроса (`EXPLAIN`), измерять и выбирать — тема
lab025.

## Две стратегии исполнения CTE

Есть два способа выполнить `WITH cte AS (...) SELECT ... FROM cte ...`:

- **Материализация (materialized).** СУБД **сначала полностью вычисляет** запрос
  CTE, складывает результат во временный буфер (в памяти или на диске), а потом
  главный запрос читает готовые строки из этого буфера. CTE считается **один
  раз**, как отдельная «промежуточная таблица».

- **Встраивание (inlining).** СУБД **подставляет** определение CTE прямо в то
  место главного запроса, где он используется, — как будто вы написали подзапрос
  в `FROM` вместо CTE. Отдельного буфера нет; планировщик оптимизирует главный
  запрос и тело CTE **вместе**, как единое целое.

Разница не в результате (он одинаков!), а в том, **что может сделать
оптимизатор** и **сколько раз** выполнится тело CTE.

## Поведение по умолчанию в PostgreSQL 12+

До PostgreSQL 12 CTE **всегда** материализовался — это был жёсткий
«оптимизационный забор» (optimization fence). Начиная с **PostgreSQL 12** правило
такое (речь про **нерекурсивные** CTE):

- CTE, **использованный в запросе РОВНО ОДИН раз** и **без побочных эффектов**
  (не изменяет данные, без «изменчивых» функций вроде `random()`), СУБД **может
  встроить** (inline) — чтобы оптимизировать сквозь границу CTE.
- CTE, **использованный несколько раз**, по умолчанию **материализуется** (чтобы
  не вычислять одно и то же по нескольку раз).
- **Рекурсивный** CTE (`WITH RECURSIVE`, lab007) всегда материализуется.

Ключевые слова позволяют **переопределить** это решение вручную:

| Запись | Что заставляет сделать |
|--------|------------------------|
| `WITH x AS (...)` | поведение **по умолчанию** (см. выше: 1 раз → возможно встроит; N раз → материализует) |
| `WITH x AS MATERIALIZED (...)` | **всегда материализовать** — вычислить один раз в буфер, даже если используется однократно |
| `WITH x AS NOT MATERIALIZED (...)` | **всегда встраивать** — подставить определение, даже если используется несколько раз |

Синтаксис — слово ставится **между `AS` и скобкой**:

```sql
WITH per_customer AS MATERIALIZED (
    SELECT o.customer_id, SUM(p.price * o.quantity) AS total
    FROM orders AS o JOIN products AS p ON p.id = o.product_id
    GROUP BY o.customer_id
)
SELECT ...
```

## `MATERIALIZED` как «оптимизационный забор»

Материализованный CTE — это **барьер для оптимизатора**. Планировщик обязан
вычислить его «как написано», целиком и отдельно, и **не может** протолкнуть
внутрь условия из главного запроса. Иногда это именно то, что нужно, иногда —
наоборот, мешает. Разберём обе стороны.

### Когда `MATERIALIZED` полезен

**1. Дорогой расчёт, переиспользуемый несколько раз.** Если один и тот же тяжёлый
шаг нужен в запросе многократно, материализация гарантирует, что он посчитается
**один раз**, а не заново на каждое использование. По умолчанию PostgreSQL и так
материализует CTE, использованный >1 раза, — но `MATERIALIZED` фиксирует это
**явно** (намерение видно в коде и не «сломается», если позже число использований
изменится):

```sql
WITH category_revenue AS MATERIALIZED (      -- посчитать выручку по категориям ОДИН раз
    SELECT p.category_id, SUM(p.price * o.quantity) AS revenue
    FROM orders AS o JOIN products AS p ON p.id = o.product_id
    GROUP BY p.category_id
)
SELECT cat.name, cr.revenue,
       ROUND(cr.revenue * 100.0 / (SELECT SUM(revenue) FROM category_revenue), 2) AS pct
FROM category_revenue AS cr                  -- использование №1
JOIN categories AS cat ON cat.id = cr.category_id
WHERE cr.revenue > (SELECT AVG(revenue) FROM category_revenue)   -- использования №2 и №3
ORDER BY cr.revenue DESC;
```
```
    name     | revenue  |  pct
-------------+----------+-------
 Электроника | 83420.00 | 60.75
 Дом         | 34950.00 | 25.45
(2 rows)
```

`category_revenue` тут нужен трижды (строки, `SUM`, `AVG`). `MATERIALIZED` явно
говорит «свернуть заказы по категориям один раз и дальше работать с готовой
маленькой табличкой». Это [задача 10](../queries/10_materialized.sql).

**2. Барьер, когда встраивание нежелательно.** Иногда встраивание меняет план на
худший, или CTE вызывает функцию, которую нельзя выполнять по многу раз.
`MATERIALIZED` жёстко отделяет шаг — «посчитай его ровно так и ровно раз».

### Когда `MATERIALIZED` мешает — и нужен `NOT MATERIALIZED`

Барьер отрезает оптимизацию. Классический случай — когда главный запрос **берёт из
CTE лишь маленький кусочек**:

```sql
WITH all_orders AS ( SELECT * FROM orders )   -- «весь» набор
SELECT * FROM all_orders WHERE customer_id = 6;
```

- Если CTE **встроен** (по умолчанию: используется 1 раз, без побочек), условие
  `customer_id = 6` уходит внутрь, к таблице `orders`, — и может использовать её
  **индекс**, прочитав только нужные строки.
- Если CTE **материализован**, СУБД сначала честно соберёт **все** заказы в буфер,
  и лишь потом отфильтрует `customer_id = 6` уже по буферу. Индекс базовой таблицы
  тут не поможет — буфер это отдельный временный набор **без индексов**.

Поэтому на больших данных лишний `MATERIALIZED` бывает **медленнее**. Если нужно,
чтобы оптимизатор «видел сквозь» CTE и проталкивал условия к таблицам, — пишут
`NOT MATERIALIZED`:

```sql
WITH all_orders AS NOT MATERIALIZED ( SELECT * FROM orders )
SELECT * FROM all_orders WHERE customer_id = 6;   -- условие дойдёт до таблицы orders
```

## Важно: на результат это не влияет

Подчеркнём ещё раз: `MATERIALIZED` / `NOT MATERIALIZED` меняют **как** считается
запрос (план, скорость, число вычислений шага), но **не меняют, что** он вернёт.
Строки на выходе одинаковы при любом варианте. Это чисто **оптимизационная**
подсказка планировщику.

> В учебной базе lab005 данных мало (десятки строк), так что разницы в скорости вы
> не заметите. `MATERIALIZED` здесь используем ради **семантики и привычки** —
> «этот шаг считаем один раз и дальше работаем с готовым результатом». На
> продакшене (миллионы строк) выбор между материализацией и встраиванием уже
> реально влияет на время; как это диагностировать через `EXPLAIN` — в lab025.

## Подводные камни

- **Материализованный CTE не индексируется.** Это временный набор строк без
  индексов и статистики. Фильтр/джойн **по CTE** идёт полным перебором его строк;
  индекс исходной таблицы к нему не применяется. Если из большого CTE нужен
  маленький срез по условию — материализация может быть дороже встраивания.
- **«Забор» работает в обе стороны.** `MATERIALIZED` мешает оптимизатору
  проталкивать условия внутрь; `NOT MATERIALIZED` — наоборот, может заставить
  пересчитать тело CTE несколько раз (по разу на каждое использование), если он
  используется многократно.
- **Не путайте с материализованным представлением.** `MATERIALIZED VIEW`
  (`CREATE MATERIALIZED VIEW`, lab022) — это **сохранённая на диске** таблица с
  результатом, живущая между запросами. `MATERIALIZED` у CTE — лишь способ
  исполнить шаг **внутри одного запроса**; наружу ничего не сохраняется. Слово
  одно, вещи разные.

## Итог

- У CTE две стратегии исполнения: **материализация** (посчитать один раз в буфер)
  и **встраивание** (подставить определение в главный запрос, оптимизировать
  вместе).
- **PostgreSQL 12+**: нерекурсивный CTE без побочных эффектов, использованный
  **один раз**, может быть **встроен**; использованный **несколько раз** — по
  умолчанию **материализуется**.
- `MATERIALIZED` — «вычислить один раз, барьер оптимизации»; `NOT MATERIALIZED` —
  «встроить всегда, разреши оптимизатору смотреть сквозь».
- На **результат** это не влияет — только на план и скорость. Материализованный
  CTE — временный набор **без индексов**; фильтр по нему не использует индексы
  таблиц.
