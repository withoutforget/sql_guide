# 05 — Область видимости, подводные камни, что за пределами лабы

[← MATERIALIZED](04-materialized.md) · [оглавление](../README.md)

---

Последний файл: собираем правила области видимости, перечисляем подводные камни,
и очерчиваем **границы темы** — какие ещё бывают CTE (и в какой лабе они).

## Область видимости — три правила

1. **CTE виден только внутри своего запроса.** Это не таблица и не представление;
   в схеме ничего не создаётся. Из соседнего запроса к нему не обратиться
   ([файл 01](01-what-is-cte.md)).
2. **Внутри секции `WITH` ссылаться можно только «назад».** CTE видит те, что
   объявлены **выше** него; сослаться «вперёд» на нижеследующий CTE — ошибка
   ([файл 03](03-multiple-and-chains.md)). Пишите шаги в порядке вычисления.
3. **В главном запросе доступны все объявленные CTE.** После `WITH` финальный
   `SELECT` может использовать любой из объявленных шагов, в любом порядке и
   сколько угодно раз.

Ещё пара тонкостей видимости:

- **Имя CTE перекрывает одноимённую таблицу.** Если назвать CTE `orders`, то
  внутри этого запроса `FROM orders` будет читать **CTE**, а не таблицу схемы.
  Чтобы не путаться, CTE обычно называют иначе, чем таблицы (`order_lines`,
  `per_customer`, `category_revenue`).
- **Внутренние алиасы наружу не видны.** Алиасы таблиц внутри CTE (`o`, `p`)
  живут только внутри него. Снаружи доступны лишь **колонки результата** CTE —
  те, что перечислены в его `SELECT`. Поэтому вычисляемым колонкам внутри CTE
  сразу дают понятные имена (`AS total`, `AS revenue`).

## Подводные камни

| Ловушка | В чём беда | Как избежать |
|---------|-----------|--------------|
| **Ссылка «вперёд»** | CTE ссылается на объявленный ниже → `relation does not exist` | Расположить шаги в порядке вычисления (базовые — выше) |
| **Обращение к CTE снаружи запроса** | CTE не существует за пределами своего запроса | Нужен результат между запросами → `VIEW` / `TEMP TABLE` (lab022) |
| **`WITH` перед каждым CTE / запятая перед `SELECT`** | Синтаксическая ошибка | Один `WITH`, запятые **между** CTE, без запятой перед главным запросом |
| **Материализованный CTE не индексируется** | Фильтр/джойн по CTE идёт перебором; индекс базовой таблицы не работает | Если нужен маленький срез по большому CTE — `NOT MATERIALIZED` (или обычный подзапрос) — [файл 04](04-materialized.md) |
| **Лишний `MATERIALIZED` как «забор»** | Мешает протолкнуть условие к таблице → может быть медленнее | Не форсировать материализацию без нужды; мерить (`EXPLAIN`, lab025) |
| **Имя CTE совпало с таблицей** | Внутри запроса CTE «затеняет» таблицу — читается не то | Называть CTE иначе, чем таблицы |
| **«Топ-1 в группе» через `revenue = MAX(...)`** | При **ничьей** на максимуме join/подзапрос вернёт **всех** рекордсменов — несколько строк на группу (см. [задачу 12](../queries/12_top_customer_per_category.sql)) | Осознанно решить, нужны ли все при ничьей; ровно один — оконным `ROW_NUMBER()` в lab010 |
| **Ожидать, что CTE ускорит запрос сам по себе** | CTE — про читаемость, а не про скорость; иногда даже медленнее из-за материализации | Оптимизировать осознанно, а не «завернуть в WITH ради скорости» |

### «CTE ускоряет запрос» — это миф

Частое заблуждение: «оберну подзапрос в CTE — станет быстрее». Нет. CTE — это про
**читаемость и структуру**, а не про производительность. С точки зрения скорости
он в лучшем случае **нейтрален** (когда встраивается — [файл 04](04-materialized.md)),
а материализованный может быть и **медленнее** обычного подзапроса, потому что его
результат — временный набор без индексов. Пишите CTE ради ясности; за скоростью —
к индексам и планам (lab024–lab025).

## Что бывает ещё (и в каких лабах)

Мы разобрали **нерекурсивные читающие** CTE. У `WITH` есть ещё два режима — за
пределами этой лабы, но полезно знать про их существование:

### Рекурсивные CTE — `WITH RECURSIVE` (это lab007)

Обычный CTE не может ссылаться сам на себя. **`WITH RECURSIVE`** — может: он
вычисляется итеративно, «наращивая» результат, пока тот растёт. Это инструмент
для **иерархий и графов**: дерево категорий (родитель → потомки на любую глубину),
цепочка сотрудник → руководитель, «все узлы, достижимые из данного», разворачивание
маршрутов, генерация последовательностей.

```sql
-- ПРЕДВЕСТИЕ (это lab007, здесь НЕ используем): числа от 1 до 5
WITH RECURSIVE nums AS (
    SELECT 1 AS n                 -- «якорь» — стартовая строка
    UNION ALL
    SELECT n + 1 FROM nums        -- «рекурсивная часть» — ссылается на сам CTE
    WHERE n < 5
)
SELECT n FROM nums;               -- 1, 2, 3, 4, 5
```

Всё, что нужно про синтаксис (`RECURSIVE`, якорь, `UNION [ALL]`, условие
остановки, защита от зацикливания) — **в lab007**. В наших задачах lab005
рекурсию **не применяем**: все шаги ссылаются только «назад», не на себя.

### Изменяющие данные CTE — `WITH ... AS (INSERT/UPDATE/DELETE ... RETURNING)` (это lab019)

В PostgreSQL внутри `WITH` может стоять не только `SELECT`, но и **команда,
изменяющая данные** (`INSERT`, `UPDATE`, `DELETE`) с `RETURNING`. Так за **один**
запрос делают, например, «перенести строки из одной таблицы в другую» (удалить и
тут же вставить возвращённое). Это требует DML, которого мы ещё не проходили, —
поэтому **см. lab019**. В lab005 все CTE только **читают**.

## Когда использовать CTE, а когда нет

**Берите CTE, когда:**

- запрос многошаговый — есть понятные промежуточные результаты, которым хочется
  дать имя (выручка по категориям, крупные клиенты, суммы по городам);
- один промежуточный результат нужен **несколько раз** (без CTE — дублирование);
- запрос перестал читаться с первого взгляда — декомпозиция на именованные шаги
  почти всегда делает его понятнее;
- нужна рекурсия по иерархии/графу — тогда `WITH RECURSIVE` (lab007).

**Можно обойтись без CTE, когда:**

- шаг крошечный и одноразовый — обычный подзапрос в `FROM`/`WHERE` не хуже
  ([файл 02](02-cte-vs-derived.md));
- вы гонитесь за скоростью и подозреваете, что материализация мешает плану, —
  тогда либо `NOT MATERIALIZED`, либо переписать без CTE (но сначала измерьте,
  lab025).

## Итог лабы

- **CTE** (`WITH имя AS (...)`) — именованный временный результат, видимый только
  внутри своего запроса. Даёт **читаемость**, **декомпозицию на шаги** и
  **переиспользование** одного результата в нескольких местах.
- CTE — читаемая эволюция **подзапроса в `FROM`** (lab004); результат тот же, но
  шаг **назван** и запрос читается сверху вниз.
- Несколько CTE — через запятую; **цепочки** — следующий ссылается на предыдущий;
  ссылаться можно только **назад**; снаружи запроса CTE не существует.
- `MATERIALIZED` / `NOT MATERIALIZED` управляют исполнением (посчитать один раз в
  буфер vs встроить и оптимизировать вместе); на **результат** не влияют.
  По умолчанию в PG12+: 1 использование → возможно встроит, N → материализует.
- CTE — про **ясность**, не про скорость. Материализованный CTE **не
  индексируется**.
- **Рекурсивные** (`WITH RECURSIVE`) — lab007; **изменяющие данные** — lab019.

## Что дальше

В `lab006` — **`LATERAL` и функции-множества** (`generate_series`, `unnest` и
другие функции, возвращающие таблицу). `LATERAL` позволяет подзапросу/функции в
`FROM` **ссылаться на колонки предыдущих таблиц того же `FROM`** — то есть считать
что-то «для каждой строки слева» (топ-N заказов на клиента, разворачивание
массивов, соединение с табличной функцией). Вместе с CTE это следующий шаг к
по-настоящему выразительным аналитическим запросам.
