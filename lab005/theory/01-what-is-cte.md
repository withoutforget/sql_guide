# 01 — Что такое CTE. Оператор `WITH`

[← оглавление лабы](../README.md) · [дальше: CTE vs подзапрос в FROM →](02-cte-vs-derived.md)

## Что это

**CTE** (Common Table Expression, «обобщённое табличное выражение») — это
**именованный подзапрос**, который вы объявляете в начале запроса оператором
`WITH`, а затем используете дальше по имени, как обычную таблицу.

Проще всего думать так: CTE — это **временный именованный результат**, который
живёт **только в пределах одного запроса**. Вы говорите «пусть `revenue` — это
вот такой `SELECT`», и ниже пишете `FROM revenue`, будто `revenue` — настоящая
таблица. Как только запрос отработал, `revenue` исчезает: в схеме ничего не
создаётся, никакой таблицы на диске не появляется.

Минимальный синтаксис:

```sql
WITH имя AS (
    SELECT ...            -- определение CTE: любой обычный запрос
)
SELECT ...                -- ГЛАВНЫЙ запрос, который пользуется CTE по имени
FROM имя;
```

- `WITH` открывает секцию определений и стоит **в самом начале**, перед главным
  `SELECT`.
- `имя AS (...)` — объявляем CTE: даём имя и в скобках пишем запрос, который его
  наполняет.
- После закрывающей скобки идёт **главный запрос** (его называют «внешний»,
  «финальный» или «основной»), и вот он уже обращается к CTE по имени.

## Первый пример — с результатом

Посчитаем суммарную выручку по каждому клиенту (выручка строки заказа = цена
товара × количество) и выведем клиентов от самых «дорогих» к «дешёвым».

```sql
WITH customer_revenue AS (
    SELECT o.customer_id,
           SUM(p.price * o.quantity) AS total
    FROM orders   AS o
    JOIN products AS p ON p.id = o.product_id
    GROUP BY o.customer_id
)
SELECT c.name, cr.total
FROM customer_revenue AS cr
JOIN customers AS c ON c.id = cr.customer_id
ORDER BY cr.total DESC;
```
```
 name  |  total
-------+----------
 Егор  | 34640.00
 Анна  | 30550.00
 Борис | 21980.00
 Вера  | 15660.00
 Дарья | 14460.00
 Глеб  |  9480.00
 Захар |  8480.00
 Жанна |  2070.00
(8 rows)
```

Что произошло по шагам:

1. Внутри `WITH` мы посчитали «клиент → его суммарная трата». Это отдельный
   агрегирующий запрос — ровно такой же, как в lab001/lab002, только теперь у
   него есть **имя** `customer_revenue`.
2. Главный запрос обратился к `customer_revenue` **как к таблице**: присоединил
   к нему справочник `customers`, чтобы подтянуть имя, и отсортировал.

Обратите внимание: `customer_revenue` в главном запросе используется как
источник в `FROM` — с алиасом `cr`, к его колонкам обращаемся `cr.customer_id`,
`cr.total`. Колонки CTE — это то, что перечислено в его `SELECT` (с учётом
алиасов): `customer_id` и `total`.

## Мысленная модель: собираем запрос из «кубиков»

Главная ценность CTE — **читаемость через декомпозицию**. Сложный запрос вы
разбиваете на понятные шаги, каждый из которых:

- имеет **говорящее имя** (`customer_revenue`, `category_totals`, `big_spenders`);
- делает **одну понятную вещь** (посчитать выручку, отобрать крупных клиентов);
- **читается сверху вниз, слева направо**, а не «изнутри наружу», как вложенные
  подзапросы.

```
WITH шаг1 AS ( ... ),      ← «кубик» 1: понятный промежуточный результат
     шаг2 AS ( ... ),      ← «кубик» 2: строится, возможно, на основе шага1
     шаг3 AS ( ... )       ← «кубик» 3
SELECT ...                 ← финальная сборка из кубиков
FROM шаг3 ...;
```

Сравните с подзапросами из lab004: там запрос рос «матрёшкой» внутрь
(`SELECT ... FROM (SELECT ... FROM (SELECT ...)))`), и чтобы понять его, глаз
должен был нырять в самую глубину и разматывать наружу. CTE «расплетает»
матрёшку в линейную последовательность именованных шагов.

## Зачем это нужно

Три главные причины использовать CTE:

1. **Читаемость.** Именованный шаг с осмысленным именем понятнее анонимного
   подзапроса. `FROM big_spenders` говорит сам за себя; `FROM (SELECT ... FROM
   ... WHERE total > (SELECT AVG ...))` — нет.

2. **Декомпозиция сложного запроса на шаги.** Многошаговую аналитику («посчитать
   выручку по категориям → найти среднюю по категориям → оставить категории выше
   средней») естественно записать тремя `WITH`-шагами, а не одним запутанным
   выражением. Каждый шаг можно осмыслить и даже проверить по отдельности.

3. **Переиспользование одного промежуточного результата в нескольких местах.**
   Это то, чего подзапрос в `FROM` (derived table из lab004) не умеет: если один
   и тот же промежуточный набор нужен дважды в запросе, derived table пришлось
   бы **скопировать целиком два раза**. CTE объявляется один раз и используется
   по имени сколько угодно раз. Подробнее — в [файле 02](02-cte-vs-derived.md) и
   [файле 03](03-multiple-and-chains.md).

## Область видимости: CTE виден только внутри своего запроса

`WITH` создаёт имена, которые существуют **исключительно внутри того запроса, в
начале которого стоят**. Это не таблица и не представление (view): в схеме ничего
не сохраняется, из другого запроса к CTE обратиться нельзя.

```sql
WITH t AS (SELECT 1 AS x)
SELECT x FROM t;      -- ✅ здесь t доступен

SELECT x FROM t;      -- ❌ ERROR: relation "t" does not exist
                      --    (это уже ДРУГОЙ запрос — t для него не существует)
```

> Если промежуточный результат нужен **между** запросами (сохранить и
> переиспользовать в разных командах) — это уже не CTE, а **представление**
> (`CREATE VIEW`) или **временная таблица** (`CREATE TEMP TABLE`). Им посвящена
> lab022; в этой лабе всё живёт в пределах одного запроса.

## Что может стоять внутри CTE

В скобках `AS (...)` — **любой полноценный запрос**: с `WHERE`, `GROUP BY`,
`HAVING`, `JOIN`, подзапросами, операциями над множествами (`UNION` и т.д.). CTE
может читать таблицы, другие CTE (об этом — [файл 03](03-multiple-and-chains.md))
и вообще всё то же, что обычный `SELECT`.

```sql
WITH expensive AS (
    SELECT p.name, p.price
    FROM products AS p
    JOIN categories AS c ON c.id = p.category_id
    WHERE p.price > 5000               -- внутри CTE — обычный WHERE
)
SELECT * FROM expensive ORDER BY price DESC;
```

> В этой лабе мы разбираем **нерекурсивные** читающие CTE. Кроме них бывают:
> — **рекурсивные** CTE (`WITH RECURSIVE`) для иерархий и графов — это **lab007**;
> — **изменяющие данные** CTE (`WITH ... AS (INSERT/UPDATE/DELETE ... RETURNING)`)
> — они требуют DML, это **lab019**.
> Здесь их не трогаем; подробнее про эти границы — в [файле 05](05-scope-pitfalls.md).

## Синтаксические мелочи, которые важно не забыть

- `WITH` пишется **один раз** в начале, даже если CTE несколько — они
  перечисляются через запятую (см. [файл 03](03-multiple-and-chains.md)), а не
  каждый со своим `WITH`.
- Тело каждого CTE **обязательно** в круглых скобках.
- После секции `WITH` **обязан** идти главный запрос (`SELECT`, а в lab019 — ещё
  `INSERT`/`UPDATE`/`DELETE`). Один `WITH` без главного запроса — ошибка.
- Точка с запятой ставится **в конце всей команды**, а не после каждого CTE.
- Можно явно переименовать колонки CTE в скобках после имени:
  `WITH cr (cust, total) AS (SELECT customer_id, SUM(...) ...)` — тогда снаружи
  колонки зовутся `cust` и `total`. Чаще колонки называют алиасами прямо в
  `SELECT` внутри CTE — как мы и делали (`AS total`).

## Итог

- **CTE** (`WITH имя AS (...)`) — именованный подзапрос, временный результат,
  видимый **только внутри своего запроса**; в схеме ничего не создаётся.
- Даёт три вещи: **читаемость**, **декомпозицию на шаги** и **переиспользование**
  одного результата в нескольких местах (чего derived table не умеет).
- Мысленная модель — «**собрать запрос из именованных кубиков**», читаемых сверху
  вниз, вместо вложенной «матрёшки» подзапросов.
- Внутри CTE — любой обычный запрос. Рекурсивные (`WITH RECURSIVE`, lab007) и
  изменяющие данные (lab019) CTE — за пределами этой лабы.
