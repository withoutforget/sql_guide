# 04 — Содержание (`@>`), существование ключей (`?`) и JSONPath

[← доступ к элементам](03-access.md) · [оглавление](../README.md) · [дальше: изменение jsonb →](05-modify.md)

---

Извлекать значения мы научились. Теперь — **искать документы по условию**: «все
товары Sony», «у каких задан экран», «заказы с товаром X», «позиции дороже
10 000». Здесь работают операторы содержания/существования (быстрые, есть индекс)
и мощный язык путей **JSONPath**. Все они — **только для `jsonb`**.

## Оператор содержания `@>` (и обратный `<@`)

`doc @> образец` — «левый документ **содержит** правый». Это самый ценный
оператор для фильтрации по шаблону. Смысл «содержит» зависит от того, что
сравнивается.

### Объекты: все пары образца есть в документе

```sql
SELECT id, name FROM products
WHERE attributes @> '{"brand":"Sony"}'
ORDER BY id;
```
```
 id |       name
----+------------------
  1 | Смартфон Nova 5
  3 | Наушники TWS Pro
  8 | Монитор 27"
```

Образец `{"brand":"Sony"}` «вкладывается» в документ, если там есть **пара**
`brand: Sony` (прочие ключи документа не мешают). Можно требовать сразу несколько
пар: `attributes @> '{"brand":"Sony","wireless":true}'` — и brand, и wireless.

**Важный подводный камень: сопоставление структурное, ОТ КОРНЯ.** `@>` не ищет
ключ на произвольной глубине — образец накладывается на документ, начиная с
корня, а вложенные значения сравниваются рекурсивно **по совпадающим ключам**:

```sql
SELECT
    '{"a":{"b":1,"c":2}}'::jsonb @> '{"a":{"b":1}}'  AS nested,    -- true: путь a→b совпал
    '{"a":{"b":1}}'::jsonb      @> '{"b":1}'         AS any_depth, -- false: b не на корне!
    '{"foo":{"bar":"baz"}}'::jsonb @> '{"bar":"baz"}' AS deep;     -- false: та же причина
```
```
 nested | any_depth | deep
--------+-----------+------
 t      | f         | f
```

`{"a":{"b":1}}` содержит `{"a":{"b":1}}` (структура совпала от корня), но **не**
содержит `{"b":1}` — ключ `b` в документе лежит внутри `a`, а образец ищет его на
верхнем уровне. Нужно «найти пару где-то вглубине» — это уже JSONPath (ниже).

### Массивы: множество как подмножество

Для массивов `A @> B` означает «каждый элемент `B` содержится в каком-то элементе
`A`» (массив трактуется как **множество**; порядок и дубли не важны). Это позволяет
искать заказы, где среди позиций есть нужный товар:

```sql
SELECT id, customer FROM orders
WHERE items @> '[{"product":"Наушники TWS Pro"}]'
ORDER BY id;
```
```
 id | customer
----+----------
  1 | Анна
  3 | Вера
  5 | Анна
```

Правый образец — массив из **одного НЕполного** объекта `{"product":"..."}`. Он
вложится в любую полную позицию `{product, qty, price}`, где `product` совпал.
Так одним оператором отбираем заказы с конкретным товаром, не разворачивая массив.

### Подводные камни `@>`

- **Числа/строки — только целиком, не частично.** `@>` не делает поиск подстроки
  и не сравнивает «больше/меньше»: `'{"ram":16}' @> '{"ram":1}'` → **false**
  (16 ≠ 1). Для диапазонов и «больше N» нужен JSONPath (ниже).
- **Скаляр в массиве** проверяется как элемент: `'[1,2,3]' @> '2'` → true,
  но `'[1,2,3]' @> '[2,4]'` → false (4 нет).
- `<@` — то же, но наоборот: `A <@ B` ⇔ `B @> A` («A содержится в B»).

> **Производительность.** `@>` и `?`-операторы (ниже) умеют использовать
> **GIN-индекс** по колонке `jsonb` — это делает фильтрацию по документам быстрой
> на больших таблицах. Устройство GIN-индексов — тема lab024; здесь важно знать,
> что `@>`-фильтр индексируем (в отличие от «вытащить `->>` и сравнить»).

## Существование ключей: `?`, `?|`, `?&`

Проверяют **наличие ключа** (в объекте) или строкового элемента (в массиве) —
именно ключа/элемента, а не значения.

| Оператор | Аргумент | true, если… |
|----------|----------|-------------|
| `?`  | text | ключ/элемент **есть** |
| `?\|` | `text[]` | есть **хотя бы один** из перечисленных |
| `?&` | `text[]` | есть **все** перечисленные |

```sql
SELECT
    name,
    attributes ? 'screen'                 AS has_screen,
    attributes ?| array['ram','ssd']      AS has_ram_or_ssd,
    attributes ?& array['brand','screen'] AS has_brand_and_screen
FROM products WHERE id IN (2, 4)
ORDER BY id;
```
```
        name        | has_screen | has_ram_or_ssd | has_brand_and_screen
--------------------+------------+----------------+----------------------
 Ноутбук AirBook 14 | t          | t              | t
 Умная колонка Mini | f          | f              | f
```

Важные детали:
- аргумент `?|` / `?&` — **массив текста** `array['a','b']` (это операнд оператора,
  а не «работа с массивами как темой» — просто список ключей);
- `?` проверяет **наличие ключа**, даже если значение — JSON null: у «Мыши»
  `attributes ? 'warranty'` = true, хотя `warranty` там `null` (см. файл
  [01](01-json-vs-jsonb.md));
- для массива `?` смотрит **строковые** элементы: `'["a","b"]' ? 'a'` → true, а
  `'[1,2]' ? '1'` → false (элементы — числа, не строки).

## JSONPath — язык путей и фильтров (PG12+)

`@>` и `?` отвечают на «содержит/есть ли», но не умеют условий вида «больше N» или
«среди элементов массива найди подходящие». Для этого есть **SQL/JSON path** —
компактный язык запросов внутри документа.

### Синтаксис пути

| Элемент | Значение |
|---------|----------|
| `$` | корень документа |
| `$.key` | поле `key` объекта |
| `$.a.b` | вложенное поле |
| `$[0]`, `$[*]` | элемент массива по индексу / **все** элементы |
| `? (предикат)` | **фильтр**: оставить элементы, где предикат истинен |
| `@` | текущий элемент (внутри предиката) |

Операторы сравнения в предикатах: `==`, `!=`, `<`, `<=`, `>`, `>=`,
логические `&&`, `||`, `!`. **Внимание:** равенство — `==` (двойное), не `=`.

### Функции, применяющие путь

| Функция | Возвращает |
|---------|------------|
| `jsonb_path_query(doc, path)` | **набор** совпадений (set-returning → в `FROM`/`LATERAL`) |
| `jsonb_path_query_array(doc, path)` | все совпадения, **упакованные в JSON-массив** |
| `jsonb_path_query_first(doc, path)` | **первое** совпадение (или NULL) |
| `jsonb_path_exists(doc, path)` | boolean: есть ли хоть одно совпадение |

И два оператора-предиката для `WHERE` (индексируемы GIN):
- `doc @? path` — «существует ли совпадение по пути» (= `jsonb_path_exists`);
- `doc @@ predicate` — «истинен ли путь-предикат целиком».

### Пример: фильтр вложенного массива

Развернуть только позиции дороже 10 000 ₽ по всем заказам — то, что `@>` не умеет
(там сравнение «больше»):

```sql
SELECT o.id, hit ->> 'product' AS product, (hit ->> 'price')::numeric AS price
FROM orders o,
     LATERAL jsonb_path_query(o.items, '$[*] ? (@.price > 10000)') AS hit
ORDER BY price DESC, o.id;
```
```
 id |      product       | price
----+--------------------+-------
  2 | Ноутбук AirBook 14 | 74990
  4 | Планшет Tab S      | 32990
  1 | Смартфон Nova 5    | 24990
  4 | Смартфон Nova 5    | 24990
```

Читается так: `$[*]` — пройди все элементы массива items; `? (@.price > 10000)`
— оставь те, где поле `price` текущего элемента больше 10000. `jsonb_path_query`
— set-returning, поэтому стоит в `FROM` через `LATERAL` (как разворачивание в
[lab006](../../lab006/)).

### Пример: предикат в `WHERE` через `@?`

Найти события-ошибки, которые система будет **повторять** (вложенный
`meta.retry == true`) — без разворачивания:

```sql
SELECT id, payload ->> 'path' AS path
FROM events
WHERE payload @? '$.meta ? (@.retry == true)'
ORDER BY id;
```
```
 id |    path
----+-------------
  3 | /api/orders
```

### `_first` и `_array`

```sql
SELECT
    jsonb_path_query_first('{"a":[5,9,2]}', '$.a[*] ? (@ > 3)') AS first_gt3,  -- 5
    jsonb_path_query_array('{"a":[5,9,2]}', '$.a[*] ? (@ > 3)') AS all_gt3;    -- [5, 9]
```
```
 first_gt3 | all_gt3
-----------+---------
 5         | [5, 9]
```

### Строгий и ленивый режим (`strict` / `lax`)

По умолчанию путь исполняется в режиме **`lax`** (ленивый): он «прощает»
структурные несоответствия — например, применение `[*]` к не-массиву или
обращение к отсутствующему полю не роняет запрос, а просто не даёт совпадений.
Режим **`strict`** требует точного соответствия структуры и в таких случаях даёт
ошибку. Режим пишут в начале пути:

```sql
jsonb_path_query(doc, 'lax   $.items[*]')   -- по умолчанию; терпим к структуре
jsonb_path_query(doc, 'strict $.items[*]')  -- требует, чтобы items был массивом
```

На практике `lax` (по умолчанию) удобен для разнородных документов, где часть
ключей может отсутствовать; `strict` берут, когда важно поймать несоответствие
формата.

## Что выбрать

| Вопрос | Инструмент |
|--------|------------|
| «Документ содержит такую пару/элемент?» | `@>` (индексируемо) |
| «Есть ли ключ(и)?» | `?` / `?\|` / `?&` |
| «Элементы массива по условию (>, <, сложный предикат)» | JSONPath `jsonb_path_query` |
| «Есть ли совпадение по пути?» в `WHERE` | `@?` / `@@` |

## Итог файла

- `@>` — содержание по образцу (объекты: пары; массивы: подмножество); **не**
  делает «больше/меньше» и подстроки; индексируется GIN.
- `?` / `?|` / `?&` — существование ключей/элементов (проверяют **ключ**, не
  значение; JSON null-ключ считается существующим).
- **JSONPath**: `$`, `.key`, `[*]`, `? (@.x > N)` — фильтры по вложенным
  массивам; `jsonb_path_query` (набор, в `FROM`), `@?`/`@@` (в `WHERE`);
  равенство — `==`; режимы `lax` (по умолчанию) / `strict`.
