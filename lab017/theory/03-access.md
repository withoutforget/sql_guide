# 03 — Доступ к элементам: `->`, `->>`, `#>`, `#>>` и приведения

[← построение JSON](02-building.md) · [оглавление](../README.md) · [дальше: содержание и jsonpath →](04-containment-jsonpath.md)

---

Достать значение из документа — самая частая операция. В PostgreSQL для этого
четыре оператора. Ключевое различие проходит по двум осям: **по ключу/индексу vs
по пути** и **вернуть JSON vs вернуть текст**.

| Оператор | Аргумент | Возвращает | Смысл |
|----------|----------|------------|-------|
| `->`  | ключ (text) или индекс (int) | **`jsonb`** | элемент как JSON |
| `->>` | ключ (text) или индекс (int) | **`text`**  | элемент как текст |
| `#>`  | путь `text[]` `'{a,b,0}'`     | **`jsonb`** | элемент по пути как JSON |
| `#>>` | путь `text[]` `'{a,b,0}'`     | **`text`**  | элемент по пути как текст |

Мнемоника: **одна** стрелка `->` возвращает JSON, **двойная** `->>` — text.
Решётка `#` — «идти по пути».

## `->` и `->>` — по ключу и по индексу

### По ключу объекта

```sql
SELECT
    attributes -> 'brand'   AS as_json,   -- jsonb
    attributes ->> 'brand'  AS as_text    -- text
FROM products WHERE id = 1;
```
```
 as_json | as_text
---------+---------
 "Sony"  | Sony
```

Разница видна на строке: `->` вернул `"Sony"` (JSON-строка, **с кавычками**),
`->>` — `Sony` (чистый текст, без кавычек). Проверим типы (`pg_typeof` из
[lab013](../../lab013/)):

```sql
SELECT pg_typeof(attributes -> 'brand'), pg_typeof(attributes ->> 'brand')
FROM products WHERE id = 1;
--  jsonb  |  text
```

### По индексу массива

Тот же оператор, но аргумент — **целое число** (индекс с нуля). Отрицательный
индекс считает с конца:

```sql
SELECT
    '[10,20,30]'::jsonb -> 0    AS first,   -- 10
    '[10,20,30]'::jsonb -> -1   AS last,    -- 30 (с конца)
    '[10,20,30]'::jsonb ->> 1   AS second;  -- '2' → нет, '20' как текст
```
```
 first | last | second
-------+------+--------
 10    | 30   | 20
```

### Цепочки

Операторы можно **сцеплять**, спускаясь вглубь. Промежуточные шаги должны быть
`->` (возвращают JSON, к которому снова применим оператор), последний — `->` или
`->>` по нужному типу:

```sql
SELECT
    attributes -> 'colors'         AS colors,      -- весь массив (jsonb)
    attributes -> 'colors' -> 0    AS first_json,  -- "black"
    attributes -> 'colors' ->> 0   AS first_text   -- black
FROM products WHERE id = 1;
```
```
     colors      | first_json | first_text
-----------------+------------+------------
 ["black", "blue"] | "black"  | black
```

## `#>` и `#>>` — по пути

Когда спускаться надо на несколько уровней, цепочка `->` длинна. `#>` / `#>>`
принимают **путь** — массив текста `'{a,b,c}'`, где элементы — ключи и/или индексы
(индекс тоже как текст):

```sql
SELECT
    prefs #>  '{address}'             AS address_json,  -- поддокумент
    prefs #>> '{address,city}'        AS city,          -- вложенный ключ текстом
    prefs #>> '{notifications,email}' AS email_enabled  -- флаг текстом
FROM users WHERE name = 'Анна';
```
```
                 address_json                  |  city  | email_enabled
-----------------------------------------------+--------+---------------
 {"zip": "101000", "city": "Москва", "street": "Тверская"} | Москва | true
```

`prefs #>> '{address,city}'` — то же самое, что `prefs -> 'address' ->> 'city'`,
но короче. Путь с индексом: `items #>> '{0,product}'` — поле `product` нулевого
элемента массива.

## Подводный камень: отсутствующий ключ/индекс → SQL `NULL`

Если ключа нет, индекс вне диапазона или путь «не находится» — операторы
возвращают **SQL `NULL`**, а не ошибку. Запрос не падает:

```sql
SELECT
    attributes ->> 'no_such_key'  AS missing_key,   -- NULL
    '[1,2]'::jsonb -> 99          AS out_of_range,   -- NULL
    prefs #>> '{address,city}'    AS gleb_city       -- NULL: у Глеба нет address
FROM users WHERE name = 'Глеб';
```

Все три — SQL NULL. Как отличить «ключа нет» от «ключ есть, но значение JSON
null» — см. файл [01](01-json-vs-jsonb.md) (через `?` и `jsonb_typeof`).

## Подводный камень: `->` vs `->>` при сравнении и приведении

Это самое частое место ошибок. Выбор оператора определяет **тип**, а тип —
как пойдёт сравнение и приведение.

### Сравнение

```sql
-- НЕВЕРНО по смыслу: сравниваем JSON-строку "8" с числом → ошибка/не то
WHERE attributes -> 'ram' > 8

-- Тоже ловушка: ->> даёт ТЕКСТ, сравнение пойдёт как со строкой
WHERE attributes ->> 'ram' > '8'      -- '16' < '8'  (посимвольно "1" < "8")!

-- ВЕРНО: ->> в текст, затем приведение к числу — числовое сравнение
WHERE (attributes ->> 'ram')::int > 8
```

Как **текст** `'16' < '8'` (первый символ `'1'` меньше `'8'`) — фильтр «ram > 8»
как строки пропустил бы не то. Поэтому для чисел: **`->>` → `::int`/`::numeric`**.

```sql
SELECT name, (attributes ->> 'ram')::int AS ram_gb
FROM products
WHERE (attributes ->> 'ram')::int > 8
ORDER BY ram_gb DESC;
```
```
        name        | ram_gb
--------------------+--------
 Ноутбук AirBook 14 |     16
 Планшет Tab S      |     12
```

Товары без ключа `ram` дают `NULL` и в фильтр `> 8` не попадают (сравнение с NULL
→ «неизвестно», строка отсеивается — трёхзначная логика из
[lab001](../../lab001/theory/02-where.md)).

### Почему именно `->>`, а не `->`, перед `::numeric`

`->` возвращает `jsonb`, и `jsonb '8'::numeric` работать не обязано так, как ждёте
(это приведение jsonb-скаляра). Надёжный и читаемый путь — **сначала в текст**
(`->>`), **потом в число** (`::numeric`/`::int`). Приведения подробно — в
[lab013](../../lab013/theory/05-casting.md).

## Итог файла

- `->` → JSON, `->>` → text; `#>` / `#>>` — то же, но по **пути** `'{a,b,0}'`.
- Индексы массива с нуля; отрицательные — с конца.
- Отсутствующий ключ/индекс/путь → **SQL NULL** (не ошибка).
- Для чисел/сравнений: **`(doc ->> 'k')::numeric`** — сначала текст, потом
  приведение; сравнивать `->>` как текст («16 < 8») — классическая ошибка.
