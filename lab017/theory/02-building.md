# 02 — Построение JSON: литералы, `build_object`/`build_array`, `to_jsonb`

[← json vs jsonb](01-json-vs-jsonb.md) · [оглавление](../README.md) · [дальше: доступ к элементам →](03-access.md)

---

Есть три способа получить значение `jsonb`: **литерал** (написать текст),
**функции-конструкторы** (собрать из значений SQL) и **`to_jsonb`** (превратить
готовое SQL-значение). Разберём все, с упором на то, почему конструкторы
безопаснее ручной склейки строк.

## Литерал: просто текст с приведением

Самый прямой способ, когда документ известен заранее (см. файл
[01](01-json-vs-jsonb.md)):

```sql
SELECT '{"brand":"Sony","ram":8}'::jsonb;
```

Годится для констант и образцов (например, правый операнд для `@>`). Но собрать
литерал **из значений колонок** склейкой строк — плохая идея:

```sql
-- ТАК НЕ ДЕЛАЙТЕ: склейка строк ломается на кавычках, NULL и числах
SELECT ('{"name":"' || name || '","price":' || price || '}')::jsonb FROM products;
```

Если в `name` встретится кавычка или спецсимвол — получится **битый JSON** и
ошибка; `NULL` в конкатенации обнулит всю строку; числа/даты придётся вручную
форматировать. Для сборки из данных есть типобезопасные функции.

## `jsonb_build_object(k1, v1, k2, v2, …)` — объект из пар

Принимает **чередующиеся** ключ-значение и собирает объект. Каждое значение
берётся **со своим SQL-типом** и корректно кодируется (кавычки, экранирование,
числа, вложенный `jsonb`) — руками ничего форматировать не надо.

```sql
SELECT jsonb_build_object(
    'sku',       'P-' || id,
    'title',     name,
    'price_rub', price,          -- numeric → JSON number
    'attrs',     attributes      -- вложенный jsonb → поддерево
) AS document
FROM products WHERE id = 5;
```
```
 {"sku": "P-5", "attrs": {"pages": 320, "author": "Иванов",
  "genres": ["tech", "education"]}, "title": "Книга «SQL за месяц»", "price_rub": 890.00}
```

Ключевые свойства:
- ключи — строки; значения — **любого** типа и из любых колонок/выражений;
- вложенный `jsonb` (здесь `attributes`) встраивается как **поддокумент**, а не
  как строка с экранированными кавычками;
- нечётное число аргументов → ошибка (ключ без значения);
- если значение — SQL `NULL`, в объекте будет ключ со значением **JSON null**.

Есть и `json`-версия `json_build_object(...)` (результат типа `json`); правила те
же, отличается лишь тип результата.

## `jsonb_build_array(v1, v2, …)` — массив из значений

Собирает JSON-массив из перечисленных значений (разных типов — можно):

```sql
SELECT jsonb_build_array('a', 1, true, NULL, '{"k":1}'::jsonb);
```
```
 ["a", 1, true, null, {"k": 1}]
```

`NULL` стал JSON null; вложенный `jsonb` встроился как элемент. Часто
`build_array` и `build_object` комбинируют, собирая сложную структуру
(см. хардкор-задачу 14).

## `to_jsonb(anyelement)` — превратить SQL-значение в JSON

`to_jsonb` берёт **любое** SQL-значение и переводит его в `jsonb`, сохраняя
структуру:

- скаляр → JSON-скаляр: `to_jsonb(42)` → `42`, `to_jsonb('hi')` → `"hi"`,
  `to_jsonb(true)` → `true`, `to_jsonb(NULL::int)` → `null`;
- **строка таблицы** → объект `{колонка: значение, …}`;
- SQL-массив → JSON-массив.

Самое ценное — сериализовать **всю запись** одним вызовом:

```sql
SELECT to_jsonb(p) AS row_document
FROM products p WHERE id = 4;
```
```
 {"id": 4, "name": "Умная колонка Mini", "price": 3490.00,
  "category": "electronics", "attributes": {"brand": "Yandex", "voice": true}}
```

Ключи объекта — имена колонок; типы сохраняются (`price` — число, `attributes` —
вложенный объект). Если нужно исключить/переименовать поля — соберите объект явно
через `jsonb_build_object`.

> Родственник — `row_to_json(row)` (даёт `json`), исторически более ранний. На
> практике для `jsonb` берут `to_jsonb`.

## Ещё конструкторы (для полноты)

- **`jsonb_object(text[])`** — собрать объект из **плоского массива** строк
  `{k1,v1,k2,v2,...}` (или из двух массивов ключей и значений). Все значения
  будут **строками** — типобезопасности `build_object` тут нет, применяют реже:

  ```sql
  SELECT jsonb_object('{brand, Sony, wireless, yes}');
  -- {"brand": "Sony", "wireless": "yes"}   ← всё строки
  ```

## Обзор: агрегаты, собирающие JSON из строк

Всё выше собирает **один** документ из значений **одной строки**. Обратная
задача — собрать JSON из **многих строк** группы — решается агрегатами:

- **`jsonb_agg(expr ORDER BY …)`** — значения группы в JSON-**массив**;
- **`jsonb_object_agg(key, value ORDER BY …)`** — пары в JSON-**объект**.

Это прямая «json-родня» `array_agg`/`string_agg` из
[lab009](../../lab009/theory/02-collections-and-boolean.md). Подробно с примерами
и правилом обязательного `ORDER BY` внутри (для детерминизма) — в файле
[06](06-unfold-aggregate.md); здесь достаточно знать, что они есть.

## Как выбрать конструктор

| Задача | Инструмент |
|--------|------------|
| Документ известен заранее (константа, образец) | литерал `'...'::jsonb` |
| Собрать объект из колонок/выражений | `jsonb_build_object(...)` |
| Собрать массив из значений | `jsonb_build_array(...)` |
| Сериализовать целую строку/значение | `to_jsonb(...)` |
| Собрать JSON из **многих строк** | `jsonb_agg` / `jsonb_object_agg` (файл 06) |

**Правило:** для сборки из данных — **`build_object`/`build_array`/`to_jsonb`**,
не склейка строк. Они кодируют значения корректно и не порождают битый JSON.

## Итог файла

- Литерал `'...'::jsonb` — для готовых/константных документов; склеивать литерал
  из колонок вручную нельзя (битый JSON, проблемы с кавычками/NULL).
- `jsonb_build_object`/`jsonb_build_array` — типобезопасная сборка из значений;
  вложенный `jsonb` встраивается как поддерево, SQL `NULL` → JSON null.
- `to_jsonb(x)` — превратить любое значение (в т.ч. целую строку) в `jsonb`.
- Сборка из **многих строк** — агрегаты `jsonb_agg`/`jsonb_object_agg` (файл 06).
