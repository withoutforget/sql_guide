# 01 — `json` vs `jsonb`, типы значений, JSON null

## Зачем вообще JSON в реляционной БД

Реляционная модель требует заранее описать **колонки**: у товара есть `price`,
`weight`, `color`… Но реальные данные часто **полуструктурированы**: у смартфона
атрибуты одни (диагональ, ОЗУ), у книги — другие (автор, число страниц), у пледа —
третьи. Заводить сотню колонок «на все случаи» неудобно, а большинство из них
будут пустыми (разреженность). Ещё есть данные, которые *приходят* готовым
документом: ответ внешнего API, строка лога, настройки пользователя, событие.

Для таких случаев в PostgreSQL есть тип, который хранит **целый JSON-документ в
одной ячейке**: гибкий набор ключей, вложенность, массивы. Это не замена
нормализации (см. «когда НЕ надо» в конце), а инструмент для **гибких/разреженных
атрибутов** и **сырых документов**.

JSON (JavaScript Object Notation) — текстовый формат: объект `{"ключ": значение}`,
массив `[1, 2, 3]`, вложенность любой глубины.

## Два типа: `json` и `jsonb`

PostgreSQL предлагает **два** типа для хранения JSON, и разница между ними —
фундамент всей темы.

| | `json` | `jsonb` |
|---|--------|---------|
| Как хранит | **текст «как есть»** | разобранное **бинарное дерево** |
| Пробелы/форматирование | сохраняет | убирает (нормализует) |
| Порядок ключей | сохраняет исходный | **сортирует** (по длине ключа, затем побайтово) |
| Дубли ключей | **сохраняет все** | **дедуплицирует** (остаётся последний) |
| Доступ к полю | каждый раз **разбирает текст** заново | быстрый (уже разобрано) |
| Операторы `@>`, `?`, `@@` | ❌ нет | ✅ есть |
| Индексы GIN | ❌ | ✅ (тема lab024) |
| Запись | чуть быстрее (просто текст) | чуть медленнее (разбор при вставке) |

### Наглядно: дедупликация и порядок

```sql
SELECT
    '{"b": 1,   "a": 2, "a": 3}'::json  AS as_json,
    '{"b": 1,   "a": 2, "a": 3}'::jsonb AS as_jsonb;
```
```
        as_json        |     as_jsonb
-----------------------+------------------
 {"b": 1,   "a": 2, "a": 3} | {"a": 3, "b": 1}
```

`json` вернул строку **буква в букву**: два пробела, порядок `b, a, a`, оба
дубля `"a"`. `jsonb` **разобрал** документ: убрал лишние пробелы, отсортировал
ключи (`a` перед `b`) и оставил от дубля `"a"` только **последнее** значение
(`3`). Это не потеря данных — это определение бинарного формата: объект по
стандарту JSON не должен иметь двух одинаковых ключей, и `jsonb` приводит к
каноничному виду.

### Что выбрать

**Практика: почти всегда `jsonb`.** Он поддерживает операторы содержания,
существования ключей, jsonpath и индексы — то, ради чего JSON и кладут в базу.
`json` берут в редком случае, когда важно сохранить **исходный текст байт в
байт**: порядок ключей, форматирование, дубли (например, документ надо отдать
обратно ровно таким, каким приняли, или это архив «как пришло»). Вся эта лаба —
про `jsonb`; операторы `@>`, `?`, `@@` (файл [04](04-containment-jsonpath.md))
для `json` попросту не определены.

> **Детерминизм — приятный бонус `jsonb`.** Раз ключи всегда отсортированы и
> дедуплицированы, один и тот же документ **всегда** печатается одинаково. Это
> делает вывод запросов воспроизводимым (важно для сравнения результатов).

## Шесть типов значений внутри JSON

Значение внутри JSON — одного из шести типов. Узнать тип **корня** документа
помогает `jsonb_typeof(doc)`:

```sql
SELECT
    jsonb_typeof('{"x":1}')  AS t_object,   -- object
    jsonb_typeof('[1,2,3]')  AS t_array,    -- array
    jsonb_typeof('"текст"')  AS t_string,   -- string
    jsonb_typeof('42')       AS t_number,   -- number
    jsonb_typeof('true')     AS t_boolean,  -- boolean
    jsonb_typeof('null')     AS t_null;     -- null
```
```
 t_object | t_array | t_string | t_number | t_boolean | t_null
----------+---------+----------+----------+-----------+--------
 object   | array   | string   | number   | boolean   | null
```

- **object** — `{...}`, набор пар «ключ-значение»; ключ всегда строка.
- **array** — `[...]`, упорядоченный список (элементы могут быть разных типов).
- **string**, **number**, **boolean** — скалярные значения.
- **null** — особый случай, см. ниже.

Обратите внимание: JSON-число одно (`number`) — нет отдельно int и float. При
извлечении в SQL мы сами приводим его к нужному числовому типу (файл
[03](03-access.md)).

## Подводный камень №1: JSON `null` ≠ SQL `NULL`

Это разные вещи, и путаница здесь — источник багов.

- **SQL `NULL`** — «значения нет» на уровне *реляционной* модели (пустая ячейка,
  отсутствующий ключ).
- **JSON `null`** — это *настоящее значение внутри документа*, тип `null`. Ключ
  **есть**, и его значение — JSON-нуль.

В нашей схеме у товара «Мышь» задан `"warranty": null` (ключ есть, значение —
JSON null), а у остальных товаров ключа `warranty` нет вовсе. Сравните:

```sql
SELECT
    attributes ? 'warranty'                 AS key_exists,   -- есть ли КЛЮЧ
    attributes ->> 'warranty'               AS as_text,      -- значение текстом
    jsonb_typeof(attributes -> 'warranty')  AS value_type,
    attributes -> 'nope'                    AS missing,      -- несуществующий ключ
    jsonb_typeof(attributes -> 'nope')      AS missing_type
FROM products WHERE name = 'Мышь беспроводная';
```
```
 key_exists | as_text | value_type | missing | missing_type
------------+---------+------------+---------+--------------
 t          |         | null       |         |
```

Разбор по колонкам:
- `key_exists = true` — ключ `warranty` **присутствует** (оператор `?`, файл 04);
- `as_text` (пусто) — `->>` от JSON null даёт **SQL NULL**;
- `value_type = 'null'` — но `jsonb_typeof` честно говорит: это **JSON null**,
  то есть заданное значение, а не «ключа нет»;
- `missing` (пусто) и `missing_type` (пусто, тоже SQL NULL) — а вот у
  **отсутствующего** ключа `nope` `-> ` возвращает SQL NULL, и `jsonb_typeof`
  от SQL NULL — тоже SQL NULL.

**Вывод.** «Отсутствие ключа» и «ключ со значением JSON null» через `->>`
неразличимы (оба дают SQL NULL). Различить их можно только оператором `?`
(есть ли ключ) или через `jsonb_typeof(... -> 'ключ') = 'null'`. Убрать все
ключи с JSON null помогает `jsonb_strip_nulls` (файл [05](05-modify.md)).

## Литерал JSON и приведение

JSON-значение записывается **строковым литералом** и приводится к типу через
`::` (приведения — [lab013](../../lab013/)):

```sql
'{"k": "v"}'::jsonb        -- объект
'[1, 2, 3]'::jsonb         -- массив
'"просто строка"'::jsonb   -- ВНИМАНИЕ: JSON-строка (в кавычках внутри!)
'42'::jsonb                -- JSON-число
'null'::jsonb              -- JSON null
```

Тонкость: `'"строка"'::jsonb` — это JSON-*string* (двойные кавычки **внутри**
одинарных). А `'строка'::jsonb` — ошибка: это не валидный JSON. Если сомневаетесь
в валидности текста — при вставке в колонку `jsonb` PostgreSQL сам проверит
синтаксис и не пропустит битый документ.

## Когда JSONB — хорошо, а когда лучше нормализация

**JSONB уместен**, когда:
- набор атрибутов **заранее неизвестен или сильно различается** между записями
  (наши `attributes`: у электроники и книг разные ключи);
- данные приходят **готовым документом** (ответ API, лог, настройки) и хранить
  их удобнее целиком;
- нужна **гибкость** без частых миграций схемы.

**JSONB — плохая идея**, когда данными надо управлять как обычными:
- сваливать в один `jsonb` то, что естественно ложится в колонки, — **антипаттерн**.
  Теряются типы (внутри JSON нет `numeric(10,2)`, `date`), **ссылочная
  целостность** (нет внешних ключей на поле внутри документа), удобная
  индексация и простые джойны;
- данные постоянно фильтруются/соединяются по конкретному полю — обычная колонка
  с индексом почти всегда быстрее и надёжнее.

Практическое правило: **стабильное и важное — в колонки; гибкое и разреженное —
в `jsonb`**. Часто их сочетают: ключевые поля (`price`, `category`) — колонки, а
«хвост» переменных характеристик — `attributes jsonb`. Моделирование и
нормализация — тема lab021.

## Итог файла

- Два типа: `json` (текст как есть) и **`jsonb`** (бинарное дерево). Почти всегда
  берут `jsonb`: он сортирует и дедуплицирует ключи, быстрый доступ, операторы,
  индексы. `json` — только когда нужен исходный текст байт в байт.
- Шесть типов значений; `jsonb_typeof(doc)` показывает тип корня.
- **JSON `null` ≠ SQL `NULL`**: JSON null — это значение (ключ есть); различайте
  через `?` и `jsonb_typeof`.
- JSONB — для гибких/разреженных атрибутов и сырых документов, **не** замена
  нормализации.
