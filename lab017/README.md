# lab017 — JSON / JSONB

Пятая лаба **модуля D «Типы и работа с данными»** — про **полуструктурированные**
данные. Не всё ложится в жёсткие колонки: у смартфона и книги разный набор
характеристик, ответ внешнего API приходит готовым документом, строка лога — это
дерево полей, настройки пользователя — вложенный объект. PostgreSQL умеет хранить
**целый JSON-документ в одной ячейке** и полноценно его запрашивать: извлекать,
фильтровать по образцу, разворачивать в строки и собирать обратно.

Тема очень востребована на практике: гибкие атрибуты товаров, EAV, конфиги,
API-ответы, событийные логи. Здесь мы учимся:

- различать **`json` и `jsonb`** и понимать, почему на практике почти всегда `jsonb`;
- **строить** документы (`jsonb_build_object`, `to_jsonb`);
- **извлекать** значения (`->`, `->>`, `#>`, `#>>`) и приводить их к числам;
- **искать по образцу** (`@>`), проверять ключи (`?`/`?|`/`?&`) и фильтровать
  вложенное языком **JSONPath**;
- **изменять** документы (`||`, `jsonb_set`, `-`, `#-`, `strip_nulls`);
- **разворачивать** JSON в строки (`jsonb_array_elements`, `jsonb_each`,
  `jsonb_to_recordset`) и **собирать** обратно (`jsonb_agg`, `jsonb_object_agg`).

Опираемся на пройденное и **не переобъясняем** его: приведения `::` и `pg_typeof`
— [lab013](../lab013/); функции, возвращающие множества, и `LATERAL` (ядро
разворачивания JSON-массивов) — [lab006](../lab006/); агрегаты-коллекции и правило
«`ORDER BY` внутри агрегата» — [lab009](../lab009/). JSON здесь **впервые**.

Теория — короткие документы в [`theory/`](theory/), читать по порядку. Общая
подборка ссылок — в [`../RESOURCES.md`](../RESOURCES.md).

## Доменная область

Интернет-магазин, где часть данных — **JSON-документы**. Лаба самодостаточна: своя
схема `lab017`, свой [`init.sql`](init.sql). Данные подобраны так, чтобы
**демонстрировать** тему (все крайние случаи — в комментариях `init.sql`).

### `products` — товары с гибкими атрибутами
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` (PK), `name`, `category`, `price` | INT/TEXT/NUMERIC | товар; категория `electronics`/`books` |
| `attributes` | **`jsonb`** | гибкие атрибуты, **разный набор ключей** по категориям |

Что заложено в `attributes`:
- **разреженность**: у электроники `{brand, ram, screen, …}`, у книг
  `{author, pages, genres:[…]}` — наборы ключей не совпадают;
- **`brand = "Sony"`** у товаров 1, 3, 8 — для фильтра по образцу `@>`;
- ключ **`ram`** у 8/16/12 ГБ (товары 1, 2, 7) — для `(->>)::int` и фильтра `> 8`;
- ключ **`screen`** есть у 1, 2, 7, 8 и **отсутствует** у прочих — для `?`/`?&`/`?|`;
- у «Мыши» (id 10) — **`"warranty": null`** (JSON null, ключ **есть**), тогда как
  у остальных ключа `warranty` **нет вовсе** — разница «JSON null vs ключа нет» и
  материал для `jsonb_strip_nulls`;
- у книг — **вложенный массив** `genres:[…]` для jsonpath/разворачивания.

### `orders` — заказы (позиции массивом в одном поле)
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` (PK), `customer` | INT/TEXT | заказ и покупатель |
| `items` | **`jsonb`** | **массив объектов** `[{product, qty, price}, …]` |

Массивы `items` — для разворачивания (`jsonb_array_elements`), сумм `qty*price`,
поиска заказа с товаром (`@>`), jsonpath-фильтра «позиции дороже N» и аналитики
«топ товаров» по всем заказам. Суммы заказов проверены (см. `init.sql`).

### `users` — профили/настройки
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` (PK), `name` | INT/TEXT | пользователь |
| `prefs` | **`jsonb`** | **вложенные** объекты (`address`), массивы (`favorite_categories`), флаги (`notifications`) |

Заложено: путь `{address,city}` для `#>`/`#>>`; у Глеба **нет** `address`
(missing → NULL) и **пустой** массив `favorite_categories`; у Бориса
`"newsletter": null`.

### `events` — события/логи
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` (PK) | INT | идентификатор |
| `payload` | **`jsonb`** | разнородные документы события (разные типы значений, вложенный `meta`) |

Для `jsonb_typeof`, jsonpath-предикатов `@?`/`@@` (события-ошибки с `meta.retry`).

> **Про хардкор.** Задача 14 — «сборка вложенного ответа»: внутренний
> `LATERAL`-подзапрос разворачивает позиции заказа (`jsonb_array_elements`),
> считает `total` и **тут же собирает их обратно** в чистый JSON-массив
> `jsonb_agg(... ORDER BY subtotal DESC)` с вычисленным `subtotal`, а внешний
> `jsonb_build_object` заворачивает всё в финальный документ. Связка «развернуть →
> посчитать → собрать» — ровно то, что делает backend, отдавая заказ клиенту.
> Задача 15 — «SQL поверх JSON-логов»: позиции спрятаны в массивах `items`, у нас
> нет нормализованной таблицы продаж — разворачиваем `items` **всех** заказов в
> единый поток строк и группируем по товару (продажи, выручка, в скольких
> заказах). Обе решаются материалом этой и прошлых лаб; детерминизм — за счёт
> `ORDER BY` внутри `jsonb_agg` и тай-брейкеров по имени товара.
