# lab016 — Полнотекстовый и нечёткий поиск

Четвёртая лаба **модуля D «Типы и работа с данными»** — про «умный» поиск по тексту.
В [lab015](../lab015/) мы искали **точно**: `LIKE`/regex сопоставляют символы буква в
букву. Но настоящий поиск должен находить по **словоформам** («ноутбук» ↔ «ноутбуки»),
понимать **релевантность** (что важнее) и прощать **опечатки** («ноудбук»). Этого
`LIKE` не умеет. Здесь два взаимодополняющих инструмента:

- **Полнотекстовый поиск (FTS)** — приводит слова к леммам, режет стоп-слова, ищет по
  смыслу и **ранжирует**: `to_tsvector`/`tsquery`, `@@`, `ts_rank`, `setweight`,
  `ts_headline`.
- **Нечёткий (fuzzy) поиск** — по **похожести**, а не по точному совпадению: триграммы
  `pg_trgm` (`similarity`, `%`, `<->`), расстояние Левенштейна и фонетика
  (`fuzzystrmatch`), снятие диакритики (`unaccent`).

Строковые функции и регулярки из [lab015](../lab015/) здесь не переобъясняем —
опираемся на них. Противопоставление простое: **regex/`LIKE` — точно и без
словоформ; FTS — по леммам и релевантности; триграммы/Левенштейн — по сходству, с
опечатками.**

Теория — короткие документы в [`theory/`](theory/), читать по порядку. Общая подборка
ссылок — в [`../RESOURCES.md`](../RESOURCES.md).

## Как запустить

```bash
# из корня репозитория один раз:
docker compose up -d

cd lab016
./run.sh                 # накатит схему и прогонит все запросы

# либо зайти руками и экспериментировать:
../psql.sh lab016
```

> **Эта лаба требует расширений** `pg_trgm`, `fuzzystrmatch`, `unaccent`. Их
> устанавливает сам [`init.sql`](init.sql) — **в схему `lab016`**, чтобы функции были
> видны при `search_path = lab016` (раннер ставит именно его, без `public`). Ничего
> ставить руками не нужно. Почему именно так и как это сделано идемпотентно —
> в шапке `init.sql` и в [теории 06](theory/06-comparison.md).

`run.sh` идемпотентен: пересоздаёт схему `lab016` (вместе с расширениями) с нуля при
каждом запуске — запускать можно сколько угодно раз. Вывод **детерминирован**:
стемминг и ранги воспроизводимы, `similarity`/`levenshtein` — чистые функции, а во
всех сортировках по рангу/сходству заданы тай-брейкеры (равные значения там часты).

## Доменная область

Каталог интернет-магазина электроники, отзывы и «грязный» справочник брендов.
Полная схема и подробная шапка о подборе данных — в [`init.sql`](init.sql). **Каждая
таблица и колонка подобрана под тему.**

### `products` — товары
| Колонка | Тип | Смысл / зачем такой вид |
|---------|-----|-------------------------|
| `id` | INT (PK) | идентификатор |
| `name` | TEXT | название (вес **A**) — заголовок для `setweight`, триграммного сходства по имени |
| `description` | TEXT | описание на русском со **словоформами** (ноутбук/ноутбуки/ноутбука, игр/игры/игровой) и **стоп-словами** (для/и/на/с) — ядро FTS |
| `price` | NUMERIC | цена, ₽ |
| `search_tsv` | tsvector | **генерируемый** столбец: `setweight(name,'A') \|\| setweight(description,'D')` + GIN-индекс — «как хранят вектор» |

### `reviews` — отзывы (тексты подлиннее)
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор |
| `product_id` | INT (FK) | ссылка на товар |
| `author` | TEXT | автор |
| `body` | TEXT | текст с повторами слова «ноутбук» — для **ранжирования** (`ts_rank`) и **подсветки** (`ts_headline`) |

### `brands` — один бренд, записанный по-разному
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор |
| `raw_name` | TEXT | написания с регистром, пробелами, опечатками и транслитерацией — для `similarity`/`%`, `<->` и **дедупа** нечётких дубликатов |

### `terms` — словарь категорий
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `term` | TEXT (PK) | канонические слова-термины — «словарь» для исправления опечаток по **Левенштейну** |

### `foreign_names` — латиница с диакритикой
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор |
| `name` | TEXT | `Nescafé`, `Škoda Octavia`, `Café Zoë`… — для **`unaccent`** (поиск без акцентов) |

> **Как данные демонстрируют тему:**
> - **Словоформы.** Запрос «ноутбук» находит товары, где в описании «ноутбуки»/
>   «ноутбука» (задача 2); запрос «игры» ловит формы «игр»/«игры» — а `LIKE` находит
>   лишь буквальное «игры» (задача 13).
> - **Стоп-слова.** «для», «и», «на» выпадают из `tsvector` (задача 1).
> - **Веса.** «клавиатура» есть и в названии (A), и в описании (D) товара 8, но только
>   в описании товара 10 → ранг товара 8 в разы выше (задача 6).
> - **Ранжирование и подсветка.** Отзыв 1 упоминает «ноутбук» трижды → топ по
>   `ts_rank` (задача 5); `ts_headline` подсвечивает все словоформы (задача 7).
> - **Опечатки/варианты.** «Самсунг»/«самсунг»/«Самунг»/«Самсунг Электроникс» —
>   похожи по триграммам (задача 8); «ксиоме» → ближайшие «Ксиоми» (задача 9);
>   «ноудбук» → «ноутбук» по Левенштейну (задача 10). «Сяоми» и латиница «Xiaomi»
>   специально далеки — честная граница метода (задачи 9, 15).
> - **Диакритика.** «cafe» находит «Nescafé», «Café Zoë»; «skoda» → «Škoda» (задача 12).

> **Про хардкор.** Задача 14 — маленький «поисковый движок»: CTE `fts` считает
> релевантность описания (`ts_rank`, ловит словоформы), CTE `trg` — сходство названия
> (`word_similarity`, терпит опечатки), а `LEFT JOIN`+`COALESCE` сводит их в общий
> `score`. Видно, как совпадение в названии поднимает «игровой ноутбук» и «игровую
> мышь» над товарами, где «игры» лишь в описании. Задача 15 схлопывает «грязные»
> написания бренда **без словаря**: self-join `brands×brands` по `similarity ≥ 0.35`
> назначает каждому написанию канон (минимальный id среди похожих), после чего
> `GROUP BY … HAVING count(*) > 1` выдаёт кластеры-дубликаты; «Сяоми» и латиница
> честно остаются в одиночках. Обе — без единой новой конструкции сверх этой и прошлых
> лаб (`similarity`/`ts_rank` + CTE/self-join/`string_agg` из lab005/lab002/lab009).
