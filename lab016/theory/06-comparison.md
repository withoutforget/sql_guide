Мы прошли четыре разных подхода к поиску по тексту. Этот файл — итог: чем они
отличаются, что выбрать под задачу, и как вообще подключить расширения.

## Сравнительная таблица

| Критерий | `LIKE`/`ILIKE`, regex `~` (lab015) | FTS (`@@`, `ts_rank`) | Триграммы (`pg_trgm`) | `levenshtein` |
|----------|-----------------------------------|-----------------------|-----------------------|---------------|
| **Что сопоставляет** | символы/шаблон буквально | **леммы** (морфология) | символьное сходство (тройки) | число правок |
| **Словоформы** («ноутбук»↔«ноутбуки») | нет | **да** (стемминг) | частично | нет |
| **Опечатки** («ноудбук») | нет | нет | **да** | **да** |
| **Стоп-слова** («для»,«и») | не отличает | **отбрасывает** | не отличает | не отличает |
| **Ранжирование релевантности** | нет | **да** (`ts_rank`) | по сходству (`similarity`) | по расстоянию |
| **Зависимость от языка** | нет | да (нужна конфигурация) | нет | нет |
| **Нужно расширение** | нет (встроено) | нет (встроено) | **`pg_trgm`** | **`fuzzystrmatch`** |
| **Ускоряющий индекс** | только префикс `'abc%'` (B-tree) | **GIN** по `tsvector` | GIN/GiST триграммный | нет (перебор) |

## Практический выбор

- **Точная подстрока/префикс, шаблон** («артикул начинается с `AB-`», «есть слово в
  тексте буквально») → `LIKE`/`ILIKE`/regex из [lab015](../../lab015/). Просто и с
  индексом на префиксе.
- **Поиск по смыслу с учётом словоформ и релевантностью** (поле поиска по каталогу,
  статьям, отзывам) → **FTS**. Нормализует язык, ранжирует, подсвечивает.
- **Опечатки, похожие названия, автодополнение, «возможно, вы имели в виду»** →
  **триграммы** (`similarity`/`%`/`<->`). Не зависят от языка, есть индекс.
- **Строгое «отличается не более чем на N символов» по короткому словарю** →
  **`levenshtein`**. Точное целое расстояние.
- **Поиск без учёта акцентов** (латиница с диакритикой) → **`unaccent`** поверх
  любого из способов.
- **Реальный «поиск как в магазине»** обычно **комбинирует**: FTS по описанию для
  релевантности + триграммы по названию на случай опечаток, а результаты сводит общим
  скором (задача 14).

## Мысленная модель одной фразой

> `LIKE`/regex — «совпадает ли **шаблон символов**»; FTS — «есть ли **слово** (в любой
> форме) и насколько документ **релевантен**»; триграммы/Левенштейн — «насколько
> строки **похожи**, даже с опечатками».

## Расширения и `search_path` (как это подключено в лабе)

FTS (`to_tsvector`, `ts_rank`, `ts_headline`, `*_to_tsquery`) **встроен** в
PostgreSQL — ничего ставить не нужно. А `pg_trgm`, `fuzzystrmatch`, `unaccent` — это
**contrib-расширения**, их включают командой `CREATE EXTENSION`. Расширение ставится
**один раз на базу**, и его функции/операторы живут в **конкретной схеме**.

Тонкость нашего тренажёра: раннер выполняет каждый запрос с `SET search_path TO
lab016;` — **без `public`**. Значит, объекты расширений должны быть найдены при
`search_path = lab016`. Поэтому в `init.sql` мы ставим расширения **прямо в схему
`lab016`**:

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm       WITH SCHEMA lab016;
CREATE EXTENSION IF NOT EXISTS fuzzystrmatch WITH SCHEMA lab016;
CREATE EXTENSION IF NOT EXISTS unaccent      WITH SCHEMA lab016;
```

Тогда `similarity()`, `levenshtein()`, `unaccent()` и операторы `%`, `<->`, `<%`
видны запросам без всяких оговорок. А чтобы прогон был **идемпотентным** при любом
исходном состоянии базы (вдруг расширение осталось в `public` от прошлых опытов),
`init.sql` сперва безусловно дропает их глобально (`DROP EXTENSION IF EXISTS ...
CASCADE`), а уже потом создаёт заново в `lab016`. Детали — в комментариях `init.sql`.

> Если вы разворачиваете такое у себя и держите всё в `public` (обычная ситуация вне
> тренажёра) — просто `CREATE EXTENSION pg_trgm;` без `WITH SCHEMA`, и при `public` в
> `search_path` всё найдётся само.

## Про индексы — одним абзацем

Всё, что мы считали «на лету», в проде **индексируют**, иначе поиск сканирует всю
таблицу. Кратко: **GIN по `tsvector`** ускоряет `@@`; **GIN/GiST по `gin_trgm_ops`/
`gist_trgm_ops`** ускоряют `LIKE '%...%'`, `ILIKE`, `~`, `%`, `<->`. В `init.sql`
лабы такие индексы созданы для демонстрации, но **ни одна задача от них не зависит** —
результат тот же и без индекса, просто медленнее. Устройство GIN/GiST, `EXPLAIN`,
частичные и покрывающие индексы — целиком тема **lab024**.
