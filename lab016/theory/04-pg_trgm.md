# 04 — Нечёткий поиск триграммами: `pg_trgm`

FTS ищет по **словам** (леммам). Но что делать с **опечатками** («ноудбук»),
**вариантами написания** («Ксиоми»/«Ксиаоми») и **автодополнением**? Тут слова не
совпадают ни как подстроки, ни как леммы. Ответ — **триграммы**: расширение
`pg_trgm` меряет, насколько две строки похожи **посимвольно**.

## Что такое триграмма

Триграмма — это последовательность из **трёх подряд идущих символов**. Строку
дополняют пробелами по краям (по два спереду, один сзади), после чего нарезают на
тройки:

```sql
SELECT show_trgm('cat');
--          show_trgm
-- -------------------------
--  {"  c"," ca","at ",cat}
--   └ «  c», « ca», «cat», «at »
```

Идея: **чем больше общих триграмм у двух строк, тем они похожее**. У «cat» и «cats»
почти все триграммы общие → строки близки; у «cat» и «dog» общих нет → далеки.

> Для кириллицы `show_trgm` печатает не буквы, а хеш-коды триграмм
> (`{0x8a0178,...}`) — так `pg_trgm` работает с многобайтовыми символами. На саму
> математику сходства это не влияет, просто вывод нечитаемый; смотрите на числа
> `similarity`, а не на коды.

## `similarity` и оператор `%`

```sql
similarity(a, b) → real          -- доля общих триграмм, 0 (ничего общего) … 1 (идентичны)
```

```sql
SELECT similarity('ноутбук', 'ноутбуки');  -- 0.700  (почти совпадают)
SELECT similarity('ноутбук', 'планшет');   -- 0.000  (общих триграмм нет)
SELECT similarity('Самсунг', 'Самунг');    -- 0.500  (опечатка — пропущена буква)
```

`similarity` **нечувствительна к регистру** и игнорирует различия в неалфавитных
символах: триграммы строятся по приведённому тексту, поэтому `similarity('Самсунг',
'самсунг') = 1.0`, и лишние пробелы по краям тоже не мешают.

Оператор **`%`** — это «похожи ли строки сильнее порога»:

```sql
a % b   ⟺   similarity(a, b) ≥ pg_trgm.similarity_threshold
```

Порог по умолчанию **0.3**, меняется на сессию:

```sql
SET pg_trgm.similarity_threshold = 0.4;      -- строже
SHOW pg_trgm.similarity_threshold;           -- 0.4
```

Типовой нечёткий поиск «найди похожие названия, несмотря на опечатку»:

```sql
SELECT raw_name, round(similarity(raw_name, 'самсунг')::numeric, 3) AS sim
FROM brands
WHERE raw_name % 'самсунг'          -- отфильтровать по порогу
ORDER BY sim DESC, id;              -- и отсортировать по сходству (тай-брейкер по id!)
```

## Оператор расстояния `<->` — сортировка «ближайших»

```sql
a <-> b   =   1 - similarity(a, b)      -- 0 — идентичны, 1 — ничего общего
```

Это «расстояние»: чем меньше, тем ближе. Незаменимо для **автодополнения** и
**«возможно, вы имели в виду»** — сортируем кандидатов по возрастанию расстояния:

```sql
SELECT raw_name, round((raw_name <-> 'ксиоме')::numeric, 3) AS distance
FROM brands
ORDER BY raw_name <-> 'ксиоме', id      -- ближайшие сверху; тай-брейкер обязателен
LIMIT 5;
--  raw_name | distance
-- ----------+----------
--  Ксиоми   |    0.444   ← предлагаем это
--  ксиоми   |    0.444
--  Ксиаоми  |    0.750
```

`<->` и `ORDER BY ... LIMIT N` — стандартный приём поиска K ближайших; при наличии
GiST-триграммного индекса он выполняется особенно быстро.

## `word_similarity` — слово внутри длинного текста

`similarity(a, b)` сравнивает строки **целиком**. Если `b` — длинный документ, а
`a` — одно слово, сходство «размывается»: триграммы слова тонут среди триграмм всего
текста, и `similarity` падает **ниже порога** (часто почти до нуля), даже когда слово
в тексте есть — то есть `a % b` не сработает, хотя слово там присутствует.

`word_similarity(a, b)` решает это: он находит в `b` **наиболее похожий непрерывный
участок** и меряет сходство только с ним:

```sql
SELECT similarity('ноутбук', 'Мощный игровой ноутбук для игр');       -- 0.286 (< порога 0.3 — «мало»)
SELECT word_similarity('ноутбук', 'Мощный игровой ноутбук для игр');  -- 1.000 (слово найдено)
```

Операторы-пороги (порог свой — `pg_trgm.word_similarity_threshold`, по умолчанию
**0.6**):

| Оператор | Смысл |
|:--------:|-------|
| `a <% b` | `word_similarity(a, b) ≥ порога` — слово `a` есть где-то в тексте `b` |
| `a %> b` | коммутатор: то же, что `b <% a` |

```sql
-- отзывы, где нечётко упоминается «ноутбук» (в т.ч. с опечаткой):
SELECT id FROM reviews WHERE 'ноутбук' <% body;
```

Есть и **`strict_word_similarity`** — как `word_similarity`, но участок обязан
совпадать с **границами слов** (не «кусок слова»), поэтому оценки строже. Полезно,
когда важно совпадение именно целого слова.

## Границы метода (обязательно понимать)

- **Разные алфавиты не пересекаются.** У латиницы и кириллицы общих триграмм нет:
  `similarity('Xiaomi', 'Ксиоми') = 0`. Триграммы ловят опечатки *внутри одного
  письма*, но не транслитерацию — «Xiaomi» и «Ксиоми» так не свяжешь (для этого
  нужен словарь соответствий).
- **Короткие строки капризны.** У очень коротких слов мало триграмм, и сходство
  скачет: `similarity('Ксиоми','Сяоми') = 0.18` — ниже порога, хотя людям это «тот
  же бренд». Триграммы — эвристика, а не понимание смысла.
- **Порог — компромисс.** Низкий порог тянет мусор (ложные срабатывания), высокий —
  теряет реальные совпадения. 0.3 — разумная отправная точка, но под свои данные его
  подбирают.

## FTS или триграммы?

| | FTS (`@@`) | Триграммы (`pg_trgm`) |
|--|-----------|------------------------|
| Ищет по | **леммам** (морфология языка) | **символьному сходству** |
| Опечатки | не терпит («ноудбук» не найдёт «ноутбук») | **терпит** |
| Словоформы | да (через стемминг) | частично (общие триграммы у форм) |
| Язык | зависит от конфигурации | не зависит (любой алфавит) |
| Ранжирование | `ts_rank` (релевантность) | `similarity`/`<->` (близость) |

Часто их **комбинируют**: FTS даёт релевантность по смыслу, триграммы страхуют от
опечаток (см. гибридный поиск в задаче 14).

## Про индексы (обзорно)

`pg_trgm` умеет ускорять то, что обычный индекс не тянет: `LIKE '%подстрока%'`,
`ILIKE`, regex-поиск `~` и сами `similarity`/`%`/`<->`. Для этого строят
триграммный индекс — **GIN** (`gin_trgm_ops`, быстрее для `%`/`LIKE`) или **GiST**
(`gist_trgm_ops`, поддерживает `<->` для K ближайших):

```sql
CREATE INDEX products_name_trgm ON products USING gin (name gin_trgm_ops);
```

В `init.sql` лабы такой индекс есть, но задачи от него не зависят — он лишь ускоряет.
Подробно про GIN/GiST и планы — lab024.
