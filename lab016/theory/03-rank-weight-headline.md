`@@` отвечает «да/нет». Но настоящий поиск должен ещё и **отсортировать** находки по
релевантности, учесть, что совпадение в **заголовке** важнее, чем в теле, и показать
пользователю **подсвеченный фрагмент**. Этим и займёмся.

## `ts_rank` — насколько документ релевантен

```sql
ts_rank([ weights, ] vector, query [, normalization ]) → real
```

Возвращает число (обычно небольшое, `0..~1`): чем **чаще** леммы запроса в документе
и чем «весомее» их позиции, тем выше. Абсолютное значение смысла не имеет — важен
**относительный порядок**, поэтому `ts_rank` живёт в `ORDER BY ... DESC`.

```sql
SELECT r.id, r.author,
       round(ts_rank(to_tsvector('russian', r.body),
                     plainto_tsquery('russian', 'ноутбук'))::numeric, 4) AS rank
FROM reviews r
WHERE to_tsvector('russian', r.body) @@ plainto_tsquery('russian', 'ноутбук')
ORDER BY rank DESC, r.id;
--  id | author |  rank
-- ----+--------+--------
--   1 | Анна   | 0.0827   ← «ноутбук» упомянут трижды
--   5 | Дарья  | 0.0760
--   8 | Зоя    | 0.0760
--   2 | Борис  | 0.0608   ← один раз
```

> **Тай-брейкер обязателен.** У отзывов 5 и 8 ранг одинаков (`0.0760`). Без второго
> ключа сортировки (`, r.id`) их взаимный порядок не определён и может «плавать».
> В FTS равные ранги — обычное дело, всегда добавляйте тай-брейкер.

> **`round(...)` — для читаемости.** `ts_rank` возвращает `real` (float). Само по
> себе значение детерминировано, но округление до `numeric` даёт стабильную ширину
> столбца. Приведение `::numeric` нужно, потому что у `round(double, n)` нет формы с
> двумя аргументами (см. lab001, файл про агрегаты).

### `ts_rank_cd` — плотность покрытия (cover density)

Тот же интерфейс, но дополнительно учитывает, насколько **близко** найденные слова
стоят друг к другу: документ, где слова запроса рядом, релевантнее, чем где они
разбросаны. Для однословного запроса разница невелика; она проявляется на запросах
из нескольких слов. На практике пробуют оба и берут тот, что лучше сортирует.

### Флаги нормализации

Длинный документ почти всегда наберёт больше совпадений, чем короткий, — и «утопит»
короткие релевантные тексты. Необязательный аргумент `normalization` (битовая
сумма) велит делить ранг на «размер» документа. Ходовые значения:

| Флаг | Что делает |
|:----:|-----------|
| `0` | без нормализации (по умолчанию) |
| `1` | делить на `1 + log(длина документа)` |
| `2` | делить на длину документа |
| `32` | делить на `ранг + 1` → загнать в диапазон `0..1` |

```sql
ts_rank(vector, query, 1)      -- сгладить влияние длины
ts_rank(vector, query, 2|32)   -- флаги комбинируются битовым ИЛИ
```

Для наших коротких текстов нормализацию не применяем, но знать про неё нужно: без
неё ранжирование по длинным полям смещено в пользу длинных документов.

## `setweight` — какое поле важнее

Совпадение в **названии** товара должно значить больше, чем в описании. FTS решает
это **весовыми метками** `A`, `B`, `C`, `D` (`A` — самый важный, `D` — наименее).
`setweight(vector, 'A')` помечает все леммы вектора буквой, а `||` склеивает векторы:

```sql
SELECT setweight(to_tsvector('russian', 'Игровой ноутбук'), 'A')
    || setweight(to_tsvector('russian', 'мощная видеокарта'), 'D');
--                     ?column?
-- -----------------------------------------------
--  'видеокарт':4 'игров':1A 'мощн':3 'ноутбук':2A
```

Буква веса стоит рядом с позицией (`ноутбук:2A`). Тонкость текстового представления:
буквами помечаются только веса `A`/`B`/`C`, а **вес `D` не печатается вовсе** — он
«по умолчанию», и его показывает именно *отсутствие* буквы (`видеокарт:4`, `мощн:3`
— это вес `D`). Вес при этом действительно хранится, просто в выводе он невидим.
Дальше при ранжировании передаём **массив весов** `{D, C, B, A}` — по одному
множителю на каждую метку:

```sql
-- стандартный массив {0.1, 0.2, 0.4, 1.0}: A даёт ×1.0, D — всего ×0.1
SELECT ts_rank('{0.1,0.2,0.4,1.0}', search_tsv,
               plainto_tsquery('russian', 'клавиатура'))
FROM products WHERE ...;
```

Практический эффект (запрос «клавиатура»):

```sql
--  id | name                                | weighted_rank
-- ----+-------------------------------------+---------------
--   8 | Механическая клавиатура TypeMaster  | 0.62313   ← слово и в названии (A), и в описании (D)
--  10 | Планшет TabMax 11                   | 0.06079   ← «клавиатуру» только в описании (D)
```

Совпадение в заголовке подняло товар 8 на порядок выше товара 10 — ровно то, что
нужно магазину. Массив весов — часть API `ts_rank` (это не работа с массивами как
типом, до которой доберёмся в lab018).

## `ts_headline` — подсветка фрагмента

```sql
ts_headline([конфиг,] исходный_текст, query [, опции]) → text
```

Возвращает **кусок исходного текста** (не леммы!) с обёрнутыми совпадениями —
готовый сниппет для выдачи. Важно: первым аргументом идёт **оригинальный текст**, а
не `tsvector` (функции нужно вернуть настоящие слова). Опции — строка `ключ=значение`
через запятую:

| Опция | Смысл | По умолчанию |
|-------|-------|--------------|
| `StartSel` / `StopSel` | чем оборачивать совпадение | `<b>` / `</b>` |
| `MaxWords` / `MinWords` | размер фрагмента (слов) | `35` / `15` |
| `ShortWord` | не начинать/кончать фрагмент словами короче N | `3` |
| `MaxFragments` | сколько отдельных кусков склеить (`0` — один непрерывный) | `0` |
| `FragmentDelimiter` | разделитель между кусками | ` ... ` |

```sql
SELECT ts_headline('russian', body,
         plainto_tsquery('russian', 'ноутбук игры'),
         'StartSel=«, StopSel=», MaxWords=12, MinWords=4')
FROM reviews WHERE id = 1;
--                ts_headline
-- -------------------------------------
--  «Ноутбук» совсем не греется, «игры»
```

Фрагмент **центрируется вокруг совпадений**, а не начинается с начала текста: при
`MaxWords=12` движок вырезал наиболее «плотный» кусок с найденными словами (начало
«Отличный игровой…» в окно не влезло). Хвостового ` ... ` тут нет, потому что
`MaxFragments` по умолчанию `0` — возвращается один непрерывный фрагмент, а не
склейка кусков. Подсвечены **все словоформы** запроса: «Ноутбук», «игры» — потому что
совпадение идёт по леммам. `ts_headline` перечитывает исходный текст, поэтому это
относительно дорогая операция — её применяют только к финальной странице выдачи
(к десятку строк после ранжирования), а не ко всей таблице.

## Как это хранят и ускоряют (обзорно)

Считать `to_tsvector(...)` на каждой строке при каждом запросе расточительно. На
практике вектор **материализуют** — держат в столбце и обновляют автоматически:

```sql
CREATE TABLE products (
    ...,
    search_tsv tsvector GENERATED ALWAYS AS (
        setweight(to_tsvector('russian', name),        'A') ||
        setweight(to_tsvector('russian', description),  'D')
    ) STORED
);
```

`GENERATED ALWAYS AS (...) STORED` — **генерируемый столбец**: его значение считается
из других колонок и физически хранится, пересчитываясь при изменении строки. Так
вектор всегда синхронен с данными, а запросы обращаются к готовому `search_tsv`
(именно его мы используем в задачах на веса). По этому столбцу строят **GIN-индекс**:

```sql
CREATE INDEX products_search_gin ON products USING gin (search_tsv);
```

GIN (Generalized Inverted Index) — «обратный индекс»: для каждой леммы хранит список
документов, где она есть, поэтому `@@` не сканирует всю таблицу. В `init.sql` лабы
такой столбец и индекс созданы — но **задачи от индекса не зависят** (он лишь
ускоряет то же самое). Устройство индексов, `EXPLAIN` и когда GIN vs GiST — тема
lab024, сюда не углубляемся.
