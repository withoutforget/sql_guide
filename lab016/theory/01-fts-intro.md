## Зачем нужен полнотекстовый поиск

В [lab015](../../lab015/) мы искали по тексту через `LIKE` и регулярные выражения.
Это **точное** сопоставление символов: `LIKE '%ноутбук%'` находит строки, где
буквально есть подстрока «ноутбук». Для настоящего поиска по тексту этого мало:

| Чего не умеют `LIKE`/regex | Пример |
|----------------------------|--------|
| **Словоформы** (морфология) | запрос «ноутбук» не найдёт «ноутбуки», «мышь» не найдёт «мыши» (подстроки не совпадают) |
| **Стоп-слова** | «для», «и», «на» ничего не значат для поиска, но `LIKE` их не отличает |
| **Ранжирование** | `LIKE` отвечает «да/нет», а не «насколько документ релевантен» |
| **Смысловой разбор языка** | не знает, что «работы» и «работа» — одно слово |

**Полнотекстовый поиск (FTS, full-text search)** решает это так: и документ, и
запрос **нормализуются** — каждое слово приводится к **лемме** (основе, словарной
форме), стоп-слова выбрасываются. Дальше сравниваются уже леммы, а не буквы.
Поэтому «ноутбуки», «ноутбука», «ноутбук» — для FTS одно и то же слово.

Два новых типа данных:

- **`tsvector`** — «переваренный» **документ**: отсортированный список лемм с
  позициями.
- **`tsquery`** — «переваренный» **поисковый запрос**: леммы, связанные
  операторами (И/ИЛИ/НЕ/фраза). Про него — [файл 02](02-tsquery.md).

Матч проверяет оператор **`@@`**: `tsvector @@ tsquery` → `true`, если документ
удовлетворяет запросу.

## `to_tsvector` — превращаем текст в леммы

```sql
to_tsvector([ конфигурация, ] текст) → tsvector
```

Функция разбивает текст на слова (токены), приводит каждое к лемме по правилам
**конфигурации** языка и выбрасывает стоп-слова. Результат — леммы с **позициями**:

```sql
SELECT to_tsvector('russian', 'Быстрые игровые ноутбуки для работы и учёбы');
--            to_tsvector
-- ----------------------------------------------------
--  'быстр':1 'игров':2 'ноутбук':3 'работ':5 'учеб':7
```

Разберём результат:

- `быстрые → быстр`, `игровые → игров`, `ноутбуки → ноутбук`, `работы → работ`,
  `учёбы → учеб` — это **стемминг** (усечение до основы). Именно он даёт поиск по
  словоформам: любая форма слова даёт одну и ту же лемму.
- **`для` (позиция 4) и `и` (позиция 6) пропали** — это **стоп-слова** русского
  языка: слишком частые, для поиска бесполезны. Позиции 4 и 6 при этом
  «зарезервированы» (нумерация исходных слов сохраняется) — видно, что леммы идут
  `1,2,3,5,7`.
- Леммы в `tsvector` хранятся **отсортированными по алфавиту**, а не по порядку в
  тексте; порядок восстанавливается по позициям.

Если одна лемма встречается несколько раз, позиции копятся в списке:

```sql
SELECT to_tsvector('russian', 'ноутбук и ещё ноутбук');
--       to_tsvector
-- -----------------------
--  'ещ':3 'ноутбук':1,4      -- одна лемма (ноутбук) — две позиции; «ещё» усечено до «ещ»
```

## Конфигурации: `russian`, `english`, `simple`

Конфигурация задаёт **набор словарей**: как токенизировать, что считать
стоп-словом, как усекать до основы. Три ходовых:

| Конфигурация | Стемминг | Стоп-слова | Когда |
|--------------|:--------:|:----------:|-------|
| `russian`    | да, по русским правилам | русские | русский текст |
| `english`    | да, по английским | английские | английский текст |
| `simple`     | **нет** | нет | только регистр вниз + разбиение на токены |

`simple` не знает ни языка, ни словоформ — просто нарезает и приводит к нижнему
регистру, **ничего не отбрасывая**:

```sql
SELECT to_tsvector('simple', 'Быстрые игровые ноутбуки для работы и учёбы');
--                        to_tsvector
-- -------------------------------------------------------------
--  'быстрые':1 'для':4 'и':6 'игровые':2 'ноутбуки':3 'работы':5 'учёбы':7
```

Сравните с `russian` выше: `simple` оставил `для`, `и` и не усёк слова. `simple`
полезен для точных названий, артикулов, кодов — там, где стемминг только навредит.

> **Как выбирается конфигурация «по умолчанию».** Форма `to_tsvector(текст)` без
> первого аргумента берёт конфигурацию из настройки `default_text_search_config`
> (обычно `pg_catalog.english`). Для русского текста **всегда указывайте `'russian'`
> явно** — иначе слова не будут лемматизироваться как надо. Во всех задачах лабы мы
> пишем конфигурацию явно.

### Подводный камень: смешанный и «не тот» язык

Стеммер применяет правила своего языка ко всему подряд. Английские слова в русском
тексте `russian` тоже попытается усечь (обычно безвредно), а вот русский текст под
`english`-конфигурацией не лемматизируется — «ноутбуки» и «ноутбук» останутся
разными токенами, и поиск по словоформам сломается. Для двуязычного контента берут
либо `simple` (без стемминга), либо отдельные векторы под каждый язык.

> **Про букву «ё».** Русский стеммер нормализует «ё»: `учёбы → учеб`. Так что по
> запросу «учеба» найдётся текст с «учёба». А для «ё=е» в *точных* (не-FTS)
> сравнениях подойдёт штатный `unaccent` — в PostgreSQL 16 он приводит «ё» к «е»
> без всяких доработок словаря ([файл 05](05-fuzzy-unaccent.md)).

## Оператор `@@` — проверка совпадения

`@@` возвращает `boolean`: удовлетворяет ли документ (`tsvector`) запросу (`tsquery`):

```sql
SELECT to_tsvector('russian', 'Купили новые ноутбуки')
       @@ plainto_tsquery('russian', 'ноутбук');   -- true
```

Слева «ноутбуки» стало леммой `ноутбук`, справа запрос «ноутбук» — та же лемма,
значит матч. Это и есть **поиск по словоформе**, недоступный `LIKE`.

Удобно, что `@@` **сам** приведёт текст к вектору, если слева дать обычную строку —
но тогда используется конфигурация по умолчанию:

```sql
-- эквивалентно, но конфигурация НЕ 'russian' → для русского так не надо:
'текст' @@ 'запрос'::tsquery
```

Поэтому в задачах мы всегда пишем `to_tsvector('russian', ...) @@ ...('russian', ...)`
явно с обеих сторон.

### Типовой запрос поиска

```sql
SELECT id, name
FROM products
WHERE to_tsvector('russian', description) @@ plainto_tsquery('russian', 'ноутбук')
ORDER BY id;
-- вернёт и товары, где в описании «ноутбуки»/«ноутбука», а не только «ноутбук».
```

Считать `to_tsvector(...)` на лету для каждой строки — нормально для учебных объёмов.
На больших таблицах вектор **хранят** в отдельном столбце и строят по нему индекс —
про это [файл 03](03-rank-weight-headline.md) и подробно lab024.

## Поведение с `NULL` и пустым текстом

- `to_tsvector('russian', NULL)` → `NULL` (как почти любая функция от `NULL`).
- `to_tsvector('russian', '')` или текст из одних стоп-слов → **пустой** `tsvector`
  (`''`), который `@@` никогда не удовлетворяет. То есть строка с пустым описанием
  просто не попадёт в выдачу — не ошибка.

## Мысленная модель

```
текст  ──to_tsvector('russian',…)──►  tsvector (леммы+позиции)  ┐
                                                                 ├─@@─► true/false
запрос ──*_to_tsquery('russian',…)──►  tsquery (леммы+операторы)┘
```

FTS — это «сравниваем не буквы, а нормализованные слова». Как строить правую часть
(`tsquery`) со всеми операторами — в следующем файле.
