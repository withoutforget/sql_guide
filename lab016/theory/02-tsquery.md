# 02 — Поисковый запрос: `tsquery`, операторы и виды `*_to_tsquery`

[← зачем FTS и tsvector](01-fts-intro.md) · [оглавление](../README.md) · [дальше: ранжирование, веса, подсветка →](03-rank-weight-headline.md)

`tsquery` — это поисковый запрос из лемм, связанных **операторами**. Строить его
руками (`to_tsquery`) можно, но чаще берут одну из функций-обёрток, которые сами
разбирают человеческий ввод. Разберём и операторы, и все четыре обёртки.

## Операторы `tsquery`

| Оператор | Смысл | Пример |
|:--------:|-------|--------|
| `&` | И — обе леммы обязательны | `'ноутбук' & 'игровой'` |
| `\|` | ИЛИ — хотя бы одна | `'ноутбук' \| 'планшет'` |
| `!` | НЕ — леммы быть не должно | `'ноутбук' & !'офисный'` |
| `<->` | ФРАЗА — вторая лемма **сразу за** первой | `'игровой' <-> 'ноутбук'` |
| `<N>` | вторая лемма ровно через `N` позиций | `'быстрый' <2> 'ноутбук'` |

Приоритет (от сильного к слабому): `!`  →  `<->`/`<N>`  →  `&`  →  `|`. Скобками
`()` группируют явно:

```sql
SELECT to_tsquery('russian', 'ноутбук & (игровой | мощный) & !офисный');
--                  результат
-- ----------------------------------------------
--  'ноутбук' & ( 'игров' | 'мощн' ) & !'офисн'
```

Обратите внимание: `to_tsquery` тоже **лемматизирует** слова (`игровой→игров`),
поэтому запрос и документ говорят на одном «языке лемм».

### `to_tsquery` — ручной, самый мощный и самый капризный

```sql
to_tsquery([конфиг,] 'строка с операторами')
```

Требует **корректного синтаксиса с операторами**: слова должны быть разделены
`& | ! <->`. Просто пробел — синтаксическая ошибка:

```sql
to_tsquery('russian', 'ноутбук игровой')   -- ОШИБКА: syntax error in tsquery
to_tsquery('russian', 'ноутбук & игровой') -- OK
```

Поэтому «сырой» пользовательский ввод в `to_tsquery` подавать нельзя — он уронит
запрос. Для ввода от людей есть три безопасные обёртки ниже.

### `plainto_tsquery` — простой текст через И

Берёт обычную фразу, лемматизирует слова и **соединяет их `&`** (все обязательны):

```sql
SELECT plainto_tsquery('russian', 'быстрый игровой ноутбук');
--            result
-- --------------------------------
--  'быстр' & 'игров' & 'ноутбук'
```

Операторы в тексте игнорируются (трактуются как слова). Идеален, когда нужно
«найти документы, где есть все эти слова».

### `phraseto_tsquery` — фраза с сохранением порядка

Как `plainto`, но соединяет леммы оператором `<->` — слова должны идти **подряд и
в том же порядке**:

```sql
SELECT phraseto_tsquery('russian', 'игровой ноутбук');
--         result
-- -----------------------
--  'игров' <-> 'ноутбук'
```

`'игров' <-> 'ноутбук'` найдёт «игровой ноутбук», но **не** «ноутбук игровой» и не
«ноутбук для игр». Стоп-слова внутри фразы превращаются в сдвиг расстояния
(`<N>`), так что «поиск по фразе» остаётся точным даже с выброшенными «для»/«и».

### `websearch_to_tsquery` — «как в поисковике»

Понимает привычный синтаксис поисковых систем и **никогда не падает** на кривом
вводе (то, что нужно для строки поиска на сайте):

| В строке | Значение |
|----------|----------|
| пробел между словами | И (`&`) |
| слово `or` / `OR` | ИЛИ (`\|`) |
| `-слово` | НЕ (`!`) |
| `"фраза в кавычках"` | фраза (`<->`) |

```sql
SELECT websearch_to_tsquery('russian', 'смартфон камера -планшет');
--                 result
-- -----------------------------------
--  'смартфон' & 'камер' & !'планшет'

SELECT websearch_to_tsquery('russian', '"игровой ноутбук" or планшет');
--                 result
-- --------------------------------------
--  'игров' <-> 'ноутбук' | 'планшет'
```

## Какую обёртку выбирать

| Ситуация | Функция |
|----------|---------|
| строка поиска для пользователей на сайте | **`websearch_to_tsquery`** |
| «найти всё, где есть эти слова» из кода | `plainto_tsquery` |
| точная фраза / порядок слов важен | `phraseto_tsquery` |
| нужен полный контроль (И/ИЛИ/НЕ/скобки) из доверенного источника | `to_tsquery` |

## Применение: `@@`

Любой `tsquery` проверяется против документа тем же оператором `@@`:

```sql
-- товары, где в названии+описании есть И «ноутбук», И «игровой»:
SELECT id, name
FROM products
WHERE to_tsvector('russian', name || ' ' || description)
      @@ to_tsquery('russian', 'ноутбук & игровой');
```

### Фразовый поиск против `plainto`

```sql
-- plainto: слова в любом порядке и месте (просто оба присутствуют)
... @@ plainto_tsquery('russian', 'игровой ноутбук')   -- 'игров' & 'ноутбук'
-- phraseto: строго «игровой<пробел>ноутбук» подряд
... @@ phraseto_tsquery('russian', 'игровой ноутбук')  -- 'игров' <-> 'ноутбук'
```

Текст «мощный ноутбук для игр» удовлетворит `plainto` (`игр` и `ноутбук` там есть),
но **не** `phraseto` (леммы не стоят подряд в нужном порядке).

## Подводные камни

- **`to_tsquery` требует операторов.** Пробел = ошибка. Пользовательский ввод — только
  через `plainto`/`phraseto`/`websearch`.
- **Конфигурация запроса и документа должна совпадать.** Строите вектор на `russian`
  — и запрос стройте на `russian`, иначе леммы не сойдутся.
- **`!` (НЕ) сам по себе не находит строки.** `!'офисн'` истинно для любого документа
  без этой леммы, включая пустые; `!` осмыслен в связке с `&`.
- **Стоп-слово как единственное слово запроса** даёт пустой `tsquery`, который не
  найдёт ничего: `plainto_tsquery('russian','для')` → пусто.
