# 05 — Расстояние Левенштейна, фонетика и `unaccent`

[← триграммы pg_trgm](04-pg_trgm.md) · [оглавление](../README.md) · [дальше: когда что выбрать →](06-comparison.md)

---

Триграммы дают «мягкое» сходство `0..1`. Иногда же нужно **точное целое** число
правок между строками (для строгого исправления опечаток по словарю) или сравнение
слов **по звучанию**. Это расширение `fuzzystrmatch`. Плюс разберём `unaccent` —
снятие диакритики.

## `levenshtein` — редакционное расстояние

```sql
levenshtein(a, b) → int
levenshtein(a, b, ins_cost, del_cost, sub_cost) → int
```

Возвращает **минимальное число односимвольных правок** (вставка, удаление, замена),
которыми одна строка превращается в другую. Ноль — строки равны.

```sql
SELECT levenshtein('ноутбук', 'ноудбук');   -- 1  (одна замена: т→д)
SELECT levenshtein('кот', 'скот');          -- 1  (одна вставка: с)
SELECT levenshtein('sitting', 'kitten');    -- 3
```

Второй вариант задаёт **стоимость** операций (по умолчанию все по 1). Например, если
вставки дешевле замен:

```sql
SELECT levenshtein('ноутбук', 'ноутбук', 1, 1, 2);  -- 0 (строки равны — правок нет)
```

Практика — **исправление опечатки по справочнику**: для введённого слова берём из
словаря ближайший термин с малым расстоянием:

```sql
SELECT d.term, levenshtein('ноудбук', d.term) AS edits
FROM terms d
WHERE levenshtein('ноудбук', d.term) <= 2      -- «достаточно близко»
ORDER BY edits, d.term;                         -- ближайшее сверху; тай-брейкер!
--   term    | edits
-- ----------+-------
--  ноутбук  |     1
```

### `levenshtein_less_equal` — быстрее, когда важна только близость

```sql
levenshtein_less_equal(a, b, max) → int
```

Считает расстояние, но **прекращает работу**, как только стало ясно, что оно больше
`max` (тогда возвращает значение `> max`). На длинных строках это ощутимо быстрее
полного `levenshtein`, когда нас интересует лишь «отличаются не более чем на `max`»:

```sql
SELECT levenshtein_less_equal('aaaaaa', 'bbbbbb', 2);  -- 3  (точное считать не стали: и так > 2)
```

### Левенштейн против триграмм

| | `levenshtein` | триграммы (`similarity`) |
|--|--------------|--------------------------|
| Тип оценки | целое число правок | доля `0..1` |
| Хорош для | коротких строк, строгого «≤ N опечаток» | подстрок, длинных названий, автодополнения |
| Индекс | нет (считается перебором) | есть (GIN/GiST) |
| Порядок символов | учитывает точно | учитывает грубо (через тройки) |

Для коротких словарных слов и «не больше двух опечаток» удобнее `levenshtein`; для
поиска похожего в большой таблице (где нужен индекс) — триграммы.

## Фонетика: `soundex`, `metaphone`, `dmetaphone`, `difference`

Эти функции кодируют слово по **звучанию**, чтобы находить похоже звучащие имена
(классика — англоязычные фамилии `Robert`/`Rupert`):

```sql
SELECT soundex('Robert'), soundex('Rupert');      -- R163 | R163  (один код → «звучат одинаково»)
SELECT difference('Robert', 'Rupert');            -- 4  (0..4: насколько похожи коды soundex)
SELECT metaphone('Thompson', 10);                 -- 0MPSN  (ведущий «0» кодирует звук «th»)
SELECT dmetaphone('Smith');                       -- SM0  (double metaphone, устойчивее)
```

- `soundex(t)` — 4-символьный код; совпал у двух слов → они «созвучны».
- `difference(a, b)` — насколько близки soundex-коды, `0` (разные) … `4` (совпали).
- `metaphone(t, max)` / `dmetaphone(t)` — более точные алгоритмы для английского.

> **⚠️ Фонетика — только для латиницы/английского.** Алгоритмы построены на
> английской фонетике и по кириллице **не работают**: `soundex('ноутбук')` возвращает
> **пустую строку**. Для русского фонетический поиск через `fuzzystrmatch` бесполезен
> — используйте триграммы (`similarity`) или `levenshtein`. Помните об этом, когда
> видите soundex в чужом коде «для поиска имён».

## `unaccent` — снятие диакритики

Расширение `unaccent` заменяет буквы с диакритическими знаками на базовые: `é→e`,
`ö→o`, `ï→i`, `š→s`, `ñ→n` и т.п. — по словарю. Это нужно, чтобы искать без учёта
акцентов: пользователь набрал «cafe», а в базе «Café».

```sql
SELECT unaccent('Nescafé');        -- Nescafe
SELECT unaccent('Škoda Octavia');  -- Skoda Octavia
SELECT unaccent('naïve café');     -- naive cafe
SELECT unaccent('Löwenbräu');      -- Lowenbrau
```

Приём для **поиска без акцентов** — нормализовать `unaccent`-ом *обе* стороны
сравнения:

```sql
SELECT name FROM foreign_names
WHERE unaccent(lower(name)) LIKE '%cafe%';   -- найдёт «Nescafé», «Café Zoë»
```

`unaccent` можно встроить и в конфигурацию FTS (как словарь-препроцессор перед
стеммером) — тогда поиск станет «безакцентным» автоматически; настройка своей
конфигурации выходит за рамки лабы.

> **Про русскую «ё».** Штатный словарь `unaccent` в PostgreSQL 16 (файл
> `unaccent.rules`) снимает диакритику не только с латиницы, но и с кириллической
> «ё», приводя её к «е»:
>
> ```sql
> SELECT unaccent('ёлка');  -- елка
> SELECT unaccent('Ё');     -- Е
> ```
>
> Так что «ё=е» вне FTS работает через обычный `unaccent` без всяких доработок
> словаря. (В FTS русский стеммер сглаживает «ё» и сам, см. [файл 01](01-fts-intro.md).)

### Важно: `unaccent` — функция, а не «просто ASCII»

`unaccent` убирает **диакритику**, но не транслитерирует и не «латинизирует»
кириллицу: `unaccent('Москва') = 'Москва'` (диакритики там нет). Это инструмент для
языков с акцентами (французский, немецкий, чешский, испанский…), а не «перевод в
латиницу».
