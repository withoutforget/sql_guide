# 06 — Арифметика дат и интервалов: `+`/`-`, `age`, `make_*`, `OVERLAPS`, попадание в период

[← форматирование](05-formatting.md) · [оглавление](../README.md) · [дальше: временные ряды и gap-filling →](07-timeseries.md)

---

Даты и интервалы можно складывать и вычитать — но **тип результата зависит от
операндов**, и это надо держать в голове, иначе получишь дни там, где ждал
интервал, и наоборот.

## Таблица операций и типов результата

| Выражение | Тип результата | Пример → результат |
|-----------|----------------|--------------------|
| `date + integer` | `date` (прибавили **дни**) | `date '2024-03-15' + 10` → `2024-03-25` |
| `date - integer` | `date` | `date '2024-03-15' - 5` → `2024-03-10` |
| `date - date` | **`integer`** (число дней!) | `date '2024-07-01' - date '2024-06-01'` → `30` |
| `date + interval` | `timestamp` | `date '2024-03-15' + interval '1 month'` → `2024-04-15 00:00:00` |
| `timestamp - timestamp` | **`interval`** | `'…18:00' - '…09:30'` → `08:30:00` |
| `timestamptz - timestamptz` | `interval` | разница двух моментов |
| `timestamp(tz) + interval` | `timestamp(tz)` | сдвиг момента |
| `interval * число` | `interval` | `interval '1 day' * 3` → `3 days` |
| `interval / число` | `interval` | `interval '1 hour' / 4` → `00:15:00` |

Два «капкана», которые надо запомнить:

- **`date - date` даёт `integer`** (количество дней), а **не** интервал. Хочешь
  интервал — приведи к timestamp: `date1::timestamp - date2::timestamp`.
- **`timestamp - timestamp` даёт `interval`** (`2 days 05:30:00`), а не число.
  Хочешь число часов — оберни в `EXTRACT(epoch …)/3600` (см. [файл 04](04-extract-trunc-bin.md)).

Прибавление **целого** к `date` — это всегда **дни** (`+ 10` = +10 дней). А
прибавление к `timestamp`/`timestamptz` требует именно `interval` (`+ interval '10
days'`) — «голое» число к моменту не прибавишь.

## `age` — «человеческий» интервал в годах/месяцах/днях

Простое вычитание `ts1 - ts2` даёт интервал в **днях и часах** (потому что не знает,
через какие месяцы прошло время). `age` же раскладывает разницу «календарно» — в
годах, месяцах и днях, как считает человек («мне 5 лет 6 месяцев»):

```sql
-- две фиксированные даты:
SELECT date '2024-09-30' - date '2019-03-10';               -- 2031        (просто дней)
SELECT age(timestamp '2024-09-30', timestamp '2019-03-10'); -- 5 years 6 mons 20 days
```

Формы:

- `age(ts1, ts2)` — «сколько прошло **от `ts2` до `ts1`**» (второй аргумент —
  «начало»). Именно эту форму используем в лабе — от **двух фиксированных** дат,
  чтобы вывод был воспроизводим.
- `age(ts)` — от `ts` до **текущей полуночи** (`current_date`). Недетерминированно
  (зависит от «сегодня») — в задачах лабы **не применяем**, только упоминаем.

Когда что: `age` — для показа «возраста»/стажа человеку (годы-месяцы-дни). Простое
вычитание или `EXTRACT(epoch …)` — когда нужно **число** для сравнения/сортировки/
усреднения (у `age` компоненты нельзя просто «сложить в дни» — в месяце разное
число дней). В задаче 7 показываем оба и объясняем разницу.

## `justify_*` — нормализовать интервал

`interval` может хранить «ненормализованные» значения (`36 hours`, `40 days`). Три
функции приводят их к привычному виду:

```sql
justify_hours(interval '36 hours')    -- 1 day 12:00:00     (24 ч → 1 день)
justify_days(interval '40 days')      -- 1 mon 10 days       (30 дн → 1 месяц)
justify_interval(interval '1 mon 40 days 30 hours')  -- 2 mons 11 days 06:00:00  (обе сразу)
```

`justify_hours` считает сутки = 24 ч, `justify_days` — месяц = 30 дней. Полезно для
единообразного отображения длительностей.

## Конструирование значений: `make_*`

Когда компоненты даты/времени лежат в отдельных колонках/числах, собрать значение
из них помогают `make_*`:

```sql
make_date(2024, 3, 15)                          -- date 2024-03-15
make_time(14, 30, 0)                            -- time 14:30:00
make_timestamp(2024, 3, 15, 14, 30, 0)          -- timestamp
make_timestamptz(2024, 3, 15, 14, 30, 0, 'UTC') -- timestamptz в указанной зоне
make_interval(days => 2, hours => 5, mins => 30)-- interval '2 days 05:30:00'
```

`make_interval` принимает **именованные** аргументы (`years`, `months`, `weeks`,
`days`, `hours`, `mins`, `secs`) — удобно строить интервал из числовой переменной:
`make_interval(days => n)` вместо конкатенации строк.

## ⚠️ DST: `+ '1 day'` ≠ `+ '24 hours'`

Для `timestamptz` в зоне с **летним временем** прибавление «дня» и «24 часов» может
дать **разные** моменты. Причина из [файла 02](02-literals-parsing.md): `interval`
хранит дни и секунды отдельно. «День» — это «тот же час на следующий календарный
день», а «24 часа» — ровно 86400 секунд. В день перевода часов вперёд сутки длятся
23 часа:

```sql
SET TIME ZONE 'America/New_York';   -- переход на летнее время 2024-03-10 02:00
SELECT timestamptz '2024-03-09 12:00:00-05' + interval '1 day';   -- 2024-03-10 12:00-04
SELECT timestamptz '2024-03-09 12:00:00-05' + interval '24 hours';-- 2024-03-10 13:00-04
```

«+1 день» сохранил настенное время 12:00 (но это на час меньше реального времени, т.к.
сутки были 23-часовые); «+24 часа» отсчитал ровно сутки — и настенные часы показали
13:00. Мораль: для **календарной** логики («тот же час завтра») берите `interval
'1 day'`, для **физической** длительности — `interval '24 hours'`. (В UTC и в зонах
без DST разницы нет — поэтому детерминированный вывод лабы держим в UTC.)

## Попадание в период: полуоткрытый интервал `>= a AND < b`

Как правильно отобрать «всё за март 2024»? Наивно тянет написать `BETWEEN`:

```sql
-- ❌ ОПАСНО для timestamp/timestamptz:
WHERE created_at BETWEEN '2024-03-01' AND '2024-03-31'
```

`BETWEEN` включает **обе** границы (`>= '2024-03-01' AND <= '2024-03-31'`). Но
`'2024-03-31'` без времени — это `2024-03-31 00:00:00`, то есть **полночь** 31-го.
Заказы, сделанные 31 марта **после полуночи** (весь день!), в диапазон **не
попадут** — потеряете почти целые сутки. Плюс момент `2024-04-01 00:00:00` (если
такой есть) в `<= '2024-03-31 23:59:59'` не поймать красиво.

**Правильный шаблон — полуоткрытый интервал:** нижняя граница включена, верхняя —
нет, а верхняя = **начало следующего периода**:

```sql
-- ✅ весь март 2024, без потерь и без «магических» 23:59:59:
WHERE created_at >= '2024-03-01'
  AND created_at <  '2024-04-01'
```

Это ловит все моменты марта до последней микросекунды и не зависит от точности
(секунды, миллисекунды — неважно). То же для месяца/квартала: верхняя граница —
`начало + interval '1 month'` / `+ interval '3 months'`. Такой подход ещё и хорошо
ложится на индекс по `created_at`. В задаче 4 разбираем это на квартале. (Для чистых
`date` без времени `BETWEEN` безопасен — там нет «хвоста» суток; проблема именно у
`timestamp`/`timestamptz`.)

## Пересечение периодов: оператор `OVERLAPS`

Проверить, **пересекаются ли два периода** (акции, брони, смены), помогает
стандартный оператор:

```sql
(начало1, конец1) OVERLAPS (начало2, конец2)   -- → boolean
```

Возвращает `true`, если периоды имеют **общее время**. Границы трактуются
**полуоткрыто**: если периоды лишь **соприкасаются** концами (конец одного = начало
другого), это **не** пересечение:

```sql
(date '2024-03-01', date '2024-03-10') OVERLAPS (date '2024-03-05', date '2024-03-07')  -- true  (вложен)
(date '2024-03-01', date '2024-03-10') OVERLAPS (date '2024-03-08', date '2024-03-20')  -- true  (частично)
(date '2024-03-01', date '2024-03-10') OVERLAPS (date '2024-03-10', date '2024-03-20')  -- false (только касаются)
```

Вместо второго значения можно передать **длину** периода интервалом:
`(start, interval '3 days') OVERLAPS (…)`. В задаче 10 ищем пересекающиеся промо-акции
через `OVERLAPS` в self-join.

> **Полноценные диапазонные типы** (`daterange`, `tsrange`) — где период это одно
> значение с операторами включения `@>` и пересечения `&&`, индексируемое GiST, —
> это отдельная тема **lab018**. Здесь обходимся скалярными сравнениями и `OVERLAPS`;
> для большинства аналитических задач их достаточно.
