# 02 — Литералы, ввод и разбор дат: `DATE '...'`, `INTERVAL '...'`, `to_date`, `DateStyle`

---

Разобрались, *какие* бывают типы — теперь как **записывать** значения: в
SQL-литералах, приведением из текста и разбором нестандартных строк.

## Типизированные литералы

Самая явная форма — имя типа перед строкой в одинарных кавычках:

```sql
DATE '2024-03-15'
TIME '14:30:00'
TIMESTAMP '2024-03-15 14:30:00'
TIMESTAMPTZ '2024-03-15 14:30:00+03'
INTERVAL '2 days 5 hours'
```

Это то же самое, что приведение (`'2024-03-15'::date`), просто читается как «вот
дата такая-то». Внутри кавычек — общепринятый **ISO 8601**: `ГГГГ-ММ-ДД` для даты,
`ЧЧ:ММ:СС` для времени, между ними пробел (или `T`). Дробные секунды — через точку:
`'14:30:00.250'`. Для `timestamptz` смещение зоны пишут в конце: `+03`, `-05:30`,
`Z` (= UTC). **Год-месяц-день в таком порядке однозначен** и работает всегда,
независимо от настроек, — предпочитайте его.

## Приведение из текста (`::`, `CAST`)

Поскольку строковый литерал имеет «неопределённый» тип (`unknown`, см. lab013),
привести его к дате можно и обычным `::`:

```sql
'2024-03-15'::date                 -- 2024-03-15
'2024-03-15 14:30'::timestamptz    -- момент (зона из сессии, т.к. смещение не указано)
'90 minutes'::interval             -- 01:30:00
```

Приведение падает, если строка — не корректное значение (как и `'abc'::int` в
lab013):

```sql
'2024-13-40'::date   -- ERROR: date/time field value out of range
'вчера'::date        -- ERROR: invalid input syntax for type date: "вчера"
```

Приёмы «не уронить запрос на битой строке» те же, что в lab013: отфильтровать плохие
строки в `WHERE` или обернуть приведение в `CASE`. Регулярных выражений для проверки
формата тут ещё нет — они в lab015.

## Опасность неоднозначных форматов и `DateStyle`

Проблема возникает, когда во входной строке день и месяц можно перепутать:
`'03/04/2024'` — это 3 апреля или 4 марта? Ответ зависит от параметра сессии
**`DateStyle`**, у которого две части: формат **вывода** и порядок полей на
**вводе** (`MDY` — месяц-день-год, как в США; `DMY` — день-месяц-год, как в РФ/ЕС):

```sql
SHOW datestyle;               -- напр. 'ISO, MDY'  (вывод ISO, ввод месяц-день-год)
SET datestyle = 'ISO, DMY';   -- переключить ввод на день-месяц-год
```

```sql
SET datestyle = 'ISO, MDY';
SELECT date '03/04/2024';     -- 2024-03-04  (истолковано как March 4)
SET datestyle = 'ISO, DMY';
SELECT date '03/04/2024';     -- 2024-04-03  (истолковано как 3 April)
```

Одна и та же строка → разные даты в зависимости от настройки. **Вывод:** не
полагайтесь на `DateStyle` для разбора внешних данных. Либо приводите строки к
однозначному ISO (`ГГГГ-ММ-ДД`), либо разбирайте явным шаблоном через `to_date`
(ниже) — тогда результат не зависит от настроек сессии.

## Разбор нестандартных строк: `to_date` и `to_timestamp`

Когда строка приходит в «своём» формате (`'15.03.2024'`, `'20240315'`,
`'15-Mar-2024'`), надёжнее всего разобрать её **явным шаблоном**:

```sql
to_date(текст, шаблон)              -- → date
to_timestamp(текст, шаблон)         -- → timestamptz
```

Шаблон описывает, где в строке что стоит. Основные коды:

| Код | Значение | Пример входа |
|-----|----------|--------------|
| `YYYY` | год, 4 цифры | `2024` |
| `YY` | год, 2 цифры | `24` |
| `MM` | месяц, 01–12 | `03` |
| `Mon` | месяц, сокр. назв. | `Mar` (по локали) |
| `Month`| месяц, полное назв. | `March` |
| `DD` | день месяца, 01–31 | `15` |
| `HH24`| час, 00–23 | `14` |
| `HH12`| час, 01–12 | `02` |
| `MI` | минуты, 00–59 | `30` |
| `SS` | секунды, 00–59 | `05` |
| `AM`/`PM` | половина суток | `PM` |

```sql
SELECT to_date('15.03.2024', 'DD.MM.YYYY');            -- 2024-03-15
SELECT to_date('2024/03/15', 'YYYY/MM/DD');            -- 2024-03-15
SELECT to_timestamp('15.03.2024 14:30', 'DD.MM.YYYY HH24:MI');
--   2024-03-15 14:30:00+00  (в UTC-сессии; to_timestamp даёт timestamptz)
```

Разделители в шаблоне (`.`, `/`, `-`, пробел) сопоставляются буквально, но
`to_date`/`to_timestamp` довольно **снисходительны** к лишним пробелам и
несоответствиям. Пара нюансов:

- те же коды используются для *вывода* в `to_char` (см. [файл 05](05-formatting.md)) —
  один язык шаблонов в обе стороны;
- `Mon`/`Month` на вводе зависят от локали (`lc_time`) — для внешних данных
  надёжнее числовой `MM`;
- `to_timestamp` возвращает **`timestamptz`**, истолковывая разобранное время в
  поясе сессии (если в строке/шаблоне нет зоны).

## Литералы интервалов

`interval` записывается очень гибко — словами, ISO 8601 или «часовой» записью:

```sql
INTERVAL '1 day'
INTERVAL '2 days 5 hours 30 minutes'
INTERVAL '90 minutes'          -- нормализуется в 01:30:00
INTERVAL '1 year 2 months'
INTERVAL '1-6' YEAR TO MONTH   -- 1 год 6 месяцев (SQL-стандартная форма)
INTERVAL '05:30:00'            -- 5 ч 30 мин (часовая запись)
INTERVAL 'P1Y2M10D'            -- ISO 8601: 1 год 2 мес 10 дней
```

Единицы: `microsecond`, `second`, `minute`, `hour`, `day`, `week`, `month`,
`year`, `decade`, `century` (и множественное число). Можно смешивать и указывать
дробные (`'1.5 hours'` → `01:30:00`). Отрицательные тоже допустимы:
`INTERVAL '-3 days'`.

> ⚠️ **Месяцы и дни в интервале — не фиксированное число секунд.** `interval` хранит
> отдельно поля *месяцы*, *дни* и *секунды* — именно потому, что в месяце бывает
> 28–31 день, а в сутках при переводе часов 23 или 25 часов. Поэтому
> `date + interval '1 month'` корректно попадает в тот же день следующего месяца, а
> не «плюс 30 дней». Подробно об арифметике — [файл 06](06-arithmetic.md).
