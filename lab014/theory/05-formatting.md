# 05 — Форматирование вывода: `to_char`

[← extract / date_trunc](04-extract-trunc-bin.md) · [оглавление](../README.md) · [дальше: арифметика и `age` →](06-arithmetic.md)

---

`to_char(значение, шаблон)` превращает дату/время/интервал (а также число) в
**текст** по заданному шаблону. Это «обратная сторона» `to_date`/`to_timestamp` из
[файла 02](02-literals-parsing.md): тот же язык шаблонов, только в другую сторону —
из значения в строку.

```sql
to_char(timestamptz '2024-03-05 09:07:03+00', 'DD.MM.YYYY')  -- '05.03.2024'
```

Результат — всегда `text`. Применяют для человекочитаемого вывода в отчётах и для
получения строкового **ключа периода** (`'2024-03'`).

## Основные коды шаблона

| Код | Значение | Пример (для `2024-03-05 09:07:03`) |
|-----|----------|------------------------------------|
| `YYYY` | год, 4 цифры | `2024` |
| `YY` | год, 2 цифры | `24` |
| `MM` | месяц, 2 цифры (01–12) | `03` |
| `DD` | день месяца, 2 цифры | `05` |
| `HH24` | час 00–23 | `09` |
| `HH12` / `HH` | час 01–12 | `09` |
| `MI` | минуты | `07` |
| `SS` | секунды | `03` |
| `MS` | миллисекунды | `000` |
| `AM` / `PM` | половина суток | `AM` |
| `Mon` | сокр. название месяца | `Mar` (зависит от локали) |
| `Month` | полное название месяца | `March` (дополнено пробелами) |
| `Dy` | сокр. название дня недели | `Tue` |
| `Day` | полное название дня недели | `Tuesday ` (дополнено пробелами) |
| `D` | день недели 1–7 (**1=вс**) | `3` |
| `ID` | день недели ISO 1–7 (1=пн) | `2` |
| `Q` | квартал | `1` |
| `WW` | номер недели года | `10` |
| `IW` | номер ISO-недели | `10` |
| `DDD` | день года | `065` |
| `TZ` / `OF` | назв. зоны / смещение | `UTC` / `+00` |

```sql
to_char(ts, 'YYYY-MM-DD HH24:MI:SS')  -- '2024-03-05 09:07:03'
to_char(ts, 'DD.MM.YYYY')             -- '05.03.2024'
to_char(ts, 'YYYY-MM')                -- '2024-03'   ← ключ месяца
to_char(ts, 'HH24:MI')                -- '09:07'
to_char(ts, 'Q "квартал" YYYY')       -- '1 квартал 2024'
```

## Три важных нюанса

### 1. `FM` — убрать набивку нулями/пробелами

По умолчанию числовые поля **дополняются нулями** до фиксированной ширины (`MM`
всегда 2 цифры), а названия (`Month`, `Day`) — **пробелами** до максимальной
длины. Префикс `FM` (fill mode) эту набивку убирает:

```sql
to_char(date '2024-03-05', 'FMDD.FMMM.YYYY')  -- '5.3.2024'  (без ведущих нулей)
to_char(date '2024-03-05', 'DD.MM.YYYY')      -- '05.03.2024'
to_char(date '2024-03-05', 'FMMonth')         -- 'March'     (без хвостовых пробелов)
to_char(date '2024-03-05', 'Month')           -- 'March    ' (дополнено до 9 символов)
```

### 2. Литеральный текст в шаблоне — в двойных кавычках

Буквы внутри шаблона PostgreSQL пытается трактовать как коды. Чтобы вставить
**свой** текст, заключите его в двойные кавычки:

```sql
to_char(ts, '"заказ от" DD.MM.YYYY')   -- 'заказ от 05.03.2024'
```

Без кавычек буквы «заказ» были бы разобраны как коды формата и дали мусор.

### 3. ⚠️ Названия месяцев/дней зависят от локали (`lc_time`)

`Month`, `Day`, `Mon`, `Dy` выводят названия на языке параметра **`lc_time`**
сессии/сервера. На одной машине это `March`/`Tuesday`, на другой (русская локаль) —
`Март`/`Вторник`, на третьей (C/POSIX) — тоже английские. Плюс есть модификатор
`TM` (`TMMonth`) — «переводить по локали».

**Для воспроизводимого вывода это ловушка:** результат зависит от окружения.
Поэтому в задачах лабы:

- для ключей и дат используем **числовые/ISO** шаблоны (`'YYYY-MM'`,
  `'DD.MM.YYYY'`) — они одинаковы всюду;
- где нужно **название дня недели по-русски**, не полагаемся на локаль, а строим
  сами через `CASE` по `isodow` (`1→'пн'`, …, `7→'вс'`) — полностью детерминированно
  (см. задачу 15). Так и надёжнее, и язык гарантированно тот, что нужен.

## `to_char` для интервалов и чисел

`to_char` работает и с `interval` — но только с «часовыми» полями (день/месяц он
как отдельные единицы не печатает, разворачивает в часы неаккуратно), поэтому для
интервалов чаще берут `EXTRACT(epoch …)` и считают часы/минуты сами:

```sql
to_char(interval '2 days 05:30:00', 'HH24:MI')   -- '05:30'  (дни он тут игнорирует!)
```

И для чисел — тот же `to_char` форматирует деньги (разделители разрядов), это уже
из lab013/числовой темы:

```sql
to_char(1234567.89, 'FM999G999G990D00')   -- '1,234,567.89'
```

## Мысленная модель: где что применять

| Задача | Инструмент |
|--------|-----------|
| достать число (месяц, час, dow) | `EXTRACT` / `date_part` ([файл 04](04-extract-trunc-bin.md)) |
| сгруппировать по периоду | `date_trunc` / `date_bin` ([файл 04](04-extract-trunc-bin.md)) |
| **красиво показать** дату строкой | `to_char` (этот файл) |
| разобрать строку в дату | `to_date` / `to_timestamp` ([файл 02](02-literals-parsing.md)) |

Правило: **группируйте и сортируйте по датам** (`date_trunc`), а `to_char`
применяйте **последним шагом** — только для отображения. Строка `'2024-10'`
сортируется по алфавиту правильно, но, например, `'DD.MM.YYYY'` уже нет — не
сортируйте по отформатированному тексту.
