# 01 — Рамка окна и рамка по умолчанию

[← оглавление](../README.md) · [дальше: ROWS / RANGE / GROUPS →](02-rows-range-groups.md)

---

В [lab010](../../lab010/) мы разобрали **окно** (раздел, partition): `PARTITION BY`
бьёт строки на группы, `ORDER BY` внутри `OVER` их упорядочивает, а функция считает
значение по разделу и приписывает его каждой строке, ничего не схлопывая. Там мы
сознательно держали агрегатные окна **без `ORDER BY`** — брали весь раздел целиком.

Эта лаба — про то, что происходит, когда у окна появляется **рамка**. Рамка — это
сердце «нарастающих» и «скользящих» вычислений. Начнём с главного различия.

## Раздел (partition) ≠ рамка (frame)

Это два **разных** понятия, и путаница между ними — источник почти всех ошибок с
окнами:

- **Раздел (partition)** — **все** строки группы, заданной `PARTITION BY` (или весь
  результат, если `PARTITION BY` нет). Он один на всю группу и от текущей строки не
  зависит.
- **Рамка (frame)** — **подмножество строк ВНУТРИ раздела**, взятое **относительно
  текущей строки**. Именно по рамке агрегат (`sum`, `avg`, `min`…) и функции
  `first_value`/`last_value`/`nth_value` реально считают своё значение. Рамка **своя
  у каждой строки** и обычно «скользит» вместе с ней.

> **Мысленная модель.** Раздел — это вся «дорожка» строк товара. А рамка — это
> «окошко-каретка», которое стои́т над текущей строкой и захватывает какие-то
> соседние строки (например, «всё от начала дорожки до меня» или «я и два соседа
> слева»). Для каждой строки каретка своя. Функция смотрит **только сквозь эту
> каретку**, а не на весь раздел.

Пока рамки не было (в lab010 у агрегатов не было `ORDER BY`), рамка молча равнялась
**всему разделу** — поэтому `sum(revenue) OVER (PARTITION BY category)` давал итог по
всей категории. Как только появляется `ORDER BY`, рамка по умолчанию **сжимается** —
и поведение меняется.

## Синтаксис рамки

Рамка приписывается в конце определения окна, после `ORDER BY`:

```sql
функция(...) OVER (
    PARTITION BY ...
    ORDER BY ...
    { ROWS | RANGE | GROUPS } BETWEEN <начало> AND <конец>
    [ EXCLUDE ... ]
)
```

- `ROWS | RANGE | GROUPS` — **режим** рамки (как отсчитывать границы). Подробно — в
  [файле 02](02-rows-range-groups.md). По умолчанию `RANGE`.
- Границы `<начало>` и `<конец>` берутся из набора:

| Граница | Что означает |
|---------|--------------|
| `UNBOUNDED PRECEDING` | от **самого начала** раздела |
| `<N> PRECEDING` | на `N` (строк / значений / групп) **назад** от текущей строки |
| `CURRENT ROW` | **текущая** строка (в `RANGE`/`GROUPS` — вместе с её «ровесниками», см. файл 02) |
| `<N> FOLLOWING` | на `N` **вперёд** от текущей строки |
| `UNBOUNDED FOLLOWING` | до **самого конца** раздела |

`<начало>` не может стоять «правее» `<конца>` (нельзя `BETWEEN CURRENT ROW AND 2
PRECEDING`). Пример полной формы: `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW` — «две
предыдущие строки и текущая» (окно из трёх строк).

**Короткая форма.** Если написать только начало — конец берётся `CURRENT ROW`:

```sql
ROWS 3 PRECEDING            -- то же, что ROWS BETWEEN 3 PRECEDING AND CURRENT ROW
ROWS UNBOUNDED PRECEDING    -- то же, что ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
```

## Рамка по умолчанию — главный подводный камень

Если рамку **не написать явно**, она **всё равно есть** — просто подразумевается. И
зависит она от того, есть ли в окне `ORDER BY`:

| Есть ли `ORDER BY` в `OVER` | Рамка по умолчанию | Смысл |
|---|---|---|
| **есть** `ORDER BY` | `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` | от начала раздела **до текущей строки** → **нарастающий итог** |
| **нет** `ORDER BY` | `RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` | **весь раздел** целиком (как в lab010) |

Отсюда — два разных результата у одного и того же `sum`:

```sql
sum(revenue) OVER ()                  -- нет ORDER BY → ВЕСЬ раздел → ОБЩИЙ итог
sum(revenue) OVER (ORDER BY sale_date) -- есть ORDER BY → до текущей строки → НАРАСТАЮЩИЙ итог
```

**Почему так задумано?** Логика простая: `ORDER BY` говорит «у строк есть порядок,
кто раньше — кто позже». Естественное значение «суммы с учётом порядка» — это сумма
**всего, что было по текущий момент включительно**, то есть нарастающий итог. А без
`ORDER BY` порядок не задан, «текущего момента» нет — и единственное осмысленное окно
это весь раздел. Стандарт SQL закрепил именно `RANGE ... CURRENT ROW` как поведение
по умолчанию при наличии `ORDER BY`.

### Пример: нарастающий итог

Возьмём дневную выручку одного товара и посчитаем нарастающий итог — просто добавив
`ORDER BY` (рамку не пишем, полагаемся на умолчание):

```sql
SELECT sale_date, revenue,
       sum(revenue) OVER (PARTITION BY product_id ORDER BY sale_date) AS running
FROM sales
WHERE product_id = 1               -- Смартфон Nova
ORDER BY sale_date;
```
```
 sale_date  | revenue  | running
------------+----------+---------
 2024-03-01 | 10000.00 | 10000.00   ← сумма по строкам с начала и до этой
 2024-03-02 | 12000.00 | 22000.00   ← 10000 + 12000
 2024-03-03 | 12000.00 | 34000.00
 2024-03-04 |  8000.00 | 42000.00
 2024-03-05 | 14000.00 | 56000.00
 2024-03-06 | 15000.00 | 71000.00
 2024-03-07 |  9000.00 | 80000.00   ← дошли до конца = полный итог 80000
```

`PARTITION BY product_id` заставляет накопление **перезапускаться** в начале каждого
товара (у другого товара `running` снова начнётся с его первого дня). Это и есть
«кумулятивная сумма с рестартом по разделу».

### Тот же агрегат без `ORDER BY` = общий итог

```sql
SELECT sale_date, revenue,
       sum(revenue) OVER (PARTITION BY product_id) AS total
FROM sales WHERE product_id = 1 ORDER BY sale_date;
```
```
 sale_date  | revenue  |  total
------------+----------+---------
 2024-03-01 | 10000.00 | 80000.00   ← во ВСЕХ строках один и тот же полный итог
 2024-03-02 | 12000.00 | 80000.00
 ...        |   ...    | 80000.00
```

Разница только в `ORDER BY` внутри `OVER`, а результат принципиально другой:
нарастающий итог против общего. Именно это и решает рамка по умолчанию.

## Важное исключение: не всем функциям рамка нужна

Рамка управляет **агрегатами** (`sum`/`avg`/`count`/`min`/`max`) и функциями
**значений** (`first_value`/`last_value`/`nth_value` — [файл 04](04-first-last-nth.md)).
А вот:

- **Ранжирующие** функции из lab010 (`row_number`/`rank`/`dense_rank`/`ntile`/…) рамку
  **игнорируют** — они работают по всему упорядоченному разделу по определению.
- Функции **смещения** `lag`/`lead` ([файл 03](03-lag-lead.md)) рамку тоже
  **игнорируют** — берут строку по смещению во всём разделе. Рамку им можно даже
  приписать в тексте запроса, но она не подействует.

То есть рамка важна ровно для «оконных агрегатов» и «значений по границам рамки».
Их-то мы дальше и разбираем.
