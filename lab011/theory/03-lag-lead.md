# 03 — Функции смещения: `LAG` и `LEAD`

[← ROWS / RANGE / GROUPS](02-rows-range-groups.md) · [оглавление](../README.md) · [дальше: FIRST/LAST/NTH_VALUE →](04-first-last-nth.md)

---

Огромный класс аналитики — «сравнить строку с **предыдущей** или **следующей**»:
рост к прошлому дню, дельта между заказами, поиск разрывов в ряду. Для этого есть
две зеркальные функции — `lag` («заглянуть назад») и `lead` («заглянуть вперёд»).

## Что делают

- **`lag(expr)`** — берёт значение `expr` из **предыдущей** строки упорядоченного
  раздела.
- **`lead(expr)`** — берёт значение `expr` из **следующей** строки.

«Предыдущая/следующая» определяется `ORDER BY` внутри `OVER`. Строки при этом, как у
всех оконных функций, **не схлопываются** — значение соседа просто появляется рядом с
текущей строкой.

> **Важно: рамка на `lag`/`lead` не влияет.** В отличие от агрегатов и
> `first_value`/`last_value`, функции смещения работают по **всему упорядоченному
> разделу**, а не по рамке. Даже если приписать им `ROWS …`, рамка будет
> проигнорирована. Думайте о них как о «шагни на N строк по разделу», без всякой
> «каретки».

## Полный синтаксис и все аргументы

```sql
lag ( expr [, offset [, default] ] )  OVER ( PARTITION BY ... ORDER BY ... )
lead( expr [, offset [, default] ] )  OVER ( PARTITION BY ... ORDER BY ... )
```

| Аргумент | Смысл | По умолчанию |
|----------|-------|--------------|
| `expr` | что вернуть из соседней строки (столбец/выражение) | — (обязателен) |
| `offset` | на **сколько строк** сместиться (2 = через одну) | `1` |
| `default` | что вернуть, если такой строки нет (вышли за край раздела) | `NULL` |

Примеры смысла аргументов:

```sql
lag(revenue)          -- выручка 1 строку назад; на первой строке → NULL
lag(revenue, 2)       -- выручка 2 строки назад; на первых двух → NULL
lag(revenue, 1, 0)    -- 1 назад, но вместо NULL на первой строке → 0
lead(revenue)         -- выручка 1 строку вперёд; на последней строке → NULL
```

## Поведение на границах и с `NULL`

- **На краю раздела** (у первой строки для `lag`, у последней для `lead`) соседней
  строки нет → возвращается `default` (по умолчанию `NULL`). Это удобный маркер
  «начала/конца ряда».
- **`NULL` в данных.** `lag`/`lead` возвращают **само значение** соседней строки,
  даже если оно `NULL`. `default` подставляется **только** когда строки-соседа не
  существует, а не когда её значение `NULL`. (PostgreSQL 16 **не** поддерживает
  `IGNORE NULLS` для оконных функций — «перепрыгнуть через NULL к ближайшему не-NULL»
  придётся руками, но в наших данных `NULL`-выручек нет.)
- **Разделы независимы**: `lag` из первой строки товара **не** заглянет в предыдущий
  товар — вернёт `default`. Именно поэтому `PARTITION BY product_id` обязателен,
  иначе «предыдущий день» одного товара утёк бы в другой.

## Паттерн 1: дельта и рост % к предыдущему периоду

Классика period-over-period. `lag(revenue)` даёт «вчера», дальше — арифметика:

```sql
WITH d AS (
  SELECT product_id, sale_date, revenue,
         lag(revenue) OVER (PARTITION BY product_id ORDER BY sale_date) AS вчера
  FROM sales
)
SELECT sale_date, revenue, вчера,
       revenue - вчера                         AS дельта,
       round(100.0*(revenue - вчера)/вчера, 1) AS рост_проц
FROM d WHERE product_id = 1 ORDER BY sale_date;
```
```
 sale_date  | revenue  |  вчера   |  дельта  | рост_проц
------------+----------+----------+----------+----------
 2024-03-01 | 10000.00 |   NULL   |   NULL   |   NULL     ← первый день: сравнивать не с чем
 2024-03-02 | 12000.00 | 10000.00 |  2000.00 |   20.0
 2024-03-03 | 12000.00 | 12000.00 |     0.00 |    0.0     ← плато: дельта 0, рост 0%
 2024-03-04 |  8000.00 | 12000.00 | -4000.00 |  -33.3     ← падение
 2024-03-05 | 14000.00 |  8000.00 |  6000.00 |   75.0     ← скачок вверх
 2024-03-06 | 15000.00 | 14000.00 |  1000.00 |    7.1
 2024-03-07 |  9000.00 | 15000.00 | -6000.00 |  -40.0
```

Обратите внимание на три вещи: на первой строке всё `NULL` (нет «вчера»); плато даёт
`дельта = 0`; отрицательная дельта — падение. Это задача 5 лабы.

> **Ловушка деления.** В `рост % = 100*(сегодня−вчера)/вчера` знаменатель `вчера`
> может быть `0` (если вчера была нулевая выручка) → деление на ноль. Тогда
> оборачивают в `CASE WHEN вчера = 0 THEN NULL ELSE ... END` или `NULLIF(вчера,0)`.
> И `100.0 * …` (а не `100 *`) — чтобы не сработало целочисленное деление (как в
> [lab001](../../lab001/)). В наших данных нулевых выручек нет, но помнить об этом
> надо.

## Паттерн 2: сравнение со следующим периодом (`lead`)

`lead` — то же, но вперёд. «Что будет завтра» и подпись тренда:

```sql
SELECT sale_date, revenue,
       lead(revenue) OVER (PARTITION BY product_id ORDER BY sale_date) AS завтра,
       CASE WHEN lead(revenue) OVER (PARTITION BY product_id ORDER BY sale_date) IS NULL
                 THEN 'последний день'
            WHEN lead(revenue) OVER (PARTITION BY product_id ORDER BY sale_date) > revenue
                 THEN 'завтра рост'
            ELSE 'завтра спад/ровно' END AS прогноз
FROM sales WHERE product_id = 1 ORDER BY sale_date;
```
На последней строке `lead` = `NULL` → «последний день». Задача 6 лабы.

## Паттерн 3: разрывы в ряду (`lag` по дате)

`lag` можно применять не только к числам. `lag(sale_date)` даёт дату предыдущей
продажи; **вычитание дат** в PostgreSQL возвращает **число дней** между ними — так
находят разрывы во временно́м ряду:

```sql
WITH d AS (
  SELECT product_id, sale_date,
         lag(sale_date) OVER (PARTITION BY product_id ORDER BY sale_date) AS прошлая
  FROM sales
)
SELECT sale_date, прошлая, sale_date - прошлая AS дней_с_прошлой
FROM d WHERE product_id = 4 ORDER BY sale_date;   -- Робот-пылесос (с пропусками)
```
```
 sale_date  |  прошлая   | дней_с_прошлой
------------+------------+---------------
 2024-03-01 |   NULL     |   NULL
 2024-03-02 | 2024-03-01 |     1
 2024-03-04 | 2024-03-02 |     2          ← был пропуск (03-03)
 2024-03-06 | 2024-03-04 |     2          ← был пропуск (03-05)
 2024-03-07 | 2024-03-06 |     1
```
Интервал `> 1` означает пропущенные дни — так детектируют «дырки» в ряду (задача 7).

## `lag`/`lead` против рамочных функций

Легко перепутать `lag`/`lead` с `first_value`/`last_value` из [файла 04](04-first-last-nth.md).
Разница принципиальная:

| | `lag`/`lead` | `first_value`/`last_value`/`nth_value` |
|--|--------------|----------------------------------------|
| Что берут | строку по **фиксированному смещению** (±N) | строку по **позиции в рамке** (первая/последняя/N-я) |
| Смотрят на | **весь** упорядоченный раздел | текущую **рамку** |
| Рамка влияет? | **нет**, игнорируется | **да**, это их суть |

Проще: `lag(x, 2)` — «ровно две строки назад». `nth_value(x, 2)` — «вторая строка
**рамки** от её начала». При рамке по умолчанию это разные вещи.
