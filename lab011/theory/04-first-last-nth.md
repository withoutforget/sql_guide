# 04 — Значения по рамке: `FIRST_VALUE`, `LAST_VALUE`, `NTH_VALUE`

[← LAG / LEAD](03-lag-lead.md) · [оглавление](../README.md) · [дальше: WINDOW и паттерны →](05-window-clause-patterns.md)

---

Три функции достают **значение из конкретной строки рамки**: с её начала, с конца или
с N-й позиции. Звучит просто, но именно здесь прячется одна из **самых частых ошибок**
в SQL — ловушка `last_value`. Разберём её досконально.

## Что делают

```sql
first_value(expr)     OVER ( ... )   -- значение expr из ПЕРВОЙ строки рамки
last_value (expr)     OVER ( ... )   -- значение expr из ПОСЛЕДНЕЙ строки рамки
nth_value (expr, n)   OVER ( ... )   -- значение expr из N-й строки рамки (n с 1)
```

Ключевое слово — **рамки**, не раздела. Эти функции берут строку из текущей **рамки**
(той самой «каретки» из [файла 01](01-what-is-frame.md)), а рамка по умолчанию — это
`RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`, то есть «от начала раздела **до
текущей строки**». Отсюда и подвох.

- `nth_value(expr, n)` — `n` отсчитывается **от начала рамки**, начиная с `1`. Если в
  рамке меньше `n` строк — вернёт `NULL`.
- Все три **не игнорируют** `NULL`-значения: если, скажем, первая строка рамки имеет
  `expr = NULL`, `first_value` вернёт `NULL`. (PostgreSQL 16 **не** поддерживает
  `IGNORE NULLS` и `FROM FIRST/FROM LAST` — это части стандарта SQL, но не
  реализованы; обходят разворотом `ORDER BY` или фильтрацией.)

## `first_value` с рамкой по умолчанию — работает как ждёшь

Первая строка рамки `UNBOUNDED PRECEDING … CURRENT ROW` — это **начало раздела**, и
оно не меняется от строки к строке. Поэтому `first_value` при рамке по умолчанию
корректно даёт «первое значение раздела» во всех строках:

```sql
first_value(revenue) OVER (PARTITION BY product_id ORDER BY sale_date)
-- у Смартфона во ВСЕХ строках вернёт 10000 (выручка первого дня) — верно
```

## `last_value` с рамкой по умолчанию — ЛОВУШКА

А вот **последняя** строка рамки `… AND CURRENT ROW` — это **сама текущая строка**!
Поэтому наивный `last_value` возвращает **не** последний день раздела, а **выручку
текущего дня** — то есть просто копию столбца:

```sql
SELECT sale_date, revenue,
       first_value(revenue) OVER w AS первый_день,
       last_value (revenue) OVER w AS last_наивно
FROM sales WHERE product_id = 3          -- Кофеварка
WINDOW w AS (PARTITION BY product_id ORDER BY sale_date)
ORDER BY sale_date;
```
```
 sale_date  | revenue  | первый_день | last_наивно
------------+----------+-------------+------------
 2024-03-01 |  3000.00 |   3000.00   |   3000.00   ← last = сама строка
 2024-03-02 |  3000.00 |   3000.00   |   3000.00
 2024-03-03 |  6000.00 |   3000.00   |   6000.00   ← «последнее» скачет вместе с revenue!
 2024-03-04 |  4500.00 |   3000.00   |   4500.00
 2024-03-05 |  4500.00 |   3000.00   |   4500.00
 2024-03-06 | 12000.00 |   3000.00   |  12000.00
 2024-03-07 |  6000.00 |   3000.00   |   6000.00   ← вот тут случайно совпало с истинным последним
```

Колонка `last_наивно` **повторяет `revenue`** — потому что рамка каждой строки
кончается на ней самой. Это почти никогда не то, что нужно. Люди пишут `last_value`,
ожидая «значение последнего дня», а получают «значение текущего дня».

### Как взять НАСТОЯЩИЙ последний: расширить рамку до `UNBOUNDED FOLLOWING`

Чтобы `last_value` видел **весь** раздел, рамку надо явно раскрыть до конца:

```sql
last_value(revenue) OVER (PARTITION BY product_id ORDER BY sale_date
                          ROWS BETWEEN UNBOUNDED PRECEDING
                                   AND UNBOUNDED FOLLOWING) AS последний_день
```
```
 sale_date  | revenue  | last_наивно | последний_день
------------+----------+-------------+---------------
 2024-03-01 |  3000.00 |   3000.00   |    6000.00     ← теперь ВЕЗДЕ выручка последнего дня
 2024-03-03 |  6000.00 |   6000.00   |    6000.00
 2024-03-06 | 12000.00 |  12000.00   |    6000.00
 2024-03-07 |  6000.00 |   6000.00   |    6000.00
```

С полной рамкой `последний_день` во всех строках равен `6000` — выручке `03-07`. Это
задача 8 лабы: колонка `last_наивно` (совпадает с `revenue`) против правильной
`последний_день`. **Правило: как только используете `last_value` (или `nth_value` от
конца), почти всегда нужна рамка `... UNBOUNDED FOLLOWING`.**

## `nth_value` — тоже считается по рамке

`nth_value(expr, n)` берёт `n`-ю строку рамки от её начала. Чтобы получить «второй
день раздела», рамку нужно раскрыть на весь раздел — иначе в ранних строках рамка ещё
короче `n` и вернётся `NULL`:

```sql
SELECT DISTINCT
  first_value(revenue)  OVER w AS открытие,
  nth_value(revenue, 2) OVER w AS второй_день,
  last_value(revenue)   OVER w AS закрытие
FROM sales WHERE product_id = 1                 -- Смартфон
WINDOW w AS (PARTITION BY product_id ORDER BY sale_date
             ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING);
```
```
 открытие | второй_день | закрытие
----------+-------------+---------
 10000.00 |  12000.00   |  9000.00
```

С полной рамкой все три функции смотрят на весь ряд: открытие `10000` (день 1), второй
день `12000`, закрытие `9000` (день 7). Это задача 9 лабы. Значения одинаковы во всех
строках раздела, поэтому берём одну через `DISTINCT`.

> **Мысленная модель.** `first_value` = «начало рамки», `last_value` = «конец рамки»,
> `nth_value(_, n)` = «n-я от начала рамки». Всё завязано на **рамку**, а рамка по
> умолчанию обрывается на текущей строке. Хотите смотреть на весь раздел —
> раскрывайте рамку до `UNBOUNDED FOLLOWING`.

## Когда что брать

- «Значение **первого** элемента раздела» → `first_value` (рамка по умолчанию годится).
- «Значение **последнего** элемента раздела» → `last_value` **+ рамка до `UNBOUNDED
  FOLLOWING`**.
- «Значение **N-го** элемента раздела» → `nth_value(_, n)` **+ полная рамка**.
- «Соседняя строка на фиксированный шаг ±N» → это **не** сюда, а `lag`/`lead`
  ([файл 03](03-lag-lead.md)): они смотрят весь раздел и не зависят от рамки.
