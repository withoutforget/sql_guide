## Клауза `WINDOW`: определить окно один раз

Когда одно и то же окно нужно нескольким функциям, определение `OVER (PARTITION BY …
ORDER BY …)` приходится повторять — длинно и легко ошибиться. SQL позволяет **дать
окну имя** в клаузе `WINDOW` и ссылаться на него:

```sql
SELECT sale_date, revenue,
       row_number()   OVER w AS день,
       sum(revenue)   OVER w AS нарастающий,
       lag(revenue)   OVER w AS вчера
FROM sales
WINDOW w AS (PARTITION BY product_id ORDER BY sale_date)   -- определили ОДИН раз
ORDER BY product_id, sale_date;
```

Все три функции считаются по окну `w`. Поведение **идентично** тому, как если бы вы
вписали `OVER (PARTITION BY product_id ORDER BY sale_date)` в каждую, — это чисто
удобство и защита от опечаток.

### Где стои́т клауза `WINDOW`

В порядке предложений — **после `HAVING` и перед `ORDER BY`**:

```
SELECT ... FROM ... WHERE ... GROUP BY ... HAVING ...
WINDOW имя AS ( определение )
ORDER BY ...
```

### Наследование окон

Одно именованное окно может **наследовать** другое: новое окно берёт `PARTITION BY`
(и `ORDER BY`, если есть) у базового и **дополняет** его. Это удобно, когда несколько
окон делят разбиение, но сортируются по-разному:

```sql
SELECT sale_date, revenue,
       sum(revenue) OVER w  AS нарастающий,      -- порядок по дате
       rank()       OVER wr AS место_по_выручке  -- порядок по выручке
FROM sales
WINDOW wp AS (PARTITION BY product_id),          -- база: только разбиение
       w  AS (wp ORDER BY sale_date),            -- наследует wp, добавляет порядок по дате
       wr AS (wp ORDER BY revenue DESC)          -- наследует wp, добавляет порядок по выручке
ORDER BY product_id, sale_date;
```

Правила наследования (важны, иначе синтаксическая ошибка):

- новое окно **берёт `PARTITION BY` базового** и **не может** задать своё
  `PARTITION BY`;
- новое окно может добавить свой `ORDER BY`, **только если у базового его нет** (нельзя
  переопределить уже заданный порядок);
- новое окно может добавить/сузить **рамку**; базовое, на которое ссылаются, рамку
  содержать не должно.

Это задача 11 лабы. Клауза `WINDOW` особенно окупается, когда 3–4 функции делят
окно, — код становится короче и читаемее.

## Свод практических паттернов лабы

Всё ниже — из этой и прошлых лаб, без единой новой конструкции. Держите как шпаргалку.

### Нарастающий итог (running total) — `ROWS`

```sql
sum(revenue) OVER (PARTITION BY product_id ORDER BY sale_date
                   ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
```
Сумма с начала раздела по текущую строку; `PARTITION BY` перезапускает накопление.
Явный `ROWS` защищает от ловушки дублей ключа ([файл 02](02-rows-range-groups.md)).
Задачи 1, 2, 14.

### Скользящее среднее / сумма за N точек — `ROWS`

```sql
avg(revenue) OVER (... ORDER BY sale_date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)
```
Среднее по текущей и двум предыдущим строкам — сглаживание тренда. Центрированный
вариант `ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING` (при желании с `EXCLUDE CURRENT
ROW` — среднее по соседям) ловит выбросы. Задачи 3, 10.

### Скользящее окно за период — `RANGE` со смещением

```sql
sum(revenue) OVER (... ORDER BY sale_date
                   RANGE BETWEEN INTERVAL '2 days' PRECEDING AND CURRENT ROW)
```
«За последние N календарных дней/единиц» — учитывает разрывы в ряду, в отличие от
`ROWS`. Задача 4.

### Дельта и рост % к прошлому периоду — `LAG`

```sql
revenue - lag(revenue) OVER w                         AS дельта
round(100.0*(revenue - lag(revenue) OVER w)/lag(revenue) OVER w, 1) AS рост_проц
```
Сравнение с предыдущей строкой; `lead` — со следующей. Помните про `NULL` на краю и
деление на ноль. Задачи 5, 6, 7, 13.

### Первое/последнее/N-е значение раздела — `FIRST_VALUE`/`LAST_VALUE`/`NTH_VALUE`

```sql
first_value(revenue) OVER w                              -- открытие (рамка по умолч. ок)
last_value (revenue) OVER (... ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING)
```
Для `last_value`/`nth_value` — **обязательно** полная рамка, иначе ловушка «до текущей
строки» ([файл 04](04-first-last-nth.md)). Задачи 8, 9.

### Доля от максимума/лидера в окне — агрегат как окно

```sql
round(100.0 * revenue / max(revenue) OVER (PARTITION BY product_id), 1)
```
Окно **без `ORDER BY`** = весь раздел, поэтому `max` — рекорд товара; делим день на
пик. Родня долям из [lab010](../../lab010/). Задача 12.

### Отбор по результату окна — обёртка CTE (из lab010)

Окно нельзя писать в `WHERE`. Считаем его в CTE, фильтруем снаружи: так решаются
«день максимального скачка» (`row_number` по дельте) и «день, когда набрали половину
итога» (`row_number` по дате среди строк, где нарастающий ≥ полусуммы). Задачи 13, 14.
