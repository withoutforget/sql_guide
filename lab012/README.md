# lab012 — Аналитические паттерны: pivot, когорты, retention, воронки, RFM, ABC

Двенадцатая лаба — **капстоун модуля C «Агрегация и аналитика»**. Она **не вводит
нового синтаксиса**: всё, что нужно, уже разобрано в lab001–lab011. Задача лабы —
научиться **собирать** знакомые кирпичи в реальные аналитические отчёты, которые
каждый день делают продуктовые и маркетинговые аналитики:

> **сводные таблицы (pivot)**, **когортный анализ**, **retention (удержание)**,
> **воронки конверсии**, **RFM-сегментация** и **доли / ABC-анализ (Парето 80/20)**.

Опираемся на весь пройденный инструментарий: агрегаты и `FILTER`
([lab001](../lab001/)), `JOIN` ([lab002](../lab002/)), подзапросы
([lab004](../lab004/)), CTE ([lab005](../lab005/)), `generate_series`/`VALUES`/
`LATERAL` ([lab006](../lab006/)), `GROUPING SETS`/`ROLLUP` ([lab008](../lab008/)),
и — центрально — **оконные функции**: разделы, ранги, `NTILE`
([lab010](../lab010/)), рамки, `LAG`, running total ([lab011](../lab011/)).

Теория разбита на короткие документы в папке [`theory/`](theory/) — читайте по
порядку. Общая подборка курсов и тренажёров — в [`../RESOURCES.md`](../RESOURCES.md).

## Доменная область

Интернет-магазин + лог воронки на сайте. Пять таблиц (полная схема и подробная шапка
о том, как подобраны данные, — в [`init.sql`](init.sql)).

> **Главное про даты.** Функции работы с датами (`extract`/`date_trunc`/…) — тема
> [lab014](../lab014/), здесь их нет. Поэтому «период» (месяц) задан **целым числом**
> в `orders.period` (1..6 = янв..июн 2024), а ярлыки лежат в календаре `periods`. Вся
> когортная арифметика («возраст = период_активности − период_когорты») делается
> вычитанием целых. В реальной жизни период извлекают из даты функциями из lab014.

### `periods` — календарь периодов
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `period` | INT (PK) | номер месяца 1..6 |
| `label` | TEXT | ярлык (`2024-01` … `2024-06`) |

### `customers` — клиенты
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор |
| `name` | TEXT | имя |
| `city` | TEXT | город |

### `orders` — покупки (главная таблица; одна строка = одна покупка)
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор |
| `customer_id` | INT → `customers` | клиент |
| `period` | INT → `periods` | **месяц покупки, целым числом** (ось времени) |
| `category` | TEXT | `Электроника` / `Книги` / `Дом` |
| `amount` | NUMERIC | сумма покупки, ₽ |

### `funnel_steps` — справочник шагов воронки
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `step` | INT (PK) | номер шага 1..5 |
| `step_name` | TEXT | Визит / Просмотр / Корзина / Оформление / Оплата |

### `funnel_events` — лог воронки (одна строка = одно событие посетителя)
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор |
| `visitor_id` | INT | посетитель сайта (**отдельная популяция**, не `customers`) |
| `step` | INT → `funnel_steps` | какой шаг |
| `seq` | INT | порядок события у посетителя (суррогат времени) |

> **Как подобраны данные (чтобы паттерны были содержательны):**
> - **Pivot.** В сводной (категория × период) есть **пустые комбинации** → там `0`:
>   «Дом» не продавался в феврале и июне, «Книги» — в апреле и июне. Итог ровно
>   **1 000 000 ₽** (Электроника 690k, Дом 191k, Книги 119k).
> - **Когорты** по первой покупке: три когорты размером **5 / 3 / 4** (старт в
>   периодах 1 / 2 / 3).
> - **Retention с «дырками».** У клиентов разные рисунки активности: **верные**
>   (активны каждый период — Анна, Егор, Игорь), **отвалившиеся** (только когортный
>   период — Вера, Зоя, Лев) и **с дырками** (пропустили период и вернулись — Борис
>   {1,3}, Жанна {2,4}, Мила {3,5}, Дарья {1,4}). Из-за дырок **classic и rolling
>   retention различаются** (задачи 6–7). Матрица **треугольная**: поздние когорты
>   ещё не прожили поздние возрасты → там `NULL`, не `0`.
> - **Воронка** монотонно убывает: **10 → 8 → 5 → 3 → 2**. Ровно одна **аномалия
>   порядка**: посетитель 4 положил в корзину раньше, чем посмотрел товар → строгая
>   воронка на шаге «Корзина» даёт 4, а не 5 (задачи 8–9).
> - **RFM / ABC.** Суммы **перекошены** (принцип Парето): 3 «кита» (Анна 300k, Егор
>   250k, Игорь 200k) дают 75% выручки. Границы классов ABC (80% и 95%) каждый раз
>   **пересекает ровно один клиент** (Глеб — 80%, Дарья — 95%) — удобно для разбора
>   «в какой класс попадает пограничный клиент». Все суммы различны (нет ничьих в M).

> **Про хардкор.** Задача 9 собирает **пивот** (собрать `seq` шагов в колонки через
> `FILTER`), проверку **возрастания цепочки** событий и **unpivot** (`LATERAL VALUES`)
> — чтобы засчитывать шаг воронки только при правильном порядке во времени. Задача 14
> совмещает `NTILE(5)`-скоры с многоветочным `CASE` (важен **порядок веток** — от
> частного к общему), назначая клиентам человекочитаемые сегменты. Обе — ровно на
> материале lab001–lab012, без новых конструкций. Матрица retention (задачи 6–7) тоже
> нетривиальна: следите за `DISTINCT`-активностью, знаменателем-размером когорты и
> различием `NULL`/`0`.
