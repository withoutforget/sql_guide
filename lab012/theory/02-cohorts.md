## Что такое когорта

**Когорта** — это группа пользователей, объединённых **моментом первого события**.
Самые частые определения когорты:

- **месяц регистрации** («все, кто зарегистрировался в январе»);
- **месяц первой покупки** («все, кто впервые купил в марте»);
- иногда — канал привлечения, тариф и т.п.

Зачем это нужно: чтобы **сравнивать сопоставимое**. Клиенты, пришедшие в январе, и
клиенты, пришедшие в июне, находятся на разных стадиях «жизни». Если смешать их в
одну кучу, метрики («сколько активны в этом месяце») будут врать: июньские физически
не могли прожить полгода. Когорты фиксируют «точку старта» и позволяют смотреть, как
каждая группа ведёт себя **с одинакового момента** — это фундамент retention (файл
03).

## Приём с периодом: почему период задан числом, а не датой

В реальных данных момент события — это **дата** (`registered`, `ordered_at`), и
«месяц первой покупки» вычисляют **функциями работы с датами**: `date_trunc('month',
ordered_at)` или `extract(...)`. Но эти функции — тема **[lab014](../../lab014/)**,
они ещё не разобраны.

Чтобы честно обойтись **без** датных функций, в этой лабе **период задан целым
числом** — колонка `orders.period` (1..6 = янв..июн 2024), а ярлыки месяцев лежат в
таблице-календаре `periods(period, label)`. Это стандартный приём: хранить заранее
вычисленную «периодность» рядом с фактом. Тогда:

- «когорта клиента» = **`min(period)`** по его заказам — обычный агрегат, без дат;
- «возраст периода» (файл 03) = **разность целых** `period − cohort_period`;
- вся когортная арифметика сводится к сложению/вычитанию целых.

> В проде период чаще всего **извлекают из даты** функциями из lab014 (или заводят
> отдельную таблицу-календарь и джойнят по ней). Здесь он просто задан колонкой —
> чтобы сосредоточиться на **паттерне**, а не на возне с датами.

## Как определить когорту каждого клиента

Когорта клиента = период его **первого** события. Два равнозначных способа.

### Способ 1: агрегат `min(period)` + `GROUP BY` (в CTE)

```sql
WITH first_purchase AS (
    SELECT customer_id, min(period) AS cohort_period
    FROM orders
    GROUP BY customer_id
)
SELECT c.name, fp.cohort_period
FROM first_purchase fp
JOIN customers c ON c.id = fp.customer_id
ORDER BY fp.cohort_period, c.id;
```

`min(period)` берёт **самый ранний** период, в котором клиент что-то купил, — это и
есть период его когорты. Дальше присоединяем имя и (при желании) ярлык месяца из
`periods`. Это **задача 4**. Результат на данных лабы:

```
клиент | когорта_период | когорта_месяц
-------+----------------+--------------
Анна   |       1        | 2024-01        ┐
Борис  |       1        | 2024-01        │ когорта 1 — 5 клиентов
...    |      ...       |   ...          ┘
Егор   |       2        | 2024-02        ← когорта 2 — 3 клиента
Игорь  |       3        | 2024-03        ← когорта 3 — 4 клиента
```

### Способ 2: `min(period) OVER (PARTITION BY customer_id)` (оконно)

Тот же смысл, но **без схлопывания строк** — когорта приписывается **каждому**
заказу клиента (пригодится, когда в одном запросе нужны и детальные строки, и
когорта рядом):

```sql
SELECT customer_id, period, amount,
       min(period) OVER (PARTITION BY customer_id) AS cohort_period
FROM orders;
```

Это ровно агрегат-как-окно из [lab010](../../lab010/theory/05-aggregate-windows.md):
`min` считается по всему разделу клиента и «раздаётся» всем его строкам. Когда нужна
**одна строка на клиента** — берите способ 1 (`GROUP BY`); когда когорта нужна
**рядом с каждым фактом** — способ 2 (`OVER`).

## Размеры когорт — «знаменатель» всей аналитики

Как только когорта клиента известна, **размер когорты** (сколько в ней клиентов) —
это `count` по когортам:

```sql
WITH first_purchase AS (
    SELECT customer_id, min(period) AS cohort_period
    FROM orders GROUP BY customer_id
)
SELECT cohort_period, count(*) AS клиентов
FROM first_purchase
GROUP BY cohort_period
ORDER BY cohort_period;
```
```
cohort_period | клиентов
--------------+---------
      1       |    5
      2       |    3
      3       |    4
```

Это **задача 5**. Размер когорты — **знаменатель** для retention: «доля вернувшихся»
= `вернулись / размер когорты`. Ошибиться в знаменателе (например, взять число
клиентов, а не размер **конкретной** когорты) — типичная причина неверных процентов
удержания. Поэтому размеры когорт считают **отдельным, проверяемым шагом**.

## Мысленная модель когортного отчёта (задел на файл 03)

Почти любой когортный отчёт строится по одной схеме из трёх шагов:

1. **На каждого клиента находим когорту** — `min(period)` (этот файл).
2. **Нумеруем периоды активности относительно старта** — `возраст = period −
   cohort_period` (файл 03).
3. **Сводим (pivot)**: строки — когорты, столбцы — возраст, значение — доля/число
   активных (файлы 01 и 03).

Держите эту тройку в голове — она превращает «страшный когортный анализ» в
последовательность знакомых операций.

## Подводные камни

- **Что считать «первым событием».** Когорта по регистрации и когорта по первой
  покупке — **разные** когорты (клиент мог зарегистрироваться в январе, а купить
  впервые в марте). Всегда фиксируйте, **какое** событие определяет старт. Здесь —
  первая **покупка** (`min(period)` по `orders`).
- **Клиент без событий** не попадёт ни в одну когорту (у него нет `min(period)`).
  В наших данных таких нет — все 12 клиентов сделали хотя бы одну покупку.
- **Возвраты/отмены.** Если бы в данных были отрицательные суммы (возвраты), «первую
  покупку» считали бы только по «настоящим» заказам (`WHERE amount > 0`). У нас
  возвратов нет.
