# 01 — Сводные таблицы: pivot (long → wide) и unpivot (wide → long)

[← оглавление](../README.md) · [дальше: когорты →](02-cohorts.md)

---

Эта лаба — **капстоун** аналитики. Мы не вводим ни одной новой языковой
конструкции: всё, что нужно — агрегаты, `CASE`, `FILTER`, оконные функции,
`NTILE`, `LAG`, CTE, `JOIN`, `ROLLUP`, `generate_series`/`VALUES` — уже разобрано
в lab001–lab011. Здесь мы учимся **собирать** эти кирпичи в реальные отчёты,
которые каждый день делают продуктовые и маркетинговые аналитики. Первый и самый
базовый отчёт — **сводная таблица** (pivot).

## Бизнес-вопрос

«Покажи выручку по **категориям** и **месяцам** — категории в строках, месяцы в
столбцах». Данные лежат в «длинном» (long) виде — одна строка на факт:

```
category    | period | amount
------------+--------+--------
Электроника |   1    | 50000
Книги       |   1    | 20000
Дом         |   1    |  5000
Электроника |   2    | 60000
...
```

А отчёт нужен в «широком» (wide) виде — измерение `period` разложено **по
столбцам**:

```
категория   |  янв   |  фев   |  мар   | ...
------------+--------+--------+--------+----
Электроника | 50000  | 155000 | 130000 | ...
Книги       | 37000  |  23000 |  17000 | ...
Дом         | 30000  |      0 |  48000 | ...
```

Превращение long → wide и называется **pivot** (сведение, «поворот»).

## Приём: условная агрегация

Идея простая и укладывается в две мысли:

1. **Строки** сводной задаёт `GROUP BY` по строковому измерению (`category`).
2. **Каждый столбец** — это отдельный агрегат, который считает **только по
   строкам нужного месяца**. «Только по нужным строкам» — это ровно то, что умеет
   `FILTER (WHERE ...)` из [lab001](../../lab001/theory/05-aggregates.md):

```sql
SELECT
    category                                      AS категория,
    SUM(amount) FILTER (WHERE period = 1)         AS янв,
    SUM(amount) FILTER (WHERE period = 2)         AS фев,
    SUM(amount) FILTER (WHERE period = 3)         AS мар
FROM orders
GROUP BY category;
```

Мысленная модель: в каждой группе (`category`) мы кладём рядом **несколько
агрегатов**, и каждый «пропускает через себя» только свою подгруппу строк
(`period = k`). Один проход по группе — сразу все столбцы.

### `FILTER` или `CASE`?

`FILTER` — современный и читаемый способ (PostgreSQL). Классический, переносимый в
любую СУБД вариант — **условие внутри агрегата через `CASE`**:

```sql
-- эквивалент SUM(amount) FILTER (WHERE period = 1):
SUM(CASE WHEN period = 1 THEN amount END) AS янв
```

`CASE` без `ELSE` возвращает `NULL` для «чужих» строк, а `SUM` игнорирует `NULL`
(см. [lab001](../../lab001/theory/05-aggregates.md)) — результат тот же. Для
**подсчёта** пишут `COUNT(*) FILTER (WHERE ...)` либо
`SUM(CASE WHEN ... THEN 1 ELSE 0 END)`. В этой лабе используем `FILTER` — он был
разобран и читается лучше; но помните про `CASE`-форму как про универсальную.

## Подводный камень №1: пустая комбинация — это `NULL`, а не `0`

Что вернёт `SUM(amount) FILTER (WHERE period = 2)` для категории «Дом», если в
феврале «Дом» **вообще не продавался**? Множество строк под фильтром пусто, а
`SUM` по пустому множеству — это **`NULL`, не 0** (важнейшее правило агрегатов из
lab001). В отчёте это выглядит как дырка:

```
категория   | фев
------------+------
Дом         | NULL   ← продаж не было
```

Чаще всего в сводной хотят увидеть **`0`**, а не пусто. Лечится обёрткой
`COALESCE(..., 0)`:

```sql
COALESCE(SUM(amount) FILTER (WHERE period = 2), 0) AS фев
```

> **`0` или `NULL` — осознанный выбор.** `0` говорит «в этой ячейке точно ноль
> рублей». `NULL` говорит «данных нет / не применимо». Для выручки уместнее `0`
> (продаж не было = 0 ₽). Но, например, в матрице retention (файл 03) `NULL` и `0`
> означают **разное** и путать их нельзя. Всегда думайте, что честнее для конкретной
> ячейки. Это **задача 1**: там пустые пары (Дом×фев, Книги×апр и т.д.) дают `0`.

## Подводный камень №2: число столбцов **фиксировано** в тексте запроса

В чистом SQL сводная таблица **не может** сама «подстроить» число столбцов под
данные: сколько месяцев — вы перечисляете руками (`FILTER (WHERE period = 1)`, `= 2`,
…). Появился седьмой месяц — надо дописать столбец в запрос. Это называется
**статический pivot**, и это неотъемлемое свойство: `SELECT` обязан знать список
колонок на этапе разбора запроса, **до** того как увидит данные.

- **«Динамический pivot»** (число столбцов зависит от данных) в чистом SQL
  **невозможен**. Его делают либо в приложении/BI-инструменте (сначала запросом
  узнают список значений, потом генерируют SQL со столбцами), либо расширением
  `tablefunc` с функцией `crosstab(...)`. **Знать, что `crosstab` существует,
  полезно**, но: он требует установки расширения (`CREATE EXTENSION tablefunc`),
  требует заранее описать выходные столбцы и синтаксически неудобен. В этой лабе мы
  его **не используем** — это отдельная тема. Наш инструмент — `FILTER`/`CASE`.

## Итоги в сводной: строка и столбец «Всего»

К сводной почти всегда хотят **итоги**:

- **Итог по строке** (сумма за все месяцы) — просто ещё одна колонка `SUM(amount)`
  (без фильтра, по всей группе):

```sql
..., SUM(amount) AS итого_по_категории
```

- **Итоговая строка** (по всем категориям) — это уже группировка «на уровень выше».
  Здесь пригождается `ROLLUP`/`GROUPING SETS` из
  [lab008](../../lab008/theory/03-rollup-cube.md): добавив к `GROUP BY` набор `()`,
  получаем дополнительную строку-итог, где строковое измерение свёрнуто в `NULL`:

```sql
SELECT
    COALESCE(pr.label, 'ВСЕГО')                          AS месяц,
    COALESCE(SUM(o.amount) FILTER (WHERE o.category='Электроника'),0) AS электроника,
    COALESCE(SUM(o.amount) FILTER (WHERE o.category='Книги'),0)       AS книги,
    SUM(o.amount)                                        AS итого_за_месяц
FROM orders o JOIN periods pr ON pr.period = o.period
GROUP BY GROUPING SETS ((o.period, pr.label), ())        -- детально + общий итог
ORDER BY o.period NULLS LAST;
```

Строка `()` даёт последнюю строку «ВСЕГО», где `period IS NULL` → подписываем её
через `COALESCE(pr.label, 'ВСЕГО')`. Это **задача 3**. Совмещение «пивот столбцами +
`GROUPING SETS` строками» — классический приём, и он целиком собран из уже
знакомого материала.

## Обратная операция: unpivot (wide → long)

Часто нужно **обратное** — «широкую» таблицу (столбец на месяц) развернуть в
«длинную» (строка на месяц). Например, план продаж пришёл из Excel широким, а для
`JOIN`/`GROUP BY` нужен длинный вид. Столбцов у результата нет «динамических» —
превращаем **колонки в строки**.

Самый чистый способ в PostgreSQL — присоединить к каждой широкой строке **набор пар
`(ярлык, значение)`** через `LATERAL` + `VALUES` (обе конструкции из
[lab006](../../lab006/theory/03-values-as-table.md)):

```sql
WITH wide(категория, янв, фев, мар) AS (
    VALUES ('Электроника', 50000, 155000, 130000),
           ('Дом',         30000,   NULL,  48000)   -- в феврале не продавался
)
SELECT w.категория, u.месяц, u.сумма
FROM wide AS w
CROSS JOIN LATERAL (
    VALUES ('янв', 1, w.янв), ('фев', 2, w.фев), ('мар', 3, w.мар)
) AS u(месяц, ном, сумма)
WHERE u.сумма IS NOT NULL          -- выкидываем «дырки» (нет продаж)
ORDER BY w.категория, u.ном;
```
```
 категория   | месяц | сумма
-------------+-------+--------
 Дом         | янв   |  30000
 Дом         | мар   |  48000       ← строка «Дом/фев» отброшена: там был NULL
 Электроника | янв   |  50000
 Электроника | фев   | 155000
 Электроника | мар   | 130000
```

Разбор: `LATERAL` позволяет во внутреннем `VALUES` **ссылаться на колонки текущей
строки** `w` (`w.янв`, `w.фев`, …). На каждую широкую строку он выдаёт 3 узкие пары;
`CROSS JOIN` их «размножает». `WHERE u.сумма IS NOT NULL` убирает пустые ячейки,
чтобы в длинной таблице не было бессмысленных строк. Столбец `ном` (1/2/3) добавлен,
чтобы отсортировать месяцы **хронологически** (иначе `ORDER BY месяц` дал бы
алфавитный порядок «мар, фев, янв»). Это **задача 2**.

Альтернативы unpivot: `UNION ALL` из нескольких `SELECT` (по одному на столбец —
многословно), либо `unnest` двух массивов через `WITH ORDINALITY` (см.
[lab006](../../lab006/theory/02-unnest-ordinality-srf.md)). `LATERAL`+`VALUES`
компактнее всего.

## Итог файла

| Операция | Направление | Инструмент |
|----------|-------------|------------|
| **pivot** | long → wide | `GROUP BY строка` + `agg(...) FILTER (WHERE столбец=k)` на каждую колонку |
| пустая ячейка pivot | — | `COALESCE(agg, 0)` (если нужен 0, а не NULL) |
| итоги в pivot | — | `SUM(...)` (по строке) + `GROUPING SETS`/`ROLLUP` (итоговая строка) |
| **unpivot** | wide → long | `CROSS JOIN LATERAL (VALUES (ярлык, значение), ...)` + `WHERE ... IS NOT NULL` |

Число столбцов сводной всегда фиксировано в запросе — «динамический pivot» живёт вне
чистого SQL.
