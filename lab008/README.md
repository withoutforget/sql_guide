# lab008 — Расширенная группировка (`GROUPING SETS` / `ROLLUP` / `CUBE`)

Восьмая лаба открывает **модуль C — «Агрегация и аналитика»**. Обычный `GROUP BY`
из [lab001](../lab001/) даёт **один** уровень группировки. Но в настоящем отчёте
нужны сразу **детализация**, **промежуточные итоги** (subtotals) и **общий итог**
(grand total) — в одном результате. Раньше это собирали громоздким `UNION ALL`
нескольких `GROUP BY` (приём из [lab003](../lab003/)); расширенная группировка
делает то же самое **короче и за один проход по данным**.

В этой лабе разберём три конструкции — `GROUPING SETS` (явный список наборов),
`ROLLUP` (иерархические подытоги), `CUBE` (все комбинации), — научимся с помощью
функции `GROUPING()` отличать «итоговый `NULL`» от настоящего `NULL` в данных и
оформлять аккуратные отчёты с подписями и правильной сортировкой итогов.

Теория разбита на короткие документы в папке [`theory/`](theory/) — читайте по
порядку. Общая подборка курсов и тренажёров — в [`../RESOURCES.md`](../RESOURCES.md).

## Как запустить

```bash
# из корня репозитория один раз:
docker compose up -d

cd lab008
./run.sh                 # накатит схему и прогонит все запросы

# либо зайти руками и экспериментировать:
../psql.sh lab008
```

`run.sh` идемпотентен: пересоздаёт схему `lab008` с нуля при каждом запуске.

> Про `NULL` в выводе. Строки-итоги содержат `NULL` в свёрнутых колонках, а `psql`
> по умолчанию печатает `NULL` **пустотой**. В теории для наглядности `NULL`
> показан значком `∅` (включается командой `\pset null '∅'`); при обычном
> `./run.sh` такие ячейки будут просто пустыми. Именно поэтому в отчётах итоги
> подписывают через `GROUPING()` — см. теорию 04–05.

## Доменная область

Тот же интернет-магазин, но взгляд «аналитика»: одна денормализованная **витрина
продаж** `sales` — факт-таблица с несколькими **измерениями** сразу. Именно много
измерений и нужно, чтобы строить многомерные итоги. Полная схема и данные — в
[`init.sql`](init.sql).

### `sales` — продажи (факт-таблица витрины)
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор строки |
| `category` | TEXT | категория: `Электроника`, `Книги`, `Дом` |
| `subcategory` | TEXT | подкатегория (входит в категорию) — иерархия для `ROLLUP` |
| `region` | TEXT | регион продажи; **может быть `NULL`** («регион не указан») |
| `channel` | TEXT | канал: `online` / `offline` |
| `amount` | NUMERIC | выручка, ₽ (мера) |
| `qty` | INT | количество, шт. (вторая мера) |

> Данные (12 строк) подобраны так, чтобы **общий итог по выручке был круглым —
> ровно `1 000 000 ₽`**, и подытоги легко проверялись вручную:
> - по категориям: `Электроника 600k`, `Книги 150k`, `Дом 250k`;
> - иерархия для `ROLLUP`: `category → subcategory` (напр. `Электроника` =
>   `Смартфоны 500k` + `Наушники 100k`);
> - два независимых измерения для `CUBE`: `category × channel`;
> - по каналам: `online 600k`, `offline 400k`;
> - **важно:** в двух строках `region IS NULL` (регион не указан) — это
>   **настоящий** `NULL` в данных. По регионам: `Москва 590k`,
>   `Санкт-Петербург 340k`, `(не указан) 70k`. Он нужен, чтобы показать, зачем
>   `GROUPING()` отличает его от «итогового» `NULL`.

> **Про хардкор.** Задача 13 — полноценная витрина: `ROLLUP (region, channel)` +
> доля каждой строки от общего итога **скалярным подзапросом** (lab004, потому что
> оконные функции — только в lab010) + три разных смысла `NULL` в колонке
> `region`, разведённые маской `GROUPING()` и `CASE` в правильном порядке ветвей.
> Изюм — не дать строке общего итога и строке «регион не указан» слиться.
> Задача 14 — понять, что нужный набор подытогов (`пары категория×канал`,
> `отдельно регионы`, `общий итог`) **не выражается** ни `ROLLUP`, ни `CUBE`
> (первый дал бы иерархию префиксов, второй — все 8 наборов): только явный
> `GROUPING SETS`. Плюс — пометить «секцию» каждой строки по маске `GROUPING`.
> Обе задачи решаются ровно материалом этой лабы и lab001–lab007.

## Что дальше

`GROUPING SETS`/`ROLLUP`/`CUBE` управляют тем, **по каким разрезам** считать итоги.
В `lab009` — про то, **что именно** считать внутри группы за
пределами `SUM`/`AVG`/`COUNT`: **статистические и сортирующие агрегаты** —
`stddev`/`variance`/`corr`, перцентили и медиана (`percentile_cont`/`_disc` через
`WITHIN GROUP`), `mode`, а также сборка значений группы в строку/массив
(`string_agg`/`array_agg` с сортировкой). Любой из них можно считать на каждом
уровне сегодняшних итогов.

А ещё дальше, в `lab010`, — **оконные функции**: те же итоги и доли, но **без
схлопывания** строк, рядом с каждой детальной записью.
