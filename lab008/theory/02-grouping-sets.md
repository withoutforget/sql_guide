# 02 — `GROUPING SETS`: явный список наборов группировки

[← зачем это нужно](01-why-extended-grouping.md) · [оглавление](../README.md) · [дальше: ROLLUP и CUBE →](03-rollup-cube.md)

---

`GROUPING SETS` — самая общая и самая явная форма расширенной группировки. Ты
буквально **перечисляешь наборы группировки**, а СУБД считает каждый и складывает
результаты вместе.

## Синтаксис

```sql
SELECT ...
FROM ...
GROUP BY GROUPING SETS ( набор1, набор2, ..., наборN );
```

Каждый **набор** (grouping set) — это список колонок в скобках, по которым
группировать на этом уровне:

- `(category)` — группировать по категории;
- `(category, channel)` — группировать по паре (категория, канал);
- `()` — **пустой набор**: не группировать ни по чему = один общий итог по всей
  таблице.

Наборы перечисляются через запятую внутри `GROUPING SETS ( ... )`. Каждый набор
даёт свой блок строк в результате, как отдельный `GROUP BY`.

## Первый пример: категории + общий итог

```sql
SELECT category, SUM(amount) AS revenue
FROM sales
GROUP BY GROUPING SETS ((category), ())
ORDER BY revenue;
```
```
  category   |  revenue
-------------+------------
 Книги       |  150000.00
 Дом         |  250000.00
 Электроника |  600000.00
 ∅           | 1000000.00
(4 rows)
```

Здесь два набора:

- `(category)` дал три строки — по одной на категорию (как обычный `GROUP BY
  category`);
- `()` дал одну строку — общий итог `1 000 000` по всей таблице.

> **Про значок `∅`.** В строке общего итога колонка `category` — `NULL`. По
> умолчанию `psql` печатает `NULL` **пустотой** (ничего), что для этой темы
> неудобно: не видно, где итог. Поэтому в примерах теории я включил показ `NULL`
> как `∅` командой `\pset null '∅'`. Запомни: **`∅` в колонке измерения = эта
> колонка на данном уровне «свёрнута» (по ней считается итог).** Почему именно
> `NULL` и как надёжно отличать его от настоящего `NULL` в данных — целиком
> [файл 04](04-grouping-function-nulls.md).

## Ключевая эквивалентность: `GROUPING SETS` = `UNION ALL` нескольких `GROUP BY`

Это самое важное для понимания. Тот же результат «руками» через `UNION ALL`
(способ из [lab003](../../lab003/theory/02-union-union-all.md)) выглядит так:

```sql
SELECT category, SUM(amount) AS revenue      -- набор (category)
FROM sales
GROUP BY category
UNION ALL
SELECT NULL, SUM(amount)                      -- набор ()  — общий итог
FROM sales
ORDER BY revenue;
```

Оба запроса дают **побайтово одинаковый** результат (4 строки: три категории +
итог с `NULL` в `category`). То есть:

> `GROUP BY GROUPING SETS ((category), ())`
> — это в точности
> `GROUP BY category` **`UNION ALL`** `GROUP BY ()`.

Отсюда правило перевода в обе стороны: **каждый набор в `GROUPING SETS`
соответствует одному `GROUP BY ... UNION ALL`**. Разница — только в записи и в
цене: `GROUPING SETS` СУБД считает за **один проход** по таблице, а `UNION ALL` —
за столько проходов, сколько веток.

Проверить равенство двух наборов строк можно, кстати, оператором `EXCEPT` из
lab003 (если `A EXCEPT B` и `B EXCEPT A` пусты — наборы совпадают); это отдельная
задача в этой лабе.

## Несколько разрезов в одном запросе

Настоящая сила `GROUPING SETS` — когда наборов много и они **по разным
колонкам**. Одним запросом посчитаем выручку по категориям, по каналам, по
регионам и общий итог:

```sql
SELECT category, channel, region, SUM(amount) AS revenue
FROM sales
GROUP BY GROUPING SETS ((category), (channel), (region), ())
ORDER BY revenue;
```
```
  category   | channel |     region      |  revenue
-------------+---------+-----------------+------------
 ∅           | ∅       | ∅               |   70000.00
 Книги       | ∅       | ∅               |  150000.00
 Дом         | ∅       | ∅               |  250000.00
 ∅           | ∅       | Санкт-Петербург |  340000.00
 ∅           | offline | ∅               |  400000.00
 ∅           | ∅       | Москва          |  590000.00
 ∅           | online  | ∅               |  600000.00
 Электроника | ∅       | ∅               |  600000.00
 ∅           | ∅       | ∅               | 1000000.00
(9 rows)
```

Разбор построчно (обрати внимание, что в каждой строке **не-`∅`** ровно те
колонки, по которым считался её набор):

- `(category)` → 3 строки: `Книги 150k`, `Дом 250k`, `Электроника 600k`;
- `(channel)` → 2 строки: `offline 400k`, `online 600k`;
- `(region)` → 3 строки: `∅ 70k` (регион не указан!), `Санкт-Петербург 340k`,
  `Москва 590k`;
- `()` → 1 строка: общий итог `1 000 000`.

Итого 3 + 2 + 3 + 1 = **9 строк** — именно столько и вернулось. То же самое через
`UNION ALL` заняло бы **четыре** отдельных `SELECT ... GROUP BY` и четыре прохода
по таблице.

⚠️ Тонкость этого примера: в наборе `(region)` есть строка `∅ | ∅ | ∅ | 70000` —
это **не** общий итог, а подытог по строкам, где регион **реально не указан**
(`region IS NULL` в данных). А общий итог — это самая нижняя `∅ | ∅ | ∅ |
1000000`. Две внешне одинаковые (`∅ ∅ ∅`) строки означают **совершенно разное**.
Пока просто заметь проблему — решаем её в [файле 04](04-grouping-function-nulls.md).

## Набор из нескольких колонок: `(a, b)`

Набор может содержать не одну колонку, а несколько — тогда это обычная
группировка по их **комбинации**. Набор `(category, subcategory)` — это ровно
`GROUP BY category, subcategory`:

```sql
SELECT category, subcategory, SUM(amount) AS revenue
FROM sales
GROUP BY GROUPING SETS ((category, subcategory))
ORDER BY category, subcategory;
```
```
  category   | subcategory |  revenue
-------------+-------------+-----------
 Дом         | Кухня       | 150000.00
 Дом         | Освещение   | 100000.00
 Книги       | Детская     |  70000.00
 Книги       | Учебная     |  80000.00
 Электроника | Наушники    | 100000.00
 Электроника | Смартфоны   | 500000.00
(6 rows)
```

`GROUPING SETS` **с единственным набором** = обычный `GROUP BY` по этому набору.
Ничего нового — это просто крайний случай. Важно **не путать вложенность
скобок**:

| Запись | Смысл | Сколько уровней |
|--------|-------|-----------------|
| `GROUPING SETS ((a, b))` | один набор по паре `(a,b)` | 1 (как `GROUP BY a, b`) |
| `GROUPING SETS ((a), (b))` | два набора: по `a` и отдельно по `b` | 2 |

Одна пара скобок разделяет наборы, вложенные скобки объединяют колонки в один
набор. Перепутаешь — получишь совсем другой отчёт.

## Пустой набор `()` — это общий итог

Стоит запомнить отдельно: `()` — это набор **без единой колонки**. Группировать
не по чему → вся таблица становится одной группой → одна строка с итогом по
всему. Это законный самостоятельный `GROUP BY`:

```sql
SELECT SUM(amount) AS revenue FROM sales GROUP BY ();
```
```
  revenue
------------
 1000000.00
(1 row)
```

`GROUP BY ()` эквивалентен запросу вообще без `GROUP BY` (агрегат по всей
таблице). Внутри `GROUPING SETS` пустой набор `()` — это и есть способ попросить
строку общего итога.

## Дубликаты наборов и пустые результаты

Пара практических деталей:

- **Повторяющиеся наборы не схлопываются.** Если один и тот же набор указан
  дважды (`GROUPING SETS ((category), (category))`), его строки появятся **два
  раза** — `GROUPING SETS` не делает дедуп (он ведёт себя как `UNION ALL`, а не
  `UNION`). Обычно повтор — это случайная ошибка.
- **Порядок наборов** в `GROUPING SETS` не влияет на *содержимое* результата,
  только потенциально на порядок строк без `ORDER BY`. Как всегда — если нужен
  предсказуемый порядок, ставь `ORDER BY` (об аккуратной сортировке отчётов с
  итогами — [файл 05](05-report-formatting-sorting.md)).

## Когда использовать `GROUPING SETS` напрямую

`ROLLUP` и `CUBE` (следующий файл) — удобные сокращения, но они задают
**фиксированные** наборы (иерархию или полный куб). Когда нужен **произвольный,
нестандартный** список подытогов, которого ни `ROLLUP`, ни `CUBE` не дают
(например, «только по `(category, channel)`, по `(region)` и общий итог — и
ничего лишнего»), — пишут именно явный `GROUPING SETS`. Это тема одной из
хардкорных задач лабы.
