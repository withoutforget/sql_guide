# 06 — `WHERE`, `HAVING`, фильтрация по уровню и подводные камни

[← оформление и сортировка](05-report-formatting-sorting.md) · [оглавление](../README.md)

---

Последний файл: как с расширенной группировкой работают фильтры (`WHERE` и
`HAVING`), как оставить строки только нужного уровня итога, и чек-лист ловушек.

## `WHERE` — фильтрует **детали до** группировки (и меняет все итоги!)

`WHERE` в запросе с `ROLLUP`/`CUBE`/`GROUPING SETS` работает точно так же, как в
обычном `GROUP BY` (порядок исполнения из [lab001](../../lab001/theory/06-group-having.md):
`FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY`): он отсеивает **отдельные
строки** таблицы **до** группировки. Агрегаты в `WHERE` нельзя, `GROUPING()` в
`WHERE` тоже нельзя (строк-итогов ещё не существует).

Важное следствие: раз `WHERE` убирает часть детальных строк **до** подсчёта, то и
**все подытоги, и общий итог считаются только по оставшимся данным**. Отфильтруем
только онлайн-продажи:

```sql
SELECT
    CASE WHEN GROUPING(category)=1 THEN 'ИТОГО (online)' ELSE category END AS категория,
    SUM(amount) AS revenue
FROM sales
WHERE channel = 'online'          -- отсекаем offline ДО группировки
GROUP BY ROLLUP (category)
ORDER BY GROUPING(category), revenue DESC;
```
```
   категория    |  revenue
----------------+-----------
 Электроника    | 360000.00
 Дом            | 130000.00
 Книги          | 110000.00
 ИТОГО (online) | 600000.00
(4 rows)
```

Общий итог здесь `600 000`, а не `1 000 000` — потому что offline-строки вырезаны
`WHERE` ещё до группировки. Это ожидаемо и правильно: «итог по онлайну». Держи в
голове: **`WHERE` в таком запросе задаёт, из чего вообще считаются все уровни
итогов**.

## `HAVING` — фильтрует **готовые строки-итоги**

`HAVING` применяется **после** группировки — к уже посчитанным строкам (деталям и
итогам). В нём разрешены и агрегаты, и `GROUPING()`. Обычные `HAVING`-условия по
агрегату работают как всегда: `HAVING SUM(amount) > 200000` оставит любые строки
(в т.ч. итоговые), где сумма больше порога.

Ничего особенного в самом `HAVING` нет — специфика в том, что теперь среди строк
есть итоги, и `HAVING` их тоже видит и может фильтровать.

## Фильтрация по **уровню итога** через `GROUPING()` в `HAVING`

Частая задача: из большого `ROLLUP`/`CUBE` оставить строки только **одного
уровня** — например, только подытоги по категории, без деталей и без общего итога.
Уровень строки однозначно задаётся `GROUPING()`, поэтому фильтруем по нему в
`HAVING`:

```sql
SELECT category, subcategory, SUM(amount) AS revenue
FROM sales
GROUP BY ROLLUP (category, subcategory)
HAVING GROUPING(subcategory) = 1     -- подкатегория свёрнута ...
   AND GROUPING(category)    = 0     -- ... а категория ещё нет  → это подытог категории
ORDER BY category;
```
```
  category   | subcategory |  revenue
-------------+-------------+-----------
 Дом         | ∅           | 250000.00
 Книги       | ∅           | 150000.00
 Электроника | ∅           | 600000.00
(3 rows)
```

Из десяти строк `ROLLUP` осталось три — ровно подытоги по категориям. Условие
`GROUPING(subcategory)=1 AND GROUPING(category)=0` — это выбор одного набора
(`(category)`); то же можно записать маской: `HAVING GROUPING(category,
subcategory) = 1` (маска `1` = «свёрнута только младшая», см.
[файл 04](04-grouping-function-nulls.md)).

Так же отбирают только общий итог (`HAVING GROUPING(category) = 1`) или, наоборот,
**только детали** (`HAVING GROUPING(category)=0 AND GROUPING(subcategory)=0`).

> Иногда те же подытоги проще получить обычным `GROUP BY category` — и это будет
> дешевле, если детали и общий итог не нужны. `ROLLUP + HAVING GROUPING` оправдан,
> когда в одном запросе одновременно нужны и полный отчёт, и его отфильтрованный
> срез, или когда так удобнее строить отчёт программно.

## Подводные камни

- **Не определяй «итоговость» через `IS NULL`.** Настоящий `NULL` в данных
  (у нас `region`) неотличим по значению от итогового. Только `GROUPING()`.
  (Повторяем намеренно — это ошибка №1.)
- **`GROUPING SETS` не убирает дубликаты.** Он ведёт себя как `UNION ALL`, а не
  `UNION`. Повторяющийся набор → повторяющиеся строки. `CUBE`/`ROLLUP` дубликатов
  наборов не порождают, но при **комбинировании** (`ROLLUP(a), ROLLUP(a)`) можно
  получить одинаковые наборы и задвоение — комбинируй осознанно.
- **`CUBE` взрывается как `2ⁿ`.** `CUBE` из 5 колонок — 32 набора, из 6 — 64.
  Прежде чем писать `CUBE` по многим колонкам, прикинь число уровней; часто нужен
  не весь куб, а явный `GROUPING SETS` с нужными наборами.
- **`COUNT(*)` в строке-итоге считает детальные строки, а не под-группы.**
  В подытоге по «Дом» `COUNT(*)` вернёт число исходных строк продаж этой
  категории, а не число подкатегорий. Чтобы посчитать, сколько под-групп попало в
  итог, нужен `COUNT(DISTINCT subcategory)`.
- **`SUM`/`AVG` по пустому набору дают `NULL`, а не `0`** (правило из
  [lab001](../../lab001/theory/05-aggregates.md) — оно в силе и здесь). Если после
  `WHERE` в какой-то группе не осталось строк, её просто не будет в результате; а
  если оборачиваешь меру и хочешь `0` вместо `NULL` — `COALESCE(SUM(x), 0)`.
- **Сортировка итогов** по колонке-измерению зависит от `NULLS FIRST/LAST` и легко
  ломается; надёжнее `ORDER BY GROUPING(...)` ([файл 05](05-report-formatting-sorting.md)).
- **`GROUPING()` без расширенной группировки** (в обычном `GROUP BY` или вообще без
  группировки по этой колонке) вернёт `0` — смысла в ней там нет.

## Когда что: расширенная группировка и соседние инструменты

| Ситуация | Инструмент |
|----------|-----------|
| Один уровень группировки | обычный `GROUP BY` (lab001) |
| Детализация + подытоги по иерархии + общий итог | `ROLLUP` |
| Итоги по всем комбинациям независимых измерений | `CUBE` |
| Произвольный набор подытогов (не префиксы, не полный куб) | `GROUPING SETS` |
| Пометить/подписать/отфильтровать уровень итога | `GROUPING()` |
| Итог **рядом с каждой детальной строкой**, без схлопывания | оконные функции, **lab010** |
| Собрать те же уровни «руками» / когда нет `GROUPING SETS` | `UNION ALL` нескольких `GROUP BY` (lab003) |

Расширенная группировка заменяет громоздкий `UNION ALL` нескольких `GROUP BY`:
короче, один проход по данным, легче править. Но она по-прежнему **схлопывает**
строки — когда нужен итог, приклеенный к детали (доля строки в группе, накопленный
итог, ранг), это уже **оконные функции** из lab010.
