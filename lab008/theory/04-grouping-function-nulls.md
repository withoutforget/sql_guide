# 04 — Функция `GROUPING()` и проблема `NULL` в итогах

[← ROLLUP и CUBE](03-rollup-cube.md) · [оглавление](../README.md) · [дальше: оформление и сортировка →](05-report-formatting-sorting.md)

---

Это, пожалуй, самый важный практический файл лабы. Расширенная группировка
порождает `NULL` в строках-итогах — и этот `NULL` легко спутать с настоящим
`NULL` в данных. Функция `GROUPING()` создана ровно для того, чтобы их различать.

## Откуда в итогах берётся `NULL`

Вспомним механику: в строке-итоге колонка, по которой на этом уровне **не
группировали** (свернули), не имеет одного конкретного значения — по ней собрано
много разных. СУБД ставит в такую ячейку `NULL` — как маркер «эта колонка здесь
свёрнута».

В строке подытога по «Дом» (`ROLLUP (category, subcategory)`) значится
`Дом | ∅ | 250000`: `subcategory` свёрнута (в «Дом» входят и «Кухня», и
«Освещение»), поэтому там `NULL`. Само по себе это удобно — `NULL` наглядно
показывает «здесь итог по нескольким значениям».

## Проблема: два `NULL`, которые значат разное

Беда в том, что `NULL` в колонке измерения теперь означает **две совершенно
разные вещи**:

1. **«итоговый `NULL`»** — колонка свёрнута, это строка-итог;
2. **«настоящий `NULL`»** — так лежит в данных (у нас `region IS NULL` —
   «регион не указан»).

Посмотрим на `ROLLUP (region, channel)` — и увидим оба вида `NULL` в одной
колонке `region`:

```sql
SELECT region, channel, SUM(amount) AS revenue
FROM sales
GROUP BY ROLLUP (region, channel)
ORDER BY region NULLS LAST, channel NULLS LAST;
```
```
     region      | channel |  revenue
-----------------+---------+------------
 Москва          | offline |  120000.00
 Москва          | online  |  470000.00
 Москва          | ∅       |  590000.00
 Санкт-Петербург | offline |  240000.00
 Санкт-Петербург | online  |  100000.00
 Санкт-Петербург | ∅       |  340000.00
 ∅               | offline |   40000.00   ← регион НАСТОЯЩИЙ NULL (не указан), канал offline
 ∅               | online  |   30000.00   ← регион НАСТОЯЩИЙ NULL,             канал online
 ∅               | ∅       |   70000.00   ← подытог по «регион не указан»
 ∅               | ∅       | 1000000.00   ← ОБЩИЙ ИТОГ
(10 rows)
```

Посмотри на колонку `region`: `∅` стоит в **четырёх** строках, и означает `∅`
в них **разное**:

- в двух строках (`∅ | offline | 40000` и `∅ | online | 30000`) — это
  **настоящий** `NULL` из данных: продажи, где регион не указан, детализированы по
  каналу;
- в строке `∅ | ∅ | 70000` — `region` тоже настоящий `NULL` (это подытог **по**
  «регион не указан»), а вот `channel` уже **свёрнут** (итоговый `NULL`);
- в строке `∅ | ∅ | 1000000` — **оба** `∅` итоговые (общий итог).

Глазами по одной колонке `region` строку `∅ | ∅ | 70000` (подытог по «не указан»)
и строку `∅ | ∅ | 1000000` (общий итог) **не отличить** — в обеих `region = NULL`.
А смысл у них противоположный. Вот эту неоднозначность и решает `GROUPING()`.

## `GROUPING(колонка)` — флаг «эта колонка свёрнута?»

`GROUPING(колонка)` — специальная функция, доступная в запросах с расширенной
группировкой. Она возвращает:

- **`1`** — если в данной строке колонка **свёрнута** (агрегирована, «итоговый
  `NULL`»);
- **`0`** — если колонка **участвует в группировке** этой строки (её значение
  настоящее — в том числе если это настоящий `NULL` из данных).

Ключевой момент: `GROUPING()` смотрит **не на значение**, а на то, **входит ли
колонка в набор группировки** данной строки. Поэтому она безошибочно отличает
свёрнутую колонку от настоящего `NULL`.

Проверим на `region` (`ROLLUP (region)` — регионы + общий итог):

```sql
SELECT region, GROUPING(region) AS g_region, SUM(amount) AS revenue
FROM sales
GROUP BY ROLLUP (region)
ORDER BY g_region, region NULLS LAST;
```
```
     region      | g_region |  revenue
-----------------+----------+------------
 Москва          |        0 |  590000.00
 Санкт-Петербург |        0 |  340000.00
 ∅               |        0 |   70000.00   ← region = NULL, но GROUPING = 0 → это ДАННЫЕ
 ∅               |        1 | 1000000.00   ← region = NULL и GROUPING = 1 → это ИТОГ
(4 rows)
```

Вот и различие в чистом виде. Две строки с `region = ∅`:

- `∅ | 0 | 70000` — `GROUPING(region) = 0`: колонка группируется, `NULL` тут
  **настоящий** (продажи с неуказанным регионом);
- `∅ | 1 | 1000000` — `GROUPING(region) = 1`: колонка **свёрнута**, это строка
  общего итога.

> **Правило.** Строка является итогом **по колонке `X`** тогда и только тогда,
> когда `GROUPING(X) = 1`. Никогда не проверяй «итоговость» через `X IS NULL` —
> ошибёшься на настоящих `NULL` в данных.

## Битовая маска: `GROUPING(a, b, ...)` для нескольких колонок

`GROUPING()` умеет принимать **сразу несколько колонок** и возвращать **число —
битовую маску**. Каждой колонке-аргументу соответствует один бит: самый левый
аргумент — старший бит. Бит равен `1`, если колонка свёрнута.

Для `GROUPING(a, b)` возможны 4 значения:

| Строка | `a` свёрнута? | `b` свёрнута? | `GROUPING(a,b)` = `2·gᴀ + gᵦ` |
|--------|:---:|:---:|:---:|
| детализация `(a,b)` | нет (0) | нет (0) | `0` (`00₂`) |
| подытог `(a)` | нет (0) | да (1) | `1` (`01₂`) |
| подытог `(b)` | да (1) | нет (0) | `2` (`10₂`) |
| общий итог `()` | да (1) | да (1) | `3` (`11₂`) |

То есть `GROUPING(a, b)` — это компактный «номер уровня». Формально
`GROUPING(a, b) = 2 * GROUPING(a) + GROUPING(b)`. Посмотрим на
`ROLLUP (category, subcategory)`:

```sql
SELECT category, subcategory,
       GROUPING(category)              AS g_cat,
       GROUPING(subcategory)           AS g_sub,
       GROUPING(category, subcategory) AS mask,
       SUM(amount) AS revenue
FROM sales
GROUP BY ROLLUP (category, subcategory)
ORDER BY category NULLS LAST, subcategory NULLS LAST;
```
```
  category   | subcategory | g_cat | g_sub | mask |  revenue
-------------+-------------+-------+-------+------+------------
 Дом         | Кухня       |     0 |     0 |    0 |  150000.00
 Дом         | Освещение   |     0 |     0 |    0 |  100000.00
 Дом         | ∅           |     0 |     1 |    1 |  250000.00   ← подытог категории (mask 1)
 Книги       | Детская     |     0 |     0 |    0 |   70000.00
 Книги       | Учебная     |     0 |     0 |    0 |   80000.00
 Книги       | ∅           |     0 |     1 |    1 |  150000.00
 Электроника | Наушники    |     0 |     0 |    0 |  100000.00
 Электроника | Смартфоны   |     0 |     0 |    0 |  500000.00
 Электроника | ∅           |     0 |     1 |    1 |  600000.00
 ∅           | ∅           |     1 |     1 |    3 | 1000000.00   ← общий итог (mask 3)
(10 rows)
```

Заметь: при `ROLLUP` маска `2` (`GROUPING(category)=1, GROUPING(subcategory)=0` —
подытог «только по подкатегории») **не встречается** — потому что `ROLLUP` не
сворачивает старшую колонку раньше младшей. А вот у `CUBE (category, subcategory)`
появилась бы и строка с маской `2`. Так по набору встречающихся масок можно
«прочитать», какие уровни итогов есть в отчёте.

Маска особенно удобна, когда надо одним выражением отличить конкретный уровень:
`WHERE ... HAVING GROUPING(category, subcategory) = 1` оставит ровно
подытоги-по-категории (об этом — [файл 06](06-having-filtering-pitfalls.md)).

## Красивые подписи итогов через `CASE` + `GROUPING()`

`∅` в отчёте — некрасиво и непонятно читателю. Правильный способ подписать
итоговые строки — `CASE`, который смотрит на `GROUPING()` (а не на `IS NULL`!):

```sql
SELECT
    CASE WHEN GROUPING(region) = 1 THEN 'ВСЕ РЕГИОНЫ'      -- строка-итог
         WHEN region IS NULL       THEN 'Регион не указан' -- настоящий NULL в данных
         ELSE region END           AS регион,
    SUM(amount) AS revenue
FROM sales
GROUP BY ROLLUP (region)
ORDER BY GROUPING(region), revenue DESC;
```
```
      регион      |  revenue
------------------+------------
 Москва           |  590000.00
 Санкт-Петербург  |  340000.00
 Регион не указан |   70000.00   ← настоящий NULL получил свою подпись
 ВСЕ РЕГИОНЫ      | 1000000.00   ← итоговый NULL — свою
(4 rows)
```

**Порядок ветвей `CASE` тут критичен.** Сначала проверяем `GROUPING(region) = 1`
(итог), и только потом `region IS NULL` (данные). Если поменять местами — строка
общего итога (где `region` тоже `NULL`) ошибочно подпишется как «Регион не
указан». Такой отчёт врёт: он спрячет общий итог под видом маленького подытога.
Это классическая ошибка, ради которой и существует `GROUPING()`.

## Мини-итог

- В строках-итогах свёрнутые колонки = `NULL` — это «итоговый `NULL`».
- Настоящий `NULL` в данных выглядит **так же** — глазами не отличить.
- `GROUPING(X)` = `1`, если `X` свёрнута (итог по `X`), иначе `0` — и это **не
  зависит от значения**, поэтому надёжно.
- `GROUPING(a, b, ...)` — битовая маска-«номер уровня»: каждый аргумент даёт свой
  бит, старший — у **первого** (`gₐ·2ⁿ⁻¹ + … + g_last·2⁰`; для двух колонок это
  `2·gₐ + g_b`).
- Подписи итогов делай через `CASE WHEN GROUPING(...) = 1`, а не через `IS NULL`,
  и ставь ветку `GROUPING` **перед** веткой про настоящий `NULL`.
