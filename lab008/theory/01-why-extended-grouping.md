# 01 — Зачем нужна расширенная группировка

[← оглавление лабы](../README.md) · [дальше: GROUPING SETS →](02-grouping-sets.md)

---

## Один `GROUP BY` — один уровень итогов

В [lab001](../../lab001/theory/06-group-having.md) мы разобрали `GROUP BY`: он
раскладывает строки по «корзинам» и считает агрегат внутри каждой. Ключевое
свойство, о котором сейчас важно вспомнить: **обычный `GROUP BY` даёт ровно один
уровень группировки**. Написали `GROUP BY category` — получили по одной строке на
категорию, и всё:

```sql
SELECT category, SUM(amount) AS revenue
FROM sales
GROUP BY category
ORDER BY category;
```
```
  category   |  revenue
-------------+-----------
 Дом         | 250000.00
 Книги       | 150000.00
 Электроника | 600000.00
(3 rows)
```

Три строки, три числа. Но в **отчёте** этого почти всегда мало.

## Чего не хватает: детализация + подытоги + общий итог сразу

Представь, что тот же результат нужно показать руководителю. Живой отчёт о
продажах выглядит так:

```
Электроника ......... 600 000
Книги ............... 150 000
Дом ................. 250 000
─────────────────────────────
ИТОГО ............. 1 000 000
```

То есть нужны **сразу**, в одной таблице:

- **детализация** (detail) — строки по категориям;
- **общий итог** (grand total) — суммарная строка «ИТОГО».

А в отчётах побольше — ещё и **промежуточные итоги** (subtotals): например,
детализация по подкатегориям **плюс** подытог по каждой категории **плюс** общий
итог. Это три разных уровня агрегации в одном результате. Обычный `GROUP BY`
так не умеет: он считает только один уровень за раз.

Вводится терминология, она понадобится дальше:

| Термин | По-русски | Что это |
|--------|-----------|---------|
| **detail** | детализация | самые дробные строки (по всем измерениям) |
| **subtotal** | промежуточный итог, подытог | итог по части измерений |
| **grand total** | общий итог | итог по всей таблице (по «нулю» измерений) |
| **super-aggregate row** | супер-агрегатная строка | любая строка-итог (subtotal или grand total) |

## Как это делали «руками»: `UNION ALL` нескольких `GROUP BY`

До расширенной группировки такой отчёт собирали **вручную** — считали каждый
уровень отдельным `GROUP BY` и склеивали результаты через `UNION ALL` (это мы
проходили в [lab003](../../lab003/theory/02-union-union-all.md): `UNION ALL`
просто дописывает строки одного запроса под строки другого).

Детализация по категориям (`GROUP BY category`) плюс общий итог по всей таблице
(`GROUP BY` по «ничему»):

```sql
SELECT category, SUM(amount) AS revenue    -- уровень 1: по категориям
FROM sales
GROUP BY category
UNION ALL
SELECT 'ВСЕ КАТЕГОРИИ', SUM(amount)         -- уровень 2: общий итог
FROM sales
ORDER BY revenue;
```
```
   category    |  revenue
---------------+------------
 Книги         |  150000.00
 Дом           |  250000.00
 Электроника   |  600000.00
 ВСЕ КАТЕГОРИИ | 1000000.00
(4 rows)
```

Работает. Но у этого «ручного» способа есть очевидные минусы:

- **Многословно.** Каждый уровень итогов — это отдельный `SELECT ... GROUP BY`.
  Хотите ещё разрез по каналу и по региону? Ещё три `SELECT`'а. Отчёт
  «детализация + подытоги по 3 измерениям + общий итог» превращается в простыню
  из пяти–шести почти одинаковых запросов, склеенных `UNION ALL`.
- **Медленно.** Каждая ветка `UNION ALL` — это **отдельный проход по таблице**.
  Пять уровней итогов = пять сканирований `sales`. СУБД считает одно и то же по
  много раз.
- **Хрупко.** Легко ошибиться: забыть ветку, разойтись в списке колонок, в одном
  месте отфильтровать в `WHERE`, а в другом — нет. И правки приходится вносить
  сразу в несколько мест.

## Решение: расширенная группировка

**Расширенная группировка** (`GROUPING SETS`, `ROLLUP`, `CUBE`) делает ровно то
же самое — считает несколько уровней итогов и складывает их в один результат, —
но **одним `GROUP BY`, за один проход по таблице**. Тот же отчёт «категории +
общий итог»:

```sql
SELECT category, SUM(amount) AS revenue
FROM sales
GROUP BY GROUPING SETS ((category), ())   -- два уровня в одном GROUP BY
ORDER BY revenue;
```

Одна строка вместо `UNION ALL` из двух запросов — а результат тот же (мы это
строго покажем в [файле 02](02-grouping-sets.md)). Добавить ещё разрезы — значит
дописать пару скобок, а не новый `SELECT`.

### Мысленная модель

Держи в голове простую картинку:

> **Расширенная группировка = «выполнить несколько `GROUP BY` за один раз и
> сложить их результаты друг под друга через `UNION ALL`».**

Всё, что ты уже знаешь про `GROUP BY` и агрегаты из lab001, остаётся в силе;
меняется только одно — теперь в одном запросе можно задать **несколько наборов
группировки** сразу.

## Три конструкции — обзор

| Конструкция | Что задаёт | Аналогия |
|-------------|-----------|----------|
| `GROUPING SETS ((a),(b),())` | **явный список** наборов группировки | «посчитай вот эти уровни» |
| `ROLLUP (a, b, c)` | **иерархию** итогов: `(a,b,c),(a,b),(a),()` | «детали → подытоги → общий» |
| `CUBE (a, b)` | **все комбинации**: `(a,b),(a),(b),()` | «полный куб по всем разрезам» |

`GROUPING SETS` — базовая, самая общая форма: ты просто перечисляешь наборы.
`ROLLUP` и `CUBE` — удобные сокращения для двух очень частых случаев (иерархия и
полный куб); оба **раскрываются** в `GROUPING SETS` (покажем в
[файле 03](03-rollup-cube.md)).

## Важная оговорка: это всё ещё «схлопывающая» группировка

Расширенная группировка — это по-прежнему `GROUP BY`: каждая группа
**схлопывается** в одну строку. В отчёте выше нет отдельных продаж — только
свёрнутые суммы по категориям и общий итог. Детальные строки как таковые
исчезают.

Есть другой инструмент — **оконные функции** (`OVER (...)`, отдельная тема
lab010) — который умеет показывать итог **рядом с каждой детальной строкой, не
схлопывая их** (например, «сумма заказа и тут же — доля от общего»). Это не
конкурент, а дополнение: когда нужны именно строки-итоги (как в бухгалтерском
отчёте с подытогами) — расширенная группировка; когда нужен итог **приклеенный к
каждой детальной строке** — оконные функции. Пока держим в голове только различие:

| | Расширенная группировка (эта лаба) | Оконные функции (lab010) |
|--|--|--|
| Строки | **схлопывает** в итоговые | **сохраняет** все детальные |
| Итог | отдельной строкой (subtotal / grand total) | в новой колонке рядом с деталью |
