# 04 — `LATERAL`: правая часть `FROM`, которая видит левую

[← VALUES как источник строк](03-values-as-table.md) · [оглавление](../README.md) · [дальше: фирменные кейсы (top-N, gap-filling) →](05-cases-topn-gapfill.md)

---

Это центральный файл лабы. `LATERAL` — небольшое ключевое слово, снимающее одно
важное ограничение `FROM` и открывающее целый класс запросов, которые иначе в SQL
не выразить (top-N на группу, соединение с табличной функцией от колонки, «для
каждой строки — свой вычисляемый блок»).

## Проблема: элементы `FROM` независимы

Обычные источники в `FROM` — таблицы, подзапросы, `VALUES` — **не видят друг
друга**. Каждый вычисляется сам по себе, а `JOIN`/`WHERE` только потом связывают
их результаты. Поэтому подзапрос в `FROM` **не может** сослаться на колонку
соседней таблицы из того же `FROM`:

```sql
-- ❌ НЕ работает: подзапрос справа пытается использовать c.id слева
SELECT c.name, s.revenue
FROM customers AS c,
     (SELECT SUM(pr.price * o.quantity) AS revenue
      FROM orders AS o JOIN products AS pr ON pr.id = o.product_id
      WHERE o.customer_id = c.id) AS s;      -- ← c.id здесь «не виден»
```
```
ERROR:  invalid reference to FROM-clause entry for table "c"
```

Ошибка прямо об этом: подзапрос `s` **не имеет права** ссылаться на `c` —
элементы `FROM` для него параллельны, а не «слева направо». Из lab004 мы знаем
обходной путь для **одного значения**: коррелированный **скалярный** подзапрос в
`SELECT` или `WHERE`. Но он умеет вернуть лишь одно число и лишь в тех местах —
не как полноценный источник строк во `FROM`.

## `LATERAL` — разрешить правой части видеть левую

**`LATERAL`** перед подзапросом или функцией в `FROM` **снимает** это ограничение:
теперь правый источник **может ссылаться на колонки таблиц, стоящих левее** в том
же `FROM`. Тот же запрос с `LATERAL` работает:

```sql
SELECT c.name, s.revenue
FROM customers AS c,
     LATERAL (SELECT SUM(pr.price * o.quantity) AS revenue
              FROM orders AS o JOIN products AS pr ON pr.id = o.product_id
              WHERE o.customer_id = c.id) AS s;   -- ✅ теперь c.id виден
```

Ключевое отличие от коррелированного скалярного подзапроса (lab004): подзапрос под
`LATERAL` — это **полноценный источник строк**. Он может вернуть **сколько угодно
строк и сколько угодно колонок**, а не одно значение. Именно это делает возможным
top-N на группу, разворачивание, соединение с функцией — всё, чего скалярный
подзапрос не умеет.

## Мысленная модель: «для каждой строки слева — выполнить правую часть»

Держите в голове такую картинку (концептуально; реальный планировщик хитрее):

1. Внешний запрос идёт по строкам левой части **по одной**.
2. Для **каждой** левой строки берутся её значения (`c.id`, `p.price`, …),
   подставляются в правый `LATERAL`-источник, и он вычисляется **заново**.
3. Строки, которые он вернул, **приклеиваются** к текущей левой строке.

Это ровно та же модель «по разу на каждую внешнюю строку», что у коррелированного
подзапроса из [lab004, файл 03](../../lab004/theory/03-exists-correlated.md) — но
результат встраивается во `FROM` как таблица, поэтому строк и колонок может быть
много.

## Синтаксис: три формы

### 1. Через запятую (неявный `CROSS JOIN`)

```sql
FROM t, LATERAL (SELECT ... WHERE ... = t.col) AS s
```

Запятая во `FROM` — это `CROSS JOIN`; с `LATERAL` правая часть видит `t`. Пары, где
правый набор пуст, **исчезают** (как во всяком `CROSS`/`INNER`).

### 2. `CROSS JOIN LATERAL`

```sql
FROM t CROSS JOIN LATERAL (SELECT ... WHERE ... = t.col) AS s
```

То же, что форма 1, но написано явным `CROSS JOIN LATERAL` — читается яснее.

### 3. `LEFT JOIN LATERAL ... ON true`

```sql
FROM t LEFT JOIN LATERAL (SELECT ... WHERE ... = t.col) AS s ON true
```

`LEFT JOIN LATERAL` **сохраняет левую строку, даже если правый набор пуст** (тогда
правые колонки = `NULL`). Условие соединения обычно не нужно — вся связь уже внутри
подзапроса (`WHERE ... = t.col`), поэтому пишут **`ON true`** («соединять всегда»).

## `CROSS JOIN LATERAL` vs `LEFT JOIN LATERAL` — ключевое различие

Разница между ними — ровно как между `INNER JOIN` и `LEFT JOIN` из lab002, только
справа стоит коррелированный `LATERAL`-подзапрос:

| Что справа вернул `LATERAL` | `CROSS JOIN LATERAL` | `LEFT JOIN LATERAL ... ON true` |
|-----------------------------|:--------------------:|:-------------------------------:|
| одну или несколько строк | строки приклеиваются | строки приклеиваются |
| **ноль строк** (пусто) | **левая строка пропадает** | **левая строка остаётся, правые колонки = `NULL`** |

Покажем на «самом дорогом заказе каждого клиента» (задачи 7 и 8). У Жанны заказов
нет, значит правый подзапрос для неё вернёт **ноль строк**.

`CROSS JOIN LATERAL` — Жанна **пропадает**:

```sql
SELECT c.name AS customer, top.product, top.revenue
FROM customers AS c
CROSS JOIN LATERAL (
    SELECT pr.name AS product, pr.price * o.quantity AS revenue
    FROM orders AS o JOIN products AS pr ON pr.id = o.product_id
    WHERE o.customer_id = c.id
    ORDER BY revenue DESC
    LIMIT 1
) AS top;
```
```
 customer |      product       | revenue
----------+--------------------+----------
 Егор     | Ноутбук AirBook 14 | 74990.00
 Анна     | Смартфон Nova 5    | 24990.00
 ...
 Глеб     | Наушники TWS Pro   |  5990.00
(6 rows)          ← Жанны здесь НЕТ
```

`LEFT JOIN LATERAL ... ON true` — Жанна **остаётся** с `NULL`:

```sql
SELECT c.name AS customer, top.product, top.revenue
FROM customers AS c
LEFT JOIN LATERAL (
    SELECT pr.name AS product, pr.price * o.quantity AS revenue
    FROM orders AS o JOIN products AS pr ON pr.id = o.product_id
    WHERE o.customer_id = c.id
    ORDER BY revenue DESC
    LIMIT 1
) AS top ON true;
```
```
 customer |      product       | revenue
----------+--------------------+----------
 Егор     | Ноутбук AirBook 14 | 74990.00
 ...
 Глеб     | Наушники TWS Pro   |  5990.00
 Жанна    |                    |          ← осталась благодаря LEFT JOIN LATERAL
(7 rows)
```

**Правило:** если правый набор **может быть пустым** и левые строки терять нельзя —
берите **`LEFT JOIN LATERAL ... ON true`**. Это та же логика «сохранить сироту»,
что в [lab002, файл 04](../../lab002/theory/04-outer-joins.md), только справа —
коррелированный подзапрос.

## Автоматический `LATERAL` для табличных функций

Есть приятное исключение: если справа во `FROM` стоит **функция** (а не подзапрос),
и она ссылается на колонку левой таблицы, `LATERAL` **включается сам** — слово
писать не нужно:

```sql
-- LATERAL не написан, но unnest(p.tags) уже видит p.tags — это неявный LATERAL
SELECT p.name, tag.value
FROM products AS p,
     unnest(p.tags) AS tag(value);
```

Именно поэтому разворачивание тегов в **задаче 5** работает
без ключевого слова. Написать `LATERAL` явно тоже можно — результат тот же:
`FROM products p, LATERAL unnest(p.tags) AS tag(value)`. Для функций это дело вкуса;
для **подзапросов** `(SELECT …)` слово `LATERAL` **обязательно**.

> Тонкость с пустым набором: `unnest('{}')` даёт 0 строк, поэтому товар без тегов
> при таком (неявно `CROSS`) соединении **выпадает** — как Жанна выше. Чтобы
> сохранить и «беститеговые» товары, берут `LEFT JOIN LATERAL unnest(p.tags) …
> ON true` — тогда у них `value = NULL`.

## `LATERAL` vs коррелированный подзапрос (lab004)

Оба «коррелируют» — ссылаются на внешнюю строку. Разница в **мощности и месте**:

| | Коррелированный **скалярный** подзапрос (lab004) | `LATERAL`-подзапрос (эта лаба) |
|--|--------------------------------------------------|-------------------------------|
| Где стоит | в `SELECT` или `WHERE` | в `FROM` (как источник строк) |
| Сколько возвращает | **ровно 1 значение** (1 строка, 1 колонка) | **сколько угодно строк и колонок** |
| Несколько колонок сразу | нет (пришлось бы писать несколько подзапросов) | да, за один проход |
| top-N (LIMIT N > 1) на группу | **невозможно** | **естественно** |
| Типичное применение | «цена vs среднее в категории», `EXISTS` | top-N, разворачивание, «свой блок на строку» |

Иными словами, `LATERAL` — это «коррелированный подзапрос, повышенный до
полноценного источника строк во `FROM`». Когда достаточно одного числа —
скалярный подзапрос короче; когда нужны **строки или несколько колонок** — `LATERAL`.

## `LATERAL` vs обычный `JOIN` (lab002)

Обычный `JOIN` соединяет **два независимых** набора по условию `ON`. `LATERAL`
нужен, когда правый набор **зависит от текущей левой строки** и его нельзя вычислить
заранее — потому что в нём есть `LIMIT`, коррелированная функция или расчёт от
колонки слева.

- «Приклеить к заказу его клиента» — обычный `JOIN` (оба набора известны).
- «Приклеить к клиенту его **3 самых дорогих** заказа» — `LATERAL` (правый набор
  = top-3 **именно этого** клиента, зависит от `c.id` и содержит `LIMIT`).

Если правую часть **можно** вычислить без ссылки на левую (обычное равенство
ключей, без `LIMIT`/агрегата «на строку») — берите обычный `JOIN`, он проще.
`LATERAL` — инструмент для «правая часть считается **для каждой** левой строки».

## Итог файла

- Обычные элементы `FROM` независимы; **`LATERAL`** разрешает правому подзапросу/
  функции **ссылаться на колонки таблиц слева**.
- В отличие от коррелированного скалярного подзапроса, `LATERAL`-источник может
  вернуть **много строк и много колонок** — отсюда top-N на группу и развороты.
- Мысленная модель — «**для каждой строки слева выполнить правую часть**».
- **`CROSS JOIN LATERAL`** теряет левую строку при пустом правом наборе;
  **`LEFT JOIN LATERAL ... ON true`** её сохраняет (правые колонки `NULL`).
- Для **функций** во `FROM` `LATERAL` включается автоматически; для **подзапросов**
  его пишут явно.
