# 02 — `unnest`, строковые SRF, `WITH ORDINALITY` и SRF в `SELECT`

[← SRF и generate_series](01-srf-generate-series.md) · [оглавление](../README.md) · [дальше: VALUES как источник строк →](03-values-as-table.md)

---

Кроме `generate_series` есть ещё несколько ходовых функций-множеств. Разберём
`unnest` (разворачивание массива), строковые SRF, приставку `WITH ORDINALITY`
(нумерация строк) и отдельно — что происходит, когда SRF ставят не в `FROM`, а в
список `SELECT`.

## `unnest(массив)` — массив в строки

`unnest` берёт массив и возвращает **по строке на каждый его элемент**. Это самый
частый способ «развернуть» массив в набор строк.

```sql
SELECT x
FROM unnest(ARRAY['красный', 'зелёный', 'синий']) AS t(x);
```
```
    x
---------
 красный
 зелёный
 синий
(3 rows)
```

Порядок строк совпадает с порядком элементов в массиве. Поведение на краях:

- **пустой массив** `'{}'` → **ноль строк**;
- **`NULL` вместо массива** → тоже **ноль строк** (не одна строка с `NULL`).

```sql
SELECT count(*) FROM unnest('{}'::text[]) AS t(x);        -- 0
SELECT count(*) FROM unnest(NULL::text[]) AS t(x);        -- 0
```

Чаще всего `unnest` применяют к **колонке-массиву** таблицы, чтобы развернуть её
построчно. В нашей схеме у товаров есть небольшое поле `tags TEXT[]`:

```sql
SELECT p.name, tag.value AS tag
FROM products AS p,
     unnest(p.tags) AS tag(value);   -- по строке на каждый тег каждого товара
```

Здесь функция `unnest(p.tags)` ссылается на колонку **соседней** таблицы `p` —
разворачивает массив тегов **отдельно для каждого товара**. Это и есть неявный
`LATERAL` (см. [файл 04](04-lateral.md)): для табличных функций PostgreSQL
включает его автоматически, поэтому слово `LATERAL` тут писать не нужно. Товар с
несколькими тегами даёт несколько строк; товар с пустым массивом (`Настольная
лампа`, `tags = '{}'`) не даёт ни одной и в результат не попадает — прямое
следствие правила «`unnest` пустого массива → 0 строк». Это **задача 5**.

> **Про массивы в целом.** Полноценная работа с массивами (конструкторы, срезы,
> операторы `@>`, `&&`, агрегат `array_agg`, многомерные массивы) — тема **lab018**.
> Здесь массив нужен ровно для одного: показать, как `unnest` превращает его в
> строки. Больше про массивы знать сейчас не требуется.

## Строковые SRF: `string_to_table` и `regexp_split_to_table`

Иногда «список» приходит не массивом, а **строкой с разделителями** («`тег1,тег2,тег3`»,
CSV-подобные данные, склеенные значения). Развернуть такую строку в строки таблицы
помогают две функции.

**`string_to_table(строка, разделитель)`** — режет строку по фиксированному
разделителю:

```sql
SELECT x FROM string_to_table('электроника,книги,дом', ',') AS t(x);
```
```
      x
-------------
 электроника
 книги
 дом
(3 rows)
```

Тонкость: **два разделителя подряд дают пустой элемент** (пустую строку), а не
пропускают её:

```sql
SELECT '[' || x || ']' AS x FROM string_to_table('a,b,,c', ',') AS t(x);
-- [a], [b], [], [c]   -- между двумя запятыми получился пустой элемент
```

**`regexp_split_to_table(строка, шаблон)`** — то же, но разделитель задаётся
**регулярным выражением**. Удобно, когда разделителей несколько или их «сколько
угодно подряд» (например, любые пробелы):

```sql
SELECT x FROM regexp_split_to_table('раз  два   три', '\s+') AS t(x);
-- раз, два, три   -- \s+ = один или несколько пробельных символов
```

Обе функции — обычные SRF: стоят в `FROM`, результат фильтруется/соединяется как
угодно. Регулярные выражения подробно — в lab015; здесь достаточно знать, что
такой инструмент есть.

## `WITH ORDINALITY` — пронумеровать строки SRF

У функций-множеств есть особая приставка **`WITH ORDINALITY`**. Она добавляет к
результату **ещё одну колонку — порядковый номер строки** (1, 2, 3, …). Номер
идёт в том порядке, в каком SRF выдаёт строки.

Синтаксис: `FROM функция(...) WITH ORDINALITY AS алиас(колонка_значения, колонка_номера)`.

```sql
SELECT val, n
FROM generate_series(10, 40, 10) WITH ORDINALITY AS t(val, n);
```
```
 val | n
-----+---
  10 | 1
  20 | 2
  30 | 3
  40 | 4
(4 rows)
```

В алиасе `t(val, n)` **первое** имя (`val`) — это значение, которое возвращает
функция, а **второе** (`n`) — добавленный номер. Работает с любой SRF, не только с
`generate_series`. Например, пронумеровать элементы массива (это **задача 3**):

```sql
SELECT step, stage
FROM unnest(ARRAY['Оформлен','Оплачен','Собран','Доставлен'])
     WITH ORDINALITY AS t(stage, step);
```
```
 step |  stage
------+----------
    1 | Оформлен
    2 | Оплачен
    3 | Собран
    4 | Доставлен
(4 rows)
```

**Зачем.** `WITH ORDINALITY` даёт «позицию элемента» без ручного счётчика: номер
шага доставки, порядковый номер строки в исходном списке, индекс элемента массива.
Это чистый способ ответить на вопрос «а каким по счёту идёт этот элемент».

> **Важное ограничение.** `WITH ORDINALITY` можно приписать **только к вызову
> функции** в `FROM`, но **не** к произвольному подзапросу `(SELECT …)`. Запись
> `(SELECT … LIMIT 3) WITH ORDINALITY` — синтаксическая ошибка. Пронумеровать
> строки произвольного `SELECT` (например, top-N из подзапроса) удобнее оконной
> функцией `ROW_NUMBER()` — но это уже lab010.

## SRF в списке `SELECT` — можно, но с оговорками

SRF разрешено ставить не только в `FROM`, но и прямо в **список `SELECT`**. Тогда
на каждую входную строку функция «размножает» её на столько строк, сколько сама
вернула:

```sql
SELECT id, generate_series(1, id) AS n
FROM (VALUES (1), (3)) AS v(id);
```
```
 id | n
----+---
  1 | 1
  3 | 1
  3 | 2
  3 | 3
(4 rows)
```

Строка `id = 1` дала одну строку, `id = 3` — три. По сути это неявное соединение
входной строки с результатом SRF (то самое `LATERAL` под капотом).

### Подводный камень: несколько SRF в одном `SELECT`

Если в списке `SELECT` стоят **две** SRF разной длины, современный PostgreSQL
(версии 10+) выстраивает их **параллельно** («в ногу»): число строк равно длине
**самой длинной**, а более короткие дополняются `NULL`.

```sql
SELECT generate_series(1, 2) AS a, generate_series(1, 4) AS b;
```
```
 a | b
---+---
 1 | 1
 2 | 2
   | 3      ← a закончилась → NULL
   | 4
(4 rows)
```

Это часто **не то**, что ожидают. Историческая деталь: в старых версиях
PostgreSQL (до 10) поведение было ещё более странным — число строк вычислялось как
**наименьшее общее кратное** длин, из-за чего значения «зацикливались». От этого
наследия и предостережений вокруг «SRF в `SELECT`» и пошла общая рекомендация.

### Рекомендация: ставьте SRF в `FROM`, а не в `SELECT`

Тот же результат почти всегда **чище и понятнее** выразить, поставив SRF в `FROM`
(при необходимости — через `LATERAL`, [файл 04](04-lateral.md)):

```sql
-- эквивалент примера выше, но SRF вынесена в FROM — читается однозначно
SELECT v.id, g.n
FROM (VALUES (1), (3)) AS v(id),
     LATERAL generate_series(1, v.id) AS g(n);
```

**Правило:** SRF — в `FROM`. В `SELECT` её кладут разве что для очень коротких
разовых выражений; для всего остального (и всегда, если SRF больше одной) —
`FROM`. Подробнее «SRF в `SELECT` против `FROM`» — в [файле 06](06-pitfalls-choosing.md).

## Итог файла

- **`unnest(массив)`** разворачивает массив в строки; пустой массив и `NULL` дают
  **ноль строк**. Массивы целиком — тема lab018.
- **`string_to_table`** и **`regexp_split_to_table`** разворачивают строку с
  разделителями (второй — по регулярному выражению).
- **`WITH ORDINALITY`** добавляет колонку-номер строки SRF; работает только с
  функциями в `FROM`, не с подзапросами.
- SRF в **`SELECT`** размножает строки, но при нескольких SRF ведёт себя неочевидно
  («в ногу», с `NULL`-добивкой) — **предпочитайте SRF в `FROM`**.
