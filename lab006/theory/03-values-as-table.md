# 03 — `VALUES` как источник строк во `FROM`

`generate_series` и `unnest` порождают строки **по правилу**. Но часто нужно
просто **перечислить несколько строк вручную** — маленький справочник, набор
порогов, список меток. Для этого есть конструктор `VALUES`.

## `VALUES` — не только для `INSERT`

`VALUES` мы уже видели в `INSERT ... VALUES (...), (...)`: это **конструктор
строк**, список кортежей. Но `VALUES` — самостоятельное выражение: он может стоять
**где угодно, где ожидается набор строк**, в том числе прямо во `FROM`.

```sql
-- VALUES сам по себе — это уже готовый набор строк
VALUES (1, 'a'), (2, 'b'), (3, 'c');
```
```
 column1 | column2
---------+---------
       1 | a
       2 | b
       3 | c
(3 rows)
```

Без алиасов колонки получают служебные имена `column1`, `column2`, … Чтобы
пользоваться таким набором как таблицей, ему дают **алиас с именами колонок** —
и ставят во `FROM`:

```sql
SELECT t.id, t.label
FROM (VALUES (1, 'a'), (2, 'b'), (3, 'c')) AS t(id, label);
```

`AS t(id, label)` именует источник (`t`) и его колонки (`id`, `label`). Дальше `t`
— обычная таблица «на лету»: с ней работают `WHERE`, `JOIN`, `ORDER BY`,
агрегаты. Такой набор **не создаётся в схеме** — он существует только в пределах
запроса, как и результат `generate_series` или подзапроса в `FROM`.

## Типы колонок и `NULL`

Тип каждой колонки PostgreSQL выводит из перечисленных значений. Если в одной
колонке смешаны разные типы, действуют обычные правила приведения; иногда тип
стоит **указать явно** — приведением `::тип` хотя бы в первой строке, чтобы не было
двусмысленности:

```sql
-- явно задаём типы, чтобы колонки были numeric и text, а не «неизвестно что»
SELECT *
FROM (VALUES
        (1::int, 'Электроника'::text, 0.20::numeric),
        (2,      'Книги',              0.10),
        (3,      'Дом',                0.20)
     ) AS vat(category_id, category, rate);
```

`NULL` в `VALUES` тоже допустим. Но если в колонке встречается **только** `NULL`,
PostgreSQL не может вывести её тип из значений и молча назначает ей тип **`text`**
(само `VALUES` при этом ошибки не даёт):

```sql
SELECT pg_typeof(x) FROM (VALUES (NULL), (NULL)) AS t(x);
-- text   -- колонка из одних NULL по умолчанию стала text
```

Пока значение никак не используется, это незаметно. Но стоит применить его как
дату или число — и всплывёт ошибка приведения:

```sql
SELECT x + 1 FROM (VALUES (NULL), (NULL)) AS t(x);
-- ERROR:  operator does not exist: text + integer
```

Поэтому, если такая колонка по смыслу дата/число, **задайте тип явно** приведением
(`NULL::date`, `NULL::numeric`) — тогда она будет нужного типа, а не `text`.

## Приём 1: справочник-классификатор на лету

Самое частое применение — **маленький справочник**, которого нет (и не нужно) в
базе, но по которому надо что-то сопоставить. Классифицируем товары по ценовым
сегментам, соединив их с `VALUES`-справочником диапазонов (это **задача 4**):

```sql
SELECT p.name, p.price, seg.label AS segment
FROM products AS p
JOIN (VALUES
        (    0::numeric,  2000::numeric, 'бюджет'),
        ( 2000::numeric, 10000::numeric, 'средний'),
        (10000::numeric,  1e9::numeric,  'премиум')
     ) AS seg(lo, hi, label)
     ON p.price >= seg.lo AND p.price < seg.hi;   -- тета-соединение по диапазону
```

Соединение здесь — **по диапазону** (`>=` и `<`), это тета-соединение из lab002:
товар попадает в сегмент, чей полуинтервал `[lo, hi)` накрывает его цену. Так
логика классификации живёт **в запросе**, а не в отдельной таблице.

Другие типичные справочники «на лету»: соответствие «код → человекочитаемая метка»
(`'A' → 'активен'`), плановые показатели по категориям, ставки/коэффициенты,
короткий whitelist значений для фильтра.

## Приём 2: развернуть одну строку в несколько

`VALUES` в паре с `CROSS JOIN` **размножает** каждую строку таблицы на набор
фиксированных вариантов. В lab002 так строили сетку «категория × размер»:

```sql
-- каждой категории — три размерных варианта
SELECT cat.name, s.size
FROM categories AS cat
CROSS JOIN (VALUES ('S'), ('M'), ('L')) AS s(size);
```

Это тот же каркас-решётка, что и `CROSS JOIN` с `generate_series`, только набор
задан **явным перечислением**, а не сгенерирован.

## `VALUES` против других источников строк

| Источник | Когда брать |
|----------|-------------|
| `VALUES (...), (...)` | строк **немного и они известны заранее** — справочник, список меток, набор порогов |
| `generate_series(...)` | строки идут **по правилу** — ряд чисел, сплошной календарь |
| `unnest(массив)` | данные уже лежат **в массиве** (литерал или колонка) |
| `UNION ALL` из `SELECT`ов | строки берутся **из разных таблиц/запросов** (lab003) |

`VALUES` и «`SELECT ... UNION ALL SELECT ...`» из lab003 взаимозаменяемы для
списка констант, но `VALUES` **компактнее и быстрее**: одна конструкция вместо
цепочки `SELECT`ов. Сравните:

```sql
-- многословно (lab003)                 -- компактно (эта лаба)
SELECT 1 AS id, 'a' AS label            (VALUES (1,'a'), (2,'b'), (3,'c')) AS t(id, label)
UNION ALL SELECT 2, 'b'
UNION ALL SELECT 3, 'c'
```

## Подводные камни

- **Нужен алиас с именами колонок.** Без `AS t(...)` колонки называются
  `column1`, `column2` — неудобно и нечитаемо. В `FROM` алиас источника
  обязателен в принципе.
- **Одинаковое число элементов в каждой строке.** Все кортежи `VALUES` должны быть
  одной «ширины» — иначе ошибка (как при `INSERT`).
- **Согласованные типы по колонкам.** Значения одной колонки должны приводиться к
  общему типу; при сомнении задавайте тип явно (`::numeric`, `::date`).
- **`VALUES` — для небольших наборов.** Это ручной список; сотни строк в нём —
  сигнал, что данные пора держать в таблице, а не в тексте запроса.

## Итог файла

- `VALUES (...), (...)` — конструктор строк; во `FROM` с алиасом `AS t(col, …)`
  он становится **маленькой таблицей на лету**, не создавая ничего в схеме.
- Главные применения: **справочник-классификатор** (join по коду или диапазону) и
  **разворачивание** строки в набор вариантов через `CROSS JOIN`.
- По сравнению с `UNION ALL` из `SELECT`ов `VALUES` компактнее; по сравнению с
  `generate_series`/`unnest` — это **явное перечисление**, а не генерация по правилу.
