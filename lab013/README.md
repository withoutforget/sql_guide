# lab013 — Типы данных и приведения типов

Первая лаба **модуля D «Типы и работа с данными»**. Тема фундаментальная и
обманчиво «простая» — её часто проскакивают, а потом ловят баги: целочисленное
деление занижает средний чек, деньги «плывут» из-за float, `char(n)` тайком
добивает пробелами, а одна битая строка роняет весь импорт. Здесь разбираем
систему типов PostgreSQL **системно**: числа, строки, `boolean`, и главное —
**приведение типов** (`CAST`/`::`), неявные и явные преобразования, `NULL`.

Многое (`::`, `COALESCE`, `NULLIF`, целочисленное деление, `round`) уже мелькало
в прошлых лабах по касательной — теперь собираем это в цельную картину и
закрываем подводные камни.

Теория разбита на короткие документы в папке [`theory/`](theory/) — читайте по
порядку. Общая подборка курсов и тренажёров — в [`../RESOURCES.md`](../RESOURCES.md).

## Как запустить

```bash
# из корня репозитория один раз:
docker compose up -d

cd lab013
./run.sh                 # накатит схему и прогонит все запросы

# либо зайти руками и экспериментировать:
../psql.sh lab013
```

`run.sh` идемпотентен: пересоздаёт схему `lab013` с нуля при каждом запуске.

## Доменная область

Интернет-магазин + импорт «сырых» данных. Четыре таблицы (полная схема и подробная
шапка о том, как подобраны данные, — в [`init.sql`](init.sql)). **Типы колонок
подобраны так, чтобы каждая демонстрировала тему.**

### `products` — товары (витрина типов)
| Колонка | Тип | Смысл / зачем такой тип |
|---------|-----|-------------------------|
| `id` | `smallint` | небольшой целочисленный ключ (int2) |
| `sku` | `varchar(8)` | код товара из **7 символов** при пределе `varchar(8)` (усечение — задачи 6–7) |
| `name` | `text` | название |
| `category` | `varchar(16)` | категория |
| `price` | `numeric(10,2)` | **деньги**: точный тип, ровно 2 знака |
| `weight_kg` | `double precision` | вес: **приближённый** float (округление — задача 4) |
| `units` | `integer` | остаток, шт. (целочисленное деление — задача 2) |
| `in_stock` | `boolean` | есть ли в наличии (агрегация булевых — задача 8) |

### `orders` — заказы (денежные ловушки)
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | `integer` | идентификатор |
| `customer` | `text` | имя клиента |
| `revenue_rub` | `integer` | сумма заказа, **целые ₽** (int-деление в «среднем чеке») |
| `discount_rub` | `integer` | скидка, ₽; **`NULL` = скидки не было** (для `COALESCE`) |
| `items` | `integer` | число товаров в заказе |
| `is_paid` | `boolean` | оплачен ли заказ |

### `channels` — маркетинговые каналы (безопасное деление)
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | `smallint` | идентификатор |
| `name` | `text` | канал |
| `visits` | `integer` | визиты; **у «Баннеров» = 0** (деление на ноль → `NULLIF`) |
| `orders_cnt` | `integer` | заказы с канала |
| `cost_rub` | `numeric(10,2)` | затраты, ₽ |

### `raw_import` — «сырой» импорт (числа как текст)
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | `integer` | идентификатор |
| `source_name` | `text` | название позиции |
| `price_text` | `text` | цена **текстом**: `'199'`, `' 590 '`, `'12.5'`, `'бесплатно'`, `''` |
| `qty_text` | `text` | количество текстом: `'10'`, `'  7 '`, `'нет'` |
| `is_valid` | `boolean` | прошла ли строка проверку (флаг для безопасного приведения) |

> **Как данные демонстрируют ловушки:**
> - **Целочисленное деление.** В `orders` суммы целые: наивный `sum/count` даёт
>   средний чек **4735** вместо **4735.56**, среднее число товаров — **1** вместо
>   **1.44** (задача 2).
> - **round double vs numeric.** Веса `2.5` и `0.5` кг подобраны так, что
>   `round(weight_kg)` (double, банковское) и `round(weight_kg::numeric)` (numeric,
>   «половина вверх») **расходятся**: 2/3 и 0/1 (задача 4).
> - **char(n) и пробелы.** `sku` из 7 символов, добитый до `char(10)`, занимает
>   **10 байт** и сравнивается **без учёта хвостовых пробелов** — в отличие от
>   `text` (задача 7).
> - **Грязный текст.** В `raw_import` четыре валидные строки (включая с пробелами
>   и дробью) и три битые (`'бесплатно'`, `''`, `'нет'`), помеченные
>   `is_valid = false`, — приводим только годные, не роняя запрос (задачи 5, 13).
> - **Деление на ноль / NULL.** Канал «Баннеры» с `visits = 0` и `NULL`-скидки в
>   `orders` — материал для `NULLIF`, `COALESCE`, `GREATEST`/`LEAST` (задачи 3, 9, 10, 14).

> **Про хардкор.** Задача 13 проходит по **всей** «сырой» таблице (и валидные, и
> битые строки) и приводит числа-как-текст к `numeric`/`int` через `CASE`-обёртку,
> которая выполняет приведение только для годных строк — запрос **не падает** даже
> на `'бесплатно'` и `''`, без единой регулярки (regexp — тема lab015). Задача 14
> собирает наивную формулу «среднего чека», которая **не падает, но ошибается в
> трёх местах сразу** (теряет `NULL`-скидки, режет целочисленным делением, считает
> по всем заказам вместо оплаченных), и рядом — правильную с
> `COALESCE`/`FILTER`/`round`. `NULLIF` нужен именно в правильной формуле: у
> клиента без оплат (Глеб) знаменатель «оплаченных» = 0, и `NULLIF(0,0) → NULL`
> спасает от деления на ноль (у Глеба `correct = NULL`). Обе — ровно на материале
> lab001–lab013.
