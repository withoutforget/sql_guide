Здесь — три главных «подводных камня» и полезные функции. Всё это причины багов
в реальных отчётах.

## Целочисленное деление — ловушка №1

Оператор `/` **сохраняет тип операндов**. Если оба целые — деление тоже целое:
дробная часть **отбрасывается** (не округляется — именно отбрасывается):

```sql
SELECT 5 / 2;        -- 2      (int / int → int, .5 отброшена!)
SELECT 7 / 2;        -- 3
SELECT 5 / 2.0;      -- 2.5000000000000000  (один операнд numeric → результат numeric)
SELECT 5.0 / 2;      -- 2.5000000000000000
SELECT 5::numeric / 2; -- 2.5000000000000000
```

> Длинный «хвост нулей» у `numeric`-деления — не случайность: деление `numeric`
> выполняется с запасом по масштабу (не менее 16 знаков после запятой), поэтому
> печатается `2.5000000000000000`, а не `2.5`. Это ровно та точность, ради которой
> `numeric` и существует; лишние нули убираются округлением: `round(5.0 / 2, 1)` → `2.5`.

Это классический источник ошибок в агрегатах: `sum(...)` от целой колонки —
`bigint`, и `sum / count` снова считается **целым делением**:

```sql
-- 42620 / 9 должно быть 4735.56, а наивно:
SELECT sum(revenue_rub) / count(*)  FROM orders;             -- 4735    ← врёт
SELECT round(sum(revenue_rub)::numeric / count(*), 2) FROM orders;  -- 4735.56 ← верно
```

**Как чинить** (любой из способов приводит к дробному делению):

```sql
x::numeric / y          -- явное приведение числителя
x * 1.0 / y             -- домножить на дробный литерал (1.0 — numeric)
avg(x)                  -- агрегат avg() СРАЗУ считает дробно (не целочисленно)
```

### Остаток и родственники

```sql
SELECT 7 % 2;        -- 1     оператор остатка от деления
SELECT mod(7, 2);    -- 1     то же функцией
SELECT div(7, 2);    -- 3     целая часть частного (явно «целое деление»)
```

## Округление — ловушка №2: `round` для `numeric` ≠ для `double`

`round(x)` округляет до целого, `round(x, n)` — до `n` знаков после запятой. Но
поведение **зависит от типа** аргумента:

| Аргумент            | Правило округления         | `round(2.5)` | `round(0.5)` |
|---------------------|----------------------------|--------------|--------------|
| `numeric`           | половина **от нуля** («вверх») | `3`      | `1`          |
| `double precision`  | половина **к чётному** (банковское) | `2` | `0`      |

```sql
SELECT round(2.5::numeric);   -- 3
SELECT round(2.5::float8);    -- 2   (! к ближайшему чётному)
SELECT round(0.5::float8);    -- 0
SELECT round(1.5::float8);    -- 2
```

Поэтому в задаче 4 вес `2.5` кг даёт `round(weight_kg)` = 2 (double, банковское),
а `round(weight_kg::numeric)` = 3 (numeric). Разные числа из одного значения!

**Второй нюанс: у `round(double, n)` НЕТ второго аргумента.**

```sql
SELECT round(123.456::float8, 2);
-- ERROR: function round(double precision, integer) does not exist
```

Функция `round(x, n)` определена **только для `numeric`**. Хотите округлить
`double` до 2 знаков — сначала приведите к `numeric`:

```sql
SELECT round(123.456::numeric, 2);   -- 123.46
```

Ещё и поэтому деньги держат в `numeric`: округление до копеек там работает
предсказуемо и с нужным числом знаков.

### `trunc`, `ceil`, `floor`

| Функция            | Что делает                       | `x = 2.7` | `x = -2.7` |
|--------------------|----------------------------------|-----------|------------|
| `trunc(x[, n])`    | отбросить дробную часть (к нулю)  | `2`       | `-2`       |
| `ceil(x)` / `ceiling` | округлить **вверх** (к +∞)     | `3`       | `-2`       |
| `floor(x)`         | округлить **вниз** (к −∞)         | `2`       | `-3`       |
| `round(x[, n])`    | округлить к ближайшему            | `3`       | `-3`       |

```sql
SELECT trunc(2.9), ceil(2.1), floor(2.9), ceil(-2.1), floor(-2.1);
--       2          3          2           -2          -3
```

## Переполнение — ловушка №3

У **целых** типов фиксированный диапазон. Выход за него — это **ошибка**, а не
«тихое» заворачивание, как в C:

```sql
SELECT 2147483647::int + 1::int;             -- ERROR: integer out of range
SELECT 32767::smallint + 1::smallint;        -- ERROR: smallint out of range
```

> ⚠️ Тонкость: `32767::smallint + 1` (без каста) **НЕ** переполняется — вернёт
> `32768` типа `integer`. Литерал `1` — это `integer`, а `smallint + integer`
> неявно повышается до `integer` (более широкого типа), и результат туда влезает.
> Чтобы получить именно `smallint + smallint` (и увидеть переполнение), оба
> операнда должны быть `smallint`: `32767::smallint + 1::smallint`.

Это хорошо: лучше явная ошибка, чем незаметно испорченные данные. Лечится
выбором более широкого типа (`bigint`) или приведением до операции:

```sql
SELECT 2147483647::bigint + 1;   -- 2147483648   (ок, bigint вмещает)
```

У `numeric` без точности переполнения нет (растёт), а у `numeric(p, s)` — ошибка
`numeric field overflow` при превышении `p` (см. файл 02). У `double` вместо
ошибки при переполнении получается `Infinity`.

## Полезные математические функции

| Функция / оператор    | Смысл                     | Пример → результат          |
|-----------------------|---------------------------|-----------------------------|
| `abs(x)`              | модуль                    | `abs(-5)` → `5`             |
| `power(a, b)` / `a ^ b` | возведение в степень    | `2 ^ 10` → `1024`           |
| `sqrt(x)`             | квадратный корень         | `sqrt(2)` → `1.4142135…`    |
| `mod(a, b)` / `a % b` | остаток                   | `mod(10, 3)` → `1`          |
| `div(a, b)`           | целая часть частного      | `div(10, 3)` → `3`          |
| `gcd(a, b)` / `lcm(a, b)` | НОД / НОК              | `gcd(12, 8)` → `4`          |
| `@ x`                 | тоже модуль (оператор)    | `@ -5` → `5`                |

```sql
SELECT abs(-42), power(2, 8), sqrt(144), mod(17, 5);
--      42        256          12         2
```

> ⚠️ `x ^ y` возвращает `double precision`, поэтому для точных денежных расчётов
> степень через `^` не берут (см. приближённость float в файле 02).
