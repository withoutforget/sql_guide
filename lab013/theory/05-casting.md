# 05 — Приведение типов: `CAST`, `::`, неявные преобразования, разрешение типов

**Приведение типа (cast)** — превращение значения одного типа в другой. Это
сердце темы: понимаешь, когда PostgreSQL приводит сам, а когда требует приведения
руками — и половина «непонятных ошибок» исчезает.

## Три формы записи

Все три делают одно и то же:

```sql
CAST(price AS text)     -- 1) стандартный ANSI-синтаксис
price::text             -- 2) короткий синтаксис PostgreSQL (самый частый)
text(price)             -- 3) функция-имя типа (работает не для всех типов)
```

Синтаксис `::` короче и читается слева направо («значение — как такой-то тип»).
`CAST(...)` переносим между СУБД. Оба эквивалентны; выбирайте по вкусу.
`::` уже эпизодически встречался в прошлых лабах (например, `count::numeric`
для дробного деления) — теперь разбираем его системно.

## Три уровня «строгости» приведения

PostgreSQL различает три вида приведений — от самого автоматического к самому
ручному. Это ключевая мысленная модель темы.

| Уровень       | Когда применяется                          | Пример |
|---------------|--------------------------------------------|--------|
| **Неявное** (implicit)   | СУБД приводит **сама**, молча, в любом выражении | `int` → `numeric` в `2 + 2.5` |
| **При присваивании** (assignment) | СУБД приводит сама, но **только** при записи в колонку (`INSERT`/`UPDATE`) | `numeric` → `varchar` при вставке |
| **Явное** (explicit)     | **только** если вы написали `CAST`/`::`    | `text` → `int` |

### Неявные приведения: где «само работает»

Внутри одного числового семейства PostgreSQL расширяет тип к «более широкому»
автоматически — чтобы не терять данные:

```sql
SELECT 2 + 2.5;          -- 4.5   (int неявно → numeric)
SELECT pg_typeof(1 + 1.0); -- numeric
```

Такое приведение безопасно (целое всегда влезает в numeric), поэтому разрешено
без спроса.

### Где неявное НЕ срабатывает — и как чинить

Между «далёкими» типами (например, текст и число) неявного приведения **нет**, и
вы получите ошибку `operator does not exist`:

```sql
SELECT '10'::text = 10;
-- ERROR: operator does not exist: text = integer
-- HINT: You might need to add explicit type casts.
```

PostgreSQL не угадывает, что вы имели в виду (сравнить как числа? как строки?),
и требует решения от вас — **явным приведением**:

```sql
SELECT '10'::int = 10;         -- true   (сравниваем как числа)
SELECT '10' = 10::text;        -- true   (сравниваем как строки)
```

Это же — причина ошибки в `WHERE sku = 5` (varchar против integer): либо
`WHERE sku = '5'`, либо `WHERE sku::int = 5` (если в sku лежат числа).

### Приведение при присваивании (assignment)

При вставке в колонку PostgreSQL приводит охотнее, чем в выражениях. Например,
`numeric` можно вставить в `varchar`-колонку без ручного cast:

```sql
CREATE TABLE t (code varchar(10));
INSERT INTO t VALUES (12345);   -- ок: 12345 (int) → '12345' (varchar) при присваивании
```

А ещё при присваивании в `numeric(p, s)` значение **округляется** до `s` знаков,
а в `varchar(n)` — вставка длинной строки **падает с ошибкой** (в отличие от
явного `::varchar(n)`, который усекает — см. файл 04):

```sql
INSERT INTO t2 (price) VALUES (12.349);   -- в numeric(10,2) ляжет 12.35 (округлилось)
```

## Литералы и вывод типа

Напомним из файла 01: строковый литерал `'123'` — это **`unknown`**, его тип
подбирается по контексту, а числовой `123` — сразу `integer`. Поэтому:

```sql
SELECT '2024-01-01'::date;   -- строку приводим к дате явно
SELECT '3.14'::numeric;      -- строку к числу
SELECT '{1,2,3}'::int[];     -- строку к массиву (массивы — lab018)
```

`unknown`-литерал легко приводится куда угодно — потому что у него ещё «нет»
типа. А вот уже типизированный `text` — только явно.

## Разрешение общего типа (type resolution)

Когда выражение **объединяет несколько значений разных типов**, PostgreSQL
выбирает **один общий тип** и приводит к нему всё. Это происходит в:

- ветках `CASE` (`THEN`/`ELSE`);
- столбцах `UNION`/`INTERSECT`/`EXCEPT`;
- аргументах `COALESCE`/`GREATEST`/`LEAST`;
- элементах массива и строках `VALUES`.

Правило: тип выбирается «наименьший, вмещающий все». Числа сходятся к более
широкому (`int` + `numeric` → `numeric`):

```sql
SELECT pg_typeof(CASE WHEN true THEN 1 ELSE 2.5 END);   -- numeric
SELECT pg_typeof(x) FROM (SELECT 1 UNION SELECT 2.5) u(x);  -- numeric
SELECT pg_typeof(coalesce(1, 2.5));                     -- numeric
```

Если общий тип найти **нельзя** — ошибка. Причём вид ошибки зависит от того,
литерал перед нами или уже типизированное значение:

```sql
-- обе ветки числовые/строковые, но 'x' — unknown-литерал: PG пытается привести
-- его к integer (по первой ветке) и спотыкается на содержимом:
SELECT CASE WHEN true THEN 1 ELSE 'x' END;
-- ERROR: invalid input syntax for type integer: "x"

-- если 'x' уже настоящий text — типы просто несовместимы:
SELECT CASE WHEN true THEN 1 ELSE 'x'::text END;
-- ERROR: CASE types text and integer cannot be matched
```

Практический вывод: **следите, чтобы все ветки/столбцы были совместимого типа**.
Хотите смешать число и текст в одной колонке — приведите всё к `text` явно
(`1::text`). В задаче 11 собираем разнотипные ветки корректно.

> ⚠️ Тонкость про деньги: `UNION` числа `numeric` и `double precision` даёт
> **`double precision`** (более «широкий» по правилам PG) — и точность денег
> теряется. Если объединяете суммы с чем-то float — приводите к `numeric` явно.

## Ошибки приведения и как их пережить

Приведение строки к числу падает, если строка — **не** корректное число:

```sql
SELECT 'abc'::int;   -- ERROR: invalid input syntax for type integer: "abc"
SELECT ''::int;      -- ERROR: invalid input syntax for type integer: ""  (пустая строка тоже!)
SELECT '12.5'::int;  -- ERROR: invalid input syntax for type integer: "12.5"  (дробь → к int нельзя)
```

При этом пробелы по краям **прощаются**, а дробь приводится к `numeric`:

```sql
SELECT ' 42 '::int;      -- 42    (крайние пробелы обрезаются)
SELECT '12.5'::numeric;  -- 12.5  (в numeric — ок)
```

**В PostgreSQL 16 «безопасного» приведения из коробки нет:** одна битая строка
роняет весь запрос. Приёмы защиты (без забегания в lab015 с регулярками):

1. **Отфильтровать** заведомо плохие строки в `WHERE` (фильтр применяется до
   вычисления `SELECT`, поэтому cast на них не выполнится) — задача 5;
2. **Обернуть cast в `CASE`**, который приводит только когда безопасно
   (`CASE WHEN is_valid THEN price_text::numeric END`) — `CASE` вычисляет лишь
   выбранную ветку, битые строки не приводятся — задача 13.

> В PostgreSQL 18+ появился синтаксис `CAST(x AS int DEFAULT NULL ON ERROR)` —
> «безопасный cast». **У нас PostgreSQL 16, его нет** — не используем.
