## Зачем вообще типы

**У каждого значения и у каждого столбца есть тип данных.** Тип — это не
формальность, он задаёт три вещи:

1. **множество допустимых значений** (в `integer` не положишь `3.5` или `'кот'`);
2. **набор операций** (числа можно складывать, строки — конкатенировать `||`,
   даты — вычитать; а вот сложить дату со строкой нельзя);
3. **представление и размер в памяти** (`integer` — 4 байта, `bigint` — 8,
   `text` — переменная длина).

Тип — это «контракт»: СУБД заранее знает, что можно делать со значением, и
ловит ошибки (сложить число с текстом) ещё до выполнения, а не выдаёт мусор.
Отсюда же берутся все «подводные камни» этой лабы: целочисленное деление,
потеря точности, переполнение — всё это следствия того, *какой тип* у значения.

## Категории типов

PostgreSQL — строго типизированная СУБД с богатым набором типов. Основные
категории (в этой лабе разбираем числа, строки и логический тип; остальные —
обзорно, с указанием, где копать):

| Категория        | Типы (примеры)                              | Где подробно |
|------------------|---------------------------------------------|--------------|
| **Числовые**     | `smallint`, `integer`, `bigint`, `numeric`, `real`, `double precision` | эта лаба, файлы 02–03 |
| **Символьные**   | `text`, `varchar(n)`, `char(n)`             | эта лаба, файл 04 |
| **Логический**   | `boolean`                                   | эта лаба, файл 04 |
| **Дата/время**   | `date`, `time`, `timestamp`, `interval`     | **lab014** |
| **Строки/поиск** | регулярные выражения, `pg_trgm`             | **lab015–lab016** |
| **JSON**         | `json`, `jsonb`                             | **lab017** |
| **Составные**    | массивы `[]`, диапазоны, `enum`, `record`   | **lab018** |
| **Прочие**       | `uuid`, `bytea`, сетевые `inet`, `money`*   | по мере надобности |

\* Отдельный тип `money` в PostgreSQL есть, но им **не пользуются** для денег:
он привязан к настройке локали и неудобен. Деньги хранят в `numeric` — почему,
разбираем в файле 02.

## Как узнать тип выражения: `pg_typeof`

`pg_typeof(выражение)` возвращает **имя типа** результата. Незаменимо, когда не
уверен, что «родилось» из выражения (а это частый источник багов):

```sql
SELECT pg_typeof(1);            -- integer
SELECT pg_typeof(1.5);          -- numeric      (число с точкой → numeric, НЕ float)
SELECT pg_typeof(1 + 1);        -- integer
SELECT pg_typeof(1 / 2);        -- integer      (! целочисленное деление, см. файл 03)
SELECT pg_typeof(1.0 / 2);      -- numeric
SELECT pg_typeof(true);         -- boolean
SELECT pg_typeof('привет');     -- unknown      (см. ниже про литералы)
SELECT pg_typeof('привет'::text); -- text
```

На колонках он показывает объявленный тип, а на выражениях — **выведенный**
(результирующий) тип:

```sql
SELECT
    pg_typeof(price)        AS t_price,     -- numeric
    pg_typeof(units / 2)    AS t_int_div,   -- integer  (int / int → int!)
    pg_typeof(price / 2)    AS t_num_div    -- numeric
FROM products LIMIT 1;
```

> Полезно завести привычку: столкнулся со странным результатом деления или
> округления — оберни выражение в `pg_typeof` и посмотри, какой у него тип.
> Половина «загадок» объясняется именно типом.

## «Псевдотип» литералов: `unknown`

Когда вы пишете **строковый литерал в кавычках** — `'123'`, `'2024-01-01'`,
`'true'` — PostgreSQL сначала НЕ знает его тип и помечает как **`unknown`**
(«неизвестный»). Конкретный тип выбирается позже — по контексту:

```sql
SELECT pg_typeof('123');            -- unknown
SELECT '123' + 0;                   -- 123     ('123' стал integer по соседству с 0)
SELECT '2024-01-01'::date;          -- дата    (мы сами указали тип)
INSERT INTO products(sku) VALUES ('EL-9'); -- '' станет varchar — по типу колонки
```

Числовые же литералы получают тип **сразу** по форме записи:

```sql
SELECT pg_typeof(123);      -- integer   (целое без точки)
SELECT pg_typeof(123.4);    -- numeric   (с точкой)
SELECT pg_typeof(1e3);      -- numeric   (точка ИЛИ экспонента → numeric, не float!)
SELECT pg_typeof(9999999999); -- bigint   (не влезло в integer → следующий размер)
SELECT pg_typeof(1e3::float8); -- double precision (float — только явным приведением)
```

Это различие — почему `'123'` (unknown/строка) и `123` (integer) ведут себя
по-разному, и почему `WHERE sku = 5` (текст против числа) даёт ошибку, а
`WHERE sku = '5'` — нет. Механику приведения разберём в файле 05.
