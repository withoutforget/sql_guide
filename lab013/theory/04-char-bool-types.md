## Символьные (строковые) типы

Здесь — только про **типы** и приведения к тексту/из текста. Строковые функции
(`substring`, `split_part`, `format`) и регулярные выражения — тема **lab015**.

| Тип                            | Что это                                            |
|--------------------------------|----------------------------------------------------|
| `text`                         | строка **любой** длины. Тип по умолчанию для текста. |
| `varchar(n)` / `character varying(n)` | строка **до** `n` символов (меньше — можно).  |
| `varchar` (без `n`)            | то же, что `text` (без ограничения длины).         |
| `char(n)` / `character(n)`     | строка **ровно** `n` символов, добивается пробелами. |

### `text` vs `varchar(n)`: производительность одинакова

Частый миф — что `varchar` «быстрее» `text`. В PostgreSQL это **не так**: все три
типа хранятся одинаково, разницы в скорости нет. `varchar(n)` отличается только
**проверкой длины**. Практическое правило: берите **`text`**, а `varchar(n)`
— когда ограничение длины действительно нужно как бизнес-правило (код из 8
символов, телефон и т.п.).

### `varchar(n)`: вставка — ошибка, приведение — усечение

Важнейший нюанс, который путают. Если строка длиннее `n`:

- **Вставка/присваивание** слишком длинной строки в колонку `varchar(n)` →
  **ошибка**:

  ```sql
  CREATE TABLE t (v varchar(5));
  INSERT INTO t VALUES ('abcdefghij');
  -- ERROR: value too long for type character varying(5)
  ```

- А вот **явное приведение** `::varchar(n)` молча **обрезает** до `n`:

  ```sql
  SELECT 'abcdefghij'::varchar(5);   -- 'abcde'   (усекло без ошибки!)
  ```

Это две разные операции с разным поведением — деталь про «cast vs assignment»
разберём в файле 05. В задаче 6 мы пользуемся именно усечением через cast.

### `char(n)`: коварная набивка пробелами

`char(n)` **хранит ровно `n` символов**, добивая справа пробелами. Эти хвостовые
пробелы почти всегда мешают:

```sql
SELECT length('abc'::char(10));         -- 3    (length хвост «прячет»)
SELECT octet_length('abc'::char(10));   -- 10   (а байтов реально 10 — набивка!)
```

Самый опасный сюрприз — **сравнение**: у типа `char` хвостовые пробелы
**незначимы**, а у `text`/`varchar` — значимы:

```sql
SELECT 'abc  '::char(5) = 'abc'::char(5);   -- true   (char игнорирует хвостовые пробелы)
SELECT 'abc  '::text    = 'abc'::text;      -- false  (text их учитывает)
```

Отсюда классические баги: одна колонка `char(n)`, другая `text` — и «одинаковые»
на вид значения не джойнятся. **Совет: не используйте `char(n)`** без веской
причины; берите `text` или `varchar(n)`. Тип `"char"` (в кавычках, 1 байт) —
внутренний, его вообще не трогаем. Демонстрация набивки — в задаче 7.

### Приведение к тексту и из текста

Почти всё приводится к `text` (для вывода, конкатенации, логов) и обратно:

```sql
SELECT 42::text;             -- '42'
SELECT true::text;           -- 'true'
SELECT 3.14::text;           -- '3.14'
SELECT (12.5)::text || ' ₽'; -- '12.5 ₽'
SELECT '42'::int;            -- 42     (из текста в число — если текст валиден)
```

Про ошибки обратного приведения (`'abc'::int`) — файл 05.

## Логический тип `boolean`

`boolean` хранит одно из трёх: `TRUE`, `FALSE` или `NULL` (неизвестно). На входе
понимает много представлений (регистр не важен):

| → `TRUE`                         | → `FALSE`                          |
|----------------------------------|------------------------------------|
| `TRUE`, `'t'`, `'true'`, `'yes'`, `'y'`, `'on'`, `'1'` | `FALSE`, `'f'`, `'false'`, `'no'`, `'n'`, `'off'`, `'0'` |

```sql
SELECT 'yes'::boolean, 't'::boolean, '1'::boolean;   -- t t t
SELECT 'no'::boolean,  'off'::boolean, '0'::boolean; -- f f f
```

### Трёхзначная логика с `NULL`

`boolean` участвует в **трёхзначной логике** (`TRUE`/`FALSE`/`NULL`): любое
сравнение с `NULL` даёт `NULL`, а не `TRUE`/`FALSE`. Это подробно разобрано в
[lab001, файл 02](../../lab001/theory/02-where.md) — здесь только напоминание:
`WHERE flag` пропускает строку, только если `flag` именно `TRUE` (не `NULL`).

### Приведения `boolean ↔ int` и `↔ text`

```sql
SELECT true::int, false::int;    -- 1  0
SELECT 1::boolean, 0::boolean;   -- t  f     (только 0 и 1 приводятся из int)
SELECT true::text, false::text;  -- 'true'  'false'
```

Приведение `boolean → int` бесценно для агрегации: **`sum(flag::int)`** считает,
у скольких строк флаг истинен, — то же самое, что `count(*) FILTER (WHERE flag)`.
А из булевых агрегатов (введены в [lab009](../../lab009/theory/02-collections-and-boolean.md))
пригодятся:

- `bool_and(усл)` — истинно, если условие верно для **всех** строк (∀);
- `bool_or(усл)` — истинно, если верно **хотя бы для одной** (∃).

Всё это собрано в задаче 8.
