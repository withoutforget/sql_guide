# 02 — Числовые типы: целые, `numeric`, `real`/`double`

[← система типов](01-type-system.md) · [оглавление](../README.md) · [дальше: операции над числами →](03-numeric-ops.md)

---

Числовых типов три семейства: **целые**, **точные дробные** (`numeric`) и
**приближённые дробные** (плавающая точка). Выбор между ними — не вкусовщина: от
него зависит, потеряете ли вы копейки и переполнится ли счётчик.

## Целые числа

| Тип                | Другое имя | Размер | Диапазон                                   |
|--------------------|-----------|--------|--------------------------------------------|
| `smallint`         | `int2`    | 2 байта | −32 768 … 32 767                          |
| `integer` / `int`  | `int4`    | 4 байта | −2 147 483 648 … 2 147 483 647 (≈ ±2,1 млрд) |
| `bigint`           | `int8`    | 8 байт | ≈ −9,2·10¹⁸ … 9,2·10¹⁸                      |

- **`integer`** — выбор по умолчанию для «обычных» целых: id, количества, счётчики.
- **`smallint`** — когда точно известно, что значения малы (возраст, номер месяца)
  и таблица огромна: экономит место.
- **`bigint`** — для больших id (миллиарды строк), денег в копейках, суммарных
  счётчиков. `sum(integer)` сам возвращает `bigint`, чтобы не переполниться.

Целочисленные литералы получают наименьший подходящий тип: `100` → `integer`,
`9999999999` (> 2,1 млрд) → `bigint`.

### `serial` / `bigserial` — не тип, а «автоинкремент»

`serial` и `bigserial` — это **синтаксическое удобство**, а не отдельные типы.
`id serial` разворачивается в `integer` со значением по умолчанию
`nextval(...)` из последовательности (счётчика). Подробно генерацию ключей
разберём в **lab019/lab020**; здесь достаточно знать, что «под капотом» это
обычный `integer`/`bigint`.

## Точные дробные: `numeric(p, s)` — тип для ДЕНЕГ

`numeric` (синоним — `decimal`) хранит числа **с произвольной точностью и точно**,
без ошибок округления. Два параметра:

```
numeric(p, s)
        │  └─ scale:     сколько знаков ПОСЛЕ запятой
        └──── precision: сколько всего значащих цифр (до и после запятой)
```

- `numeric(10, 2)` — всего 10 значащих цифр, из них 2 после запятой → до
  `99 999 999.99`. Классика для цены в рублях с копейками.
- `numeric` **без параметров** — «сколько угодно» цифр, хранит ровно то, что
  положили. Удобно для промежуточных вычислений.

```sql
SELECT 12345.678::numeric(10,2);   -- 12345.68   (округляется до scale=2)
SELECT 1.0 / 3;                    -- 0.33333333333333333333 (numeric, длинный точный хвост)
SELECT pg_typeof(890.00);          -- numeric
```

**Почему деньги — только `numeric`.** Плавающая точка (ниже) хранит числа
приближённо, и `0.1 + 0.2` там ≠ `0.3`. Для денег это недопустимо: копейки
«поплывут». `numeric` считает точно, как на бумаге. Правило: **деньги, курсы,
всё, где важна каждая копейка — `numeric`.**

### Округление в `numeric` — «половина вверх»

`numeric` при округлении использует правило **round half away from zero**
(половину округляем ОТ нуля, попросту «вверх» по модулю):

```sql
SELECT round(0.5::numeric);   -- 1
SELECT round(1.5::numeric);   -- 2
SELECT round(2.5::numeric);   -- 3   (а не 2!)
SELECT round(2.675::numeric, 2); -- 2.68
```

Это **отличается** от округления `double precision` (там «к чётному») — важный
нюанс, разбираем в файле 03.

### Переполнение `numeric`

Само число `numeric` (без точности) **не переполняется** — оно просто растёт:

```sql
SELECT 99999999999999999999::numeric + 1;  -- 100000000000000000000
```

Но если задан `numeric(p, s)` и результат не влезает в `p` значащих цифр — будет
ошибка:

```sql
SELECT 12345.67::numeric(5,2);
-- ERROR: numeric field overflow
-- DETAIL: A field with precision 5, scale 2 must round to an absolute value less than 10^3.
```

## Приближённые дробные: `real` и `double precision`

| Тип                | Другое имя | Размер | Точность                     |
|--------------------|-----------|--------|------------------------------|
| `real`             | `float4`  | 4 байта | ~6 значащих десятичных цифр  |
| `double precision` | `float8`  | 8 байт | ~15 значащих десятичных цифр |

Это числа с **плавающей точкой** по стандарту IEEE 754. Они быстрые и компактные,
но хранят значения **приближённо** — как двоичную дробь, в которую не все
десятичные дроби ложатся точно. Отсюда знаменитое:

```sql
SELECT 0.1::float8 + 0.2::float8;             -- 0.30000000000000004  (!)
SELECT (0.1::float8 + 0.2::float8) = 0.3::float8;  -- false
```

> ⚠️ Обратите внимание: `SELECT 0.1 + 0.2;` в PostgreSQL даёт **ровно** `0.3` —
> потому что литералы `0.1` и `0.2` без явного приведения имеют тип `numeric`
> (точный!), а не float. «Поплывёт» именно когда тип — `real`/`double`.

Когда использовать float: научные измерения, координаты, вес, проценты — где
пятнадцатый знак не важен и нужна скорость. **Никогда — для денег.**

### Спецзначения: `Infinity`, `-Infinity`, `NaN`

Плавающая точка (а также `numeric`) умеет хранить три особых значения:

```sql
SELECT 'Infinity'::float8;    -- Infinity   (+бесконечность)
SELECT '-Infinity'::float8;   -- -Infinity
SELECT 'NaN'::float8;         -- NaN        (Not a Number, «не число»)
```

`NaN` возникает при неопределённых операциях (`0.0/0.0` даёт ошибку, но
`'NaN'` можно задать явно). В PostgreSQL у `NaN` **нестандартная семантика**,
о которой надо знать:

```sql
SELECT 'NaN'::float8 = 'NaN'::float8;   -- true   (в математике было бы false!)
SELECT 'NaN'::float8 > 1e308;           -- true   (NaN считается БОЛЬШЕ всего)
```

Причина: чтобы `NaN` можно было сортировать, группировать и индексировать,
PostgreSQL договорился считать `NaN` равным самому себе и большим любого числа.
Практическое следствие — в `ORDER BY value` строки с `NaN` уезжают **в самый
конец** (после `+Infinity`). То же правило действует и для `numeric`. Смотрите
задачу 12.

## Сводка: что выбрать

| Задача                        | Тип                    |
|-------------------------------|------------------------|
| id, количество, счётчик       | `integer` (или `bigint`) |
| **деньги, цена, курс**        | **`numeric(p, s)`**    |
| измерение, вес, координата, % | `real` / `double precision` |
| очень большие целые           | `bigint`               |
