# 02 — Условие соединения: `ON`, `USING`, `NATURAL`, алиасы

[← зачем джойны](01-why-joins.md) · [оглавление](../README.md) · [дальше: INNER JOIN →](03-inner-join.md)

---

Соединять две таблицы можно, а вот **по какому условию** их склеивать — надо
как-то указать. В SQL для этого есть три способа: `ON`, `USING` и
`NATURAL JOIN`. Плюс отдельная важная тема — **квалификация имён** и **алиасы
таблиц**, без которых джойны просто не пишутся. Разберём всё по порядку.

## `ON <условие>` — универсальный и главный способ

`ON` — это условие, которое проверяется для каждой пары строк (см. мысленную
модель из [файла 01](01-why-joins.md)). Пара попадает в результат, только если
`ON` истинно.

```sql
SELECT p.name, c.name AS category
FROM products  AS p
JOIN categories AS c  ON c.id = p.category_id;
```

Ключевое, что нужно понять: **`ON` — это не обязательно равенство.** Туда можно
писать любое логическое выражение, как в `WHERE`:

```sql
-- соединение по равенству (99% случаев):
... ON c.id = p.category_id

-- по нескольким условиям сразу:
... ON o.product_id = p.id AND o.ordered_at >= '2024-06-01'

-- по неравенству (нужно, например, чтобы соединить строку с более дорогими
-- товарами — редкий, но законный приём):
... ON p2.price > p1.price
```

`ON` с равенством по колонкам — это **эквисоединение** (equi-join), самый частый
случай. `ON` с `>`, `<`, `BETWEEN` — **тета-соединение** (theta-join); оно
пригодится в задачах «на подумать».

## `USING (колонка)` — сокращение, когда имена колонок совпадают

Если соединяемые колонки в **обеих таблицах называются одинаково**, можно
записать коротко через `USING`:

```sql
-- Пусть в обеих таблицах ключ называется category_id.
SELECT *
FROM products
JOIN categories USING (category_id);
```

`USING (category_id)` — это ровно то же, что
`ON products.category_id = categories.category_id`, но с двумя отличиями в
поведении:

1. **Колонка из `USING` в результате одна, не две.** При обычном `ON` в выборке
   `SELECT *` были бы *обе* колонки `category_id` (из products и из categories) —
   с одинаковыми именами. `USING` их **объединяет в одну**, поэтому к ней можно
   обращаться без префикса таблицы: просто `category_id`.
2. Требует, чтобы имена **и типы** колонок совпадали.

В нашей схеме имена ключей *разные* (`products.category_id` и `categories.id`),
поэтому `USING` здесь не применить — нужен `ON`. `USING` красив, когда схему
проектировали единообразно (везде `category_id`).

## `NATURAL JOIN` — «соединить по всем одноимённым колонкам»

`NATURAL JOIN` идёт ещё дальше: он **сам** находит все колонки с одинаковыми
именами в обеих таблицах и соединяет по равенству *всех* их сразу — условие
писать вообще не надо.

```sql
SELECT *
FROM orders
NATURAL JOIN products;   -- соединит по ВСЕМ совпадающим именам колонок
```

Звучит удобно, но на практике **`NATURAL JOIN` опасен и его избегают**. Почему:

- Он соединяет по *именам*, а имена могут совпасть случайно. У нас в `orders` и
  `products` совпадают `id` (у обоих есть колонка `id`!). `NATURAL JOIN` попытается
  соединить `orders.id = products.id` — полная бессмыслица, но синтаксически
  всё «правильно», ошибки не будет — будет тихо неверный результат.
- Стоит кому-то добавить в таблицу новую колонку с «популярным» именем
  (`created_at`, `name`, `status`) — и условие джойна молча меняется, запрос
  начинает возвращать другое. Это **хрупко**.

Правило: **`NATURAL JOIN` — почти никогда.** Явный `ON` (или `USING`) всегда
понятнее и надёжнее. Знать про `NATURAL` надо, чтобы узнавать его в чужом коде.

### Сравнение трёх способов

| Способ | Условие | Дубль колонки в результате | Когда использовать |
|--------|---------|:--:|--------------------|
| `ON a.x = b.y` | пишете явно, любое | да (обе колонки) | **всегда** — универсально |
| `USING (x)` | равенство по одноимённой `x` | нет (одна общая) | имена ключей совпадают и хочется короче |
| `NATURAL JOIN` | равенство по *всем* одноимённым | нет | **избегать** — слишком неявно и хрупко |

## Квалификация имён: `таблица.колонка`

Как только в запросе больше одной таблицы, возникает вопрос: если в обеих есть
колонка `id` (а у нас она есть везде), то что означает просто `id`?

Ответ: **ничего определённого — это ошибка неоднозначности.**

```sql
-- ❌ ОШИБКА: column reference "id" is ambiguous
SELECT id, name
FROM customers c
JOIN orders o ON o.customer_id = c.id;
-- Какой id имелся в виду — customers.id или orders.id? БД не угадывает.
```

Лечится **квалификацией** — указанием таблицы перед колонкой через точку:

```sql
SELECT c.id, o.id AS order_id, o.ordered_at
FROM customers c
JOIN orders o ON o.customer_id = c.id;
```

Практическое правило: **в запросе с джойнами квалифицируйте вообще все колонки**,
даже те, что есть только в одной таблице. Так запрос читается однозначно (сразу
видно, откуда колонка), и он не сломается, если завтра в другую таблицу добавят
колонку с тем же именем.

## Алиасы таблиц — короткие имена

Писать `customers.id`, `customers.name`, `customers.city` длинно. Поэтому
таблицам дают **алиасы** — короткие псевдонимы на время запроса (в lab001 мы уже
видели алиасы колонок; здесь то же самое для таблиц):

```sql
FROM customers AS c        -- дальше обращаемся как c
JOIN orders    AS o  ON o.customer_id = c.id
```

- `AS` можно опустить: `FROM customers c` — то же самое.
- Алиас обычно делают из первой буквы (`c`, `o`, `p`) или короткого слова
  (`cust`, `ord`). Главное — читаемо.
- **Если задан алиас, использовать нужно именно его**, а полное имя таблицы
  становится недоступно: `FROM customers c ... WHERE customers.city = ...` —
  ошибка, только `c.city`.

### Когда алиас таблицы ОБЯЗАТЕЛЕН

Есть случай, где без алиаса вообще никак — **self-join**, когда одна и та же
таблица участвует в запросе дважды. Тогда её нужно как-то различать:

```sql
-- сотрудник и его руководитель — обе строки из employees
SELECT e.name AS сотрудник, m.name AS руководитель
FROM employees AS e
JOIN employees AS m  ON m.id = e.manager_id;   -- e и m — две «копии» одной таблицы
```

Без алиасов `e` и `m` СУБД не поняла бы, о какой из двух «копий» `employees` идёт
речь в каждом месте. Подробно self-join — в [файле 05](05-cross-self-join.md).

## Итог

- Условие соединения почти всегда пишем через **`ON`** — он универсален и явен.
- `USING (col)` — приятное сокращение, когда ключи одноимённые; склеивает
  колонку в одну.
- `NATURAL JOIN` — знать, но **не использовать**: слишком неявно и ломко.
- В джойнах **квалифицируйте колонки** (`t.col`) и давайте таблицам **алиасы** —
  это не украшательство, а защита от неоднозначности и будущих поломок.
