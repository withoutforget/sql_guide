## Зачем вообще соединять таблицы

В lab001 почти все задачи решались на одной таблице. Но реальные данные почти
никогда не лежат в одной таблице — и это не случайность, а осознанный подход,
который называется **реляционной моделью**.

Посмотрите на наш магазин. У нас есть заказы. В каждом заказе — кто заказал,
что заказал, сколько, когда. Можно было бы хранить всё в одной «плоской» таблице:

| id | клиент | город клиента | товар | категория | цена | кол-во |
|----|--------|---------------|-------|-----------|------|--------|
| 1  | Анна   | Москва        | Наушники TWS Pro | Электроника | 5990 | 1 |
| 2  | Анна   | Москва        | Книга "SQL за месяц" | Книги | 890 | 2 |

Что здесь плохо:

- **Дублирование.** «Анна» и «Москва» повторяются в каждом её заказе. Цена
  наушников хранится в каждой строке, где их купили.
- **Аномалии обновления.** Анна переехала в Казань — нужно найти и поправить
  *все* её строки. Пропустили одну — данные противоречат сами себе.
- **Аномалии удаления.** Удалили последний заказ товара — потеряли и сведения о
  самом товаре (его цену, название).
- **Некуда деть «одинокие» сущности.** Клиент, который зарегистрировался, но
  ничего не купил, в такой таблице просто не появится — для него нет строки.

Решение реляционной модели: **разложить данные по отдельным таблицам, каждая про
одну сущность**, а связи между ними хранить через **ссылки на идентификаторы**.
Именно так устроена схема этой лабы:

- `customers` — по одной строке на клиента (имя, город — один раз);
- `products` — по одной строке на товар (цена — один раз);
- `categories` — по одной строке на категорию;
- `orders` — по строке на «товар в заказе», где вместо повторения имён и цен
  стоят **ссылки**: `customer_id` и `product_id`.

Такое разделение называют **нормализацией** (подробно — в отдельной лабе). А
операция, которая снова «склеивает» разложенные таблицы в одну широкую выборку
по мере надобности — это и есть **соединение**, `JOIN`.

## Первичные и внешние ключи

Чтобы ссылки работали, нужны две вещи.

**Первичный ключ (primary key, PK)** — колонка (или набор колонок), которая
**уникально** идентифицирует строку в таблице. У нас это `id` в каждой таблице.
Двух клиентов с одинаковым `id` быть не может — БД это гарантирует
(`PRIMARY KEY`).

**Внешний ключ (foreign key, FK)** — колонка, которая **ссылается на первичный
ключ другой таблицы**. В `orders` таких две:

```sql
CREATE TABLE orders (
    id           INT  PRIMARY KEY,
    customer_id  INT  NOT NULL REFERENCES customers(id),  -- FK → customers
    product_id   INT  NOT NULL REFERENCES products(id),   -- FK → products
    ...
);
```

`REFERENCES customers(id)` говорит БД: «в `customer_id` может стоять только тот
`id`, который реально существует в `customers`». Это **ссылочная целостность** —
БД не даст создать заказ на несуществующего клиента или удалить клиента, у
которого есть заказы (без явного указания, что делать со связанными строками).

> Важно: **джойн НЕ требует внешнего ключа.** FK — это про *целостность* данных.
> Соединять можно любые две таблицы по любому условию равенства (и не только
> равенства) колонок. Просто чаще всего соединяют именно по паре PK–FK, потому
> что связи в данных обычно так и устроены.

Связи бывают разной **кратности** (cardinality) — это полезно держать в голове,
потому что от неё зависит, «размножатся» строки при джойне или нет:

| Тип связи | Пример | Что значит |
|-----------|--------|------------|
| один-к-одному (1:1) | пользователь ↔ его профиль | одной строке слева соответствует не более одной справа |
| один-ко-многим (1:N) | категория → её товары | одной строке слева — много строк справа |
| многие-ко-многим (M:N) | клиенты ↔ товары (через заказы) | каждый клиент купил много товаров, каждый товар купили многие; реализуется через промежуточную таблицу (`orders`) |

## Декартово произведение — фундамент всех джойнов

Прежде чем разбирать `JOIN`, нужно понять, что происходит, когда в `FROM`
указано **несколько таблиц**. Ответ: получается **декартово произведение** —
**каждая строка первой таблицы склеивается с каждой строкой второй**.

Возьмём две крошечные таблицы. Пусть в `categories` только 2 строки, а в
`products` — 3:

```
categories          products
id | name           id | name           | category_id
---+------------    ---+----------------+-----------
 1 | Электроника     1 | Наушники       | 1
 2 | Книги          10 | Конструктор     | 4
                     5 | Книга SQL       | 2
```

Запрос со старым синтаксисом «таблицы через запятую» (или явный `CROSS JOIN`):

```sql
SELECT c.name AS category, p.name AS product
FROM categories c, products p;      -- 2 × 3 = 6 строк, БЕЗ всякого условия
```

Результат — **все возможные пары** (2 × 3 = 6 строк):

```
 category     | product
--------------+-------------
 Электроника  | Наушники
 Электроника  | Конструктор
 Электроника  | Книга SQL
 Книги        | Наушники
 Книги        | Конструктор
 Книги        | Книга SQL
```

Здесь много бессмысленных пар («Книги» + «Наушники»). Чтобы оставить только
осмысленные — где категория товара действительно совпадает с категорией — нужно
**условие соединения**:

```sql
SELECT c.name AS category, p.name AS product
FROM categories c, products p
WHERE p.category_id = c.id;          -- ← вот оно, условие соединения
```

```
 category     | product
--------------+-------------
 Электроника  | Наушники
 Книги        | Книга SQL
```

Из 6 пар осталось 2 — только те, где `category_id` товара равен `id` категории.

**Это и есть суть любого джойна:** взять декартово произведение и отфильтровать
его условием. `INNER JOIN`, `LEFT JOIN` и прочие — это просто *удобный и явный*
синтаксис для той же идеи (плюс `OUTER`-варианты добавляют строки-«сироты»,
которым пары не нашлось — об этом в файле 04).

### Мысленная модель джойна

Держите в голове такую картинку исполнения `a JOIN b ON <условие>`:

1. мысленно строим декартово произведение `a × b` (все пары строк);
2. для каждой пары проверяем `<условие>`;
3. оставляем только пары, где условие `TRUE`.

Реально СУБД так «в лоб» не делает — она использует умные алгоритмы (hash join,
merge join — это в лабе про производительность), но **результат** ровно такой,
как будто перебрали все пары и отфильтровали. Эта модель никогда вас не подведёт.

## Современный синтаксис: `JOIN ... ON`

Писать таблицы через запятую и условие в `WHERE` — старый стиль (SQL-89). Он
работает, но у него два минуса: (1) легко забыть условие и случайно получить
декартово произведение на миллионы строк; (2) для внешних соединений
(`LEFT`/`RIGHT`) этого синтаксиса просто не хватает.

Современный стандарт (SQL-92 и новее) — **явный `JOIN` с условием в `ON`**:

```sql
SELECT c.name AS category, p.name AS product
FROM categories AS c
JOIN products  AS p  ON p.category_id = c.id;
```

Читается почти по-русски: «взять `categories`, соединить с `products` по
условию, что `category_id` товара равен `id` категории». Условие соединения
(`ON ...`) стоит **рядом** с тем, что соединяем — не потеряется.

Дальше в этой лабе мы используем **только явный синтаксис** — он яснее, безопаснее
и обязателен для внешних соединений.

## Виды соединений — карта на всю лабу

Чтобы не запутаться, вот все виды джойнов сразу, одной таблицей. Дальше каждый
разберём подробно.

| Вид | Что возвращает | Файл |
|-----|----------------|------|
| `INNER JOIN` | только совпавшие пары строк (пересечение) | [03](03-inner-join.md) |
| `LEFT [OUTER] JOIN` | все строки левой таблицы + совпавшие правой (нет пары → `NULL`) | [04](04-outer-joins.md) |
| `RIGHT [OUTER] JOIN` | зеркально: все строки правой + совпавшие левой | [04](04-outer-joins.md) |
| `FULL [OUTER] JOIN` | все строки обеих таблиц; где пары нет — `NULL` | [04](04-outer-joins.md) |
| `CROSS JOIN` | декартово произведение (все пары, без условия) | [05](05-cross-self-join.md) |
| self-join | таблица соединяется сама с собой (иерархии) | [05](05-cross-self-join.md) |

Слова `INNER` и `OUTER` — необязательные: `JOIN` без уточнения означает `INNER
JOIN`, а `LEFT JOIN` — это то же самое, что `LEFT OUTER JOIN`. Их часто опускают.
