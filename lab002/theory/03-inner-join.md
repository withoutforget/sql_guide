## Что делает `INNER JOIN`

`INNER JOIN` (или просто `JOIN`) возвращает **только те пары строк, для которых
условие `ON` истинно**. Строки, которым пары не нашлось, в результат **не
попадают** — с обеих сторон. Это соединение «по пересечению».

```sql
SELECT
    o.id          AS order_id,
    c.name        AS customer,
    p.name        AS product,
    o.quantity
FROM orders    AS o
JOIN customers AS c  ON c.id = o.customer_id
JOIN products  AS p  ON p.id = o.product_id
ORDER BY o.id;
```

Здесь мы соединили три таблицы и получили «человекочитаемый» заказ: вместо
`customer_id = 1` видим `Анна`, вместо `product_id = 5` — `Книга "SQL за месяц"`.

Фрагмент результата:

```
 order_id | customer | product              | quantity
----------+----------+----------------------+----------
        1 | Анна     | Наушники TWS Pro     |        1
        2 | Анна     | Книга "SQL за месяц" |        2
        3 | Борис    | Ноутбук AirBook 14   |        1
        ...
```

## Что именно отсеивается — главное про `INNER`

Ключевое свойство `INNER JOIN`: **если у строки нет пары по условию — она
исчезает.** И это работает в обе стороны. Разберём на наших данных.

В `orders` 14 строк. Соединим клиентов с заказами:

```sql
SELECT c.name, o.id AS order_id
FROM customers AS c
JOIN orders    AS o  ON o.customer_id = c.id;
```

Результат — **14 строк** (по строке на каждый заказ). Но клиентов у нас **7**, а
в выборке участвуют только **6**: **Жанна (id 7) заказов не делала — и её в
результате нет вообще.** У её строки в `customers` не нашлось ни одной пары в
`orders`, поэтому `INNER JOIN` её выкинул.

Симметрично: товары `Умная колонка`, `Плед`, `Скакалка`, `Йога-коврик` никто не
заказывал — при `products JOIN orders` их тоже не будет.

> Запомните формулировку: **`INNER JOIN` показывает только тех, у кого есть
> пара.** Если вопрос звучит «покажи всех клиентов, включая тех, кто ничего не
> купил» — `INNER JOIN` не подходит, нужен `LEFT JOIN` (следующий файл).

## «Размножение» строк при `INNER JOIN` (1:N)

Сколько строк вернёт джойн? Зависит от кратности связи. Рассмотрим
`customers JOIN orders`:

- у Анны 3 заказа → её строка в `customers` даст **3** строки в результате
  (склеится с каждым её заказом);
- у Бориса 2 заказа → **2** строки;
- у Жанны 0 заказов → **0** строк.

То есть строка «одной» стороны (клиент) **повторяется столько раз, сколько у неё
пар** на «многой» стороне (заказы). Это нормально и ожидаемо — но именно отсюда
растёт **ловушка задвоения при агрегатах** (её разберём в [файле 06](06-multi-fanout-antijoin.md)):
если после такого джойна неаккуратно взять `SUM(c.something)`, значение клиента
просуммируется несколько раз.

Мысленная модель прежняя: декартово произведение, отфильтрованное по `ON`. Просто
у клиента с 3 заказами условию `ON` удовлетворяют 3 пары, у клиента с 0 заказами
— ни одной.

## Порядок таблиц в `INNER JOIN` не влияет на результат

`A JOIN B` и `B JOIN A` для `INNER JOIN` дают **один и тот же набор строк**
(порядок колонок в `SELECT` вы задаёте сами, а набор строк одинаков). `INNER
JOIN` симметричен:

```sql
FROM customers c JOIN orders o ON o.customer_id = c.id   -- то же множество строк,
FROM orders o JOIN customers c ON c.id = o.customer_id   -- что и это
```

Для **внешних** соединений (`LEFT`/`RIGHT`) порядок уже важен — там левая и
правая таблицы играют разные роли. Но об этом в файле 04.

## `INNER JOIN` + `WHERE`: обычная фильтрация

После соединения к результату можно применять всё, что мы знаем из lab001:
`WHERE`, `ORDER BY`, `GROUP BY`, агрегаты. `WHERE` в `INNER JOIN` работает
интуитивно — фильтрует уже склеенные строки:

```sql
-- заказы москвичей на сумму дороже 5000 ₽ за строку
SELECT c.name, p.name AS product, o.quantity, p.price * o.quantity AS сумма
FROM orders    AS o
JOIN customers AS c  ON c.id = o.customer_id
JOIN products  AS p  ON p.id = o.product_id
WHERE c.city = 'Москва'
  AND p.price * o.quantity > 5000
ORDER BY сумма DESC;
```

Порядок логического исполнения (из lab001) с джойнами расширяется так:

```
FROM + JOIN ... ON   — построить соединённый набор строк
  ↓
WHERE                — отфильтровать строки
  ↓
GROUP BY / HAVING    — сгруппировать / отфильтровать группы
  ↓
SELECT → DISTINCT → ORDER BY → LIMIT
```

> Тонкость: для `INNER JOIN` без разницы, писать условие в `ON` или в `WHERE` —
> результат одинаков. А вот для внешних соединений это **радикально разные вещи**
> — самая частая ловушка джойнов, ей посвящён отдельный раздел в [файле 04](04-outer-joins.md).

## Когда `INNER JOIN`, а когда нет

**Берите `INNER JOIN`, когда** вам нужны только строки, у которых есть
соответствие в обеих таблицах: «заказы с именами клиентов и товаров», «товары
вместе с их категорией» — тут строки без пары и не нужны.

**Не берите `INNER JOIN`, когда** вопрос включает «в том числе тех, у кого пары
нет»: «все клиенты, включая ничего не купивших», «сколько заказов у каждого
клиента (у кого-то ноль)», «товары, которые никто не купил». Для таких вопросов —
внешние соединения, следующий файл.
