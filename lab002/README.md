# lab002 — JOIN'ы (соединения таблиц)

Вторая лаба — про то, как **склеивать данные из разных таблиц**. В lab001 почти
все задачи жили в одной таблице; в реальной жизни данные разложены по многим
таблицам (клиенты отдельно, заказы отдельно, товары отдельно), и чтобы ответить
на осмысленный вопрос — «сколько потратил каждый клиент», «какие товары никто не
купил», «кто чей руководитель» — таблицы нужно **соединять**. Этим занимается
`JOIN`.

Цель лабы — не просто выучить синтаксис `JOIN`, а понять **мысленную модель**
(джойн = отфильтрованное декартово произведение), увидеть разницу между
`INNER`/`LEFT`/`RIGHT`/`FULL`/`CROSS`, освоить self-join и — главное — не
попадаться на две классические ловушки: **`ON` vs `WHERE`** во внешних джойнах и
**задвоение строк (fan-out)** при агрегатах.

Теория разбита на короткие документы в папке [`theory/`](theory/) — читайте по
порядку. Общая подборка курсов и тренажёров — в [`../RESOURCES.md`](../RESOURCES.md).

## Как запустить

```bash
# из корня репозитория один раз:
docker compose up -d

cd lab002
./run.sh                 # накатит схему и прогонит все запросы

# либо зайти руками и экспериментировать:
../psql.sh lab002
```

`run.sh` идемпотентен: пересоздаёт схему `lab002` с нуля при каждом запуске.

## Доменная область

Тот же интернет-магазин, что и в lab001, но категория вынесена в отдельную
таблицу и добавлена таблица сотрудников для self-join. Полная схема — в
[`init.sql`](init.sql). Данные специально подобраны так, чтобы **демонстрировать**
все виды соединений: есть клиент без заказов, товары без заказов, категория без
товаров и иерархия сотрудников с вершиной (`manager_id = NULL`).

### `categories` — категории товаров
| Колонка | Тип      | Смысл                              |
|---------|----------|------------------------------------|
| `id`    | INT (PK) | идентификатор категории            |
| `name`  | TEXT     | название (`Электроника`, `Книги`, …); «Сад» — без товаров |

### `products` — товары
| Колонка       | Тип           | Смысл                             |
|---------------|---------------|-----------------------------------|
| `id`          | INT (PK)      | идентификатор товара              |
| `name`        | TEXT          | название                          |
| `category_id` | INT (FK)      | ссылка на `categories.id`         |
| `price`       | NUMERIC(10,2) | цена за штуку, ₽                   |
| `in_stock`    | INT           | остаток на складе, шт.            |

> Товары «Спорт» (`Скакалка`, `Йога-коврик`), а также `Умная колонка` и `Плед`
> никто ни разу не заказывал — видны в анти-джойне (задача 6).

### `customers` — клиенты
| Колонка      | Тип      | Смысл             |
|--------------|----------|-------------------|
| `id`         | INT (PK) | идентификатор     |
| `name`       | TEXT     | имя               |
| `city`       | TEXT     | город             |
| `registered` | DATE     | дата регистрации   |

> Клиент **Жанна** (id 7) не сделал ни одного заказа — виден в `LEFT`/анти-джойне
> (задачи 4, 5). Клиент **Захар** (id 8) заказывал **только в июле** — на нём
> выпукло видна разница `ON` vs `WHERE` (задача 8).

### `orders` — заказы (одна строка = один товар в заказе)
| Колонка       | Тип      | Смысл                    |
|---------------|----------|--------------------------|
| `id`          | INT (PK) | идентификатор строки      |
| `customer_id` | INT (FK) | ссылка на `customers.id`  |
| `product_id`  | INT (FK) | ссылка на `products.id`   |
| `quantity`    | INT      | количество                |
| `ordered_at`  | DATE     | дата заказа               |

### `employees` — сотрудники (иерархия для self-join)
| Колонка      | Тип           | Смысл                                   |
|--------------|---------------|-----------------------------------------|
| `id`         | INT (PK)      | идентификатор                           |
| `name`       | TEXT          | имя                                     |
| `position`   | TEXT          | должность                               |
| `manager_id` | INT (FK→self) | руководитель (ссылка на `employees.id`); `NULL` у директора |
| `salary`     | NUMERIC(10,2) | оклад, ₽/мес                            |

## Что дальше

`JOIN` соединяет таблицы **вширь** — добавляет колонки. В `lab003` разберём
**операции над множествами** (`UNION`, `INTERSECT`, `EXCEPT`) — способ
комбинировать результаты запросов **ввысь**, складывая строки друг под друга, а
также увидим, как через них решаются задачи «найти общее / найти разницу» между
двумя выборками.
