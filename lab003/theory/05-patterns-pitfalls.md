Последний файл собирает практику: типовые приёмы, частые ошибки и — главное —
как выбрать между операциями над множествами, `JOIN` и подзапросами (последние —
в lab004).

## Готовые паттерны

### 1. Слить однотипные источники (с меткой)

Несколько таблиц/выборок одинаковой формы → один поток. Метка-константа
показывает происхождение строки:

```sql
SELECT id, сумма, 'июнь' AS период FROM orders_june
UNION ALL
SELECT id, сумма, 'июль' AS период FROM orders_july;
```

Берём `UNION ALL`: источники не пересекаются, дедуп не нужен и только замедлил бы.

### 2. «И то, и другое» → `INTERSECT`; «одно, но не другое» → `EXCEPT`

Строим множество-«кто удовлетворяет условию A» и множество-«кто удовлетворяет
условию B», затем пересекаем (нужны оба) или вычитаем (первое без второго). См.
задачи 8 и 9. Так выражаются вопросы «купили и X, и Y», «были в марте и в апреле»,
«активны на вебе, но не в приложении».

### 3. Разность / сверка датасетов → `EXCEPT`

«Что есть в A, но потерялось в B»: `SELECT ключ FROM A EXCEPT SELECT ключ FROM B`.
Классика — «товары каталога без заказов» (задача 7), «id из выгрузки, которых нет
в таблице», «ожидаемые записи, которых не хватает». Это альтернатива анти-джойну
из lab002.

### 4. Симметрическая разность → `(A EXCEPT B) UNION (B EXCEPT A)`

«Строки, которые есть ровно в одном из двух наборов» — расхождения в обе стороны
сразу. См. задачу 12. Полезно, когда важны различия и там, и там (что добавилось и
что пропало).

### 5. Проверка равенства двух наборов

Два запроса дают **одинаковое множество** строк ⟺ обе разности пусты:

```sql
-- если ЭТО не вернуло ни одной строки — наборы идентичны
(SELECT ... FROM A  EXCEPT  SELECT ... FROM B)
UNION
(SELECT ... FROM B  EXCEPT  SELECT ... FROM A);
```

Если результат непустой — строки в нём и есть все расхождения. Одного `A EXCEPT B`
недостаточно: оно пусто и когда `A` — лишь **подмножество** `B` (в `B` может быть
что-то сверх `A`). Этот приём часто применяют в тестах: сравнить «фактический»
результат запроса с «эталонным».

### 6. Дедупликация и подсчёт дублей

`UNION` (и `INTERSECT`/`EXCEPT` без `ALL`) убирают повторы. Разница числа строк
между `UNION ALL` и `UNION` на одних данных = **сколько было дубликатов** (задача 3).

## Подводные камни (чек-лист)

| Ловушка | В чём беда | Как избежать |
|---------|-----------|--------------|
| **Порядок колонок** | Сопоставление идёт по **позиции**, а не по имени. Перепутал порядок — данные «переехали» в чужую колонку, ошибки нет | Явный список колонок, одинаковый порядок во всех ветках; не `SELECT *` |
| **Несовместимые типы** | Число ↔ дата и т.п. → ошибка; либо неожиданное приведение | Следить за типами по позициям; при нужде явно приводить (`CAST`) |
| **Забыл `ALL`** | Молчаливый дедуп там, где он не нужен → лишняя сортировка/хеш и потеря кратностей | По умолчанию `UNION ALL`; `UNION` — только когда повторы реально мешают |
| **Лишний `UNION`** | Дедуп по-настоящему **дорогой** на больших данных | Если ветки не пересекаются — всегда `UNION ALL` |
| **`EXCEPT` перепутал стороны** | `A EXCEPT B` ≠ `B EXCEPT A` — несимметричен | Проговаривать «из какого множества вычитаю какое» |
| **`ORDER BY`/`LIMIT` не там** | У ветки не может быть своего `ORDER BY` без скобок; сортировка — одна, в конце | Один `ORDER BY` после последней ветки; для ветки — скобки |
| **Приоритет в цепочке** | `INTERSECT` считается раньше `UNION`/`EXCEPT` → без скобок другой ответ | В смешанных цепочках **всегда** ставить скобки |
| **Сюрприз с `NULL`** | При дедупе `NULL` = `NULL` (в отличие от `JOIN`/`WHERE`) | Помнить про разницу семантики (файлы 02–03) |
| **Имена из первой ветки** | Заголовки и цели `ORDER BY` берутся из первого запроса | Алиасы задавать в первой ветке |

## Когда операции над множествами, а когда `JOIN` или подзапрос

Разные инструменты для разных форм вопроса. Держите в голове такую развилку.

**Берите операции над множествами (`UNION`/`INTERSECT`/`EXCEPT`), когда** у вас
**два результата одинаковой формы** и нужно скомбинировать их **как наборы строк**:
слить, найти общее, найти разность. Особенно когда ветки — это «разные срезы
одного и того же» (два периода, два канала, два условия над той же таблицей).

**Берите `JOIN` (lab002), когда** нужно **добавить колонки** — приставить к строке
данные из другой таблицы по ключу. `JOIN` строит **широкие** строки; операции над
множествами оставляют ширину прежней. Это ортогональные инструменты, и они часто
работают вместе: внутри ветки объединения спокойно стоит многотабличный `JOIN`
(как в задачах 8–13).

**Берите подзапросы `IN`/`EXISTS` (это уже lab004), когда** вопрос — про
**принадлежность по ключу**, а формы строк у вас **разные**. «Клиенты, у которых
**есть** заказ», «товары, которых **нет** в заказах» отлично выражаются и через
`EXCEPT`/анти-джойн, и через `NOT EXISTS` — но `EXCEPT` требует, чтобы обе ветки
возвращали **сопоставимые колонки**, а `EXISTS` проверяет наличие связи, не
выравнивая структуры.

Важная связь тем: **один и тот же вопрос часто выразим по-разному.**

| Вопрос | Через множества | Через `JOIN` (lab002) | Через подзапрос (lab004) |
|--------|-----------------|------------------------|--------------------------|
| «купили и X, и Y» | `INTERSECT` двух выборок | самосоединение `orders` по клиенту | два `EXISTS` |
| «купили X, но не Y» | `EXCEPT` | `LEFT JOIN ... IS NULL` | `EXISTS` X + `NOT EXISTS` Y |
| «товары без заказов» | каталог `EXCEPT` заказанное | анти-джойн `LEFT JOIN ... IS NULL` | `NOT EXISTS` / `NOT IN` |
| «слить два списка» | `UNION [ALL]` | — (не задача `JOIN`) | — |

Нет «единственно правильного» варианта — есть более читаемый под конкретную
задачу. `INTERSECT`/`EXCEPT` выигрывают, когда вопрос буквально про **два
множества строк одинаковой формы**: запись получается короткой и близкой к
формулировке. Когда же речь про «просто проверить, есть ли связь» и структуры
разные — часто чище `EXISTS` (lab004).

> Тонкость про `NULL` на будущее: `NOT IN (подзапрос)` ведёт себя коварно, если в
> подзапросе встречается `NULL` (может «съесть» весь результат). У `EXCEPT` этой
> ловушки нет — он корректно работает с `NULL` (как мы видели, `NULL` = `NULL` при
> сравнении строк). Это ещё один аргумент за `EXCEPT` в задачах-разностях. Детали
> — в lab004.

## Итог лабы

- Операции над множествами комбинируют **результаты запросов** «по вертикали»:
  `UNION` (объединение), `INTERSECT` (пересечение), `EXCEPT` (разность).
- Ветки должны быть **совместимы**: одинаковое число колонок, совместимые типы
  **по позициям**; имена — из первой ветки.
- `UNION` убирает дубликаты (дорого), `UNION ALL` — нет (быстро и по умолчанию).
  Дубликат — совпадение по **всем** колонкам; при дедупе **`NULL` = `NULL`**.
- `INTERSECT` — «в обоих», `EXCEPT` — «в первом, но не во втором» (несимметричен).
  Формы `ALL` сохраняют кратности (`min` и `max(m−n,0)`).
- `INTERSECT` приоритетнее `UNION`/`EXCEPT`; в смешанных цепочках — **скобки**.
  `ORDER BY`/`LIMIT` — один раз, в конце, ко всему результату.
