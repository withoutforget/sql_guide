## О чём эта лаба

В lab002 мы соединяли таблицы через `JOIN` — **добавляли колонки**, приставляя к
строке одной таблицы подходящие данные из другой. Это работа «**вширь**».

Здесь — другой инструмент. Операции над множествами (`UNION`, `INTERSECT`,
`EXCEPT`) берут **результаты двух (и более) запросов `SELECT`** и комбинируют их
**«по вертикали»** — складывают строки одного результата под строки другого,
находят их общую часть или разность. Это работа «**ввысь**»: число колонок не
меняется, меняется набор строк.

Всего операторов три (у каждого есть вариант `ALL`, о нём дальше):

| Оператор | Что делает | Аналог из теории множеств |
|----------|------------|----------------------------|
| `UNION` | строки, которые есть **хотя бы в одном** из запросов | объединение `A ∪ B` |
| `INTERSECT` | строки, которые есть **в обоих** запросах сразу | пересечение `A ∩ B` |
| `EXCEPT` | строки **первого** запроса, которых **нет во втором** | разность `A \ B` |

«Множество» здесь — это просто **набор строк-результатов** запроса. Операторы
работают со строками целиком, сравнивая их по **всем** колонкам сразу.

## `JOIN` вширь против `UNION` ввысь — наглядно

Возьмём два маленьких результата: список из онлайн-канала и список из
офлайн-канала.

```
online              offline
email        name   email         name
------------ -----  ------------- ------
anna@…       Анна   anna@…        Анна
boris@…      Борис  egor@…        Егор
```

**`JOIN` (вширь)** искал бы пары «строка слева ↔ строка справа» по какому-то
условию и **приклеивал колонки** одной к другой. Результат стал бы **шире**
(колонок больше):

```
email_l | name_l | email_r | name_r      ← колонки обеих таблиц в одной строке
--------+--------+---------+--------
anna@…  | Анна   | anna@…  | Анна
```

**`UNION ALL` (ввысь)** не ищет никаких пар — он просто **кладёт строки второго
результата под строки первого**. Результат стал **длиннее** (строк больше), но
**той же ширины** (те же 2 колонки):

```
email    | name
---------+------
anna@…   | Анна     ← из online
boris@…  | Борис    ← из online
anna@…   | Анна     ← из offline
egor@…   | Егор     ← из offline
```

Запомните противопоставление:

| | `JOIN` | Операции над множествами |
|--|--------|--------------------------|
| Направление | вширь (добавляет **колонки**) | ввысь (добавляет/убирает **строки**) |
| Что нужно от запросов | условие соединения (`ON`) | одинаковая **форма** строк (число и типы колонок) |
| Число колонок в итоге | сумма колонок таблиц | столько же, сколько в каждом запросе |
| Типичный вопрос | «приставь к заказу имя клиента» | «слей два списка», «что общего / чем различаются» |

## Общий вид запроса

```sql
SELECT <колонки>  FROM ...          -- первый запрос (ветка)
<ОПЕРАТОР>                          -- UNION / UNION ALL / INTERSECT / EXCEPT
SELECT <колонки>  FROM ...          -- второй запрос (ветка)
[ ORDER BY ... ]                    -- необязательно, ОДИН раз, в самом конце
```

Каждый `SELECT` называют **веткой** (или «плечом») операции. Веток может быть и
больше двух — операторы выстраиваются в цепочку (об этом в
[файле 04](04-precedence-order-by.md)). Каждая ветка — это самостоятельный
`SELECT` со своими `FROM`, `WHERE`, `JOIN`, `GROUP BY`. А вот `ORDER BY`
относится ко **всему** результату и пишется один раз в конце — почему так, тоже в
[файле 04](04-precedence-order-by.md).

## ⚠️ Правила совместимости: что можно объединять

Нельзя объединить как попало любые два запроса. Ветки должны быть
**совместимы по объединению** (union-compatible). Правил всего два, но нарушать
их — классика ошибок.

### Правило 1. Одинаковое число колонок

У всех веток должно быть **одно и то же число колонок**. Иначе — ошибка ещё на
этапе разбора запроса:

```sql
SELECT email, name FROM online_customers      -- 2 колонки
UNION
SELECT email        FROM offline_customers;    -- 1 колонка
-- ОШИБКА: each UNION query must have the same number of columns
```

### Правило 2. Совместимые типы — ПО ПОЗИЦИЯМ

Колонки сопоставляются **по порядку**: 1-я колонка первой ветки с 1-й колонкой
второй, 2-я со 2-й и т.д. Типы сопоставляемых колонок должны быть **совместимы**
(приводиться к общему типу). Числовые типы приводятся друг к другу, `text`/`varchar`
между собой — тоже, а вот «число ↔ дата» уже несовместимы.

Если типы разные, но приводимые, PostgreSQL сам выберет общий тип. Например,
`integer` и `numeric` сольются в `numeric`:

```sql
SELECT 1 AS n
UNION ALL
SELECT 2.5;
```
```
  n
-----
   1
 2.5     ← общий тип колонки стал numeric, единица показана как 1
```

### ❗ Сопоставление идёт по ПОРЯДКУ, а не по именам колонок

Это **самая частая ловушка**. Оператор смотрит только на **позицию** колонки, а
её **имя игнорирует**. Если перепутать порядок — синтаксической ошибки не будет
(типы-то совпали), а данные молча «переедут» не в ту колонку:

```sql
-- ❌ логическая ошибка: колонки перепутаны местами
SELECT name, city FROM online_customers      -- (name, city)
UNION ALL
SELECT city, name FROM offline_customers;     -- (city, name)  ← порядок обратный!
```

В первой колонке результата окажется мешанина: сверху имена (`Анна`, `Борис`),
снизу города (`Москва`, `Екатеринбург`). Ошибки нет — есть тихо неверный
результат. **Всегда перечисляйте колонки во всех ветках в одном и том же
порядке** и лучше — явным списком, а не `SELECT *` (при `*` порядок задаётся
порядком колонок в таблице, а он может отличаться).

### Откуда берутся имена колонок результата

Имена (заголовки) всего результата берутся из **первой** ветки. Имена во второй
и следующих ветках ни на что не влияют — можете называть их как угодно (или
никак). Практический вывод: **алиасы колонок задавайте в первом запросе** —
именно они станут заголовками и по ним же можно будет сортировать в финальном
`ORDER BY`.

```sql
SELECT email AS адрес, city AS город FROM online_customers
UNION ALL
SELECT email, city FROM offline_customers;      -- заголовки будут «адрес», «город»
```

## Мысленная модель

Держите в голове такую картинку исполнения `A <ОПЕРАТОР> B`:

1. **отдельно** выполняется запрос `A` → получается набор строк;
2. **отдельно** выполняется запрос `B` → получается другой набор строк;
3. проверяется совместимость (число и типы колонок), типы приводятся к общим;
4. два набора **комбинируются** по правилу оператора (сложить / пересечь /
   вычесть);
5. если это не `...ALL` — из результата **убираются дубликаты**;
6. в самом конце, если есть, применяется `ORDER BY` (и `LIMIT`) — ко всему
   результату.

Ключевое: ветки считаются **независимо**, а оператор работает уже с их готовыми
результатами как с наборами строк.
