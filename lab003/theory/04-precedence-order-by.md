# 04 — Приоритет операций, скобки, `ORDER BY` и `LIMIT`

[← INTERSECT и EXCEPT](03-intersect-except.md) · [оглавление](../README.md) · [дальше: паттерны и подводные камни →](05-patterns-pitfalls.md)

---

Пока у нас было по две ветки. Но операторы можно **выстраивать в цепочку** из
трёх и более запросов. И тут возникают два вопроса: в каком **порядке** они
считаются и куда девать **сортировку**, если она одна на весь результат.

## Приоритет: `INTERSECT` связывает сильнее, чем `UNION` и `EXCEPT`

Когда в запросе несколько разных операторов без скобок, порядок вычисления
определяется приоритетом:

1. **`INTERSECT`** имеет **более высокий** приоритет — считается **первым**.
2. **`UNION`** и **`EXCEPT`** имеют **одинаковый** приоритет и считаются **слева
   направо** (в порядке записи).

То есть без скобок

```sql
A UNION B INTERSECT C          -- читается как:  A UNION (B INTERSECT C)
A EXCEPT B INTERSECT C         -- читается как:  A EXCEPT (B INTERSECT C)
A EXCEPT B UNION  C            -- читается как:  (A EXCEPT B) UNION C
```

Аналогия — школьная арифметика: `INTERSECT` как умножение (`×`), а `UNION` и
`EXCEPT` как сложение и вычитание (`+`, `−`). Сначала «умножения», потом
«сложения/вычитания» слева направо.

## Скобки: явно задают порядок

**Скобки переопределяют приоритет** — как в арифметике. И это не роскошь: очень
часто именно скобки решают, правильный ответ вы получите или нет.

Разберём на реальном вопросе (задача 11): «клиенты, купившие **хотя бы одну** из
книг 5 или 6 **и при этом** хоть что-то из электроники». По смыслу это:

```
(покупатели книги 5  ∪  покупатели книги 6)  ∩  покупатели электроники
```

Сначала объединяем покупателей обеих книг, потом пересекаем с покупателями
электроники. Записываем **со скобками**:

```sql
(
    SELECT c.id, c.name FROM orders o JOIN customers c ON c.id = o.customer_id
    WHERE o.product_id = 5
  UNION
    SELECT c.id, c.name FROM orders o JOIN customers c ON c.id = o.customer_id
    WHERE o.product_id = 6
)
INTERSECT
SELECT c.id, c.name
FROM orders o
JOIN customers  c   ON c.id  = o.customer_id
JOIN products   p   ON p.id  = o.product_id
JOIN categories cat ON cat.id = p.category_id
WHERE cat.name = 'Электроника'
ORDER BY name;
```
```
 id | name
----+------
  1 | Анна
  3 | Вера
(2 rows)
```

А теперь **уберём скобки** — и из-за приоритета `INTERSECT` запрос молча посчитает
другое: `книга5 UNION (книга6 INTERSECT электроника)`:

```sql
SELECT ... WHERE o.product_id = 5      -- книга 5
UNION
SELECT ... WHERE o.product_id = 6      -- книга 6
INTERSECT
SELECT ... WHERE cat.name = 'Электроника'
ORDER BY name;
```
```
 id | name
----+-------
  1 | Анна
  3 | Вера
  5 | Дарья    ← лишняя: Дарья купила книгу 5, но электронику НЕ покупала
(3 rows)
```

Появилась Дарья, которой в правильном ответе быть не должно. Ошибки СУБД не
выдала — просто ответ другой. **Мораль: как только в цепочке смешаны разные
операторы — расставляйте скобки явно, даже если приоритет и так «на вашей
стороне».** Это и защита от ошибок, и читаемость.

> Сравните с задачей 13: там цепочка `(книги ∩ электроника) EXCEPT дом`. Поскольку
> `INTERSECT` и так считается первым, скобки вокруг него **необязательны** — но мы
> их всё равно ставим для ясности. Правило одно: **сомневаешься — ставь скобки.**

## `ORDER BY` и `LIMIT` — один раз, ко всему результату

Сортировка и ограничение числа строк относятся к **итоговому** результату всей
операции, а не к отдельной ветке. Поэтому:

- `ORDER BY` пишется **один раз**, в **самом конце**, после последней ветки;
- он сортирует **весь** объединённый результат.

```sql
SELECT email, city FROM online_customers
UNION
SELECT email, city FROM offline_customers
ORDER BY city NULLS LAST, email;      -- ← один ORDER BY на весь UNION
```
```
     email      |      city
----------------+-----------------
 zhanna@mail.ru | Екатеринбург
 anna@mail.ru   | Москва
 egor@mail.ru   | Москва
 vera@mail.ru   | Москва
 darya@mail.ru  | Новосибирск
 boris@mail.ru  | Санкт-Петербург
 gleb@mail.ru   |                  ← город NULL, ушёл в конец из-за NULLS LAST
(7 rows)
```

### По каким именам сортировать

Заголовки всего результата берутся из **первой** ветки (см.
[файл 01](01-set-operations-intro.md)). Поэтому в `ORDER BY` ссылаться нужно на
имена колонок **первого** запроса — либо по имени, либо по **номеру позиции**:

```sql
... ORDER BY city NULLS LAST, email   -- по именам из первой ветки
... ORDER BY 2 NULLS LAST, 1          -- то же самое по номерам колонок (2=city, 1=email)
```

Сортировка по номеру (`ORDER BY 2`) особенно удобна в операциях над множествами:
имена во второй ветке могут отличаться, а номер позиции однозначен всегда. Оба
варианта выше дают идентичный результат.

### Нельзя приписать `ORDER BY` к ветке без скобок

Внутри объединения у отдельной ветки **не может быть своего `ORDER BY`** — он
воспринимается как финальный и должен стоять после всех веток. Вот так — **ошибка**:

```sql
SELECT email FROM online_customers
ORDER BY email            -- ❌ ORDER BY здесь относится ко всему, а дальше идёт UNION
UNION
SELECT email FROM offline_customers;
-- ОШИБКА: syntax error
```

Если действительно нужно упорядочить или ограничить **одну ветку** (например,
взять топ-N из неё перед объединением), ветку заключают в **скобки** — тогда
`ORDER BY`/`LIMIT` относятся к ней:

```sql
(SELECT email FROM online_customers  ORDER BY email LIMIT 3)   -- топ-3 из онлайна
UNION ALL
(SELECT email FROM offline_customers ORDER BY email LIMIT 2)   -- топ-2 из офлайна
ORDER BY email;                                                -- сортировка итога
```
```
     email
---------------
 anna@mail.ru        ← топ-3 онлайна: anna, boris, darya
 anna@mail.ru        ← топ-2 офлайна: anna, egor (anna попала в обе ветки —
 boris@mail.ru          UNION ALL сохранил оба вхождения)
 darya@mail.ru
 egor@mail.ru
(5 rows)
```

> Тонкость: `ORDER BY` **внутри** ветки без `LIMIT`/`OFFSET` смысла почти не имеет
> — операция над множествами всё равно может переупорядочить и продедуплицировать
> строки, так что порядок «внутренней» сортировки в итоге не сохраняется. Порядок
> в конечном результате гарантирует только **внешний** `ORDER BY`. Внутренняя
> сортировка осмысленна именно в паре с `LIMIT` — «сначала отбери нужные строки,
> потом объединяй».

## `LIMIT` ко всему результату

Финальный `LIMIT` (как и `ORDER BY`) применяется ко всему объединению и пишется
после него:

```sql
SELECT email, city FROM online_customers
UNION
SELECT email, city FROM offline_customers
ORDER BY email
LIMIT 5;                 -- первые 5 строк итогового отсортированного результата
```

Всё про `ORDER BY`/`LIMIT`/`NULLS FIRST|LAST` из lab001 работает здесь без
изменений — просто применяется к результату операции над множествами.
