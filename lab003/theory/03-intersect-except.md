# 03 — `INTERSECT` и `EXCEPT`: общее и разность

[← UNION и UNION ALL](02-union-union-all.md) · [оглавление](../README.md) · [дальше: приоритет и ORDER BY →](04-precedence-order-by.md)

---

Два оператора отвечают на вопросы «что **общего** у двух выборок» и «чем они
**различаются**»:

- **`INTERSECT`** — строки, которые есть **в обоих** запросах (пересечение).
- **`EXCEPT`** — строки **первого** запроса, которых **нет во втором** (разность).

Как и `UNION`, оба сравнивают строки **целиком** (по всем колонкам) и по
умолчанию **убирают дубликаты**. И у обоих есть форма `ALL` — про кратности,
разберём ниже.

## `INTERSECT` — что есть в обоих запросах

Вопрос: «кто зарегистрирован **и** онлайн, **и** офлайн» — то есть чьи строки
присутствуют в **обоих** списках:

```sql
SELECT email, name, city FROM online_customers
INTERSECT
SELECT email, name, city FROM offline_customers
ORDER BY email;
```
```
    email     | name |  city
--------------+------+--------
 anna@mail.ru | Анна | Москва
 gleb@mail.ru | Глеб |
(2 rows)
```

В пересечение попали только Анна и Глеб — их одинаковые строки есть и там, и там.

Обратите внимание на **Глеба**: его строки совпали, **хотя город у него `NULL`**.
Это ровно то правило из [файла 02](02-union-union-all.md): при сопоставлении
строк в операциях над множествами `NULL` считается равным `NULL`. Обычный `JOIN`
с условием `on1.city = on2.city` двух Глебов бы **не** сматчил (`NULL = NULL` →
`UNKNOWN`), а `INTERSECT` — матчит. Это принципиальное отличие семантики.

`INTERSECT` симметричен: `A INTERSECT B` и `B INTERSECT A` дают один и тот же
набор строк (в теории множеств `A ∩ B = B ∩ A`).

## `EXCEPT` — что есть в первом, но не во втором

Вопрос: «кто есть в онлайн-списке, но **отсутствует** в офлайн-списке»:

```sql
SELECT email, name, city FROM online_customers
EXCEPT
SELECT email, name, city FROM offline_customers
ORDER BY email;
```
```
     email     | name  |      city
---------------+-------+-----------------
 boris@mail.ru | Борис | Санкт-Петербург
 darya@mail.ru | Дарья | Новосибирск
 vera@mail.ru  | Вера  | Москва
(3 rows)
```

Остались те, кого нет в офлайне: Борис, Вера, Дарья. Анна и Глеб выбыли — они
есть и во второй ветке.

### `EXCEPT` несимметричен — порядок веток важен

В отличие от `UNION` и `INTERSECT`, у `EXCEPT` **порядок запросов меняет
результат**. `A EXCEPT B` («есть в A, нет в B») и `B EXCEPT A` («есть в B, нет в
A») — разные вопросы. Перевернём предыдущий:

```sql
SELECT email, name FROM offline_customers
EXCEPT
SELECT email, name FROM online_customers
ORDER BY email;
```
```
     email      | name
----------------+-------
 egor@mail.ru   | Егор
 zhanna@mail.ru | Жанна
(2 rows)
```

Теперь это «есть в офлайне, но нет в онлайне» → Егор и Жанна. Совсем другой
список. Всегда думайте, **из какого** множества вычитаете **какое**.

## Формы `ALL`: `INTERSECT ALL` и `EXCEPT ALL` и кратности

По умолчанию `INTERSECT` и `EXCEPT` (как и `UNION`) схлопывают дубликаты. Формы
`INTERSECT ALL` и `EXCEPT ALL` работают с **кратностями** (сколько раз строка
встречается). Правила простые — считаем для каждой строки её число копий слева
(`m`) и справа (`n`):

| Оператор | Сколько копий строки в результате |
|----------|-----------------------------------|
| `INTERSECT` | 1, если строка есть в обоих (иначе 0) |
| `INTERSECT ALL` | `min(m, n)` |
| `EXCEPT` | 1, если строка есть слева и **нет** справа (иначе 0) |
| `EXCEPT ALL` | `max(m − n, 0)` |

Проще на «чистом» примере. Пусть левое множество `A = {a, a, a, b}` (три `a` и
одна `b`), правое зависит от оператора. В примеры добавлен `ORDER BY x` — сам по
себе результат множественной операции **не отсортирован** (порядок строк не
гарантирован), а сортировка делает вывод воспроизводимым.

**`INTERSECT` vs `INTERSECT ALL`** (правое `B = {a, a, c}`):

```sql
SELECT * FROM (VALUES ('a'),('a'),('a'),('b')) t(x)
INTERSECT            -- дедуп: просто «есть в обоих»
SELECT * FROM (VALUES ('a'),('a'),('c')) t(x)
ORDER BY x;
--  x
-- ---
--  a               → 1 строка
```
```sql
SELECT * FROM (VALUES ('a'),('a'),('a'),('b')) t(x)
INTERSECT ALL        -- min(копий слева, копий справа): min(3, 2) = 2
SELECT * FROM (VALUES ('a'),('a'),('c')) t(x)
ORDER BY x;
--  x
-- ---
--  a
--  a               → 2 строки
```

**`EXCEPT` vs `EXCEPT ALL`** (правое `B = {a, c}`):

```sql
SELECT * FROM (VALUES ('a'),('a'),('a'),('b')) t(x)
EXCEPT               -- дедуп: «есть слева и нет справа»
SELECT * FROM (VALUES ('a'),('c')) t(x)
ORDER BY x;
--  x
-- ---
--  b               → 1 строка (все 'a' убраны, т.к. 'a' есть справа)
```
```sql
SELECT * FROM (VALUES ('a'),('a'),('a'),('b')) t(x)
EXCEPT ALL           -- max(копий слева − копий справа, 0): для 'a' это 3−1=2
SELECT * FROM (VALUES ('a'),('c')) t(x)
ORDER BY x;
--  x
-- ---
--  a
--  a               → для 'a': 3 − 1 = 2 копии
--  b               → для 'b': 1 − 0 = 1 копия  (итого 3 строки)
```

На наших данных `EXCEPT ALL` виден в задаче 6: в офлайне Егор записан дважды, в
онлайне его нет (0 копий), поэтому `offline EXCEPT ALL online` оставляет **обоих**
Егоров (2 − 0 = 2), тогда как обычный `EXCEPT` — только одного.

На практике формы `ALL` для `INTERSECT`/`EXCEPT` нужны редко (обычно работают с
множествами-без-повторов). Но знать про кратности полезно — иначе легко удивиться
«лишним» или «пропавшим» строкам.

### Что если одна ветка пуста

Пустой результат (`∅` — ноль строк) — законный вход, и операторы ведут себя с ним
предсказуемо, ровно по смыслу множеств:

| Выражение | Результат | Почему |
|-----------|-----------|--------|
| `A INTERSECT ∅` | `∅` | у пустого множества нет общих строк ни с чем |
| `A EXCEPT ∅` | `A` (уникальные строки `A`) | вычитать нечего — остаётся всё `A` |
| `∅ EXCEPT A` | `∅` | из пустого множества вычитать нечего |
| `A UNION ∅` | `A` (уникальные строки `A`) | добавлять нечего |

Пустой ветка становится, например, когда её `WHERE` не выпустил ни одной строки.
Полезно как раз в проверках: `A EXCEPT B` возвращает **пусто** ровно тогда, когда
все строки `A` нашлись в `B` (см. проверку равенства наборов ниже).

## Практические кейсы

Сила `INTERSECT`/`EXCEPT` в том, что многие «человеческие» вопросы переводятся на
них почти дословно.

### «Купили И X, И Y» → `INTERSECT`

Клиенты, купившие **обе** книги — «SQL за месяц» (id 5) **и** «Чистый код» (id 6).
Строим два множества покупателей и пересекаем:

```sql
SELECT c.id, c.name
FROM orders AS o JOIN customers AS c ON c.id = o.customer_id
WHERE o.product_id = 5
INTERSECT
SELECT c.id, c.name
FROM orders AS o JOIN customers AS c ON c.id = o.customer_id
WHERE o.product_id = 6
ORDER BY name;
```
```
 id | name
----+-------
  1 | Анна
  5 | Дарья
(2 rows)
```

> Почему не одним `WHERE product_id = 5 AND product_id = 6`? Потому что это
> условие на **одну строку** заказа, а одна строка — это один товар; `5 AND 6`
> никогда не истинно. Нам нужны клиенты, у которых **есть строка с 5** и (в другой
> строке) **есть строка с 6**. Это условие на **два множества**, и `INTERSECT`
> выражает его напрямую.

### «Купили X, но НЕ Y» → `EXCEPT`

Клиенты, купившие «SQL за месяц» (5), но **не** купившие «Чистый код» (6):

```sql
SELECT c.id, c.name
FROM orders AS o JOIN customers AS c ON c.id = o.customer_id
WHERE o.product_id = 5
EXCEPT
SELECT c.id, c.name
FROM orders AS o JOIN customers AS c ON c.id = o.customer_id
WHERE o.product_id = 6
ORDER BY name;
```
```
 id | name
----+------
  3 | Вера
(1 row)
```

Вера купила первую книгу и не покупала вторую. Анна и Дарья отпали — они брали обе.

### «Есть в справочнике, но нет в фактах» → `EXCEPT`

Товары каталога, которые **ни разу не заказывали**. Из всех товаров вычитаем те,
что встречались в заказах:

```sql
SELECT id, name FROM products
EXCEPT
SELECT p.id, p.name
FROM orders AS o JOIN products AS p ON p.id = o.product_id
ORDER BY id;
```
```
 id |        name
----+--------------------
  3 | Умная колонка Mini
  8 | Плед тёплый
 13 | Скакалка Speed
 14 | Йога-коврик
(4 rows)
```

Это тот же вопрос, что решался **анти-джойном** в lab002 (`LEFT JOIN ... WHERE
правая IS NULL`). `EXCEPT` часто записывает его короче и нагляднее. Такой приём —
основа **сверки датасетов**: «какие id есть в файле A, но потерялись в таблице B»,
«каких товаров из прайса нет на складе» и т.п.

### Проверка на дубликаты и на равенство наборов

- **Дедуп-проверка.** Если `SELECT ... UNION SELECT ...` (с дедупом) даёт меньше
  строк, чем `... UNION ALL ...`, — в данных были повторы. Разница чисел — это
  количество дубликатов.
- **Равенство двух наборов.** Два запроса возвращают **одинаковое множество**
  строк тогда и только тогда, когда `A EXCEPT B` **и** `B EXCEPT A` оба пусты.
  Одного направления мало (`A EXCEPT B` пусто ещё и когда `A` — подмножество `B`).
  Этот приём — популярный способ сравнить «ожидаемый» и «фактический» результат в
  тестах. Подробнее — в [файле 05](05-patterns-pitfalls.md).
