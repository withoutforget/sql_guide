# 02 — `WHERE`: фильтрация строк

[← SELECT/FROM](01-select-from.md) · [оглавление](../README.md) · [дальше: сортировка →](03-order-distinct-limit.md)

## Что это

`WHERE` оставляет из таблицы только те строки, для которых условие истинно.
Выполняется **после `FROM`, но до `GROUP BY` и `SELECT`** — то есть фильтрует
*исходные* строки, ещё до всякой группировки и вычисления агрегатов.

```sql
SELECT name, price
FROM products
WHERE price < 10000;
```

## Операторы сравнения

| Оператор | Значение              | Пример                    |
|----------|-----------------------|---------------------------|
| `=`      | равно                 | `category = 'books'`      |
| `<>` или `!=` | не равно         | `category <> 'toys'`      |
| `<` `>`  | меньше / больше       | `price > 1000`            |
| `<=` `>=`| меньше/больше или равно | `in_stock >= 10`        |

Сравнивать можно числа, строки (лексикографически), даты (хронологически).

## Логические операторы: `AND`, `OR`, `NOT`

Условия комбинируются:

```sql
WHERE category = 'electronics' AND price < 10000     -- оба условия
WHERE category = 'books' OR category = 'toys'        -- хотя бы одно
WHERE NOT (category = 'home')                        -- отрицание
```

**Приоритет: `NOT` → `AND` → `OR`.** `AND` «сильнее» `OR`, поэтому

```sql
-- НЕ то, что кажется: читается как  A AND (B OR C-нет)... точнее A AND B, ИЛИ C
WHERE category = 'books' AND price < 1000 OR price > 50000

-- явные скобки убирают двусмысленность:
WHERE (category = 'books' AND price < 1000) OR price > 50000
```

Правило: **сомневаешься — ставь скобки.**

## Специальные предикаты

### `BETWEEN a AND b` — диапазон

Границы **включаются**: `BETWEEN 500 AND 1500` == `>= 500 AND <= 1500`.

```sql
WHERE price BETWEEN 500 AND 1500
WHERE price NOT BETWEEN 500 AND 1500     -- вне диапазона
```

Работает и для дат: `WHERE ordered_at BETWEEN '2024-06-01' AND '2024-06-30'`.
Осторожно с датами-и-временем: `BETWEEN ... AND '2024-06-30'` не включит
события 30-го числа после полуночи, если в колонке есть время.

### `IN (...)` — принадлежность списку

```sql
WHERE category IN ('books', 'toys')        -- то же, что = 'books' OR = 'toys'
WHERE id NOT IN (1, 2, 3)
```

Внутри `IN` может быть и подзапрос: `WHERE customer_id IN (SELECT id FROM ...)`
(это уже lab003).

⚠️ **Ловушка `NOT IN` + `NULL`:** если в списке окажется `NULL`, `NOT IN`
вернёт «ничего» для всех строк (из-за трёхзначной логики, см. ниже). С `NOT IN`
по подзапросу это классический баг — предпочитайте `NOT EXISTS`.

### `LIKE` — поиск по шаблону

Два подстановочных символа:

| Символ | Значение                        |
|--------|---------------------------------|
| `%`    | любое количество любых символов (включая ноль) |
| `_`    | ровно один любой символ         |

```sql
WHERE name LIKE 'Книга%'      -- начинается с «Книга»
WHERE name LIKE '%Pro%'       -- содержит «Pro»
WHERE name LIKE '_ниги'       -- 5 символов, заканчивается на «ниги»
```

- `LIKE` **чувствителен к регистру**. Для регистронезависимого поиска в
  PostgreSQL есть `ILIKE`: `name ILIKE '%pro%'`.
- Чтобы искать сам символ `%` или `_` буквально, используют экранирование:
  `LIKE '100\%'` (по умолчанию escape-символ — обратный слэш) или
  `LIKE '100!%' ESCAPE '!'`.
- Для сложных шаблонов есть регулярные выражения (`~`, `~*`) — позже.

## `NULL` и трёхзначная логика — очень важно

`NULL` — это не ноль и не пустая строка, а **«значение неизвестно»**. Любое
сравнение с `NULL` даёт не `TRUE`/`FALSE`, а третье состояние — `UNKNOWN`. А
`WHERE` пропускает строку, только если условие именно `TRUE`.

```sql
WHERE discount = NULL      -- ❌ никогда не TRUE, вернёт 0 строк
WHERE discount <> NULL     -- ❌ тоже не работает
```

Правильно — специальные предикаты:

```sql
WHERE discount IS NULL
WHERE discount IS NOT NULL
```

Следствия трёхзначной логики:
- `NULL = NULL` → `UNKNOWN` (не `TRUE`!);
- `TRUE OR UNKNOWN` → `TRUE`, но `TRUE AND UNKNOWN` → `UNKNOWN`;
- поэтому `NOT IN (список с NULL)` и «съедает» все строки.

> В нашей учебной схеме `NULL`-ов нет (все колонки `NOT NULL`), но в реальных
> данных они повсюду — привыкайте думать о них сразу.

## `WHERE` vs `HAVING`

- `WHERE` фильтрует **строки** до группировки и **не может** содержать
  агрегаты (`COUNT`, `SUM`, ...).
- `HAVING` фильтрует **группы** после `GROUP BY` и работает с агрегатами.
