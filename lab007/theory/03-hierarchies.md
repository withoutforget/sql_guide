# 03 — Иерархии: спуск, подъём, уровни, путь, дерево с отступами

[← синтаксис и исполнение](02-syntax-and-execution.md) · [оглавление](../README.md) · [дальше: графы и циклы →](04-graphs-cycles.md)

---

Разберём **деревья** (иерархии) во всех типичных формах. Дерево — иерархия, где у
узла один родитель и нет циклов ([файл 01](01-why-recursion.md)), поэтому здесь
обход **всегда завершается сам** и о защите от зацикливания думать не нужно (это
головная боль графов, [файл 04](04-graphs-cycles.md)).

Работаем на дереве сотрудников `employees(id, name, position, manager_id)`, корень
— Иван (`manager_id IS NULL`):

```
Иван(1)
├─ Пётр(2) ──── Сергей(4) ── Алексей(6), Дмитрий(7)
│               Мария(5)  ── Елена(8)
└─ Ольга(3) ─── Никита(9), Анна(10)
```

## Спуск вниз: все потомки узла

«Найти всех, кто под узлом X» — самое частое. Направление задают две вещи:
**якорь** (откуда стартуем) и **условие соединения** в рекурсии (куда шагаем).

Для спуска: якорь — прямые дети X, рекурсия присоединяет **детей уже найденных**
(«ребёнок — тот, чей `manager_id` равен id текущего»):

```sql
WITH RECURSIVE subs AS (
    SELECT id, name, 1 AS depth
    FROM employees
    WHERE manager_id = 2                      -- ЯКОРЬ: прямые подчинённые Петра
    UNION ALL
    SELECT e.id, e.name, s.depth + 1
    FROM subs s
    JOIN employees e ON e.manager_id = s.id   -- шаг ВНИЗ: e — ребёнок s
)
SELECT depth, name FROM subs ORDER BY depth, name;
```
```
 depth |  name
-------+---------
     1 | Мария
     1 | Сергей
     2 | Алексей
     2 | Дмитрий
     2 | Елена
(5 rows)
```

`depth` (несём как `s.depth + 1`, [файл 02](02-syntax-and-execution.md)) показывает
дистанцию **от стартового узла**: тимлиды — 1, разработчики — 2. Это
[задача 3](../queries/03_subordinates_depth.sql).

> Хотите включить **сам** стартовый узел в результат? Сделайте якорь по нему
> (`WHERE id = 2`, `0 AS depth`), а рекурсию оставьте прежней. Тогда в ответе будет
> и Пётр (уровень 0), и все под ним.

## Подъём вверх: путь до корня

Обратное направление — «от узла к его начальству, до самого верха». Меняем и
якорь, и условие соединения на **зеркальные**: якорь — сам узел, рекурсия идёт к
**родителю** (`e.id = текущий.manager_id`):

```sql
WITH RECURSIVE chain AS (
    SELECT id, name, manager_id, 1 AS step, name::text AS path
    FROM employees
    WHERE id = 6                               -- ЯКОРЬ: сам Алексей
    UNION ALL
    SELECT e.id, e.name, e.manager_id,
           c.step + 1,
           c.path || ' → ' || e.name           -- копим цепочку строкой
    FROM chain c
    JOIN employees e ON e.id = c.manager_id     -- шаг ВВЕРХ: e — начальник c
)
SELECT step, name, path FROM chain ORDER BY step;
```
```
 step |  name   |              path
------+---------+--------------------------------
    1 | Алексей | Алексей
    2 | Сергей  | Алексей → Сергей
    3 | Пётр    | Алексей → Сергей → Пётр
    4 | Иван    | Алексей → Сергей → Пётр → Иван
(4 rows)
```

Останавливается **само**: у корня Ивана `manager_id = NULL`, `JOIN ... ON e.id =
NULL` не находит начальника → 0 строк → стоп. Это
[задача 4](../queries/04_path_to_root.sql).

> **Спуск vs подъём — вся разница в двух местах:**
>
> | | Якорь | Условие соединения (шаг) |
> |--|-------|--------------------------|
> | **Спуск** (потомки) | стартовый узел / его дети | `child.manager_id = current.id` |
> | **Подъём** (предки) | стартовый узел | `parent.id = current.manager_id` |
>
> Данные те же, направление обхода задаёте вы.

## Уровень (глубина) узла

Мы уже видели: **`level`/`depth` несётся как `+ 1` на каждом шаге**. В якоре
задаёте стартовое значение (`1` для «считать от 1» или `0`, чтобы корень был
нулевым уровнем), в рекурсии прибавляете единицу. Это настолько частый приём, что
считайте его частью «скелета» рекурсивного обхода дерева.

## Полный обход от корня + вывод «деревом» с отступами

Чтобы показать **всю** иерархию от корня и красиво, нужны две вещи: **уровень**
(для отступа) и **правильный порядок строк** (чтобы потомки шли под своим
родителем). [Задача 5](../queries/05_org_tree_indented.sql):

```sql
WITH RECURSIVE tree AS (
    SELECT id, name, 1 AS level, ARRAY[id] AS path
    FROM employees
    WHERE manager_id IS NULL                   -- ЯКОРЬ: корень
    UNION ALL
    SELECT e.id, e.name, t.level + 1, t.path || e.id   -- путь от корня массивом
    FROM tree t
    JOIN employees e ON e.manager_id = t.id
)
SELECT level,
       repeat('    ', level - 1) || name AS org_chart,   -- 4 пробела на уровень
       path
FROM tree
ORDER BY path;                                 -- сортировка по пути = обход в глубину
```
```
 level |      org_chart      |    path
-------+---------------------+------------
     1 | Иван                | {1}
     2 |     Пётр            | {1,2}
     3 |         Сергей      | {1,2,4}
     4 |             Алексей | {1,2,4,6}
     4 |             Дмитрий | {1,2,4,7}
     3 |         Мария       | {1,2,5}
     4 |             Елена   | {1,2,5,8}
     2 |     Ольга           | {1,3}
     3 |         Никита      | {1,3,9}
     3 |         Анна        | {1,3,10}
(10 rows)
```

Два приёма вывода:

1. **Отступ** — `repeat('    ', level - 1) || name`: функция `repeat(str, n)`
   повторяет строку `n` раз; корень (`level 1`) без отступа, каждый уровень глубже
   — плюс 4 пробела. Имя визуально «уезжает» вправо по глубине.
2. **Порядок** — сортировка по **накопленному пути-массиву** `path` (`{1}`,
   `{1,2}`, `{1,2,4}`, …). Массивы сравниваются поэлементно, поэтому дети идут
   **сразу под родителем**, а вся ветвь — единым блоком. Это и есть **обход в
   глубину** (сначала вся ветвь Петра до листьев, потом ветвь Ольги).

> Если вместо `path` отсортировать по `level` — получите обход **в ширину** (все
> первого уровня, потом все второго …), но потомки «оторвутся» от родителей.
> Порядок обхода подробнее — в [файле 04](04-graphs-cycles.md).

## Путь строкой: «хлебные крошки»

Иногда нужен не массив, а читаемая строка пути от корня к узлу — «хлебные крошки»
`Электроника / Компьютеры / Ноутбуки`. Тот же приём, что при подъёме, только вниз
от корня и с конкатенацией имён. На дереве категорий
`categories(id, name, parent_id)` — [задача 6](../queries/06_category_breadcrumbs.sql):

```sql
WITH RECURSIVE cats AS (
    SELECT id, name, 1 AS level, name::text AS breadcrumb
    FROM categories
    WHERE parent_id IS NULL                    -- корневые категории
    UNION ALL
    SELECT c.id, c.name, p.level + 1,
           p.breadcrumb || ' / ' || c.name      -- дописываем имя ребёнка
    FROM cats p
    JOIN categories c ON c.parent_id = p.id
)
SELECT level, breadcrumb FROM cats ORDER BY breadcrumb;
```
```
 level |                      breadcrumb
-------+-------------------------------------------------------
     1 | Дом и сад
     2 | Дом и сад / Инструменты
     ...
     4 | Электроника / Компьютеры / Комплектующие / Видеокарты
     4 | Электроника / Компьютеры / Комплектующие / Процессоры
     ...
```

## Транзитивное замыкание: «все потомки каждого узла»

Иногда нужен не один стартовый узел, а сразу **для каждого** узла — все его
потомки (чтобы, например, посчитать «сколько человек под каждым руководителем»).
Приём — построить **пары «(предок, потомок)» для всех пар сразу**
([задача 7](../queries/07_subordinates_count.sql)):

```sql
WITH RECURSIVE closure AS (
    SELECT id AS boss_id, id AS emp_id         -- ЯКОРЬ: каждый сам себе пара
    FROM employees
    UNION ALL
    SELECT cl.boss_id, e.id                     -- начальник начальника — тоже начальник
    FROM closure cl
    JOIN employees e ON e.manager_id = cl.emp_id
)
SELECT b.name AS boss, count(*) - 1 AS total_subordinates
FROM closure cl JOIN employees b ON b.id = cl.boss_id
GROUP BY b.id, b.name
HAVING count(*) - 1 > 0
ORDER BY total_subordinates DESC, boss;
```
```
  boss  | total_subordinates
--------+--------------------
 Иван   |                  9
 Пётр   |                  5
 Ольга  |                  2
 Сергей |                  2
 Мария  |                  1
(5 rows)
```

Здесь якорь порождает пары «узел сам с собой», а рекурсия «протягивает» связь на
уровень глубже. Получив **все** пары предок-потомок, снаружи просто группируем и
считаем (`count(*) - 1` — минус пара «сам с собой»). Это и есть транзитивное
замыкание из [файла 01](01-why-recursion.md): агрегат **поверх** рекурсивного CTE,
не внутри него ([файл 02](02-syntax-and-execution.md) — агрегаты только снаружи).

## Итог

- Дерево обходится рекурсией **без риска зациклиться** (один родитель, циклов нет).
- **Спуск** (потомки) и **подъём** (предки) отличаются только **якорем** и
  **условием соединения** — данные и структура запроса те же.
- **Уровень** несём как `+ 1` на шаг; стартовое значение задаёт якорь.
- **Путь** копим массивом (`path || узел`) — для сортировки и защиты в графах — или
  строкой (`path || ' / ' || имя`) — для «хлебных крошек».
- **Вывод деревом**: отступ `repeat(' ', level-1) || name` + сортировка по
  **пути-массиву** (обход в глубину, потомки под родителем).
- **Транзитивное замыкание** («все потомки каждого узла») — рекурсией строим пары
  предок-потомок, агрегируем **снаружи**.
