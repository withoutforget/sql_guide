# 06 — Подводные камни, выбор инструмента и границы темы

[← разузлование BOM](05-bom.md) · [оглавление](../README.md)

---

Собираем грабли темы, чек-лист «чем решать» и явные границы: что из смежного сюда
**не** входит и где это лежит.

## Ловушка 1: забыли `RECURSIVE`

Слово `RECURSIVE` пишется **сразу после `WITH`** и **один раз** на всю секцию.
Забыли — и ссылка CTE на самого себя даёт `relation "имя" does not exist` (движок
не ждёт рекурсии и не «видит» имя внутри тела). Даже если рекурсивный CTE один
среди нескольких обычных, `WITH RECURSIVE` всё равно нужен один раз в начале.

## Ловушка 2: нет условия остановки → бесконечная рекурсия

Самая опасная ошибка. Рекурсивная часть **обязана** рано или поздно давать 0 новых
строк ([файл 02](02-syntax-and-execution.md)). Три типовых механизма остановки:

- **счётчик/предикат** (`WHERE n < 5`) — для порождающих рядов;
- **исчерпание данных** — в дереве/DAG у листьев нет детей, `JOIN` возвращает 0
  строк (само собой);
- **защита от повторов** — в графе с циклом: `UNION`-дедуп (достижимость) или
  путь-массив / `CYCLE` (пути).

Забыли защиту в **графе с циклом** — запрос крутится, пока не упрётся в память или
его не снимут. **Признаки риска:** источник — граф (не дерево/DAG), обход через
`UNION ALL`, и нет ни путь-guard'а, ни `CYCLE`. При отладке незнакомого графа
временно ставьте предел глубины (`WHERE level < 20`), чтобы не повесить сервер.

## Ловушка 3: `UNION` там, где нужен `UNION ALL` (и наоборот)

Разобрано в [файле 04](04-graphs-cycles.md) и [файле 05](05-bom.md), но соберём:

| Симптом | Причина | Лечение |
|---------|---------|---------|
| Запрос по графу с циклом **не завершается** | `UNION ALL` без защиты | `UNION` (достижимость) или путь-guard/`CYCLE` (пути) |
| В BOM **недосчёт** одинаковых вхождений детали | `UNION` схлопнул нужные повторы | `UNION ALL` — для сумм нужны все вхождения |
| Пути в один узел «пропали» | `UNION` схлопнул разные маршруты | `UNION ALL` + защита путём |

Правило: **`UNION` (дедуп) — только для достижимости; всё, где важны повторы/пути —
`UNION ALL` + отдельная защита от цикла.**

## Ловушка 4: агрегат/`ORDER BY`/`LIMIT` внутри рекурсии

`SUM`, `COUNT`, `GROUP BY`, `DISTINCT`, оконные функции, `ORDER BY`, `LIMIT` над
самим рекурсивным CTE **внутри** его тела — нельзя. Всё это делается **снаружи**, в
главном запросе поверх готового CTE: сумма BOM ([задача 9](../queries/09_bom_total_qty.sql)),
минимум рёбер ([задача 13](../queries/13_shortest_path_edges.sql)), сортировка
дерева по пути. Рекурсия только **порождает** строки; аналитику по ним — сверху.

## Ловушка 5: путать «умножить по пути» и «сложить»

В BOM ([файл 05](05-bom.md)) вдоль **одного** пути количества **перемножаются**
(`e.qty_total * b.qty`), а по **разным** путям к одной детали — **складываются**
(`SUM` снаружи). Спутать легко: пропустишь умножение — недосчитаешь спицы, забудешь
суммировать вхождения — недосчитаешь болты. **Сверяйте BOM руками** на маленьком
примере.

## Ловушка 6: неверное направление обхода

Спуск и подъём по дереву различаются **якорем** и **условием соединения**
([файл 03](03-hierarchies.md)). Легко перепутать `child.parent_id = cur.id` (вниз)
и `parent.id = cur.parent_id` (вверх) и получить не тех. Проверка: посмотрите на
пару строк результата — идёте вы к потомкам или к предкам того, что в якоре.

## Ловушка 7: тип накопителя пути/строки

- `name::text AS path` в якоре — приведение к `text` нужно, иначе тип накопителя
  может стать слишком «узким» (например, длины исходной колонки) и конкатенация в
  рекурсии обрежет длинные пути. Явный `::text` в **якоре** задаёт правильный тип
  накопителя для всех шагов.
- Путь-массив: следите, чтобы тип элементов совпадал (`ARRAY[id]` из `int` — и
  дописываем `int`).

## Чек-лист: чем решать типовую задачу

| Нужно | Как |
|-------|-----|
| ряд чисел/дат | **не рекурсия** — `generate_series` (lab006) |
| все потомки узла (вниз) | рекурсия: якорь = узел/его дети, шаг `child.parent = cur.id` ([ф.03](03-hierarchies.md)) |
| путь до корня (вверх) | рекурсия: якорь = узел, шаг `parent.id = cur.parent` ([ф.03](03-hierarchies.md)) |
| уровень/глубина | нести `level + 1` ([ф.02](02-syntax-and-execution.md)) |
| дерево с отступами | `repeat(' ', level-1)‖name` + `ORDER BY path`-массив ([ф.03](03-hierarchies.md)) |
| «хлебные крошки» | копить путь строкой `path ‖ ' / ' ‖ name` ([ф.03](03-hierarchies.md)) |
| «всех потомков каждого узла» / замыкание | пары предок-потомок + агрегат снаружи ([ф.03](03-hierarchies.md)) |
| разузлование состава + количества | спуск с `qty_total * qty` + `SUM` снаружи ([ф.05](05-bom.md)) |
| достижимость в графе | `UNION` (дедуп гасит цикл) ([ф.04](04-graphs-cycles.md)) |
| пути в графе | `UNION ALL` + путь-guard `NOT next = ANY(path)` **или** `CYCLE` ([ф.04](04-graphs-cycles.md)) |
| кратчайший путь по рёбрам | нести `hops`, снаружи `MIN(hops)` ([ф.04](04-graphs-cycles.md)) |
| порядок обхода DFS/BFS | `ORDER BY path` (DFS) / `ORDER BY level` (BFS), либо `SEARCH` (PG14+) |

Главный водораздел: **дерево/DAG или граф с циклами?** Дерево/DAG (оргштат,
категории, BOM) — обход завершается сам, берём `UNION ALL`, о циклах не думаем.
Граф (маршруты, ссылки) — обязательна защита: `UNION`-дедуп для достижимости,
путь-guard или `CYCLE` для путей.

## Границы темы — что сюда НЕ входит

- **Оконные функции** (`OVER`, `ROW_NUMBER`, `RANK`, накопительные итоги) —
  **lab010–lab011**. Они дали бы другой способ, например, ранжировать пути или
  «топ-1 на группу», но это отдельная тема; здесь их **не используем**.
- **Ряды чисел и дат** — это **`generate_series` (lab006)**, а не рекурсия
  ([файл 01](01-why-recursion.md)). Рекурсивный счётчик в задачах 1–2 — только
  ради механики.
- **Изменяющие данные CTE** (`WITH ... AS (INSERT/UPDATE/DELETE ... RETURNING)`) —
  **lab019**. Здесь все CTE только читают.
- **Расширенная группировка** (`GROUPING SETS`, `ROLLUP`, `CUBE`) для сводок по
  иерархии измерений — **lab008** (следующая лаба). Это не обход дерева данных, а
  агрегирование по комбинациям колонок.
- **Производительность рекурсии** (индексы на колонку связи `manager_id` /
  `from_city`, планы) — концептуально относится к **lab024–lab025**. Практический
  ориентир: рекурсивному обходу нужен индекс по колонке, по которой идёт шаг
  (`employees(manager_id)`, `routes(from_city)`), иначе каждый шаг — полный
  просмотр таблицы.

## Итог лабы

- **`WITH RECURSIVE имя AS ( ЯКОРЬ UNION[ALL] РЕКУРСИВНАЯ_ЧАСТЬ ) SELECT ...`** —
  единственный способ в SQL обойти иерархию/граф на **заранее неизвестную**
  глубину. `RECURSIVE` — один раз после `WITH`.
- Модель исполнения: **якорь → повторять рекурсивную часть над строками прошлого
  шага, добавляя новое, пока шаг не даст 0 строк**. Нужно **условие остановки**.
- **Деревья/DAG** (сотрудники, категории, BOM): `UNION ALL`, обход завершается сам.
  Спуск/подъём — якорь + условие соединения; несём `level`, `path`; агрегаты и
  сортировку — снаружи.
- **Графы с циклами** (маршруты): защита обязательна — `UNION`-дедуп для
  достижимости; `UNION ALL` + путь-guard `NOT = ANY(path)` **или** штатный `CYCLE`
  для путей. Порядок обхода — `path` (DFS) / `level` (BFS), либо `SEARCH` (PG14+).
- **BOM**: разузлование с **умножением** количеств по уровням + `SUM` снаружи.
- Ряды чисел/дат — **`generate_series` (lab006)**, не рекурсия.

## Что дальше

В `lab008` — **расширенная группировка**: `GROUPING SETS`, `ROLLUP`, `CUBE`. Это
способ за один запрос получить агрегаты сразу по нескольким разрезам и их
комбинациям (промежуточные и общие итоги, «по категории», «по городу», «по обоим»,
«всего» — одной выборкой). Прямое развитие `GROUP BY` из lab001 и естественный
следующий шаг к настоящей аналитической отчётности.
