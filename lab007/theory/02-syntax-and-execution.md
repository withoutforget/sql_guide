# 02 — Синтаксис `WITH RECURSIVE` и как он исполняется

[← зачем рекурсия](01-why-recursion.md) · [оглавление](../README.md) · [дальше: иерархии →](03-hierarchies.md)

---

Это ключевой файл темы. Разберём **синтаксис** рекурсивного CTE и, главное,
**пошаговую модель** того, как СУБД его исполняет — без неё рекурсивные запросы
кажутся магией, а с ней пишутся и читаются механически.

## Синтаксис: три обязательные части

```sql
WITH RECURSIVE имя(колонки) AS (
    <ЯКОРЬ>                       -- 1. начальные строки (не ссылается на «имя»)
    UNION [ALL]                   -- 2. склейка
    <РЕКУРСИВНАЯ ЧАСТЬ>           -- 3. ссылается на «имя» — на результат прошлого шага
)
SELECT ...                        -- главный запрос, читает CTE как таблицу
FROM имя;
```

Разберём по частям.

- **`WITH RECURSIVE`.** Слово `RECURSIVE` пишется **сразу после `WITH`** и **один
  раз** на всю секцию — даже если рекурсивный CTE в ней один (и даже если рядом
  есть обычные, нерекурсивные CTE). Без `RECURSIVE` ссылка CTE на самого себя —
  ошибка.
- **`имя(колонки)`.** Имя, под которым CTE виден и в рекурсивной части, и в
  главном запросе. Список колонок в скобках указывать **не обязательно** (имена
  можно задать алиасами в `SELECT`), но с рекурсией его часто пишут явно — так
  нагляднее, какие колонки «несёт» рекурсия.
- **Якорь (anchor member).** Обычный `SELECT`, который даёт **стартовые строки** и
  **не ссылается** на `имя`. Это «точка входа»: прямые подчинённые начальника,
  корень дерева, число 1, рёбра из стартового города.
- **`UNION` или `UNION ALL`.** Соединяет якорь с рекурсивной частью. Разница —
  ровно та же, что в [lab003](../../lab003/theory/02-union-union-all.md): `UNION
  ALL` дописывает всё как есть, `UNION` дополнительно **убирает дубликаты**. В
  рекурсии этот выбор критичен (особенно для графов с циклами) — подробно в
  [файле 04](04-graphs-cycles.md).
- **Рекурсивная часть (recursive member).** `SELECT`, который **ссылается на
  `имя`** — то есть на строки, полученные на **предыдущем** шаге, — и строит из
  них новые. Именно эта ссылка «на самого себя» и делает CTE рекурсивным.

Колонки якоря и рекурсивной части должны **совпадать по числу и типам** (это
требование `UNION`, lab003). Порядок частей фиксирован: **сначала якорь, потом
`UNION`, потом рекурсивная часть**.

## Пошаговая модель исполнения (главное!)

Как PostgreSQL это вычисляет. Держите в голове **три «ведёрка»**:

- **результат** — сюда накапливается всё, что вернёт CTE (итог);
- **рабочая таблица** (working table) — строки, которые «в работе прямо сейчас»,
  то есть добытые на **последнем** шаге;
- **промежуточная таблица** — временный буфер для строк текущего шага.

Алгоритм:

1. **Вычислить якорь.** Его строки кладём и в **результат**, и в **рабочую
   таблицу**.
2. **Пока рабочая таблица не пуста, повторять:**
   а. Выполнить **рекурсивную часть**, подставив вместо `имя` **текущую рабочую
      таблицу** (то есть строки только с прошлого шага, а не весь результат!).
      Полученные строки — в **промежуточную таблицу**.
   б. *(Только для `UNION`, без `ALL`.)* Выкинуть из промежуточной таблицы строки,
      которые **уже есть в результате** (дедупликация).
   в. Добавить промежуточную таблицу в **результат**.
   г. Сделать промежуточную таблицу новой **рабочей таблицей** (буфер очистить).
3. Когда рекурсивная часть вернула **0 строк** — рабочая таблица опустела,
   **цикл останавливается**. Итог — накопленный **результат**.

> **Два важнейших следствия этой модели:**
>
> 1. Рекурсивная часть на каждом шаге видит **только строки предыдущего шага**
>    (рабочую таблицу), а **не весь** накопленный результат. Это ровно то, что
>    нужно: «к тем, кого нашёл на прошлом шаге, добавь их детей».
> 2. Рекурсия завершается **сама**, когда очередной шаг не даёт новых строк.
>    Значит, в рекурсивной части **обязано** быть условие, которое рано или поздно
>    перестаёт давать новое (`WHERE n < 5`, достижение листа дерева, защита от
>    повторов узлов в графе). Нет такого условия → **бесконечный цикл**.

## Разбор на счётчике 1..5 — по шагам

Возьмём самый простой рекурсивный CTE — числа от 1 до 5
([задача 1](../queries/01_counter_1_to_5.sql)):

```sql
WITH RECURSIVE nums AS (
    SELECT 1 AS n                 -- ЯКОРЬ
    UNION ALL
    SELECT n + 1 FROM nums        -- РЕКУРСИВНАЯ ЧАСТЬ
    WHERE n < 5                   -- условие остановки
)
SELECT n FROM nums;
```

Прокрутим модель (здесь `UNION ALL`, дедупа нет):

| Шаг | Рабочая таблица (вход) | Рекурсивная часть даёт | Результат после шага |
|-----|------------------------|------------------------|----------------------|
| Якорь | — | — | `{1}` |
| 1 | `{1}` | `1+1 = 2` (1<5 ✓) | `{1,2}` |
| 2 | `{2}` | `2+1 = 3` (2<5 ✓) | `{1,2,3}` |
| 3 | `{3}` | `3+1 = 4` (3<5 ✓) | `{1,2,3,4}` |
| 4 | `{4}` | `4+1 = 5` (4<5 ✓) | `{1,2,3,4,5}` |
| 5 | `{5}` | `5<5` — **ложь**, строк нет | `{1,2,3,4,5}` |
| — | пусто → **стоп** | | итог: 1,2,3,4,5 |

Ключевые наблюдения:

- На шаге 2 рабочая таблица — это `{2}`, **не** `{1,2}`. Рекурсивная часть работает
  только со «свежей» строкой прошлого шага. Поэтому `n` в ней — это `2`, и
  получается `3`, а не `2` снова.
- Остановка наступила на шаге 5: строка `5` в рабочей таблице не прошла `WHERE
  n < 5`, рекурсивная часть вернула 0 строк, рабочая таблица опустела.
- Убери `WHERE n < 5` — счётчик никогда не остановится: 6, 7, 8, … до переполнения
  или пока не убьют запрос. **Условие остановки — не украшение, а необходимость.**

## Разбор на спуске по дереву — по шагам

Тот же механизм на иерархии. «Все подчинённые Петра (id 2)»
([задача 3](../queries/03_subordinates_depth.sql)), дерево:

```
Пётр(2)
├─ Сергей(4) ── Алексей(6), Дмитрий(7)
└─ Мария(5)  ── Елена(8)
```

```sql
WITH RECURSIVE subs AS (
    SELECT id, name FROM employees WHERE manager_id = 2      -- ЯКОРЬ: прямые подчинённые
    UNION ALL
    SELECT e.id, e.name
    FROM subs s JOIN employees e ON e.manager_id = s.id      -- РЕКУРСИЯ: их подчинённые
)
SELECT * FROM subs;
```

| Шаг | Рабочая таблица | Рекурсивная часть: «дети» этих строк | Результат |
|-----|-----------------|--------------------------------------|-----------|
| Якорь | — | — | Сергей, Мария |
| 1 | Сергей, Мария | дети Сергея (Алексей, Дмитрий) + дети Марии (Елена) | + Алексей, Дмитрий, Елена |
| 2 | Алексей, Дмитрий, Елена | у них подчинённых нет → 0 строк | без изменений |
| — | пусто → **стоп** | | Сергей, Мария, Алексей, Дмитрий, Елена |

Условие остановки тут **не пишется явно** — оно «встроено» в данные: у листьев
дерева (Алексей, Дмитрий, Елена) нет подчинённых, `JOIN` для них не находит строк,
рекурсивная часть возвращает 0 строк, рабочая таблица пустеет. **Дерево конечно и
без циклов — значит, обход гарантированно завершится.** (В графе с циклом такой
гарантии нет — там условие остановки придётся обеспечивать руками, [файл 04](04-graphs-cycles.md).)

## Как «нести состояние» сквозь рекурсию

Рекурсивная часть может тащить **дополнительные колонки**, вычисляя их из строки
предыдущего шага. Это главный рабочий приём темы. Три типовых «носимых» значения:

```sql
WITH RECURSIVE t AS (
    SELECT ..., 1 AS level, ARRAY[id] AS path, name::text AS chain   -- стартовые значения
    UNION ALL
    SELECT ...,
           t.level + 1,               -- УРОВЕНЬ: глубина обхода (+1 на шаг)
           t.path || child.id,        -- ПУТЬ массивом: дописываем узел (|| — конкатенация, lab006)
           t.chain || ' → ' || child.name   -- ПУТЬ строкой: копим читаемую цепочку
    FROM t JOIN ... ON ...
)
```

- **`level`** — счётчик глубины: в якоре `1`, в рекурсии `+ 1`. Даёт «на сколько
  уровней вглубь» (глубина дерева, число рёбер маршрута).
- **`path` массивом** — накопленный путь `ARRAY[...] || следующий`. Нужен и для
  вывода маршрута, и как **защита от циклов** в графах (проверка «этот узел уже в
  пути?», [файл 04](04-graphs-cycles.md)).
- **`chain` строкой** — читаемая цепочка `'A → B → C'` через конкатенацию.

Мы уже пробовали это в разминочной [задаче 2](../queries/02_powers_of_two.sql):
там рекурсия несёт номер шага `k` и накопленное `value = value * 2`. Идея одна —
**новая строка вычисляется из старой**.

## Правила и ограничения рекурсивной части

Несколько ограничений, которые полезно знать заранее (иначе получите непонятную
ошибку):

- В рекурсивной части `имя` CTE должно упоминаться **ровно один раз** и **не** под
  `LEFT/RIGHT/FULL JOIN` со стороны, где оно может дать `NULL`, не в подзапросе и
  т.п. На практике: `FROM имя JOIN таблица ON ...` — можно; хитрые конструкции с
  `имя` внутри — часто нельзя. Для наших задач стандартной формы этого достаточно.
- В рекурсивной части **нельзя** использовать агрегаты (`SUM`, `COUNT` …),
  `GROUP BY`, `HAVING`, `DISTINCT` над самим рекурсивным `имя`, оконные функции,
  `ORDER BY`/`LIMIT` **внутри** рекурсии. Всё это делают **снаружи** — в главном
  запросе поверх готового CTE (как мы суммируем BOM в
  [задаче 9](../queries/09_bom_total_qty.sql) и берём минимум в
  [задаче 13](../queries/13_shortest_path_edges.sql)).
- Порядок строк внутри рекурсии не определён; нужен конкретный порядок вывода —
  задавайте `ORDER BY` в **главном** запросе (часто по накопленному `path`).

## Мини-словарь терминов

| Термин | Что значит |
|--------|-----------|
| Якорь (anchor) | стартовый `SELECT`, не ссылается на сам CTE |
| Рекурсивная часть | `SELECT`, ссылающийся на имя CTE (на строки прошлого шага) |
| Рабочая таблица | строки, добытые на **последнем** шаге (вход рекурсии на следующем) |
| Итерация (шаг) | одно применение рекурсивной части |
| Условие остановки | предикат/исчерпание данных, при котором шаг даёт 0 новых строк |
| Уровень (`level`) | носимый счётчик глубины обхода |

## Итог

- `WITH RECURSIVE имя AS ( ЯКОРЬ UNION[ALL] РЕКУРСИВНАЯ_ЧАСТЬ ) SELECT ...`.
  `RECURSIVE` — один раз после `WITH`. Якорь не ссылается на себя; рекурсивная
  часть ссылается на `имя` = строки прошлого шага.
- Модель исполнения: **якорь → повторять рекурсивную часть над рабочей таблицей
  (строками прошлого шага), добавляя новое в результат, пока шаг не даст 0 строк**.
- Рекурсивная часть видит **только предыдущий шаг**, а не весь результат.
- Обязано быть **условие остановки** (`WHERE`, исчерпание детей, защита от
  повторов) — иначе бесконечный цикл.
- Через рекурсию удобно **нести состояние**: `level + 1`, `path || узел`, строку
  цепочки.
- Агрегаты/`DISTINCT`/`ORDER BY`/`LIMIT` — **снаружи**, в главном запросе, не
  внутри рекурсии.
