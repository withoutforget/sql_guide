# 04 — Графы: достижимость, пути, циклы и защита от зацикливания

[← иерархии](03-hierarchies.md) · [оглавление](../README.md) · [дальше: разузлование BOM →](05-bom.md)

---

Деревья обходятся без риска: один родитель, циклов нет. **Графы** — сложнее: ветви
сходятся, и бывают **циклы** (из узла ведёт путь обратно в него). Это добавляет
одну большую новую проблему — **зацикливание рекурсии** — и требует **защиты**.
Разберём и достижимость с путями, и все способы защиты.

Работаем на графе маршрутов `routes(from_city, to_city, distance)` — направленном
(ребро `A → B` не подразумевает `B → A`). В нём **нарочно заложен цикл**:

```
Москва ─→ Казань ──→ Самара ──→ Уфа ──→ Екатеринбург ─→ Новосибирск
   │         ↑                    │
   └→ Нижний Новгород ────────────┘ (Уфа → Казань замыкает цикл Казань→Самара→Уфа→Казань)
```

Цикл: `Казань → Самара → Уфа → Казань`. На нём и покажем, что бывает без защиты.

## Достижимость: куда можно попасть из узла

«Все города, достижимые из Москвы». Здесь важен **сам факт** достижимости, а не
маршрут, поэтому решение простое — и защита от цикла берётся **бесплатно** за счёт
`UNION` (с дедупликацией):

```sql
WITH RECURSIVE reach AS (
    SELECT to_city AS city
    FROM routes WHERE from_city = 'Москва'      -- ЯКОРЬ: прямые рейсы из Москвы
    UNION                                        -- ← UNION, а не UNION ALL!
    SELECT r.to_city
    FROM reach re JOIN routes r ON r.from_city = re.city
)
SELECT city FROM reach ORDER BY city;
```
```
      city
-----------------
 Екатеринбург
 Казань
 Нижний Новгород
 Новосибирск
 Самара
 Уфа
(6 rows)
```

Это [задача 10](../queries/10_graph_reachability.sql). Почему не зациклилось,
несмотря на цикл в графе, — в следующем разделе.

## `UNION` vs `UNION ALL` в рекурсии — вопрос жизни и смерти

Разницу мы знаем из [lab003](../../lab003/theory/02-union-union-all.md): `UNION
ALL` дописывает всё подряд, `UNION` **убирает дубликаты**. В рекурсии по графу с
циклом это уже не про «дороже/дешевле», а про **завершится запрос или нет**.

Вспомним модель исполнения ([файл 02](02-syntax-and-execution.md)): для `UNION`
(без `ALL`) на каждом шаге строки, **уже присутствующие в результате**,
**выбрасываются**. Прокрутим достижимость от Москвы:

| Шаг | Рабочая таблица | Рекурсия даёт | Новое (после дедупа) |
|-----|-----------------|---------------|----------------------|
| Якорь | — | — | Казань, Нижний Новгород |
| 1 | Казань, Нижний Новгород | Самара (из Казани), Казань (из НН) | **Самара** (Казань уже есть — выкинута) |
| 2 | Самара | Уфа | **Уфа** |
| 3 | Уфа | Казань, Екатеринбург | **Екатеринбург** (Казань уже есть — выкинута) |
| 4 | Екатеринбург | Новосибирск | **Новосибирск** |
| 5 | Новосибирск | — | — |
| — | пусто → **стоп** | | |

Ключевой момент — **шаг 3**: рекурсия честно выдала `Казань` (из ребра `Уфа →
Казань`), но Казань **уже в результате** → `UNION` её выбросил → она не попала в
рабочую таблицу → цикл не пошёл на второй круг. **Дедупликация `UNION` и есть
защита от зацикливания** для задач достижимости.

А что было бы с `UNION ALL`? Казань на шаге 3 **не выбросилась** бы, попала в
рабочую таблицу, породила Самару, та — Уфу, Уфа — снова Казань… **бесконечно**.

> **Демонстрация зацикливания (безопасно, с ограничителем).** Чтобы **увидеть**, а
> не повесить сервер, добавим искусственный предел глубины `hops < 7`:
>
> ```sql
> WITH RECURSIVE walk AS (
>     SELECT from_city, to_city, 1 AS hops
>     FROM routes WHERE from_city = 'Казань'
>     UNION ALL                                   -- ALL: дедупа НЕТ
>     SELECT w.from_city, r.to_city, w.hops + 1
>     FROM walk w JOIN routes r ON r.from_city = w.to_city
>     WHERE w.hops < 7                            -- ← БЕЗ этого предела — вечный цикл
> )
> SELECT hops, to_city FROM walk ORDER BY hops, to_city;
> ```
> ```
>  hops |   to_city
> ------+--------------
>     1 | Самара
>     2 | Уфа
>     3 | Екатеринбург
>     3 | Казань          ← цикл вернул нас в Казань
>     4 | Новосибирск
>     4 | Самара          ← и снова пошёл по кругу: Казань→Самара…
>     5 | Уфа
>     6 | Екатеринбург
>     6 | Казань          ← опять Казань
>     7 | Новосибирск
>     7 | Самара
> ```
>
> Видно, как `Казань → Самара → Уфа → Казань` крутится по кругу: без предела
> `hops < 7` это никогда бы не остановилось. Предел здесь — **костыль ради показа**;
> в реальных запросах защиту делают правильно (см. ниже).

### Когда какой оператор в рекурсии

| Задача | Оператор | Почему |
|--------|----------|--------|
| Множество достижимых узлов (без путей) | **`UNION`** | дедуп сам гасит циклы; повторный узел не нужен |
| Дерево / DAG без циклов (BOM, оргштат) | **`UNION ALL`** | циклов нет, дедуп не нужен — и он быстрее |
| **Пути** в графе (маршруты, с повторами узлов в разных путях) | **`UNION ALL` + ручная защита** | разные пути в один узел — разные строки, дедупить нельзя; от цикла защищаемся путём (ниже) |

## Пути в графе: `UNION ALL` + защита путём-массивом

Как только нужны **сами маршруты** (а не просто «достижим/нет»), `UNION` не
годится: два разных пути в Екатеринбург — это **две разные строки**, дедуп их
схлопнет. Значит `UNION ALL` — но тогда цикл зациклит. Защита — **ручная**, через
накопленный **путь-массив**: не заходить в узел, который **уже есть** в текущем
пути.

[Задача 11](../queries/11_graph_all_paths.sql) — все маршруты Москва → Екатеринбург:

```sql
WITH RECURSIVE paths AS (
    SELECT from_city, to_city,
           ARRAY[from_city, to_city] AS path,     -- путь копим массивом
           distance AS total_dist
    FROM routes WHERE from_city = 'Москва'
    UNION ALL
    SELECT p.from_city, r.to_city,
           p.path || r.to_city,
           p.total_dist + r.distance
    FROM paths p JOIN routes r ON r.from_city = p.to_city
    WHERE NOT r.to_city = ANY(p.path)             -- ← ЗАЩИТА: узла ещё нет в пути
)
SELECT array_to_string(path, ' → ') AS route, total_dist
FROM paths WHERE to_city = 'Екатеринбург'
ORDER BY total_dist;
```
```
                              route                              | total_dist
-----------------------------------------------------------------+------------
 Москва → Нижний Новгород → Казань → Самара → Уфа → Екатеринбург |       2530
 Москва → Казань → Самара → Уфа → Екатеринбург                   |       2550
(2 rows)
```

Как работает защита:

- `path` — массив городов маршрута; `p.path || r.to_city` дописывает следующий
  город (оператор `||` для массивов — из [lab006](../../lab006/theory/02-unnest-ordinality-srf.md)).
- `NOT r.to_city = ANY(p.path)` — «следующего города **ещё нет** в пройденном
  пути». `x = ANY(массив)` истинно, если `x` равен хоть одному элементу массива;
  `NOT ...` — значит «не входит».
- Когда маршрут доходит до Уфы и хочет прыгнуть в Казань (ребро `Уфа → Казань`),
  Казань **уже в пути** `{Москва, …, Казань, Самара, Уфа}` → условие ложно →
  переход отсекается → **цикл не замыкается**. Маршруты получаются **простые** (без
  повторов узлов), рекурсия конечна.

Такой путь-массив решает **сразу три задачи**: и защищает от цикла, и даёт маршрут
для вывода, и (через `array_length` или счётчик рёбер) — длину пути.

## Штатная защита: предложение `CYCLE` (PostgreSQL 14+)

Ручной путь-массив надёжен, но многословен, и его легко забыть. Стандарт SQL (и
PostgreSQL с версии 14) даёт встроенную защиту — предложение **`CYCLE`** после тела
рекурсивного CTE:

```sql
WITH RECURSIVE paths AS (
    SELECT from_city, to_city,
           ARRAY[from_city, to_city] AS route,   -- свой массив — только для печати
           distance AS total_dist
    FROM routes WHERE from_city = 'Москва'
    UNION ALL
    SELECT p.from_city, r.to_city,
           p.route || r.to_city, p.total_dist + r.distance
    FROM paths p JOIN routes r ON r.from_city = p.to_city
    -- ручного WHERE-условия против цикла НЕТ
) CYCLE to_city SET is_cycle USING cyc_path
SELECT array_to_string(route, ' → ') AS route, total_dist
FROM paths
WHERE to_city = 'Екатеринбург' AND NOT is_cycle   -- отбросить «замыкающие» строки
ORDER BY total_dist;
```

Синтаксис: `CYCLE список_колонок SET флаг USING путь`.

- **`список_колонок`** (`to_city`) — по чему определять повтор узла. Как только
  очередной шаг даёт узел, чьи значения этих колонок **уже встречались** на пути,
  строка помечается как «цикл».
- **`SET флаг`** (`is_cycle`) — имя новой булевой колонки: `TRUE`, если этот шаг
  замкнул цикл. Такую строку CYCLE **выдаёт**, но дальше **не разворачивает** —
  этим и достигается конечность. Отфильтровать их — `WHERE NOT is_cycle`.
- **`USING путь`** (`cyc_path`) — имя новой колонки, куда CYCLE **сам** складывает
  пройденный путь (массив строк-кортежей). Для красивого вывода он неудобен
  (печатается как `{(Казань),(Самара),…}`), поэтому в [задаче 12](../queries/12_graph_paths_cycle_clause.sql)
  мы держим **свой** массив `route` для печати, а `CYCLE` используем только ради
  защиты.

Что `CYCLE` кладёт во флаг — видно, если вывести все строки обхода от Казани
(порядок строк без `ORDER BY` не гарантирован; «замыкающая» строка здесь показана
последней лишь для наглядности):

```
   to_city    | is_cycle |               cyc_path
--------------+----------+---------------------------------------
 Самара       | f        | {(Самара)}
 Уфа          | f        | {(Самара),(Уфа)}
 Казань       | f        | {(Самара),(Уфа),(Казань)}
 Екатеринбург | f        | {(Самара),(Уфа),(Екатеринбург)}
 Новосибирск  | f        | {(Самара),(Уфа),(Екатеринбург),(Новосибирск)}
 Самара       | t        | {(Самара),(Уфа),(Казань),(Самара)}   ← замкнулся цикл, дальше не идём
```

> **`CYCLE ... SET ... USING ...` — это ровно замена ручного `WHERE NOT ... =
> ANY(path)`**, только его пишет за вас движок. Оба подхода корректны; ручной —
> работает в любой версии и даёт полный контроль (и путь для вывода «даром»),
> `CYCLE` — короче и его труднее забыть. До PostgreSQL 14 (или для переносимости
> между СУБД) — только ручной способ.

## Порядок обхода: в глубину (DFS) vs в ширину (BFS)

Два фундаментальных порядка обхода:

- **В глубину** (depth-first, DFS): идём по одной ветви **до конца**, потом
  возвращаемся и берём следующую. Так печатают дерево «ветвь за ветвью».
- **В ширину** (breadth-first, BFS): обходим уровень за уровнем — сначала все
  соседи старта, потом их соседи и т.д. Так находят **кратчайший по числу рёбер**
  путь (первый раз доходим до узла — по кратчайшему маршруту).

Сама рекурсия и так «наращивает» результат по уровням (шаг = ещё одно ребро),
поэтому число рёбер = номер шага. Нужный **порядок вывода** задаётся снаружи:

- **DFS** — сортировкой по **пути-массиву** (как дерево в [файле 03](03-hierarchies.md):
  `ORDER BY path`): вся ветвь идёт единым блоком.
- **BFS** — сортировкой по **уровню/числу рёбер** (`ORDER BY hops`).
- **Кратчайший по рёбрам** — посчитать `hops` и снаружи взять минимум
  ([задача 13](../queries/13_shortest_path_edges.sql)): `WHERE hops = (SELECT
  MIN(hops) FROM paths WHERE to_city = 'Новосибирск')`.

### Штатное предложение `SEARCH` (PostgreSQL 14+)

Симметрично `CYCLE`, для **порядка обхода** есть предложение **`SEARCH`**:

```sql
) SEARCH DEPTH FIRST BY name SET ord       -- обход в глубину, порядок веток по name
) SEARCH BREADTH FIRST BY name SET ord     -- обход в ширину
...
ORDER BY ord                               -- сортируем по служебной колонке ord
```

- `DEPTH FIRST` / `BREADTH FIRST` — какой порядок;
- `BY колонки` — чем упорядочивать «братьев» (детей одного родителя);
- `SET ord` — имя новой служебной колонки, по которой затем делают `ORDER BY ord`.

Под капотом `SEARCH DEPTH FIRST` — тот же приём «сортировка по накопленному пути»,
что мы делаем руками; `BREADTH FIRST` — сортировка по уровню. Пользоваться можно
любым: ручной способ нагляднее и работает везде, `SEARCH` короче (PG14+).

## Итог

- В графе — ветви сходятся и бывают **циклы**; без защиты рекурсия по циклу
  **бесконечна**.
- **Достижимость** («куда можно попасть»): `UNION` (с дедупом) — дедупликация
  **сама гасит цикл**, потому что повторный узел выбрасывается.
- **Пути** («какими маршрутами»): нужен `UNION ALL` (разные пути — разные строки) +
  **защита**:
  - **ручная** — путь-массив и `WHERE NOT next = ANY(path)` (работает везде, даёт
    путь для вывода);
  - **штатная** — `CYCLE колонки SET флаг USING путь` + `WHERE NOT флаг` (PG14+,
    короче).
- **Порядок обхода**: DFS — сортировка по **пути**, BFS — по **уровню/числу рёбер**;
  кратчайший по рёбрам — `MIN(hops)` снаружи. Штатно — `SEARCH DEPTH/BREADTH FIRST
  BY ... SET ...` (PG14+).
