# 05 — Сортирующие агрегаты: перцентили, медиана, `mode`

[← корреляция и регрессия](04-correlation-regression.md) · [оглавление](../README.md) · [дальше: гипотетические агрегаты (rank WITHIN GROUP) →](06-hypothetical-set-rank.md)

---

Некоторым агрегатам, чтобы дать ответ, недостаточно **просто перебрать** строки —
им нужно, чтобы значения были **упорядочены**. «Значение в середине» (медиана),
«граница, ниже которой 25% данных» (квартиль), «самое частое значение» — все они
определяются **через порядок**. Такие агрегаты называют **сортирующими**
(ordered-set aggregates), и у них особый синтаксис — `WITHIN GROUP (ORDER BY ...)`.

## Синтаксис `WITHIN GROUP (ORDER BY ...)`

```sql
функция(аргумент) WITHIN GROUP (ORDER BY выражение_сортировки)
```

Читается так: «посчитай `функция` **внутри группы**, предварительно упорядочив её
по `выражение_сортировки`». `ORDER BY` здесь — **обязательная часть вызова**
(агрегату нужен порядок), а не сортировка всего запроса.

> Не путайте с `ORDER BY` внутри `array_agg`/`string_agg` из [файла
> 02](02-collections-and-boolean.md). Там `ORDER BY` — необязательное
> «украшение» (влияет лишь на порядок в списке). Здесь, у сортирующих агрегатов,
> `WITHIN GROUP (ORDER BY ...)` — **обязательный** синтаксис, без него функция не
> существует. Разное написание — разный смысл.

## `percentile_cont(доля)` — непрерывный перцентиль (и «настоящая» медиана)

**Перцентиль (percentile)** уровня `p` — это значение, **ниже которого лежит доля
`p` данных**. Перцентиль `0.5` — это середина (половина ниже, половина выше),
т.е. **медиана**. Перцентиль `0.25` — первый квартиль (ниже него четверть данных),
`0.9` — «90-й перцентиль» (ниже него 90% данных) и т.д.

`percentile_cont(p)` — **непрерывный** (continuous) перцентиль: если нужное место
попадает **между** двумя реальными значениями, он **интерполирует** (берёт точку
на прямой между ними). Поэтому результат может быть числом, которого **нет** в
данных, — зато это «настоящее» математическое положение перцентиля.

**Медиана** — это `percentile_cont(0.5)`:

```sql
SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY price) AS медиана
FROM products WHERE category = 'Электроника';
--  медиана
-- ---------
--     8000
```

Цены `Электроники`: `3000, 5000, 7000, 9000, 15000, 90000` (6 штук, чётное).
Середина — **между** 3-м (7000) и 4-м (9000) значениями. `percentile_cont`
интерполирует: `(7000 + 9000) / 2 = 8000`. Значения `8000` в прайсе нет — это
вычисленная середина.

### Как `percentile_cont` считает (пошагово)

Пусть в группе `n` отсортированных значений, индексируем их с 0: `v[0], v[1], ...,
v[n-1]`. Тогда:

1. позиция `pos = p · (n − 1)` (дробная);
2. если `pos` целая — ответ ровно `v[pos]`;
3. если дробная — линейная интерполяция между соседями:
   `v[floor(pos)] + (pos − floor(pos)) · (v[ceil(pos)] − v[floor(pos)])`.

Для медианы `Электроники`: `pos = 0.5 · 5 = 2.5` → между `v[2]=7000` и `v[3]=9000`
→ `7000 + 0.5·2000 = 8000`. Сходится.

## `percentile_disc(доля)` — дискретный перцентиль

`percentile_disc(p)` — **дискретный** (discrete): он **не интерполирует**, а
возвращает **реальное значение из набора** — первое, у которого накопленная доля
достигает `p`.

```sql
SELECT percentile_disc(0.5) WITHIN GROUP (ORDER BY price) AS медиана_disc
FROM products WHERE category = 'Электроника';
--  медиана_disc
-- -------------
--       7000.00
```

### Как `percentile_disc` считает

Идём по отсортированным значениям и берём **первое**, у которого доля
`номер_строки / n` (нумерация с 1) **достигает или превышает** `p`. Для медианы
(`p = 0.5`) при `n = 6`: нужно `k/6 ≥ 0.5` → `k ≥ 3` → берём 3-е значение =
`7000`. Никакой интерполяции — это **настоящая** цена из прайса.

## `cont` vs `disc` — в чём разница (на одних данных)

Разберём на маленьком чётном наборе `10, 20, 30, 40`, чтобы всё было видно:

```sql
SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY x) AS cont,
       percentile_disc(0.5) WITHIN GROUP (ORDER BY x) AS disc
FROM (VALUES (10),(20),(30),(40)) v(x);
```
```
 cont | disc
------+------
   25 |   20
```

- `percentile_cont(0.5) = 25` — интерполировал середину между 20 и 30. **Такого
  значения в данных нет.**
- `percentile_disc(0.5) = 20` — взял **реальное** 2-е значение (первое, где
  накопленная доля `2/4 = 0.5` достигла 0.5).

| | `percentile_cont` | `percentile_disc` |
|---|-------------------|-------------------|
| Интерполирует между соседями | **да** | нет |
| Результат всегда есть в данных | нет | **да** |
| Тип результата | `double precision` (для чисел) | **тот же тип**, что у сортируемого выражения |
| «Настоящая» медиана | **да** | приблизительная |

**Когда что.** Нужна математически корректная медиана/перцентиль (метрика,
статистика) — `percentile_cont`. Нужно обязательно **реально существующее**
значение (например, «типичный **реальный** прайс», категория, дата из набора, а
не выдуманная середина) — `percentile_disc`.

### Когда `cont` и `disc` совпадают

При **нечётном** числе элементов медиана попадает **ровно на средний элемент** —
интерполировать нечего, и обе функции дают **одинаковый** результат. Цены `Дома`:
`1500, 2000, 3000, 4000, 12000` (5 штук):

```sql
SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY price) AS cont,
       percentile_disc(0.5) WITHIN GROUP (ORDER BY price) AS disc
FROM products WHERE category = 'Дом';
--  cont | disc
-- ------+---------
--  3000 | 3000.00
```

Оба = `3000` (средний, 3-й элемент). Различие `cont`/`disc` проявляется в основном
на **чётном** числе элементов (и на «неудобных» долях вроде 0.9).

## Квартили одним вызовом: перцентиль от массива долей

Обеим функциям вместо одной доли можно передать **массив долей** — тогда они
вернут **массив** перцентилей за один проход. Классика — **квартили**
(`0.25, 0.5, 0.75`):

```sql
SELECT percentile_cont(ARRAY[0.25, 0.5, 0.75]) WITHIN GROUP (ORDER BY price) AS квартили
FROM products WHERE category = 'Электроника';
--       квартили
-- -------------------
--  {5500,8000,13500}
```

Читаем: Q1 = `5500` (ниже этой цены — четверть товаров), Q2 = `8000` (медиана),
Q3 = `13500` (ниже — три четверти). Разброс между Q1 и Q3 (**межквартильный
размах**, IQR = `13500 − 5500 = 8000`) — устойчивая к выбросам мера разброса
«центральной половины» данных. Достать элемент массива можно по индексу
(`(...)[1]` — первый) или развернуть `unnest`-ом из [lab006](../../lab006/).

`percentile_disc(ARRAY[...])` работает так же, но берёт реальные значения:

```sql
SELECT percentile_disc(ARRAY[0.25,0.5,0.75]) WITHIN GROUP (ORDER BY x)
FROM (VALUES (10),(20),(30),(40)) v(x);
--  {10,20,30}    -- реальные значения набора, без интерполяции
```

## `mode()` — самое частое значение

`mode()` (мода) возвращает **самое часто встречающееся** значение группы. Аргумент
у неё пустой — что именно считать «частым», задаёт `WITHIN GROUP (ORDER BY ...)`:

```sql
SELECT mode() WITHIN GROUP (ORDER BY rating) AS частый_рейтинг
FROM products;
--  частый_рейтинг
-- ----------------
--               5
```

Среди рейтингов всех 20 товаров оценка `5` встречается чаще всех (10 раз против 8
четвёрок и 2 троек) — `mode()` вернула `5`. Это **реальное** значение из данных
(мода всегда существует в наборе).

> **Ничья (tie) → решает порядок сортировки.** Если у нескольких значений
> частота одинаковая, `mode()` берёт **первое по заданному `ORDER BY`**:
> ```sql
> SELECT mode() WITHIN GROUP (ORDER BY x)      AS по_возр,
>        mode() WITHIN GROUP (ORDER BY x DESC) AS по_убыв
> FROM (VALUES (4),(4),(5),(5),(3)) v(x);
> --  по_возр | по_убыв
> -- ---------+---------
> --        4 |       5
> ```
> Здесь `4` и `5` встречаются по два раза — ничья. При `ORDER BY x` победит
> меньшее (`4`), при `ORDER BY x DESC` — большее (`5`). Если ничьи возможны и вам
> важно, кто победит, — задайте сортировку осознанно.

`mode` игнорирует `NULL`; на пустом входе → `NULL`.

## Медиана против среднего: диагностика перекоса

Вернёмся к мысли из [файла 01](01-why-and-overview.md). **Среднее** тянется за
выбросами, **медиана** — нет (её не сдвинуть, добавив один сколь угодно дорогой
товар: середина остаётся серединой). Поэтому **разрыв между средним и медианой —
готовый индикатор перекоса (skew)** распределения:

```sql
SELECT category,
       round(avg(price), 2)                               AS среднее,
       percentile_cont(0.5) WITHIN GROUP (ORDER BY price) AS медиана,
       round(avg(price) - percentile_cont(0.5)
             WITHIN GROUP (ORDER BY price)::numeric, 2)    AS разрыв
FROM products
GROUP BY category
ORDER BY разрыв DESC;
```
```
  category   | среднее  | медиана | разрыв
-------------+----------+---------+----------
 Электроника | 21500.00 |    8000 | 13500.00
 Дом         |  4500.00 |    3000 |  1500.00
 Игры        |  2000.00 |    2000 |     0.00
 Книги       |   800.00 |     800 |     0.00
```

- **`среднее ≫ медиана`** (большой положительный разрыв) → перекос **вправо**:
  есть дорогие «хвосты», задирающие среднее. «Электроника» (разрыв 13 500!) —
  один ноутбук за 90 000 утянул среднее до 21 500, хотя типичный товар стоит
  8 000. Среднее тут **вводит в заблуждение**, ориентироваться надо на медиану.
- **`среднее ≈ медиана`** (разрыв ≈ 0) → распределение **симметрично** («Книги»,
  «Игры»): выбросов нет, среднее честно описывает центр.
- **`среднее ≪ медиана`** → перекос **влево** (редкие очень **дешёвые**
  «хвосты»).

Отсюда практическое правило аналитика: **для денег, зарплат, времён отклика — почти
всегда сообщайте медиану, а не среднее** (эти величины обычно перекошены вправо).
Одно среднее без медианы и разброса легко вводит в заблуждение.

## Итоги файла

| Функция | Что возвращает | Особенности |
|---------|----------------|-------------|
| `percentile_cont(p)` | перцентиль уровня `p`, **интерполируя** | «настоящая» медиана; значения может не быть в данных |
| `percentile_disc(p)` | перцентиль — **реальное** значение из набора | без интерполяции; тип = тип данных |
| `percentile_*(ARRAY[...])` | **массив** перцентилей за один проход | квартили `[0.25,0.5,0.75]` |
| `mode()` | самое частое значение | ничью решает `ORDER BY` |

Все — с обязательным `WITHIN GROUP (ORDER BY ...)`, игнорируют `NULL`, на пустом
входе → `NULL`, отлично сочетаются с `GROUP BY` (по одному значению на группу).
