# 02 — Агрегаты-коллекции и булевы агрегаты

[← зачем это нужно](01-why-and-overview.md) · [оглавление](../README.md) · [дальше: разброс (дисперсия/стандартное отклонение) →](03-spread-variance-stddev.md)

---

Обычные агрегаты сворачивают группу в **число**. Но иногда нужно свернуть её не в
число, а **собрать сами значения** группы в один объект — список через запятую
или массив. Для этого есть агрегаты-коллекции: `string_agg` и `array_agg`.

## `string_agg(выражение, разделитель)` — склеить в одну строку

`string_agg` берёт значения группы и **склеивает их в одну текстовую строку**
через заданный разделитель. Классическая задача: «состав заказа одной строкой».

```sql
SELECT o.id AS заказ,
       string_agg(p.name, ', ') AS товары
FROM orders o
JOIN order_items oi ON oi.order_id = o.id
JOIN products    p  ON p.id = oi.product_id
GROUP BY o.id
ORDER BY o.id;
```
```
 заказ |                           товары
-------+-------------------------------------------------------------
     1 | Наушники TWS, Книга «SQL с нуля», Плед тёплый
     2 | Ноутбук ProBook, Мышь беспроводная, Клавиатура механическая
     ...
```

> Порядок товаров **внутри** каждой строки здесь показан условно: без `ORDER BY`
> внутри агрегата он **не гарантирован** и может меняться от запуска к запуску.
> Как сделать его предсказуемым — в разделе «`ORDER BY` внутри агрегата» ниже; в
> реальных запросах сортировку внутри задают всегда.

- Первый аргумент — **что склеивать** (текстовое выражение; не-текст приводите
  сами: `x::text`).
- Второй аргумент — **разделитель** (строка между элементами; ставится **между**,
  а не после — в конце разделителя нет).
- Как все агрегаты, `NULL`-значения **пропускаются** (не попадают в строку и не
  создают лишних разделителей).

## `array_agg(выражение)` — собрать в массив

`array_agg` собирает значения группы в **массив** PostgreSQL — тип `type[]`.
Массивы — отдельная большая тема (будет в lab018), но собрать их агрегатом просто:

```sql
SELECT category, array_agg(price) AS цены
FROM products
WHERE category = 'Книги'
GROUP BY category;
```
```
 category |          цены
----------+-------------------------
 Книги    | {500.00,700.00,900.00,1100.00}
```

> Как и у `string_agg`, порядок элементов в массиве без `ORDER BY` внутри **не
> определён** — здесь он показан условно. Детерминизм даёт `ORDER BY` внутри
> агрегата (следующий раздел).

Массив удобнее строки, когда результат пойдёт **дальше в SQL** (его можно
развернуть обратно в строки через `unnest` из [lab006](../../lab006/), измерить
`array_length`, проверить вхождение оператором `= ANY(...)`). Строка (`string_agg`)
удобнее, когда результат идёт **человеку на экран или в отчёт**.

## `ORDER BY` **внутри** агрегата — детерминизм

Важнейшая деталь. Без явной сортировки порядок элементов внутри `string_agg` /
`array_agg` **не определён** — СУБД склеивает их в том порядке, в каком ей удобно
читать строки, и этот порядок может меняться от запуска к запуску. Чтобы список
был **предсказуемым**, сортировку задают **прямо в вызове агрегата**:

```sql
string_agg(p.name, ', ' ORDER BY p.name)     -- по алфавиту
array_agg(price ORDER BY price DESC)          -- по цене, от дорогих
```

Это специальный синтаксис агрегатов: внутри скобок, **после аргументов**, перед
закрывающей скобкой, можно написать `ORDER BY`. Он сортирует **только вход этого
агрегата**, независимо от `ORDER BY` всего запроса.

```sql
SELECT category,
       string_agg(name, ', ' ORDER BY price DESC) AS от_дорогих_к_дешёвым
FROM products
GROUP BY category
ORDER BY category;
```
```
  category   |                    от_дорогих_к_дешёвым
-------------+-------------------------------------------------------------
 Электроника | Ноутбук ProBook, Монитор 27", Веб-камера HD, Клавиатура ...
 ...
```

> **Правило.** Если `string_agg`/`array_agg` идёт в отчёт или сравнивается —
> **всегда** пишите `ORDER BY` внутри. Список без заданного порядка — источник
> «плавающих» результатов и ложных различий при сравнении.

## `DISTINCT` внутри агрегата — только уникальные

Как и обычные агрегаты ([lab001](../../lab001/theory/05-aggregates.md)),
коллекции умеют `DISTINCT` — собрать только **уникальные** значения:

```sql
SELECT c.name,
       array_agg(DISTINCT p.category ORDER BY p.category) AS категории
FROM customers   c
JOIN orders      o  ON o.customer_id = c.id
JOIN order_items oi ON oi.order_id = o.id
JOIN products    p  ON p.id = oi.product_id
GROUP BY c.name
ORDER BY c.name;
```
```
 name  |       категории
-------+-------------------------
 Анна  | {Дом,Книги,Электроника}
 Борис | {Электроника}
 ...
```

Анна купила много товаров, но **разных категорий** у неё три — `DISTINCT` убрал
повторы. Так собирают «из каких категорий покупал клиент», «какие теги у статьи»
и т.п.

> **Ограничение при `DISTINCT`.** Если в агрегате есть `DISTINCT`, то `ORDER BY`
> внутри может сортировать **только по тому же выражению**, что стоит под
> `DISTINCT` (иначе PostgreSQL выдаст ошибку: непонятно, как упорядочивать
> схлопнутые дубликаты). То есть `array_agg(DISTINCT category ORDER BY category)`
> — можно, а `array_agg(DISTINCT category ORDER BY price)` — нельзя.

## Джойн размножает строки — помните про фан-аут

Коллекции собирают **строки, которые дошли до группировки**. Если джойны
[размножили](../../lab002/theory/06-multi-fanout-antijoin.md) строки (фан-аут),
в список попадут дубликаты. Два лекарства: `DISTINCT` внутри агрегата (убрать
повторы значений) или предварительная агрегация в CTE
([lab005](../../lab005/)) до джойна. Осознавайте, на каком «зерне» (grain) данных
работает агрегат.

## Короткий forward-ссылка: `json_agg` / `jsonb_agg`

У `array_agg` есть «json-родня»: **`json_agg(x)`** и **`jsonb_agg(x)`** собирают
значения группы в **JSON-массив**, а `json_object_agg(k, v)` — в JSON-объект.
Работают по тем же правилам (`ORDER BY` внутри, игнор `NULL`), но результат — не
массив PostgreSQL, а значение типа `json`/`jsonb`. Это удобно, когда данные
отдаются в API «как есть».

```sql
SELECT jsonb_agg(name ORDER BY price DESC) AS json_список
FROM products WHERE category = 'Книги';
-- ["Книга «Алгоритмы»", "Книга «PostgreSQL»", "Книга «Чистый код»", "Книга «SQL с нуля»"]
```

Подробно про JSON, его операторы и построение — отдельная тема
**lab017**. Здесь достаточно знать, что такие агрегаты существуют.

## Булевы агрегаты: `bool_and` / `bool_or`

Иногда группу нужно свернуть в **один флаг** «да/нет». Для булевых выражений есть:

| Агрегат | Возвращает `true`, если… | Аналог логики |
|---------|--------------------------|---------------|
| `bool_and(усл)` | условие истинно **для всех** строк группы | «И» / ∀ (для всех) |
| `bool_or(усл)`  | условие истинно **хотя бы для одной** строки | «ИЛИ» / ∃ (существует) |

```sql
SELECT category,
       bool_or(rating = 5)       AS есть_пятизвёздочный,
       bool_and(units_sold > 50) AS все_продавались_больше_50
FROM products
GROUP BY category
ORDER BY category;
```
```
  category   | есть_пятизвёздочный | все_продавались_больше_50
-------------+---------------------+---------------------------
 Дом         | t                   | f
 Игры        | t                   | t
 Книги       | t                   | t
 Электроника | t                   | f
```

`bool_or(rating = 5)` = «в категории **есть** товар с 5 звёздами». В «Доме»
`bool_and(units_sold > 50)` = `false`, потому что робот-пылесос продан всего 10
раз — не «все больше 50». Как и прочие агрегаты, `NULL`-строки игнорируются;
пустой вход → `NULL`. (У `bool_and`/`bool_or` есть SQL-стандартные синонимы
`every(...)` и — для `bool_or` — без общепринятого второго имени; на практике
пишут `bool_and`/`bool_or`.)

Родня для битов — `bit_and`/`bit_or` (побитовые И/ИЛИ по целочисленной колонке) —
встречается редко, упомянем для полноты: они складывают все значения группы
одним побитовым оператором.

## Напоминание: `FILTER` — агрегат по подмножеству

Из [lab001](../../lab001/theory/05-aggregates.md) вы помните `FILTER (WHERE ...)` —
он считает агрегат **только по строкам, удовлетворяющим условию**, не трогая
остальные. Он работает **с любым** агрегатом этой лабы — и со статистическими, и
с коллекциями:

```sql
SELECT category,
       count(*)                          AS всего,
       count(*) FILTER (WHERE rating = 5) AS пятизвёздочных,
       string_agg(name, ', ') FILTER (WHERE rating = 5) AS какие_именно
FROM products
GROUP BY category
ORDER BY category;
```

`FILTER` и `bool_or(усл)` иногда решают близкие задачи: `count(*) FILTER (WHERE
усл) > 0` — то же самое, что `bool_or(усл)`. Выбирайте, что читается яснее.

## Итоги файла

| Функция | Собирает | Сортировка внутри | `DISTINCT` |
|---------|----------|-------------------|------------|
| `string_agg(x, sep)` | строку через `sep` | `ORDER BY` — обязательно для порядка | да |
| `array_agg(x)` | массив `x[]` | то же | да |
| `json_agg` / `jsonb_agg` | JSON-массив (см. lab017) | то же | да |
| `bool_and` / `bool_or` | флаг ∀ / ∃ | — | — |

Главное, что стоит унести: **`string_agg`/`array_agg` без `ORDER BY` внутри —
недетерминированы**; `DISTINCT` убирает повторы; всё это по-прежнему обычные
агрегаты (игнор `NULL`, пустой вход → `NULL`, работают с `GROUP BY` и `FILTER`).
