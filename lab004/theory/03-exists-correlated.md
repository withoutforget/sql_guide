# 03 — `EXISTS` / `NOT EXISTS` и коррелированные подзапросы

[← IN / ANY / ALL](02-in-any-all.md) · [оглавление](../README.md) · [дальше: подзапрос в FROM и строчные →](04-from-derived-row.md)

---

Это центральный файл лабы. Сначала разберём различие **коррелированных** и
**некоррелированных** подзапросов, затем — `EXISTS`/`NOT EXISTS`, самый
выразительный инструмент для вопросов «есть ли связанная строка».

## Коррелированный vs некоррелированный подзапрос

До сих пор все подзапросы были **некоррелированными**: внутренний запрос
самодостаточен, не упоминает внешний и мог бы выполниться сам по себе.

```sql
-- некоррелированный: подзапрос ничего не знает про внешнюю строку
SELECT name FROM products
WHERE price > (SELECT AVG(price) FROM products);
--                              └── можно выполнить отдельно, всегда даёт 9643.33
```

**Коррелированный** подзапрос — это подзапрос, который **ссылается на колонку
строки внешнего запроса**. Сам по себе он выполниться не может: в нём есть «дырка»,
которую заполняет текущая строка снаружи.

```sql
-- коррелированный: p.category_id — это колонка ВНЕШНЕЙ строки
SELECT p.name, p.price
FROM products AS p
WHERE p.price > (SELECT AVG(p2.price) FROM products AS p2
                 WHERE p2.category_id = p.category_id);   -- ← ссылка на p снаружи
```

Здесь подзапрос считает среднюю цену **в категории текущего товара** `p`. Для
разных `p` он даёт разное число — потому что зависит от `p.category_id`. Отдельно,
без внешнего `p`, такой подзапрос не имеет смысла (`p.category_id` неизвестен).

### Как это исполняется — мысленная модель

Для коррелированного подзапроса держите в голове такую картинку (концептуально;
реальный планировщик хитрее — это в lab025):

1. Внешний запрос идёт по своим строкам **по одной**.
2. Для **каждой** внешней строки берётся её значение (`p.category_id`),
   подставляется в подзапрос, и подзапрос выполняется **заново**.
3. Результат подзапроса используется в условии для этой строки.

То есть коррелированный подзапрос выполняется **по разу на каждую внешнюю строку**,
а некоррелированный — **один раз на весь запрос**. Это ключевое различие и по
смыслу, и по стоимости.

| | Некоррелированный | Коррелированный |
|--|-------------------|-----------------|
| Ссылается на внешнюю строку | нет | да |
| Можно выполнить отдельно | да | нет |
| Сколько раз считается (концептуально) | 1 раз | по разу на внешнюю строку |
| Типичные конструкции | `IN`, `= (…)`, `ANY`/`ALL` | `EXISTS`, коррелированный `= (…)` |

### Коррелированный скалярный подзапрос — «дороже среднего в своей категории»

Пример выше — рабочий и очень практичный. Полностью:

```sql
SELECT p.name, p.price, p.category_id
FROM products AS p
WHERE p.price > (SELECT AVG(p2.price) FROM products AS p2
                 WHERE p2.category_id = p.category_id)
ORDER BY p.category_id, p.price DESC;
```
```
        name         |  price   | category_id
---------------------+----------+-------------
 Ноутбук AirBook 14  | 74990.00 |           1
 Книга "Алгоритмы"   |  2490.00 |           2
 Робот-пылесос Clean | 12990.00 |           3
 Кофеварка Barista   |  8990.00 |           3     ← в «Доме» ДВА товара выше своего среднего
 Конструктор "Город" |  2790.00 |           4
 Йога-коврик         |  1490.00 |           5
(6 rows)
```

Обратите внимание: порог сравнения **свой в каждой категории**. В «Доме» среднее
≈ 7990 ₽, и выше него сразу два товара (Кофеварка 8990 и Робот-пылесос 12990);
в «Электронике» среднее ≈ 27365 ₽, и выше только Ноутбук. Одним фиксированным
числом (как в скалярном подзапросе из [файла 01](01-what-is-subquery.md)) это не
выразить — нужна **корреляция** с категорией текущей строки.

> Обязательны разные алиасы: внешняя таблица — `p`, внутренняя — `p2`. Иначе
> `category_id` внутри подзапроса было бы неоднозначным. Это та же дисциплина
> алиасов, что в self-join из lab002 — только здесь одна «копия» стоит внутри
> подзапроса.

## `EXISTS (подзапрос)` — есть ли хоть одна строка

`EXISTS` — предикат, который отвечает на вопрос **«вернул ли подзапрос хотя бы
одну строку?»**. Возвращает `TRUE`, если подзапрос дал ≥ 1 строки, и `FALSE`, если
0 строк. Почти всегда используется **коррелированно** — подзапрос проверяет
наличие связанной строки для текущей внешней строки.

```sql
-- клиенты, купившие хоть что-то дороже 20 000 ₽
SELECT c.id, c.name
FROM customers AS c
WHERE EXISTS (
    SELECT 1
    FROM orders AS o
    JOIN products AS p ON p.id = o.product_id
    WHERE o.customer_id = c.id       -- ← корреляция: заказы именно этого клиента
      AND p.price > 20000
)
ORDER BY c.id;
```
```
 id | name
----+-------
  1 | Анна
  2 | Борис
(2 rows)
```

Как это читается: «оставь клиента `c`, если **существует** его заказ на товар
дороже 20 000 ₽». Для каждого клиента подзапрос ищет такой заказ; нашёл хотя бы
один — `EXISTS` истинно, клиент в выборке. Анна купила Смартфон (24990), Борис —
Ноутбук (74990); больше ни у кого дорогих покупок нет.

### `EXISTS` не важно, что в `SELECT` — пишут `SELECT 1`

`EXISTS` смотрит только на **факт наличия строк**, а не на их содержимое. Что бы
ни стояло в `SELECT` подзапроса — `*`, `1`, `o.id`, `NULL` — результат `EXISTS`
одинаков. По соглашению пишут **`SELECT 1`**: это подчёркивает «мне не нужны
значения, только само наличие строки», и не вводит в заблуждение, будто колонки
важны.

```sql
WHERE EXISTS (SELECT 1   FROM orders o WHERE o.customer_id = c.id)   -- канонично
WHERE EXISTS (SELECT *   FROM orders o WHERE o.customer_id = c.id)   -- то же самое
WHERE EXISTS (SELECT o.id FROM orders o WHERE o.customer_id = c.id)  -- то же самое
```

Все три эквивалентны. Планировщик всё равно не вычисляет этот `SELECT` — он
останавливается, как только находит первую подходящую строку.

## `NOT EXISTS` — нет ни одной связанной строки

`NOT EXISTS` — отрицание: `TRUE`, если подзапрос вернул **0 строк**. Это
**анти-соединение** — «оставить тех, у кого нет пары».

```sql
-- клиенты, не купившие ни одной книги (категория 2)
SELECT c.id, c.name
FROM customers AS c
WHERE NOT EXISTS (
    SELECT 1
    FROM orders AS o
    JOIN products AS p ON p.id = o.product_id
    WHERE o.customer_id = c.id
      AND p.category_id = 2          -- книги
)
ORDER BY c.id;
```
```
 id | name
----+-------
  2 | Борис
  4 | Глеб
  6 | Егор
  8 | Захар
(4 rows)
```

«Оставь клиента, для которого **не существует** ни одного заказа книги». Борис,
Глеб, Егор книг не покупали; Захар вообще ничего не заказывал (для него подзапрос
пуст → `NOT EXISTS` истинно). Анна, Вера, Дарья, Жанна книги брали — их нет.

## `EXISTS` и полу-/анти-соединения из lab002 — один вопрос, другой инструмент

В lab002 те же задачи мы решали джойнами, в lab003 — операциями над множествами.
Полезно видеть все формы рядом.

**Полу-соединение** («у кого есть пара»): клиенты, сделавшие хоть один заказ.

```sql
-- через EXISTS (коррелированно)                    -- через JOIN + DISTINCT (lab002)
SELECT c.id, c.name FROM customers c                SELECT DISTINCT c.id, c.name
WHERE EXISTS (SELECT 1 FROM orders o                FROM customers c
             WHERE o.customer_id = c.id);           JOIN orders o ON o.customer_id = c.id;
-- через IN (файл 02)
SELECT id, name FROM customers
WHERE id IN (SELECT customer_id FROM orders);
```

**Анти-соединение** («у кого нет пары»): клиенты без единого заказа.

```sql
-- через NOT EXISTS                                 -- через LEFT JOIN ... IS NULL (lab002)
SELECT c.id, c.name FROM customers c                SELECT c.id, c.name FROM customers c
WHERE NOT EXISTS (SELECT 1 FROM orders o            LEFT JOIN orders o ON o.customer_id = c.id
                 WHERE o.customer_id = c.id);       WHERE o.id IS NULL;
```

Все дают один результат (Захар в анти-случае). Что выбрать:

- **`EXISTS`/`NOT EXISTS`** обычно **читаются лучше всего** для «есть/нет связи»:
  условие стоит рядом с проверкой, дубликаты не возникают (не нужен `DISTINCT`),
  и — важно — **`NOT EXISTS` NULL-безопасен**, в отличие от `NOT IN`
  ([файл 02](02-in-any-all.md)).
- **`JOIN`** выбирают, когда из связанной таблицы нужны **колонки** (а не только
  факт наличия связи).
- Подробное сравнение «когда что» — в [файле 05](05-choosing-pitfalls.md).

### Почему `NOT EXISTS` не боится `NULL`

Вспомним ловушку из [файла 02](02-in-any-all.md): `NOT IN` ломается от `NULL`,
потому что сравнивает **значения** (`x <> NULL` = `UNKNOWN`). А `EXISTS`
сравнивает **строки на существование** — он проверяет условие внутри подзапроса и
считает строки. Условие `o.customer_id = c.id` для конкретных строк либо истинно,
либо нет; `NULL` в несвязанных строках просто не даёт совпадения, но не «отравляет»
весь результат. Поэтому «кто никого не приглашал» (пример с `referrer_id` из
[файла 02](02-in-any-all.md)) правильно решается именно через `NOT EXISTS`:

```sql
SELECT c.id, c.name
FROM customers AS c
WHERE NOT EXISTS (SELECT 1 FROM customers AS r WHERE r.referrer_id = c.id)
ORDER BY c.id;
-- корректно: Глеб, Дарья, Егор, Жанна, Захар — несмотря на NULL в referrer_id
```

## `EXISTS` vs `IN` — в чём разница

Оба выражают «есть ли связь», и часто взаимозаменяемы. Отличия:

| | `IN (подзапрос)` | `EXISTS (подзапрос)` |
|--|------------------|----------------------|
| Что сравнивает | значение с **колонкой** значений | наличие строк (условие внутри) |
| Форма подзапроса | ровно **1 колонка** | любая (обычно `SELECT 1`) |
| Обычно | некоррелированный | коррелированный |
| `NOT`-версия и `NULL` | **`NOT IN` опасен** при `NULL` | **`NOT EXISTS` безопасен** |
| Несколько условий связи | неудобно | естественно (`WHERE a=… AND b=…`) |

Практика: для «принадлежит списку значений» берут `IN`; для «существует связанная
строка (возможно, по нескольким условиям)» и **всегда** для отрицания — `EXISTS` /
`NOT EXISTS`.

## Итог

- **Некоррелированный** подзапрос не зависит от внешней строки (считается один
  раз); **коррелированный** ссылается на неё (концептуально — по разу на строку).
- Коррелированный скалярный подзапрос даёт «свой порог для каждой строки» —
  например, «дороже среднего **в своей категории**».
- `EXISTS` — «есть ли хоть одна строка»; `NOT EXISTS` — «нет ни одной». В `SELECT`
  подзапроса пишут `SELECT 1` — содержимое не важно.
- `EXISTS`/`NOT EXISTS` — самый чистый способ выразить полу-/анти-соединение; в
  отличие от `NOT IN`, **`NOT EXISTS` не ломается на `NULL`**.
