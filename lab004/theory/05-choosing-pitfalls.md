## Реляционное деление: вопросы «купил ВСЕ …» через двойной `NOT EXISTS`

Есть класс вопросов с квантором **всеобщности** («для всех»): «клиенты, купившие
**все** товары категории», «товары, заказанные **каждым** клиентом из города»,
«студенты, сдавшие **все** экзамены». В реляционной алгебре это операция
**деления**, и в SQL для неё нет отдельного оператора — её выражают через
**двойное отрицание** с `NOT EXISTS`.

Ключ — логическое преобразование. Фраза

> «клиент купил **все** книги»

эквивалентна

> «**не существует** книги, которую этот клиент **не** покупал».

Двойное «не» и есть два `NOT EXISTS`, вложенных друг в друга. Смысл: перебираем
все книги; если нашлась хоть одна, которой у клиента нет, — он не подходит; если
такой книги **не существует** — купил все.

```sql
-- клиенты, купившие ВСЕ книги (категория 2)
SELECT c.id, c.name
FROM customers AS c
WHERE NOT EXISTS (                               -- нет такой книги,
    SELECT 1
    FROM products AS p
    WHERE p.category_id = 2
      AND NOT EXISTS (                           -- которую этот клиент НЕ покупал
          SELECT 1
          FROM orders AS o
          WHERE o.customer_id = c.id
            AND o.product_id  = p.id
      )
)
ORDER BY c.id;
```
```
 id | name
----+------
  1 | Анна
(1 row)
```

Разберём по слоям (изнутри наружу):

1. **Внутренний `NOT EXISTS`** — «клиент `c` **не** покупал книгу `p`»: нет заказа,
   связывающего этого клиента с этой книгой.
2. **Внешний `NOT EXISTS`** — «**не существует** книги `p`, для которой верно
   пункт 1», то есть нет ни одной непокупенной клиентом книги.
3. Значит, клиент купил каждую книгу. У нас это только Анна (взяла все три книги;
   Дарья купила две из трёх — не проходит).

> Обратите внимание: обе внешние таблицы участвуют в корреляции. Внутренний
> подзапрос ссылается и на `p.id` (книга из среднего слоя), и на `c.id` (клиент из
> самого внешнего запроса). Это коррелированный подзапрос «на два уровня вверх» —
> обычное дело для деления.

### Тот же паттерн с другого ракурса — «заказан каждым …»

Деление работает в любую сторону — надо лишь понять, что на что «делим». «Товары,
которые заказал **каждый** клиент из Москвы» = «нет такого москвича, который бы
этот товар **не** заказывал»:

```sql
-- товары, заказанные ВСЕМИ клиентами из Москвы
SELECT p.id, p.name
FROM products AS p
WHERE NOT EXISTS (                               -- нет такого москвича,
    SELECT 1
    FROM customers AS c
    WHERE c.city = 'Москва'
      AND NOT EXISTS (                           -- который НЕ заказывал этот товар
          SELECT 1
          FROM orders AS o
          WHERE o.customer_id = c.id
            AND o.product_id  = p.id
      )
)
ORDER BY p.id;
```
```
 id |       name
----+------------------
  1 | Наушники TWS Pro
(1 row)
```

Москвичи — Анна, Вера, Егор; единственный товар, который заказали все трое, —
Наушники. Структура запроса **идентична** предыдущему, поменялись только роли
(«делимое» и «делитель»). Освоив один, вы решаете весь класс.

### Мысленный шаблон деления

```
SELECT <кандидат>                                -- напр. клиент
WHERE NOT EXISTS (
    SELECT 1 FROM <множество, которое надо покрыть целиком>   -- напр. все книги
    WHERE <ограничение множества>
      AND NOT EXISTS (
          SELECT 1 FROM <связь>                  -- напр. заказы
          WHERE <связь.кандидат = кандидат>
            AND <связь.элемент  = элемент>
      )
);
```

Читается как «нет элемента, не покрытого кандидатом» = «кандидат покрывает все
элементы». Этот шаблон стоит просто запомнить.

> ⚠️ Край «пустой делитель → вакуумная истина». Если множество, по которому
> делим, **пусто** (например, категория вообще без товаров), то внешний
> `NOT EXISTS` тривиально истинен для **каждого** кандидата — формально «купил все
> товары категории» выполняется у всех, включая тех, кто не купил ничего. Это не
> баг SQL, а логика: «для всех элементов пустого множества…» истинно всегда. Если
> такой ответ не нужен, добавьте условие, что множество непусто (например,
> `EXISTS (SELECT 1 FROM products WHERE category_id = …)`).

> Почему не `HAVING COUNT`? Деление часто решают и так: посчитать, сколько разных
> нужных элементов купил клиент, и сравнить с общим числом элементов
> (`HAVING COUNT(DISTINCT …) = (SELECT COUNT(*) …)`). Это рабочая альтернатива, но
> она требует аккуратности с дубликатами и `DISTINCT`. Двойной `NOT EXISTS` —
> каноничная, устойчивая к дубликатам формулировка деления, и она отлично
> тренирует мышление через `EXISTS`.

## Один вопрос — разные инструменты

За четыре лабы мы набрали несколько способов отвечать на «связочные» вопросы.
Часто **один и тот же вопрос** выражается через `JOIN` (lab002), операции над
множествами (lab003) и подзапросы (эта лаба). Единственно верного нет — есть более
читаемый под конкретную задачу.

| Вопрос | `JOIN` (lab002) | Множества (lab003) | Подзапрос (lab004) |
|--------|-----------------|--------------------|--------------------|
| «есть связанная строка» (полу-джойн) | `JOIN … DISTINCT` | — | `IN` / `EXISTS` |
| «нет связанной строки» (анти-джойн) | `LEFT JOIN … IS NULL` | `EXCEPT` | `NOT EXISTS` (не `NOT IN`!) |
| «купили и X, и Y» | самосоединение по клиенту | `INTERSECT` | два `EXISTS` |
| «купили X, но не Y» | `LEFT JOIN … IS NULL` | `EXCEPT` | `EXISTS` X + `NOT EXISTS` Y |
| «дороже среднего по категории» | — (неудобно) | — | коррелированный скалярный |
| «максимум в каждой группе» | тета-self-join | — | `(k, MAX) IN (…)` / корр. |
| «купили ВСЕ из набора» | (сложно) | (сложно) | двойной `NOT EXISTS` |
| «агрегат от агрегата» | — | — | подзапрос в `FROM` |

### Как выбирать

- **`JOIN`** — когда нужны **колонки** из связанной таблицы (имя, цена, дата), а не
  только факт связи. `JOIN` строит широкие строки.
- **`IN` / `EXISTS`** — когда важен **факт наличия** связи, а не значения. Не
  размножают строки (не нужен `DISTINCT`), читаются близко к формулировке. Для
  «есть ли» — `IN` или `EXISTS`; для «нет ли» — **`NOT EXISTS`**.
- **Операции над множествами** — когда сравниваете **два результата одинаковой
  формы** как наборы строк (слить, пересечь, вычесть).
- **Скалярный / коррелированный подзапрос** — когда нужно сравнить строку с
  **вычисленным показателем** (среднее, максимум, в том числе «в своей группе»).
- **Подзапрос в `FROM`** — когда нужен **агрегат от агрегата** или
  предварительная свёртка перед соединением.

### Дубликаты, `NULL`, читаемость — коротко

| Критерий | `JOIN` | `IN` | `EXISTS` |
|----------|--------|------|----------|
| Может **размножить** строки | да (fan-out) | нет | нет |
| `NOT`-форма и `NULL` | безопасна (`IS NULL`) | **`NOT IN` опасен** | **`NOT EXISTS` безопасен** |
| Нужны колонки связанной таблицы | да | нет | нет |
| Несколько условий связи | удобно (`ON …`) | неудобно | удобно (`WHERE …`) |

## Коротко про исполнение (детали — в lab025)

- **Некоррелированный** подзапрос (`= (…)`, `IN (…)`) обычно считается **один
  раз**: его результат вычисляется и используется как готовое значение/список.
- **Коррелированный** (`EXISTS`, коррелированный скаляр) концептуально
  выполняется **на каждую внешнюю строку**. Но не пугайтесь «квадратичности»:
  планировщик PostgreSQL часто переписывает `EXISTS`/`IN` в **полу-соединение**
  (hash/merge semi-join), а `NOT EXISTS` — в **анти-соединение**, и по скорости
  это сопоставимо с `JOIN`. Иными словами, `EXISTS` и `IN` — это не «медленно», а
  «другой способ записать ту же идею». **Как именно** СУБД это исполняет и как
  читать план — тема lab025 (`EXPLAIN`); здесь важна семантика, а не план.

## Чек-лист подводных камней

| Ловушка | В чём беда | Как избежать |
|---------|-----------|--------------|
| **`NOT IN` + `NULL`** | Один `NULL` в подзапросе → пустой/неверный результат, без ошибки | `NOT EXISTS`; либо `WHERE … IS NOT NULL` в подзапросе |
| **Скалярный подзапрос вернул >1 строки** | Ошибка выполнения | Агрегат (`MAX`/`MIN`) или переход на `IN`/`ANY` |
| **Скалярный подзапрос вернул 0 строк** | Тихо `NULL` → может «съесть» строку в `WHERE` | Помнить про `NULL`; `COALESCE((…), 0)` при нужде |
| **`> ALL` со `NULL` в списке** | Как `NOT IN` — `NULL` всё «отравляет» | Отфильтровать `NULL`; помнить, что `ANY` безопаснее |
| **Подзапрос в `FROM` без алиаса** | Ошибка (в стандарте и PG < 16) | Всегда давать алиас производной таблице |
| **`IN` с многоколоночным подзапросом** | `IN (SELECT a, b …)` без кортежа — ошибка | Одна колонка; либо строчная форма `(x,y) IN (…)` |
| **Путать корреляцию алиасов** | Внутри и снаружи один алиас → неоднозначность/не тот смысл | Разные алиасы для внешней и внутренней таблиц (`p` / `p2`) |
| **`NOT IN`/деление вместо `JOIN`, когда нужны колонки** | Подзапрос не отдаёт колонки связанной таблицы | Нужны поля — берите `JOIN` |

## Итог лабы

- Подзапрос — `SELECT` внутри запроса; место определяет форму результата (скаляр /
  список / наличие / таблица / кортеж).
- **Скалярный** — одно значение (>1 строки → ошибка, 0 → `NULL`).
- **`IN`/`ANY`/`ALL`** — сравнение со списком; помни ловушку **`NOT IN` + `NULL`**
  и тождества `= ANY ≡ IN`, `<> ALL ≡ NOT IN`, `> ANY ≡ >min`, `> ALL ≡ >max`.
- **`EXISTS`/`NOT EXISTS`** и **коррелированные** подзапросы — «есть/нет связанной
  строки», NULL-безопасное анти-соединение, «свой порог для каждой строки».
- **Подзапрос в `FROM`** — агрегат от агрегата; **строчный** — сравнение кортежей.
- **Двойной `NOT EXISTS`** — реляционное деление, вопросы «покрывает **все**».
