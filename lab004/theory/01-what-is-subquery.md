# 01 — Что такое подзапрос. Скалярный подзапрос

[← оглавление лабы](../README.md) · [дальше: IN / ANY / ALL →](02-in-any-all.md)

## Что такое подзапрос

**Подзапрос** (subquery, вложенный запрос) — это `SELECT`, заключённый в круглые
скобки и стоящий **внутри** другого SQL-запроса. Общая идея — «**запрос внутри
запроса**»: внешний запрос пользуется результатом внутреннего как готовым
значением, списком или таблицей.

```sql
SELECT name, price
FROM products
WHERE price > (SELECT AVG(price) FROM products);   -- ← подзапрос в скобках
--             └───────── внутренний запрос ──────┘
```

Здесь внутренний `SELECT AVG(price) FROM products` сначала считает среднюю цену
(одно число), а внешний запрос сравнивает с ним цену каждого товара. Мы **не
знаем** это число заранее и не хотим вписывать его руками — подзапрос вычисляет
его прямо в момент выполнения.

Зачем это нужно: подзапрос позволяет в **одном** запросе сослаться на результат
другого запроса — «отобрать товары дороже среднего», «клиентов, у которых есть
заказ», «самый дорогой товар в каждой категории». Без подзапросов пришлось бы
делать это в два приёма (сначала узнать число/список, потом подставить руками),
а то и вовсе невозможно выразить.

## Где может стоять подзапрос

Подзапрос — это выражение, и стоять он может в разных местах запроса. От **места**
зависит, что он должен вернуть и как называется. Вот карта на всю лабу:

| Где стоит | Что должен вернуть | Как называют | Файл |
|-----------|--------------------|--------------|------|
| в `SELECT` (список колонок) | **одно значение** (1 строка × 1 колонка) | скалярный подзапрос | этот файл |
| в `WHERE` / `HAVING` с `=`,`>`,`<` … | одно значение | скалярный подзапрос | этот файл |
| в `WHERE` / `HAVING` с `IN`,`ANY`,`ALL` | **столбец значений** (N строк × 1 колонка) | подзапрос-список | [02](02-in-any-all.md) |
| в `WHERE` / `HAVING` с `EXISTS` | важно лишь **есть ли строки** | предикатный подзапрос | [03](03-exists-correlated.md) |
| в `FROM` | **таблицу** (N строк × M колонок) | производная таблица | [04](04-from-derived-row.md) |
| сравнение кортежей `(a,b) = (…)` | **одну строку** из нескольких колонок | строчный подзапрос | [04](04-from-derived-row.md) |

Ключевое наблюдение: **сколько данных должен вернуть подзапрос, диктует
контекст**, в котором он стоит. Скалярному контексту (`price > (...)`) нужно ровно
одно значение; `IN (...)` ждёт столбец; `FROM (...)` — целую таблицу. Нарушишь
ожидание — получишь либо ошибку, либо (реже) тихо неверный результат. Каждый
случай разберём подробно.

> Ещё подзапросы делят на **некоррелированные** (внутренний запрос
> самодостаточен, не зависит от внешнего — как `AVG(price)` выше) и
> **коррелированные** (ссылается на строку внешнего запроса). Это различие — в
> [файле 03](03-exists-correlated.md); пока все примеры этого файла
> некоррелированные.

## Скалярный подзапрос: ровно одно значение

**Скалярный подзапрос** — это подзапрос, который возвращает **ровно одно
значение**: одну строку и одну колонку (таблицу 1×1). Такой подзапрос можно
ставить всюду, где синтаксис ждёт одиночное значение — как будто это обычное
число, строка или дата.

### Три исхода: одно значение, ошибка, `NULL`

Что вернёт скалярный подзапрос, зависит от того, сколько строк он выдал:

| Сколько строк вернул подзапрос | Результат |
|--------------------------------|-----------|
| **ровно 1 строка** (1 колонка) | это самое значение — нормальный случай |
| **больше 1 строки** | **ошибка выполнения** |
| **0 строк** (пусто) | `NULL` |

**Больше одной строки — ошибка.** Если подзапрос в скалярном контексте вернул
несколько строк, СУБД не знает, какую из них взять, и падает:

```sql
SELECT name FROM products
WHERE price = (SELECT price FROM products WHERE category_id = 2);
-- ERROR: more than one row returned by a subquery used as an expression
```

Книг у нас три, поэтому подзапрос вернул три цены, а `price = (…)` ждёт одну.
Лечится либо агрегатом (`MAX`/`MIN`/`AVG` свернут в одно число), либо переходом
на `IN`/`ANY` (они как раз работают со **списком** — [файл 02](02-in-any-all.md)).

**Ноль строк — `NULL`.** Пустой подзапрос в скалярном контексте даёт `NULL` (а не
ошибку и не ноль):

```sql
SELECT (SELECT price FROM products WHERE name = 'НЕТ ТАКОГО') AS missing;
```
```
 missing
---------
          ← NULL: подзапрос не нашёл ни одной строки
```

Это важно помнить: `NULL` из пустого подзапроса дальше поедет по правилам
трёхзначной логики из lab001 (сравнение с `NULL` даёт `UNKNOWN`), и внешнее
условие может «съесть» строку. Если нужен ноль вместо `NULL` — оборачивайте:
`COALESCE((SELECT ...), 0)`.

### Скалярный подзапрос в `WHERE` — сравнение с агрегатом

Самый частый случай: сравнить каждую строку с **одним общим числом** — средним,
максимумом, суммой по всей таблице.

```sql
-- товары дороже средней цены по всему каталогу
SELECT name, price
FROM products
WHERE price > (SELECT AVG(price) FROM products)   -- средняя ≈ 9643.33 ₽
ORDER BY price DESC;
```
```
        name         |  price
---------------------+----------
 Ноутбук AirBook 14  | 74990.00
 Смартфон Nova 5     | 24990.00
 Робот-пылесос Clean | 12990.00
(3 rows)
```

Почему нельзя без подзапроса? Потому что `AVG(price)` — это агрегат по **всей**
таблице, а `WHERE` фильтрует **отдельные** строки и агрегаты в нём запрещены
(правило из lab001: `WHERE COUNT(*) > 1` — ошибка). Подзапрос обходит это
элегантно: он считает среднее **отдельным** запросом, а внешний `WHERE`
сравнивает с готовым числом.

> Мысленная модель: сначала выполняется внутренний запрос и превращается в число
> `9643.33`, затем условие читается как `WHERE price > 9643.33`. Для
> некоррелированного подзапроса это буквально так — он считается **один раз**.

### Скалярный подзапрос в `SELECT` — добавить общий показатель в каждую строку

Скалярный подзапрос можно поставить и в список `SELECT` — тогда его единственное
значение **подставится в каждую строку** результата. Удобно показать рядом со
строкой некий общий ориентир (среднее, максимум) и посчитать отклонение от него:

```sql
SELECT
    name,
    price,
    (SELECT ROUND(AVG(price), 2) FROM products)              AS avg_all,
    ROUND(price - (SELECT AVG(price) FROM products), 2)      AS diff_from_avg,
    ROUND(price / (SELECT MAX(price) FROM products) * 100, 1) AS pct_of_max
FROM products
ORDER BY price DESC;
```
```
        name         |  price   | avg_all | diff_from_avg | pct_of_max
---------------------+----------+---------+---------------+------------
 Ноутбук AirBook 14  | 74990.00 | 9643.33 |    65346.67   |      100.0
 Смартфон Nova 5     | 24990.00 | 9643.33 |    15346.67   |       33.3
 Робот-пылесос Clean | 12990.00 | 9643.33 |     3346.67   |       17.3
 ...
```

Каждый скалярный подзапрос в `SELECT` вычисляется и подставляется одинаковым
числом во все строки. Здесь их три — средняя, средняя (для `diff`) и максимум.

> Про производительность (без деталей — они в lab025): некоррелированный
> скалярный подзапрос СУБД обычно считает **один раз** и кэширует, а не заново на
> каждую строку. Так что три подзапроса в `SELECT` выше — это три вычисления, а
> не «три на каждую из 15 строк».

### Вложенность: подзапрос внутри подзапроса

Подзапрос сам может содержать подзапрос — вложенность не ограничена одним
уровнем. Классический приём — найти **второе** по величине значение:

```sql
-- второй по дороговизне товар: максимум среди тех, что дешевле общего максимума
SELECT name, price
FROM products
WHERE price = (
    SELECT MAX(price) FROM products
    WHERE price < (SELECT MAX(price) FROM products)   -- ← подзапрос внутри подзапроса
);
```
```
       name       |  price
------------------+----------
 Смартфон Nova 5  | 24990.00
```

Читается изнутри наружу: самый внутренний `SELECT MAX(price)` = 74990 (Ноутбук);
средний находит максимум среди цен строго меньше 74990 → 24990 (Смартфон);
внешний отбирает товар(ы) с этой ценой.

## Скобки обязательны

Подзапрос **всегда** заключается в круглые скобки — это часть синтаксиса. Без них
парсер не поймёт, где кончается внутренний `SELECT` и начинается внешний.

```sql
WHERE price > (SELECT AVG(price) FROM products)     -- ✅
WHERE price >  SELECT AVG(price) FROM products       -- ❌ синтаксическая ошибка
```

## Когда скалярный подзапрос — хороший выбор

- **Сравнить каждую строку с общим показателем** (средним, максимумом, суммой по
  всей таблице или по связанной таблице) — как в примерах выше.
- **Подтянуть одно значение** из другой таблицы «по ключу», когда `JOIN` был бы
  избыточен ради единственной колонки (но чаще для этого всё же берут `JOIN` —
  сравнение инструментов в [файле 05](05-choosing-pitfalls.md)).

Чего скалярный подзапрос **не** умеет: вернуть список или таблицу. Как только
ответом должно быть несколько значений — это уже `IN`/`ANY`/`ALL`
([файл 02](02-in-any-all.md)), `EXISTS` ([файл 03](03-exists-correlated.md)) или
подзапрос в `FROM` ([файл 04](04-from-derived-row.md)).

## Итог

- Подзапрос — это `SELECT` в скобках внутри другого запроса; «запрос внутри
  запроса».
- Где он стоит, то и определяет, что он должен вернуть: одно значение (скаляр),
  столбец, наличие строк или целую таблицу.
- **Скалярный** подзапрос возвращает ровно одно значение: 1 строка → это
  значение, **больше 1 строки → ошибка**, 0 строк → `NULL`.
- Ставится в `SELECT` (подставится в каждую строку) и в `WHERE`/`HAVING` со
  сравнением (`=`, `>`, `<`, …). Всегда в скобках.
