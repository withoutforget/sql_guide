# 02 — Подзапрос-список: `IN` / `NOT IN`, `ANY` / `ALL`

[← что такое подзапрос](01-what-is-subquery.md) · [оглавление](../README.md) · [дальше: EXISTS и коррелированные →](03-exists-correlated.md)

---

Скалярный подзапрос из [файла 01](01-what-is-subquery.md) возвращал одно
значение. Здесь подзапрос возвращает **столбец значений** (N строк × 1 колонка), а
внешний запрос проверяет, как значение относится к этому столбцу: входит ли в него
(`IN`), больше ли хотя бы одного (`> ANY`), больше ли всех (`> ALL`).

## `IN (подзапрос)` — принадлежит списку

В lab001 мы уже видели `IN` со списком-константой: `WHERE category_id IN (1, 2)`.
Теперь вместо перечисления вручную список **вычисляет подзапрос**:

```sql
-- клиенты, сделавшие хотя бы один заказ
SELECT id, name
FROM customers
WHERE id IN (SELECT customer_id FROM orders)   -- ← список id из заказов
ORDER BY id;
```
```
 id | name
----+-------
  1 | Анна
  2 | Борис
  ...
  7 | Жанна
(7 rows)
```

Подзапрос `SELECT customer_id FROM orders` вернул столбец id всех, кто что-то
заказывал; `id IN (…)` оставил клиентов, чей id в этом столбце есть. Захар (id 8)
заказов не делал — его нет. Это **полу-соединение** (semi-join): «оставить строки,
у которых есть пара» — мы решали его джойном в lab002, теперь тем же вопросом
занимается `IN` (сравнение инструментов — в [файле 05](05-choosing-pitfalls.md)).

Дубликаты в подзапросе `IN` **не мешают**: неважно, сколько раз `customer_id`
встретился в заказах — `IN` проверяет лишь **факт наличия**. Поэтому, в отличие от
`JOIN`, клиент не «размножится» и `DISTINCT` не нужен.

> Требование к форме: подзапрос в `IN` должен возвращать **одну** колонку
> (тип которой сравним с левой частью). `WHERE id IN (SELECT id, name FROM …)` —
> ошибка (кроме строчной формы `(a,b) IN (…)`, о ней в [файле 04](04-from-derived-row.md)).

## `NOT IN (подзапрос)` — не принадлежит списку

`NOT IN` — отрицание: оставить значения, которых в списке **нет**. Это
**анти-соединение** (найти строки без пары):

```sql
-- товары, которые ни разу не заказывали
SELECT id, name
FROM products
WHERE id NOT IN (SELECT product_id FROM orders)   -- id, которых нет среди заказанных
ORDER BY id;
```
```
 id |         name
----+-----------------------
  3 | Умная колонка Mini
  8 | Настольная лампа Lumi
 12 | Пазл 1000 деталей
 13 | Мяч футбольный
 14 | Скакалка Speed
 15 | Йога-коврик
(6 rows)
```

Тот же вопрос мы решали анти-джойном в lab002 (`LEFT JOIN … WHERE o.id IS NULL`) и
через `EXCEPT` в lab003. Здесь он записан ещё короче. **Но у `NOT IN` есть
коварная ловушка с `NULL`** — и это, пожалуй, самый важный подводный камень всей
темы.

## ⚠️ Ловушка `NOT IN` + `NULL`

Запомнить её надо накрепко: **если подзапрос в `NOT IN` вернул хотя бы один
`NULL`, весь `NOT IN` перестаёт находить строки — результат пустой (или
усечённый)**. Ошибки при этом **нет** — просто тихо неверный ответ.

Причина — трёхзначная логика из lab001. `x NOT IN (a, b, c)` разворачивается в

```
x <> a  AND  x <> b  AND  x <> c
```

Подставим в список `NULL`: `x NOT IN (a, b, NULL)` = `x <> a AND x <> b AND
x <> NULL`. Но `x <> NULL` — это **`UNKNOWN`**, а не `TRUE`/`FALSE`. И вся
конъюнкция уже не может стать `TRUE`: `TRUE AND TRUE AND UNKNOWN` = `UNKNOWN`, а
`WHERE` пропускает только `TRUE`. Значит, ни одна строка не пройдёт.

Проверим на «чистом» примере:

```sql
SELECT 5 WHERE 5 NOT IN (1, 2, NULL);
```
```
 ?column?
----------
(0 rows)              ← 5 очевидно не 1 и не 2, но из-за NULL результат пуст
```

### На наших данных

В таблице `customers` есть колонка `referrer_id` — «кто пригласил клиента»; у тех,
кто пришёл сам, там `NULL`. Зададим естественный вопрос: **«кто из клиентов
никого не приглашал»** — то есть чьих id нет среди `referrer_id`. Наивно:

```sql
-- ❌ СЛОМАНО: подзапрос возвращает referrer_id, среди которых есть NULL
SELECT id, name
FROM customers
WHERE id NOT IN (SELECT referrer_id FROM customers)
ORDER BY id;
```
```
 id | name
----+------
(0 rows)              ← пусто, хотя «никого не приглашавших» пятеро!
```

Подзапрос `SELECT referrer_id FROM customers` вернул `{NULL, 1, 1, NULL, 2, NULL,
3, NULL}` — там есть `NULL` (клиенты без пригласившего). Один этот `NULL`
обнуляет весь `NOT IN`. Результат — пустая выборка, хотя правильный ответ — пятеро
клиентов (Глеб, Дарья, Егор, Жанна, Захар).

### Как обойти ловушку

**Способ 1 — отфильтровать `NULL` в подзапросе.** Раз проблема в `NULL`, уберём
его:

```sql
-- ✅ фильтруем NULL прямо в подзапросе
SELECT id, name
FROM customers
WHERE id NOT IN (SELECT referrer_id FROM customers WHERE referrer_id IS NOT NULL)
ORDER BY id;
```
```
 id | name
----+-------
  4 | Глеб
  5 | Дарья
  6 | Егор
  7 | Жанна
  8 | Захар
(5 rows)
```

**Способ 2 (рекомендуемый) — `NOT EXISTS`.** Он вообще не страдает от этой
проблемы, потому что проверяет **наличие строк**, а не сравнивает значения (см.
[файл 03](03-exists-correlated.md)):

```sql
-- ✅ NULL-безопасно: NOT EXISTS корректен независимо от NULL
SELECT c.id, c.name
FROM customers AS c
WHERE NOT EXISTS (
    SELECT 1 FROM customers AS r WHERE r.referrer_id = c.id
)
ORDER BY c.id;
-- тот же правильный результат: Глеб, Дарья, Егор, Жанна, Захар
```

**Правило: `NOT IN` по подзапросу — только если абсолютно уверены, что подзапрос
не вернёт `NULL`** (например, колонка `NOT NULL`, как `orders.product_id` в
примере выше). Во всех остальных случаях предпочитайте **`NOT EXISTS`**. Это
настолько частый источник багов, что многие команды просто договариваются «`NOT
IN` по подзапросу не пишем никогда».

> Заметьте: у **`IN`** (без `NOT`) `NULL` в подзапросе такой беды не вызывает —
> он просто не даёт дополнительных совпадений. Проблема именно у **`NOT IN`**,
> потому что там `NULL` превращает «не нашлось» в «неизвестно».

## `ANY` / `SOME` и `ALL` — сравнение со списком через оператор

`IN` проверяет только равенство («входит в список»). А если нужно «дороже хотя бы
одного» или «дороже всех»? Для этого есть `ANY` и `ALL` — они ставят **любой
оператор сравнения** (`>`, `<`, `>=`, `=`, `<>`, …) между значением и **каждым**
элементом списка-подзапроса.

- `x <оператор> ANY (подзапрос)` — `TRUE`, если сравнение верно **хотя бы для
  одного** элемента. (`SOME` — полный синоним `ANY`.)
- `x <оператор> ALL (подзапрос)` — `TRUE`, если сравнение верно **для всех**
  элементов.

Мысленная модель: `ANY` — «**существует** такой элемент, что…»; `ALL` — «**для
всех** элементов…». Это кванторы «существует» (∃) и «для всех» (∀).

### `> ANY` — больше хотя бы одного (больше минимума)

```sql
-- игрушки дороже хотя бы одной книги
SELECT name, price
FROM products
WHERE category_id = 4                                    -- Игрушки
  AND price > ANY (SELECT price FROM products WHERE category_id = 2)  -- цены книг
ORDER BY price;
```
```
        name         |  price
---------------------+---------
 Мяч футбольный      |  990.00
 Конструктор "Город" | 2790.00
(2 rows)
```

Цены книг — `{890, 1290, 2490}`. `price > ANY (…)` истинно, если цена больше
**хотя бы одной** книги, то есть больше самой дешёвой (890). Мяч (990) и
Конструктор (2790) проходят; Пазл (690) — нет. Отсюда полезное тождество:
**`> ANY` = «больше минимума» списка**, а `< ANY` = «меньше максимума».

### `> ALL` — больше всех (больше максимума)

```sql
-- игрушки дороже ВСЕХ книг
SELECT name, price
FROM products
WHERE category_id = 4
  AND price > ALL (SELECT price FROM products WHERE category_id = 2)
ORDER BY price;
```
```
        name         |  price
---------------------+---------
 Конструктор "Город" | 2790.00
(1 row)
```

`price > ALL (…)` истинно, только если цена больше **каждой** книги, то есть
больше самой дорогой (2490). Проходит один Конструктор (2790). Тождество:
**`> ALL` = «больше максимума»**, `< ALL` = «меньше минимума». Сравните с
предыдущим примером: тот же список книг, `ANY` дал 2 товара, `ALL` — 1.

`>= ALL` удобно для поиска **максимума** по всей таблице (значение, которое не
меньше всех):

```sql
-- самый дорогой товар каталога
SELECT name, price
FROM products
WHERE price >= ALL (SELECT price FROM products);
-- Ноутбук AirBook 14, 74990.00
```

### `= ANY` ≡ `IN`, а `<> ALL` ≡ `NOT IN`

Два тождества, которые полезно узнавать:

| Запись | Эквивалент | Смысл |
|--------|------------|-------|
| `x = ANY (…)`  | `x IN (…)`     | равно **хотя бы одному** элементу |
| `x <> ALL (…)` | `x NOT IN (…)` | не равно **ни одному** элементу |

То есть `IN` — это частный случай `= ANY`, а `NOT IN` — частный случай `<> ALL`.
На практике пишут короткое `IN`/`NOT IN`, но понимать связь важно — в том числе
чтобы не удивляться, что **`<> ALL` наследует ту же ловушку с `NULL`**, что и `NOT
IN` (см. ниже).

### Сводная таблица тождеств

| Через `ANY`/`ALL` | Короче / смысл |
|-------------------|----------------|
| `x = ANY (s)`  | `x IN (s)` |
| `x <> ALL (s)` | `x NOT IN (s)` |
| `x > ANY (s)`  | `x > (наименьший из s)` |
| `x < ANY (s)`  | `x < (наибольший из s)` |
| `x > ALL (s)`  | `x > (наибольший из s)` |
| `x < ALL (s)`  | `x < (наименьший из s)` |

## `ANY` / `ALL` и `NULL`

С `NULL` в списке `ANY` и `ALL` ведут себя по-разному, и это стоит понять, а не
заучивать.

- **`ALL` строгий.** `NULL` среди элементов даёт `x <оператор> NULL = UNKNOWN`, а
  `ALL` требует `TRUE` для **всех** — одно `UNKNOWN` не даёт всей конъюнкции стать
  `TRUE`. Итог: `NULL` в списке **ломает** `> ALL` ровно так же, как ломал `NOT
  IN` (`<> ALL` — это и есть `NOT IN`).
- **`ANY` снисходительный.** Ему достаточно **одного** `TRUE`. `NULL` добавит
  `UNKNOWN`, но не помешает: если какой-то не-`NULL` элемент дал `TRUE`, весь
  `ANY` — `TRUE`.

Наглядно (в списке специально подмешан `NULL`):

```sql
-- > ANY со списком {1000, NULL}: NULL не мешает, отбираются все дороже 1000
SELECT count(*) FROM products WHERE price > ANY (VALUES (1000::numeric), (NULL));
-- → 11 (все товары дороже 1000)

-- > ALL с тем же списком: NULL всё «съедает», результат пуст
SELECT count(*) FROM products WHERE price > ALL (VALUES (1000::numeric), (NULL));
-- → 0
```

Вывод тот же, что и для `NOT IN`: **как только в подзапросе возможен `NULL`, будьте
особенно осторожны с `ALL` (и `NOT IN`)**; `ANY`/`IN` в этом смысле безопаснее.

## Когда что использовать

| Вопрос | Инструмент |
|--------|------------|
| «значение входит в список из подзапроса» | `IN` |
| «значения нет в списке» (и подзапрос без `NULL`!) | `NOT IN`, иначе `NOT EXISTS` |
| «больше/меньше хотя бы одного» | `> ANY` / `< ANY` |
| «больше/меньше всех» | `> ALL` / `< ALL` |
| «просто есть ли связанная строка» | `EXISTS` ([файл 03](03-exists-correlated.md)) |

## Итог

- Подзапрос-список возвращает столбец значений; `IN`/`ANY`/`ALL` сравнивают с ним
  левое значение.
- `IN` — «входит в список» (полу-соединение); дубликаты в подзапросе не мешают.
- **`NOT IN` + `NULL` в подзапросе → пустой/неверный результат.** Обход: убрать
  `NULL` (`WHERE … IS NOT NULL`) или, надёжнее, `NOT EXISTS`.
- `ANY` = «хотя бы для одного», `ALL` = «для всех». `= ANY` ≡ `IN`, `<> ALL` ≡
  `NOT IN`; `> ANY` ≡ «больше минимума», `> ALL` ≡ «больше максимума».
- `ALL` (и `NOT IN`) страдают от `NULL` в списке; `ANY`/`IN` — нет.
