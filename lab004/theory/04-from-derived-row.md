# 04 — Подзапрос в `FROM` (производная таблица) и строчные подзапросы

[← EXISTS и коррелированные](03-exists-correlated.md) · [оглавление](../README.md) · [дальше: выбор инструмента и подводные камни →](05-choosing-pitfalls.md)

---

Два оставшихся места для подзапроса: в секции `FROM` (там подзапрос играет роль
**таблицы**) и в сравнении **кортежей** `(a, b) = (…)` / `(a, b) IN (…)`.

## Подзапрос в `FROM` — производная таблица

Подзапрос может стоять в `FROM` — тогда его результат используется как **обычная
таблица** (её называют **производной таблицей**, derived table, или подзапросом в
`FROM`). Внешний запрос выбирает из неё колонки, фильтрует, группирует — как из
любой таблицы.

```sql
SELECT ...
FROM (
    SELECT ... FROM ... GROUP BY ...   -- подзапрос-источник строк
) AS t                                  -- ← алиас производной таблицы
WHERE ...;
```

### Зачем это нужно: агрегат от агрегата

Главный повод — **посчитать агрегат от результата другого агрегата**. Прямо
вложить агрегаты друг в друга нельзя: `AVG(SUM(...))` — ошибка («aggregate function
calls cannot be nested»). Решение: сначала посчитать внутренний агрегат в
подзапросе, а внешний запрос агрегирует уже его результат.

Задача: «какова **средняя** сумма покупок в расчёте на клиента». Это среднее **от
сумм** — двухуровневая агрегация:

```sql
SELECT
    ROUND(AVG(total), 2) AS avg_total,   -- среднее по клиентским суммам
    MAX(total)           AS max_total,
    MIN(total)           AS min_total
FROM (
    -- внутренний шаг: сумма покупок КАЖДОГО клиента
    SELECT o.customer_id, SUM(p.price * o.quantity) AS total
    FROM orders   AS o
    JOIN products AS p ON p.id = o.product_id
    GROUP BY o.customer_id
) AS per_customer;                        -- ← производная таблица «суммы по клиентам»
```
```
 avg_total | max_total | min_total
-----------+-----------+-----------
  23332.86 |  74990.00 |     890.00
(1 row)
```

Внутренний запрос дал таблицу «клиент → его суммарная трата» (по строке на
клиента с заказами). Внешний усреднил колонку `total` по этой таблице. Одним
уровнем агрегатов так не сделать.

> Тонкость: `per_customer` содержит только клиентов, **у которых есть заказы**
> (Захар без заказов в `GROUP BY` не попал), поэтому среднее считается по семи
> клиентам. Если нужно учитывать и «нулевых» клиентов, внутренний запрос строят на
> `customers LEFT JOIN orders` с `COALESCE(SUM(...), 0)` — приём из lab002.

### Другие поводы для производной таблицы

- **Предварительная группировка/фильтрация перед `JOIN`** — чтобы избежать
  задвоения (fan-out из lab002): сначала свернуть «многую» сторону в подзапросе до
  одной строки на ключ, потом джойнить готовые итоги. Надёжный способ соединить
  «две ветви 1:N», не перемножив их.
- **Отфильтровать по агрегату, вычисленному во внешнем `SELECT`** — то, что
  `WHERE` не умеет; иногда удобнее вынести вычисление в подзапрос и фильтровать
  снаружи.
- **Упростить сложный запрос**, разбив его на понятные шаги.

### Алиас производной таблицы

По стандарту SQL производной таблице **обязательно** дают алиас (`AS t`) — иначе к
ней нельзя обратиться, и большинство СУБД выдадут ошибку «subquery in FROM must
have an alias».

```sql
SELECT t.customer_id, t.cnt
FROM (SELECT customer_id, COUNT(*) AS cnt FROM orders GROUP BY customer_id) AS t
WHERE t.cnt > 1;
```

> Нюанс версий PostgreSQL: начиная с **PostgreSQL 16**, алиас у подзапроса в
> `FROM` можно **опустить** — движок это допускает. Но полагаться на это не стоит:
> в стандарте SQL и в PostgreSQL до 16 алиас **обязателен**, он нужен, чтобы
> **квалифицировать колонки** (`t.total`) и различать источники в сложных
> запросах, и делает запрос переносимым. **Правило: всегда давайте производной
> таблице алиас.**

Колонки производной таблицы — это то, что перечислено в `SELECT` подзапроса
(с учётом их алиасов). Поэтому во внутреннем запросе удобно сразу назвать
вычисляемые колонки (`AS total`, `AS cnt`), чтобы обращаться к ним снаружи.

### Предвестие CTE (lab005)

Когда подзапросов в `FROM` становится несколько или один и тот же нужен дважды,
запрос превращается в «матрёшку» и читается тяжело. Для этого есть **CTE** —
конструкция `WITH`, которая позволяет **назвать** подзапрос и переиспользовать его,
писав слева направо, а не вложенно:

```sql
-- та же задача через CTE (это lab005) — читается как последовательность шагов
WITH per_customer AS (
    SELECT o.customer_id, SUM(p.price * o.quantity) AS total
    FROM orders o JOIN products p ON p.id = o.product_id
    GROUP BY o.customer_id
)
SELECT ROUND(AVG(total), 2) AS avg_total FROM per_customer;
```

Смысл тот же, что у производной таблицы, но нагляднее. **CTE — тема lab005**; пока
достаточно знать, что подзапрос в `FROM` — его прямой предшественник, и многие
«матрёшки» позже перепишутся через `WITH`.

## Строчный подзапрос и сравнение кортежей

Обычно сравнивают **одно** значение (`price = (…)`). Но SQL умеет сравнивать и
**кортежи** — несколько значений сразу, записанных в скобках: `(a, b)`. Такой
подзапрос называют **строчным** (row subquery) — он возвращает одну или несколько
строк из нескольких колонок.

### `(a, b) = (SELECT x, y …)` — равенство кортежей

Кортежи равны, если **все** их компоненты попарно равны. Правый подзапрос при этом
должен вернуть **ровно одну** строку (это скалярный контекст, но «широкий»):

```sql
-- товар, у которого (категория, цена) совпадает с (категория, максимум) Электроники
SELECT name
FROM products
WHERE (category_id, price) = (
    SELECT category_id, MAX(price) FROM products
    WHERE category_id = 1 GROUP BY category_id
);
-- → Ноутбук AirBook 14
```

### `(a, b) IN (SELECT x, y …)` — принадлежность списку кортежей

Гораздо полезнее строчная форма `IN`: она проверяет, входит ли **пара** значений в
список пар из подзапроса. Классический приём — «**самый дорогой товар в каждой
категории**»:

```sql
SELECT p.name, p.category_id, p.price
FROM products AS p
WHERE (p.category_id, p.price) IN (
    SELECT category_id, MAX(price)      -- пары (категория, её максимальная цена)
    FROM products
    GROUP BY category_id
)
ORDER BY p.category_id;
```
```
        name         | category_id |  price
---------------------+-------------+----------
 Ноутбук AirBook 14  |           1 | 74990.00
 Книга "Алгоритмы"   |           2 |  2490.00
 Робот-пылесос Clean |           3 | 12990.00
 Конструктор "Город" |           4 |  2790.00
 Йога-коврик         |           5 |  1490.00
(5 rows)
```

Подзапрос строит по одной паре `(категория, макс. цена)` на категорию. Внешний
запрос оставляет товары, чья пара `(категория, цена)` есть в этом списке, — то
есть чья цена равна максимуму **своей** категории. Одной колонкой так не
получится: сравнивать нужно цену **вместе с** категорией, иначе перепутаются
максимумы разных категорий.

> Сравните с коррелированным подзапросом «дороже среднего в своей категории» из
> [файла 03](03-exists-correlated.md): там для каждого товара свой порог считался
> коррелированно; здесь мы, наоборот, заранее собрали все пороги (максимумы) в
> список и матчим по кортежу. Часто одну и ту же задачу можно выразить обоими
> способами — а позже, в lab010, для «топ-1/топ-N в группе» появится ещё и более
> прямой инструмент — **оконные функции**.

### Строчные подзапросы и `NULL`

Сравнение кортежей наследует трёхзначную логику: если в сравниваемых кортежах
встречается `NULL`, результат может стать `UNKNOWN`, а строка — выпасть. В наших
данных участвующие колонки не `NULL`, но в реальных данных об этом стоит помнить
(та же природа, что у ловушки `NOT IN`).

## Итог

- Подзапрос в `FROM` — это **производная таблица**: результат подзапроса
  используется как обычная таблица. Главный повод — **агрегат от агрегата**
  (`AVG` от `SUM`) и предварительная свёртка перед `JOIN`.
- Давайте производной таблице **алиас** (в PostgreSQL 16 не обязателен, но это
  правильная привычка и требование стандарта).
- CTE (`WITH`, lab005) — более читаемая эволюция подзапроса в `FROM`.
- **Строчный подзапрос** сравнивает кортежи: `(a, b) = (…)` и особенно
  `(a, b) IN (…)` — удобно для «максимум в каждой группе».
