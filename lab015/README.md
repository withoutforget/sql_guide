# lab015 — Строки и регулярные выражения

Третья лаба **модуля D «Типы и работа с данными»**. Текст — самый «грязный» тип
данных: имена приходят в разном регистре и с лишними пробелами, телефоны — в пяти
форматах, адреса слеплены запятыми, e-mail пополам с мусором. Аналитик и инженер
данных постоянно **чистят, парсят, валидируют и переформатируют** строки — этим и
займёмся. До сих пор строковые операции ограничивались `||`, `length`, `upper`,
`lower` (мелькали в [lab001](../lab001/) и [lab013](../lab013/)); здесь раскрываем
**весь арсенал** — от `concat_ws`/`format`/`split_part` до **POSIX-регулярных
выражений** (`~`, `regexp_replace`, `regexp_matches`).

Тип `text` как таковой (`text`/`varchar`/`char`, приведения) — это
[lab013](../lab013/theory/04-char-bool-types.md); здесь — работа со **значениями**
строк. Форматирование **дат** (`to_char`) осталось в [lab014](../lab014/); сюда его
не тащим.

Теория — короткие документы в [`theory/`](theory/), читать по порядку. Общая
подборка ссылок — в [`../RESOURCES.md`](../RESOURCES.md).

## Как запустить

```bash
# из корня репозитория один раз:
docker compose up -d

cd lab015
./run.sh                 # накатит схему и прогонит все запросы

# либо зайти руками и экспериментировать:
../psql.sh lab015
```

`run.sh` идемпотентен: пересоздаёт схему `lab015` с нуля при каждом запуске. Вывод
**детерминирован**: локаль базы (`en_US.utf8`) стабильно обрабатывает регистр
кириллицы (`upper`/`lower`/`initcap`/`ILIKE`), а во всех сортировках заданы
тай-брейкеры.

## Доменная область

«Сырой» импорт CRM и каталога интернет-магазина — специально **грязные** текстовые
данные. Четыре таблицы (полная схема и подробная шапка о подборе данных — в
[`init.sql`](init.sql)). **Каждая колонка подобрана под тему.**

### `clients` — карточки клиентов из импорта CRM
| Колонка | Тип | Смысл / зачем такой вид |
|---------|-----|-------------------------|
| `id` | INT (PK) | идентификатор |
| `raw_name` | TEXT | ФИО «как ввели»: разный регистр, лишние пробелы, часть без отчества/имени — для **очистки** (задача 2) и **разбора на инициалы** (задача 3) |
| `phone` | TEXT | телефон в 5 форматах; у клиента 4 — **`NULL`** — для **нормализации** (10) и **NULL-безопасной** сборки (3) |
| `email` | TEXT | валидные и **не**валидные адреса — для **валидатора** (13) и **извлечения домена** (11) |
| `address` | TEXT | «город, улица, дом» через `', '` — для **`split_part`** (5) |

### `products` — каталог товаров
| Колонка | Тип | Смысл / зачем |
|---------|-----|--------------|
| `id` | INT (PK) | идентификатор |
| `sku` | TEXT | внешний артикул: часть в формате `AB-1234`, часть «битая» (`ab-99`, `Z-12`) — **regex-валидация** (9), **префикс LIKE** (7) |
| `name` | TEXT | название (кириллица + латиница) — **`length` vs `octet_length`** (1) |
| `art_no` | INT | числовой артикул — **выравнивание нулями** `lpad` → `ART-0042` (4) |
| `tags` | TEXT | теги через запятую — **разбиение и подсчёт** (6) |
| `label` | TEXT | метка; часть со знаком `%` — **LIKE с ESCAPE** (7) |
| `note` | TEXT | свободный текст с числами — **извлечение чисел** (12) |

### `logs` — строки лога событий
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор |
| `line` | TEXT | строка `user=… action=… amount=…` — для **мини-ETL** парсинга `key=value` (14) |

### `companies` — контрагенты
| Колонка | Тип | Смысл |
|---------|-----|-------|
| `id` | INT (PK) | идентификатор |
| `name` | TEXT | одна фирма, записанная **по-разному** (регистр, пробелы, «кавычки», форма ООО/ЗАО) — для **канонизации и дедупликации** (15) |

> **Как данные демонстрируют тему:**
> - **Грязные имена.** `raw_name` вида `'  иванова   анна сергеевна '` — лишние
>   пробелы (по краям и внутри) и вперемешку регистр. Чистятся `btrim` +
>   `regexp_replace('\s+',' ')` + `initcap`; часть имён без отчества/имени —
>   инициалы собираются только по существующим частям.
> - **Телефоны в 5 форматах.** `'+7 (999) 123-45-67'`, `'8 (916) 000-11-22'`,
>   `'89031234567'`, `'8-800-555-35-35'`, `'7 495 000 00 00'` — все приводятся к
>   `+7XXXXXXXXXX` одним приёмом (оставить цифры → взять последние 10 → дописать `+7`).
>   У клиента 4 телефона нет (`NULL`) — проверяем, что склейка это переживает.
> - **Валидные и битые e-mail.** Четыре корректных (`anna@mail.ru`,
>   `petr.sidorov@example.com`, `maria_ivanova@corp.co`, `elena.novikova@shop.io`)
>   и четыре «мусорных» (`bad-email@`, `no-at-sign.ru`, `@nouser.ru`,
>   `morozov@site`) — валидатор делит их надвое, а запрос извлечения домена
>   **не падает** ни на одном.
> - **Битые артикулы.** `AB-1234`/`CD-0007`/`XY-4567`/`EF-0500` проходят
>   `^[A-Z]{2}-[0-9]{4}$`, а `ab-99` (строчные) и `Z-12` (одна буква) — нет.
> - **Двойники фирм.** «ООО Ромашка», «ооо  ромашка», «ООО «Ромашка»» и «Ромашка» —
>   четыре записи одной фирмы; канонизация схлопывает их в `'ромашка'`.

| 15 | 🔥 `15_dedup_companies.sql`    | [06](theory/06-regexp-functions.md) | **Канонизация и дедуп**: свести разные написания фирмы к канону и найти двойников. |

> **Про хардкор.** Задача 14 — маленький **ETL** целиком на пройденном: сперва CTE
> вытаскивает из строки лога поля `user`/`action`/`amount` регулярками с группами и
> сразу приводит сумму к `int` (приведение — [lab013](../lab013/)), затем обычная
> агрегация с `FILTER` и `COALESCE` считает покупки и потраченное по каждому юзеру.
> Задача 15 показывает, как побеждают «грязные дубликаты»: цепочка
> `lower`→убрать кавычки→схлопнуть пробелы→`btrim`→срезать форму собственности
> приводит «ООО Ромашка», «ооо  ромашка», «ООО «Ромашка»» и «Ромашка» к единому
> канону `'ромашка'`, после чего `GROUP BY canonical HAVING count(*) > 1` выдаёт
> записи, которые «на глаз» разные, а по сути — одна фирма. Обе — без единой новой
> конструкции сверх этой и прошлых лаб.
