# 04 — Сопоставление с образцом: `LIKE`, `ILIKE`, `SIMILAR TO`

[← разбиение строк](03-splitting.md) · [оглавление](../README.md) · [дальше: POSIX-регулярные выражения →](05-posix-regex.md)

---

В SQL есть три «встроенных» способа проверить строку на соответствие образцу, по
возрастанию мощности: `LIKE` (простые маски), `SIMILAR TO` (regex-lite из стандарта
SQL) и POSIX-регулярные выражения (`~`, следующий файл). Здесь — первые два.

`LIKE` мы уже коротко трогали в [lab001](../../lab001/theory/02-where.md) — здесь
разбираем полностью, включая `ESCAPE`.

## `LIKE` / `NOT LIKE` — простые маски

`LIKE` проверяет, соответствует ли строка **маске** из двух подстановочных символов:

| Символ | Значение |
|--------|----------|
| `%` | любая последовательность символов, **в том числе пустая** |
| `_` | ровно **один** любой символ |

Все остальные символы маски трактуются **буквально**. Результат — `boolean`.

```sql
'AB-1234'  LIKE 'AB%'      -- true   (начинается с 'AB')
'Ноутбук'  LIKE '%бук'     -- true   (заканчивается на 'бук')
'Ноутбук'  LIKE '%утб%'    -- true   (содержит 'утб')
'коты'     LIKE 'кот_'     -- true   ('кот' + ровно один символ)
'кот'      LIKE 'кот_'     -- false  (нужен ещё один символ)
'AB-1234'  LIKE 'ab%'      -- false  (LIKE чувствителен к регистру!)
```

Типичные маски: `'AB%'` — **префикс**, `'%.ru'` — **суффикс**, `'%abc%'` —
**подстрока**, `'___'` — строка ровно из 3 символов.

## `ILIKE` — то же, но без учёта регистра

`ILIKE` (PostgreSQL-расширение, `I` = *insensitive*) — `LIKE`, игнорирующий регистр:

```sql
'Ноутбук' ILIKE '%бук'     -- true
'Новинка' ILIKE '%нов%'    -- true   (регистр не важен)
'AB-1234' ILIKE 'ab%'      -- true
```

⚠️ Регистронезависимость `ILIKE` для кириллицы опирается на **локаль базы**
(`LC_CTYPE`) — как и `upper`/`lower` из [файла 01](01-basic-functions.md). В нашей
базе (`en_US.utf8`) кириллица сравнивается корректно; на локали `C` `ILIKE` не «увидит»
`Н`=`н`. Для гарантированно регистронезависимых **сравнений** существует тип `citext`
и явные приравнивания через `lower(...)` (упомянуто в конце файла 06).

## `ESCAPE` — искать сам символ `%` или `_`

Раз `%` и `_` особенные, как найти строку, в которой они есть **буквально**
(например метку `'Скидка 50%'`)? Экранированием. По умолчанию escape-символ —
обратный слэш `\`:

```sql
'Скидка 50%' LIKE '%\%'     -- true   ('\%' = буквальный процент; маска = «что угодно, затем %»)
'50% off'    LIKE '50\%%'   -- true   ('50', затем буквальный '%', затем что угодно)
```

`\%` означает «символ `%`, не подстановка», `\_` — «символ `_`». Свой escape-символ
задаётся через `ESCAPE`:

```sql
'100%' LIKE '100!%' ESCAPE '!'   -- true   ('!%' = буквальный процент)
```

Это удобно, когда в самих данных много обратных слэшей и `\` неудобен как escape.

> `~~` / `~~*` — внутренние операторные псевдонимы для `LIKE` / `ILIKE`
> (`!~~` / `!~~*` — для `NOT LIKE` / `NOT ILIKE`). В коде почти всегда пишут
> человеко-читаемые `LIKE`/`ILIKE`; операторные формы встречаются в планах запросов.

## `SIMILAR TO` — «regex-lite» из стандарта SQL

`SIMILAR TO` — гибрид: маска, где `%` и `_` работают как в `LIKE`, но **вдобавок**
доступны некоторые метасимволы регулярных выражений: `|` (или), `*` `+` `?` `{n,m}`
(повторы), `()` (группы), `[...]` (классы символов).

```sql
'abc' SIMILAR TO 'a__'          -- true   ('a' + два любых)
'abc' SIMILAR TO '%(b|d)%'      -- true   (где-то есть 'b' или 'd')
'abc' SIMILAR TO '(a|z)+bc'     -- true
'AB-1234' SIMILAR TO '[A-Z]{2}-[0-9]{4}'  -- true
```

Ключевое отличие от POSIX-regex: `SIMILAR TO`, как и `LIKE`, требует **совпадения
всей строки** целиком (как будто шаблон неявно обёрнут в `^...$`), а `%`/`_` тут
означают то же, что в `LIKE`, — не то, что в POSIX (где `%` — литерал, а `.` —
любой символ). Из-за этой смеси двух миров `SIMILAR TO` путает и читается плохо.

**На практике `SIMILAR TO` почти не используют:** для простого — берут `LIKE`, для
сложного — POSIX-регулярки (`~`), которые мощнее, стандартны в других языках и
быстрее. `SIMILAR TO` полезно знать, чтобы узнавать его в чужом коде и не путать его
`%` с POSIX-смыслом.

## Сводка: что когда брать

| Задача | Инструмент |
|--------|-----------|
| префикс / суффикс / подстрока, регистр важен | `LIKE` |
| то же, регистр не важен | `ILIKE` |
| искать буквальный `%` или `_` | `LIKE ... ESCAPE` |
| сложный шаблон (группы, повторы, классы) | **POSIX-regex `~`** ([файл 05](05-posix-regex.md)) |
| `SIMILAR TO` | почти никогда — предпочесть `~` |

## Про производительность (кратко)

- `LIKE 'префикс%'` (шаблон «прикреплён к началу») **может использовать индекс** по
  колонке — СУБД ищет диапазон строк, начинающихся с префикса. Это быстро.
- `LIKE '%подстрока%'` и `LIKE '%суффикс'` (маска начинается с `%`), а также
  `ILIKE` и regex-поиск `~` — обычный индекс, как правило, **использовать не могут**
  и приводят к полному сканированию. Для быстрого поиска «по подстроке» и нечёткого
  поиска есть триграммные индексы `pg_trgm` — это тема [lab016](../../lab016/).

Подробно про индексы — модуль F (lab024).
