# 01 — Базовые строковые функции: сборка, длина, регистр

[← оглавление лабы](../README.md) · [дальше: извлечение, поиск, обрезка, замена →](02-extract-trim-replace.md)

---

Строки (`text`) как **тип** мы разобрали в [lab013](../../lab013/theory/04-char-bool-types.md):
`text` / `varchar(n)` / `char(n)`, приведения к тексту и обратно. Здесь начинается
работа со **значениями** строк — весь арсенал функций для их сборки, измерения и
преобразования. Всё это — обычные функции и операторы: их можно ставить в `SELECT`,
`WHERE`, `ORDER BY`, `GROUP BY` — где угодно, где допустимо выражение.

Тип текста в примерах — всегда `text`; функции работают одинаково и для `varchar`.

## Сборка строк: `||`, `concat`, `concat_ws`, `format`

### `||` — конкатенация (склейка)

Оператор `||` склеивает две строки в одну:

```sql
SELECT 'Заказ №' || 42;         -- 'Заказ №42'   (число приводится к тексту само)
SELECT city || ', ' || street;  -- 'Москва, Тверская'
```

Не-текстовые аргументы приводятся к `text` автоматически (`42` → `'42'`).

⚠️ **Главный подводный камень `||` — `NULL` «съедает» весь результат.**
Если хоть один операнд `NULL`, вся склейка становится `NULL`:

```sql
SELECT 'a' || NULL || 'b';   -- NULL  (не 'ab'!)
```

Это логично (`NULL` = «неизвестно», а склейка с неизвестным неизвестна), но на
практике болезненно: одно пустое поле — и вся собранная строка пропадает.

### `concat(...)` и `concat_ws(sep, ...)` — склейка, игнорирующая `NULL`

Две функции решают проблему `NULL` в `||`:

- **`concat(a, b, c, ...)`** — склеивает любое число аргументов, а `NULL`
  **пропускает** (трактует как пустую строку):

  ```sql
  SELECT concat('a', NULL, 'b');       -- 'ab'    (NULL просто выпал)
  SELECT concat('id=', 42, ' x', 3.5); -- 'id=42 x3.5'  (не-текст приводится сам)
  ```

- **`concat_ws(sep, a, b, c, ...)`** — то же, но ещё и вставляет **разделитель**
  `sep` **между** аргументами (`ws` = *with separator*). `NULL`-аргументы
  пропускаются и **лишних разделителей не создают**:

  ```sql
  SELECT concat_ws('-', 2024, 07, 21);          -- '2024-7-21'
  SELECT concat_ws('; ', 'Анна', NULL, 'anna@mail.ru'); -- 'Анна; anna@mail.ru'
  ```

  Обратите внимание: во втором примере `NULL` (телефон) выпал **вместе** со своим
  разделителем — нет «висящего» `; ;`. Именно поэтому `concat_ws` — рабочая лошадка
  для сборки строк из полей, часть которых может быть пустой (адрес, ФИО, контакты).

| Инструмент | `NULL`-аргумент | Разделитель | Когда брать |
|------------|-----------------|-------------|-------------|
| `a || b`   | делает весь результат `NULL` | нет | простая склейка, где `NULL` заведомо нет |
| `concat(...)` | пропускает | нет | склейка полей, часть которых может быть `NULL` |
| `concat_ws(sep, ...)` | пропускает (без лишнего `sep`) | да | список полей через разделитель |

> Если по смыслу `NULL` должен превратиться в конкретный текст (например `'—'`),
> используйте `COALESCE(x, '—')` из [lab013](../../lab013/theory/06-null-functions.md)
> перед склейкой.

### `format(шаблон, аргументы...)` — строка по шаблону

`format` собирает строку по шаблону с **спецификаторами** `%` — как `printf` в
других языках. Спецификаторов три, и разница между ними важна:

| Спец. | Что подставляет | Пример |
|-------|-----------------|--------|
| `%s`  | значение **как есть** (string) | `format('%s руб', 500)` → `500 руб` |
| `%I`  | как **идентификатор** SQL (identifier) — в двойных кавычках при необходимости | `format('%I', 'col name')` → `"col name"` |
| `%L`  | как **литерал** SQL (literal) — в одинарных кавычках, с экранированием | `format('%L', 'text''s')` → `'text''s'` |

```sql
SELECT format('%s %I %L', 'val', 'col name', 'text''s');
--  →  val "col name" 'text''s'
```

- `%s` — обычная подстановка (эквивалент `||`, но читается яснее и не «ломается»
  от `NULL`: `format('%s', NULL)` даёт пустую строку, а не `NULL`).
- `%I` и `%L` — **защита от SQL-инъекций** при построении динамического SQL:
  `%I` корректно закавычит имя таблицы/колонки, `%L` — строковый литерал, экранировав
  внутренние кавычки. Это ключевой приём безопасной генерации запросов (подробно —
  в lab031 про инъекции). Порядковый доступ: `%1$s` — первый аргумент, и т.д.

## Длина строки: символы vs байты

Три разные функции «длины» — и на кириллице (UTF-8) они **расходятся**:

| Функция | Что считает | `'привет'` | `'hello'` |
|---------|-------------|-----------|-----------|
| `length(s)` / `char_length(s)` | **символы** | `6` | `5` |
| `octet_length(s)` | **байты** | `12` | `5` |
| `bit_length(s)` | **биты** (= байты × 8) | `96` | `40` |

```sql
SELECT length('привет'), octet_length('привет');   -- 6   12
SELECT length('hello'),  octet_length('hello');     -- 5   5
```

Причина: в UTF-8 латинская буква — **1 байт**, а кириллическая — **2 байта**.
Поэтому `length` (сколько *символов* видит человек) и `octet_length` (сколько
*байтов* занимает на диске) совпадают только для ASCII-текста. Смешанное название
`'Мышь USB'` — 8 символов, но 12 байтов (4 кириллических × 2 + 4 ASCII × 1).

- `length` = `char_length` — полные синонимы.
- `octet_length` нужен, когда важен **физический размер** (лимиты хранения, ширина
  в байтах). Ловушка `char(n)`: `octet_length` считает и хвостовую набивку пробелами
  (см. [lab013](../../lab013/theory/04-char-bool-types.md)).
- `length('')` = `0`; `length(NULL)` = `NULL`.

## Регистр: `upper`, `lower`, `initcap`

```sql
SELECT upper('привет мир');   -- 'ПРИВЕТ МИР'
SELECT lower('ПРИВЕТ');       -- 'привет'
SELECT initcap('иван ПЕТРОВ');-- 'Иван Петров'  (каждое слово — с большой, остальное — строчными)
```

- `upper` / `lower` — перевод в верхний/нижний регистр.
- `initcap` — **каждое слово** с заглавной, остальные буквы строчные. «Слово» —
  последовательность буквенно-цифровых символов; разделителем считается всё
  остальное, включая `-` и `_`:

  ```sql
  SELECT initcap('привет-мир по_имени');  -- 'Привет-Мир По_Имени'
  SELECT initcap('ооо "ромашка"');        -- 'Ооо "Ромашка"'
  ```

⚠️ **Ловушка локали.** Как именно `А`↔`а` меняют регистр, определяет **локаль базы**
(`LC_CTYPE`). В нашей базе это `en_US.utf8`, где кириллица обрабатывается корректно
и детерминированно (`upper('привет')` = `'ПРИВЕТ'` при любом запуске). Но на базе с
локалью `C`/`POSIX` регистр меняется **только у ASCII**, а кириллица останется как
есть — типичный источник «то работает, то нет» между окружениями. В этой лабе
регистр кириллицы стабилен, потому что локаль базы это поддерживает.

## Прочие полезные функции

| Функция | Что делает | Пример → результат |
|---------|-----------|--------------------|
| `reverse(s)` | переворачивает строку | `reverse('abcГ')` → `'Гcba'` |
| `repeat(s, n)` | повторяет `s` `n` раз | `repeat('ab', 3)` → `'ababab'` |
| `left(s, n)` | первые `n` символов (отрицательное `n` — «все, кроме последних `|n|`») | `left('привет', 3)` → `'при'`; `left('hi', -1)` → `'h'` |
| `right(s, n)` | последние `n` символов | `right('привет', 3)` → `'вет'` |
| `lpad(s, len, fill)` | дополнить слева до длины `len` символами `fill` (по умолч. пробел); если длиннее — **обрезать** | `lpad('7', 3, '0')` → `'007'` |
| `rpad(s, len, fill)` | то же справа | `rpad('7', 3, '0')` → `'700'` |
| `ascii(s)` | код первого символа | `ascii('A')` → `65` |
| `chr(n)` | символ по коду | `chr(1103)` → `'я'` |
| `md5(s)` | MD5-хеш (32 hex-символа) | `md5('привет')` → `'608333adc72f545078ede3aad71bfe74'` |

`lpad`/`rpad` — самый частый инструмент **выравнивания**: добить число нулями до
фиксированной ширины (`lpad('42', 5, '0')` → `'00042'`), выровнять колонку в
текстовом отчёте. Важно: если строка **длиннее** `len`, `lpad`/`rpad` её **урежут**
(`lpad('abcdef', 3, '0')` → `'abc'`) — это не только «дополнение», но и «подгонка под
ширину». `left`/`right` — быстрый способ взять префикс/суффикс без `substring`.

`md5` детерминирован (от одной строки — всегда один хеш), поэтому годится для
стабильного «отпечатка» текста; для этого он и используется в самопроверке лаб.
