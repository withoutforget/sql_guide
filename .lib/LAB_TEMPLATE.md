# Конвенции автора лабы (внутренний документ)

Каждая лаба повторяет структуру `lab001`. Прежде чем делать новую лабу —
**прочитай `lab001` целиком** (README, `theory/*`, `init.sql`, `queries/*`,
`run.sh`) и держи тот же уровень качества и стиль.

## Структура папки labNNN/

```
labNNN/
├── README.md          # ОГЛАВЛЕНИЕ: как запустить, оглавление теории,
│                      #   домен (таблицы), список задач с колонкой «теория»,
│                      #   «что дальше». НЕ свалка теории — теория в theory/.
├── init.sql           # схема + данные. Начинается с:
│                      #   DROP SCHEMA IF EXISTS labNNN CASCADE;
│                      #   CREATE SCHEMA labNNN;  SET search_path TO labNNN;
│                      #   Полностью идемпотентна.
├── theory/            # короткие тематические .md, по одной теме на файл,
│                      #   пронумерованы 01-...md, 02-...md.
├── queries/           # эталонные решения, по файлу на задачу:
│                      #   NN_snake_name.sql. ПЕРВАЯ строка — «-- Задача N. ...»
│                      #   (её печатает run.sh как заголовок), ВТОРАЯ — 
│                      #   «SET search_path TO labNNN;»
└── run.sh             # ровно:
                       #   #!/usr/bin/env bash
                       #   source "$(dirname "$0")/../.lib/common.sh"
                       #   LAB_DIR="$(cd "$(dirname "$0")" && pwd)"
                       #   run_lab "$LAB_DIR"
                       # + chmod +x
```

## Требования к теории (theory/*.md)

- Русский язык. Объясняем **как работает и зачем**, а не только синтаксис.
- **Самодостаточность — главное правило.** Текст должен быть таким, чтобы
  человек понял тему **полностью, не переходя ни по одной ссылке**. Ссылки —
  только для «копнуть глубже», а не замена объяснения. Не отсылай «читай в
  доке» вместо того, чтобы объяснить самому. Если вводишь термин — тут же
  разъясни его простыми словами.
- Для каждой конструкции/функции: назначение, полный синтаксис, **все
  значимые аргументы и их смысл**, как ведёт себя (в т.ч. с `NULL`, с пустым
  входом, с дубликатами), подводные камни, и примеры **с показанным
  результатом** (что вернётся). Примеров должно быть много и разнообразных.
- Где уместно — мысленная модель / пошаговый разбор «как СУБД это исполняет»,
  сравнительные таблицы, «когда использовать, а когда нет».
- Отбор материала осмысленный: в теме даём то, что действительно нужно, и явно
  отсекаем то, что сюда не относится (со ссылкой «это в labNNN»).
- Навигация в начале и конце файла: `← предыдущая · оглавление · следующая →`.
- В конце каждого файла — раздел «Документация»: ссылки на **русскую**
  документацию Postgres Pro по теме (`https://postgrespro.ru/docs/postgresql/16/...`)
  и, где полезно, на англ. первоисточник / хороший учебный материал. Ссылки
  дополняют текст, но материал должен быть понятен и без них.
- Не дублировать то, что уже разобрано в предыдущих лабах — ссылаться на них.

## Требования к задачам (queries/*)

- **10–14 задач**, по нарастанию сложности, покрывающие тему разнообразно
  (разные функции, разные формы, реалистичные аналитические вопросы).
- **1–2 «хардкорные» задачи на подумать** в конце: решаются РОВНО тем, что
  разобрано в этой (и прошлых) лабе, но нетривиальны — требуют собрать приёмы
  вместе. Не вводить новые конструкции ради них. Пометить их в README (🔥).
- Каждое решение — рабочий SQL, дающий осмысленный результат на данных лабы.
- В README — таблица задач с колонками: #, файл, теория (ссылка), описание.

## Данные (init.sql)

- Можно продолжать домен «интернет-магазин» из lab001 или расширять его —
  но **лаба самодостаточна**: своя схема `labNNN`, свой `init.sql`.
- Данные должны быть подобраны так, чтобы **демонстрировать** тему: например,
  для `LEFT JOIN` нужен клиент без заказов и товар, который никто не купил; для
  оконных функций — несколько строк в каждой партиции; для дат — разброс во
  времени. Небольшие, читаемые, осмысленные (рубли, реальные названия).
- Комментарии на русском.

## Обязательная самопроверка

Перед сдачей: `cd labNNN && ./run.sh` должен отработать **без ошибок** и вывести
осмысленные результаты по всем задачам. Проверить глазами, что вывод
соответствует условиям.

## Что НЕ трогать

Не менять `lab001`, другие лабы, `.lib/common.sh`, `docker-compose.yml`,
корневые README/RESOURCES (кроме случая, когда явно поручено обновить роадмап).
